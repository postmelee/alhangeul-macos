# Issue #17 수행 계획서

## 목적

`AGENTS.md`의 현재 Git 워크플로우를 `devel` 대상 PR 기반으로 전환하고, 관련 요약 문서와 정합성을 맞춘다.

- 기존 `local/task -> local/devel -> devel` 흐름을 `devel` 대상 PR 중심 흐름으로 바꾼다.
- PR 생성용 원격 브랜치 이름 규칙을 제안한다.
- 변경으로 생기는 단점과 대응 전략을 문서로 정리한다.
- `README.md`의 Git 워크플로우 요약도 `AGENTS.md`와 정합화한다.

## 범위

### 포함

- 현재 `AGENTS.md` 워크플로우 문구 분석
- `devel` 대상 PR 기반 개편안 설계 및 반영
- 원격 브랜치 명명 규칙 제안
- 예상 단점과 대응 전략 설계
- `README.md` 워크플로우 요약 정합화

### 제외

- 새 워크플로우에 맞춘 PR/브랜치 운영 실제 적용
- 기존 Issue #15 정리 작업과의 병합

## 설계 초안

### 목표 워크플로우

기존:

`local/task{N} -> local/devel -> devel -> main PR`

검토 대상:

`local/task{N} -> publish/task{N} push -> devel 대상 PR -> devel 누적 -> main PR`

핵심 의도:

- 작업 단위별 리뷰 기록을 GitHub PR로 남긴다.
- `devel`에 들어가기 전 검토 게이트를 추가한다.
- 릴리즈용 `devel -> main` PR과 작업 단위 PR을 분리한다.

### 원격 브랜치 이름 추천

권장안: `publish/task{issue번호}`

예:

- `publish/task17`
- `publish/task42`

선정 이유:

- `local/task17`와 역할이 명확히 분리된다.
- 로컬 작업 브랜치와 원격 게시 브랜치를 이름만 보고 구분할 수 있다.
- 이슈 번호 기반이라 작업 추적과 PR 연결이 쉽다.

대안:

- `pr/task{issue번호}`
- `review/task{issue번호}`

## 예상 단점

1. 브랜치 수가 증가한다.
2. 문서 수정 같은 소규모 작업도 PR 절차를 타면 속도가 느려진다.
3. `local/task -> publish/task -> devel`로 관리 포인트가 늘어난다.
4. 혼자 작업하는 저장소에서는 운영 부담이 커질 수 있다.
5. merge 전략을 정하지 않으면 `devel` 히스토리가 복잡해질 수 있다.

## 단점 대응 전략

### 1. 브랜치 수 증가 대응

- 로컬 작업 브랜치는 계속 `local/task{N}`로 고정한다.
- 원격 게시 브랜치는 `publish/task{N}` 한 패턴으로 통일한다.
- PR merge 후 `publish/task{N}`는 즉시 삭제한다.

### 2. 소규모 작업 속도 저하 대응

- 오탈자, 단순 문구 수정, 조사 메모처럼 `AGENTS.md`의 예외 처리 대상은 경량 절차를 허용한다.
- 단, Issue와 최종 보고서는 유지해 추적성은 보존한다.
- 문서 전용 타스크는 draft PR 생성 후 자가 리뷰로 빠르게 닫는 운영 기준을 둔다.

### 3. 관리 포인트 증가 대응

- 작업 절차를 `Issue -> local/task -> publish/task -> devel PR -> merge`로 짧게 명문화한다.
- `AGENTS.md`에 브랜치 역할 표를 추가해 각 브랜치의 수명과 목적을 명확히 적는다.
- 오늘 할 일, 수행 계획서, 최종 보고서에 같은 Issue 번호를 일관되게 유지한다.

### 4. 1인 저장소 운영 부담 대응

- `devel` 대상 PR은 기본적으로 draft로 생성하고, 최종 보고 작성 후 ready 전환 여부를 판단한다.
- 리뷰 주체가 동일하더라도 PR 본문에 변경 이유와 검증 결과를 남겨 자기 검토 기록으로 사용한다.
- merge 기준을 단순화해 불필요한 형식 부담을 줄인다.

### 5. `devel` 히스토리 복잡화 대응

- `devel` 대상 PR merge 전략을 한 가지로 고정한다.
- 권장안은 `--no-ff` merge 또는 GitHub merge commit 유지다.
- squash merge는 단계별 커밋 의미가 사라질 수 있으므로 기본값으로 두지 않는다.
