# Task M020 #88 수행계획서

## 목적

Quick Look preview를 현재 data-based PDF/PNG reply 중심 구조에서 view-based preview 구조로 전환할 수 있는지 확정하고, 다중 페이지 HWP/HWPX 문서에서 첫 페이지 즉시 표시와 visible page 중심 background rendering을 구현한다.

완료 후에는 다중 페이지 문서 preview에서 최초 표시 지연을 줄이고, 스크롤로 실제 보이는 page를 중심으로 render/cache/cancellation이 동작해야 한다. 기존 Finder Thumbnail extension의 첫 페이지 thumbnail 동작은 유지한다.

작업 중 작업지시자가 현재 Quick Look PDF preview UI 유지를 우선 요구했다. 따라서 실제 구현 경로는 `PDFView + PDFThumbnailView` lazy 가능성을 먼저 검증하고, 현재 PDF UI를 충분히 보존할 수 없거나 lazy 근거가 부족하면 현행 data-based PDF preview UI를 유지한 채 생성 비용을 최적화하는 방향으로 보정한다.

## 배경

Task #85에서 Quick Look preview는 다중 페이지 문서를 Quick Look 표시용 PDF로 반환하도록 확장했다. 그러나 현재 `QLPreviewReply(dataOfContentType:)` 경로는 Quick Look에 완성된 data blob을 넘기는 구조라서 첫 페이지만 먼저 표시하고 나머지 page를 사용자가 볼 때 그리는 true lazy UX에는 한계가 있다.

Task #87은 `QLPreviewReply(forPDFWithPageSize:documentCreationBlock:)`와 custom `PDFDocument`/`PDFPage` 조합을 검증했다. 최종 결론은 Quick Look runtime이 최초 preview load 중 전체 page를 요청하고 `PDFDocument.dataRepresentation()`을 통해 전체 PDF를 materialize한다는 것이었다. 따라서 PDFKit document reply는 visible page 중심 lazy rendering 후보로 채택하지 않는다.

Task #256은 `HwpPageImageRenderer`에 Skia optional backend와 CoreGraphics fallback contract를 추가했다. 다만 기본 제품 동작은 아직 `coreGraphicsOnly`이며, #88의 핵심은 renderer backend 선택이 아니라 Quick Look preview container와 page scheduling 구조 전환이다.

## 설계 방향

먼저 Quick Look preview extension이 view-based provider로 전환될 때 필요한 principal class, protocol, Info.plist/project 설정을 실제 SDK와 현재 `project.yml` 기준으로 확인한다. 전환 범위가 `QLExtension` 내부에 머무는지, `Shared`에 page render/cache helper를 분리해야 하는지를 Stage 1에서 고정한다.

view-based preview는 첫 page surface를 가장 먼저 준비하고, 이후에는 scroll viewport에서 보이는 page와 가까운 page만 background task로 render하는 구조를 목표로 한다. 각 request마다 generation token 또는 cancellation handle을 둬서 Finder selection이 빠르게 바뀌면 이전 render 결과가 새 preview UI에 반영되지 않게 한다.

page rendering은 기존 `RhwpDocument`와 `HwpPageImageRenderer.renderPage` contract를 재사용한다. #256의 `skiaOptIn`은 별도 정책 승인 전까지 기본값으로 켜지 않는다. 이번 작업의 제품 기본 동작은 visible-page scheduling 전환이며, renderer backend 전환은 후속 task와 충돌하지 않도록 분리한다.

fallback 정책은 보수적으로 둔다. view-based 경로가 파일 크기 제한, empty document, invalid page size, render failure를 만나면 기존 text fallback 수준의 명확한 사용자 메시지를 제공하고, Thumbnail extension은 기존 `HwpThumbnailProvider`/`HwpThumbnailRenderCache` 경로를 유지한다.

## 잠정 단계

1. Stage 1: View-based Quick Look API와 현행 구조 inventory
   - 현재 `HwpPreviewProvider`, `Info.plist`, `project.yml`, Quick Look runtime 설정을 정리한다.
   - macOS SDK 기준 view-based provider 요구사항과 data-based provider와의 전환 차이를 확인한다.
   - `NSScrollView`와 `PDFView` 후보를 lazy scheduling 관점에서 비교한다.
2. Stage 2: 구현계획서 작성과 preview architecture 고정
   - chosen UI/container, request lifecycle, cancellation token, page cache key, fallback contract를 `task_m020_88_impl.md`에 고정한다.
   - Stage별 변경 파일과 검증 명령을 확정한다.
3. Stage 3: View-based preview skeleton과 첫 페이지 즉시 표시
   - `NSViewController` 기반 preview를 붙이고 첫 page render 결과를 즉시 표시한다.
   - 단일/다중 페이지 문서 모두 기존 fallback 정책과 title/size 처리가 유지되는지 확인한다.
4. Stage 4: Visible page background rendering, cache, cancellation 구현
   - scroll viewport 변화에 따라 보이는 page와 주변 page를 background render한다.
   - 빠른 selection 전환 시 이전 작업 cancel 또는 stale result discard가 동작하도록 한다.
5. Stage 5: Quick Look/Thumbnail smoke와 성능 관측
   - Release package 기준 Quick Look preview smoke, thumbnail smoke, extension registration hygiene를 수행한다.
   - HWP/HWPX 다중 페이지 샘플에서 first preview latency와 scroll 시 page render 순서를 기록한다.
6. Stage 6: 최종 정리와 보고
   - log/probe 잔여물, 문서, 오늘할일을 정리하고 최종 결과보고서를 작성한다.
   - PR 게시 전 미커밋 산출물과 개발 extension 등록 잔존 여부를 확인한다.

## 리스크

- Quick Look extension에서 view-based provider와 data-based provider의 lifecycle/Info.plist 요구사항이 달라서 project 설정 전환 범위가 예상보다 커질 수 있다.
- `NSScrollView` 기반 직접 page layout은 lazy scheduling control이 좋지만, page reuse, zoom, accessibility, selection 표시 같은 책임을 extension 쪽에서 더 많이 져야 한다.
- `PDFView`는 page UI를 재사용할 수 있지만, #87처럼 내부에서 전체 document materialization을 유도하면 lazy 효과가 약해질 수 있다.
- extension process lifetime이 짧아 background render task가 중간에 종료될 수 있으므로 stale result discard와 cancellation을 반드시 방어해야 한다.
- Quick Look runtime smoke는 등록된 provider path에 민감하다. 기존 설치본과 개발 산출물이 동시에 등록되면 결과가 흔들릴 수 있다.
- 다중 페이지 문서에서 page image cache를 과하게 잡으면 extension memory pressure가 커질 수 있다.
