# Task M050 #247 수행계획서

## 목적

`native-viewer-editor` 브랜치를 최신 제품 개발 라인에 최대한 가깝게 forward-port해, Swift native viewer/editor 후속 개발을 오래된 프로젝트 구조와 운영 문서 위에서 계속하지 않도록 한다.

이번 작업의 직접 목표는 신규 기능 구현이 아니라 장기 native 라인의 기반 정렬이다. `devel`에 누적된 프로젝트 구조, CI, 검증 스크립트, 문서 정책, HostApp shell 변경을 `native-viewer-editor`에 반영하되, 기존 native renderer/editor 작업의 의미 있는 구현은 보존한다.

## 배경

#244에서 제품 개발 기본 브랜치는 `devel`로 전환되었고, 기존 Swift native viewer/editor 장기 작업은 `native-viewer-editor` 브랜치로 보존되었다. 현재 `native-viewer-editor`는 #109 merge commit 기준의 native renderer 라인이고, `devel`은 v0.1.x WebView-backed 제품/배포 라인과 브랜치 전환 문서를 포함한다.

사전 확인 결과 `native-viewer-editor <- devel` 단순 병합은 다음 범위에서 충돌한다.

- `project.yml`과 구 `AlhangeulMac.xcodeproj` 삭제/변경 충돌
- `Sources/RhwpCoreBridge/CGTreeRenderer.swift`
- `Sources/RhwpCoreBridge/FontFallback.swift`
- `Sources/RhwpCoreBridge/FontResourceRegistry.swift`
- `Sources/HostApp/Resources/rhwp-studio/fonts/FONTS.md`
- font fallback 관련 문서와 일부 과거 작업 문서

또한 `devel-webview`에는 `devel` 이후 #243 종료/닫기 저장 확인 변경이 추가로 존재한다. 다만 #244 이후 정책상 기본 기준은 `devel`이므로, `devel-webview`는 legacy alias로 취급하고 #243은 별도 단계에서 선별 반영 여부를 판단한다.

## 범위

포함한다.

- `native-viewer-editor` 기준 작업 브랜치에서 `devel` 변경 반영
- 충돌 파일별 보존 원칙 정리와 수동 해결
- 최신 제품 라인의 프로젝트 이름, XcodeGen 설정, CI/release/helper script, 문서 정책 반영
- native renderer/editor 관련 `Sources/RhwpCoreBridge` 변경의 의도 보존
- `devel-webview`에만 있는 #243 저장 확인 변경의 cherry-pick 또는 native 포팅 가능성 평가
- 단계별 보고서, 검증 결과, 최종 보고서 작성

제외한다.

- Swift native viewer/editor 신규 기능 구현
- WebView viewer 기능 확장
- public release, tag 생성, GitHub Release 게시
- branch protection/default branch repository setting 변경
- `devel-webview` legacy alias 삭제

## 설계 방향

기준 반영 순서는 `devel`을 먼저 두고, `devel-webview`의 추가 변경은 후속 선별 대상으로 분리한다. 이렇게 해야 #244 이후의 브랜치 정책과 PR base 안내가 다시 흐려지지 않는다.

충돌 해결 원칙은 다음과 같다.

- 프로젝트/빌드/CI/배포 스크립트/문서 정책은 최신 `devel`을 우선한다.
- `AlhangeulMac.xcodeproj`에서 `Alhangeul.xcodeproj`로 이동한 구조는 최신 제품 라인 기준으로 정리한다.
- `Sources/RhwpCoreBridge`의 renderer, font fallback, resource registry 변경은 native 라인의 기존 parity 보강 의도를 먼저 확인한 뒤 최신 제품 라인 변경을 수동 통합한다.
- `mydocs/orders/*`와 과거 stage/report 문서 충돌은 최신 작업 추적 문맥을 훼손하지 않는 방향으로 정리하되, 필요한 경우 이번 #247 보고서에 충돌 처리 판단을 남긴다.
- #243 저장 확인 변경은 WebView 구현 세부에 묶인 부분과 native editor에도 재사용 가능한 dirty-state/termination coordination 개념을 분리해 판단한다.

## 잠정 단계

1. Stage 1: `native-viewer-editor`, `devel`, `devel-webview` 차이와 충돌 파일을 상세 inventory로 고정한다.
2. Stage 2: `devel` 반영 전략과 충돌 파일별 보존 원칙을 구현 계획서에 확정한다.
3. Stage 3: `devel` 변경을 `native-viewer-editor` 작업 브랜치에 반영하고 프로젝트/문서/스크립트 충돌을 해결한다.
4. Stage 4: `Sources/RhwpCoreBridge`와 font fallback 관련 충돌을 native renderer 보존 원칙에 맞게 해결한다.
5. Stage 5: `devel-webview` 전용 #243 변경의 선별 반영 여부를 판단하고 필요한 경우 native 라인에 맞게 포팅한다.
6. Stage 6: 빌드/스크립트/문서 검증, 최종 보고서, PR 게시를 정리한다.

## 검증 계획

- `git status --short --branch`
- `git diff --check`
- `git merge-tree --write-tree native-viewer-editor devel`
- `git merge-tree --write-tree native-viewer-editor devel-webview`
- `xcodegen generate`
- `./scripts/check-no-appkit.sh`
- `./scripts/build-rust-macos.sh` 또는 변경 범위에 따른 `--verify-lock`
- `xcodebuild -project Alhangeul.xcodeproj -scheme HostApp -configuration Debug -derivedDataPath build.noindex/DerivedData CODE_SIGNING_ALLOWED=NO build`
- workflow/script 변경 시 `for script in scripts/*.sh scripts/ci/*.sh; do bash -n "$script"; done`
- renderer 변경 시 native render smoke 또는 기존 stage render 검증 명령을 변경 범위에 맞게 실행

검증 명령은 구현 계획서에서 실제 변경 범위와 현재 브랜치의 스크립트 존재 여부를 다시 확인해 확정한다.

## 리스크

- `devel` 반영 범위가 커서 단일 cherry-pick보다 merge 성격의 충돌 해결이 많다.
- 제품 라인 WebView shell과 native viewer/editor 라인이 같은 HostApp 파일을 건드리므로, 잘못 해결하면 native 후속 개발 기반이 오히려 흐려질 수 있다.
- `Sources/RhwpCoreBridge` 충돌은 렌더링 parity와 직접 연결되므로 자동 해결보다 의도 확인이 필요하다.
- `devel-webview` 전용 #243은 WebView bridge 코드와 연결되어 있어 그대로 cherry-pick하면 native 라인에 불필요한 의존을 늘릴 수 있다.
- `Alhangeul.xcodeproj`는 generated project이므로, `project.yml` 정리와 XcodeGen 재생성을 일관되게 수행해야 한다.
