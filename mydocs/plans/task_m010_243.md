# Task M010 #243 수행계획서

## 목적

HostApp에서 편집된 문서가 있는 상태로 앱 종료 또는 문서 창 닫기를 요청했을 때 저장 여부를 묻는 모달을 표시한다.

완료 후 사용자는 변경사항이 있는 문서를 닫거나 앱을 종료하려 할 때 `저장`, `저장하지 않음`, `취소` 중 하나를 선택할 수 있어야 한다. `저장`을 선택하면 기존 HWP 저장/export bridge를 사용해 저장을 완료한 뒤 종료 또는 창 닫기를 이어가고, `취소` 또는 저장 취소/실패 시에는 종료 흐름이 중단되어야 한다.

## 배경

Issue #243은 현재 HostApp에서 문서를 편집한 뒤 앱 종료 또는 문서 창 닫기를 요청해도 저장 여부 확인 모달이 표시되지 않는 문제를 기록한다.

현재 앱은 SwiftUI `WindowGroup`과 수동 `DocumentWindowPresenter` 창을 함께 사용한다. 저장 동작은 `RhwpStudioWebView.Coordinator`가 WKWebView 안의 `rhwp-studio` export bridge를 호출하고, 결과 bytes를 `DocumentSavePanel` 또는 원본 파일에 기록하는 구조다. 반면 종료/창 닫기 경로에는 dirty state 확인이나 AppKit termination response 제어가 연결되어 있지 않다.

따라서 이번 작업은 HostApp 소유 영역에서 WKWebView 기반 문서 변경 상태를 추적하고, AppKit 창 닫기/앱 종료 훅에서 기존 저장 경로를 안전하게 재사용하는 데 초점을 둔다.

## 범위(포함·제외)

포함:

- 개별 문서 창 닫기 요청 시 저장 여부 확인 모달 표시

- `저장`, `저장하지 않음`, `취소` 선택에 따른 창 닫기/앱 종료 제어
- 기존 `file:save` / `file:save-as` / HWP export bridge와의 책임 경계 정리
- 저장 취소 또는 저장 실패 시 종료/창 닫기 중단 처리

- release signing, notarization, Homebrew Cask 반영

## 설계 방향

dirty state의 source of truth는 HostApp의 `DocumentViewerStore`에 둔다. WKWebView 안에서는 `rhwp-studio`의 문서 변경 신호를 감지해 native message로 전달하고, HostApp은 현재 문서 revision과 연결된 변경 여부를 보관한다. 새 문서를 load하거나 저장이 완료되면 dirty state를 명시적으로 해제한다.

종료 확인 UI는 AppKit 표준 `NSAlert` sheet/modal을 사용한다. 개별 창 닫기에서는 해당 창에 sheet로 붙일 수 있으면 sheet를 우선하고, 앱 종료처럼 특정 sheet presenter가 애매한 흐름은 main/key window 기준 또는 app-modal fallback을 검토한다. 문구는 macOS 문서 앱 관례에 맞춰 저장하지 않은 변경사항이 있음을 직접적으로 안내한다.

창 닫기 제어는 SwiftUI `WindowGroup`으로 생성된 기본 창과 `DocumentWindowPresenter`가 만든 수동 창 모두에 적용되어야 한다. 기존 `WindowAccessor`/`DocumentWindowLifecycle` 또는 수동 `NSWindowDelegate` 경로를 조사해, 창별 store와 window delegate 사이의 소유 관계를 중복 없이 연결한다.

앱 종료 제어는 `NSApplicationDelegate.applicationShouldTerminate(_:)`에서 변경된 문서가 있는지 확인하고, 필요하면 `.terminateLater`로 전환해 비동기 저장 확인을 끝낸 뒤 `reply(toApplicationShouldTerminate:)`를 호출하는 방향을 우선 검토한다. 여러 문서 창이 동시에 dirty인 경우 순차 확인을 기본값으로 두고, 사용자가 어느 하나에서 `취소`하면 앱 종료 전체를 중단한다.

저장 동작은 기존 WebView export bridge를 재사용한다. 이번 작업에서 저장 로직을 별도 파일 포맷 계층으로 복제하지 않는다. 다만 현재 저장 command가 fire-and-forget 성격이면, 종료 확인 흐름에서 저장 완료/취소/실패를 await할 수 있는 좁은 completion API를 추가한다.

## 잠정 단계(3~6단계)

1. Stage 1: 현행 종료/창 닫기와 저장 bridge 조사
   - SwiftUI 기본 창, `DocumentWindowPresenter` 수동 창, `AppDelegate` 종료 경로를 inventory한다.
   - `rhwp-studio` 문서 변경 이벤트와 현재 저장 completion 가능 여부를 확인한다.
   - 구현계획서에 dirty state, modal, termination response 설계를 고정한다.
2. Stage 2: dirty state bridge와 store 상태 구현
   - WebView host bridge에서 문서 변경/저장 완료 상태를 native로 전달한다.
   - `DocumentViewerStore`가 문서 revision별 dirty state를 보관하고 새 문서 load/save 시 상태를 정리한다.
3. Stage 3: 창 닫기 저장 확인 흐름 구현
   - 기본 SwiftUI 창과 수동 문서 창에서 close request를 가로채 저장 확인 모달을 표시한다.
   - `저장`, `저장하지 않음`, `취소` 선택에 따라 창 닫기 흐름을 제어한다.
4. Stage 4: 앱 종료 저장 확인 흐름 구현
   - `applicationShouldTerminate`에서 dirty 문서 창을 순차 확인한다.
   - 저장 성공 후 종료 재개, 저장 취소/실패 또는 사용자 취소 시 종료 중단을 처리한다.
5. Stage 5: 검증과 보고
   - HostApp build, 정적 검색, 저장/닫기/종료 수동 smoke를 수행한다.
   - 단계 보고서와 최종 보고서, 오늘할일 완료 갱신을 정리한다.

각 단계 완료 시 단계별 완료보고서를 작성하고 작업지시자 승인 후 다음 단계로 진행한다.

## 리스크

- `rhwp-studio` 내부 변경 이벤트가 안정적인 public API가 아니면 dirty state 감지가 upstream asset 구조에 의존할 수 있다.
- 저장 command가 현재 fire-and-forget이면 종료 흐름에서 저장 완료를 정확히 기다리도록 completion 경계를 새로 만들어야 한다.
- SwiftUI `WindowGroup` 창과 수동 `NSWindow` 창의 delegate ownership을 잘못 연결하면 기존 창 정리, redundant empty window close, toolbar validation이 깨질 수 있다.
- 앱 종료 중 여러 dirty 문서를 순차 처리할 때 sheet 표시 순서와 `reply(toApplicationShouldTerminate:)` 호출 타이밍이 꼬일 수 있다.
- HWPX 직접 저장 제한 정책 때문에 저장 버튼이 Save As 또는 HWP export 경로로 이어질 수 있으며, 사용자가 취소한 경우 종료 중단 처리가 명확해야 한다.
- dirty state가 false positive이면 불필요한 저장 확인이 표시되고, false negative이면 변경사항 유실 가능성이 남는다.
