# Issue #113 수행 계획서

## 목적

`task-final-report` Skill의 최종 보고서 작성 규칙을 보강해, `변경 파일 목록과 영향 범위` 섹션이 단순 파일명 목록으로 작성되지 않도록 한다.

완료 후 최종 보고서는 다음 기준을 따른다.

- `변경 파일 목록과 영향 범위` 섹션은 반드시 Markdown table로 작성한다.
- 기본 컬럼은 `파일 | 내용`으로 한다.
- 단순 파일 목록만 나열하지 않는다.

## 배경

Task #108 최종 보고서의 `변경 파일` 섹션은 변경 파일명을 bullet list로만 나열했다. 이 형식은 변경 파일이 무엇인지는 보여주지만, 각 파일이 왜 바뀌었고 어떤 내용을 담는지 바로 추적하기 어렵다.

`task-final-report` Skill은 표준 섹션으로 `변경 파일 목록과 영향 범위`를 요구하지만, table 형식이나 기본 컬럼을 명시하지 않는다. 따라서 에이전트가 최종 보고서를 작성할 때 단순 파일 목록으로도 규칙을 충족했다고 판단할 여지가 있다.

이번 작업은 Task #112의 넓은 PR 본문 구조 개편과 분리해, 최종 보고서의 변경 파일 섹션 표준만 좁게 강화한다.

## 범위

포함:

- `mydocs/skills/task-final-report/SKILL.md`의 최종 보고서 작성 규칙 보강
- 필요 시 최종 보고서 검증 항목의 최소 보정
- 단계별 완료보고서와 최종 보고서 작성
- 오늘할일 완료 처리

제외:

- 기존 완료 보고서 일괄 재작성
- `.github/pull_request_template.md` 구조 개편
- `관련 이슈`, `대상 타스크`, `작업 문서` 같은 PR 본문 용어 재정의
- PR body lint 스크립트 신규 추가
- 앱, Swift, Rust, Xcode project 변경

## 설계 방향

변경은 `task-final-report` Skill의 최종 보고서 작성 절차에 한정한다. 기존 표준 섹션 목록은 유지하되, `변경 파일 목록과 영향 범위` 항목 아래에 작성 형식 규칙을 추가한다.

규칙은 다음처럼 짧고 검증 가능한 문장으로 둔다.

- 반드시 Markdown table로 작성
- 기본 컬럼은 `파일 | 내용`
- 단순 파일 목록 금지

필요하면 검증 섹션에도 "최종 보고서의 변경 파일 섹션이 table인지 확인" 수준의 항목을 추가한다. PR 템플릿이나 PR 처리 가이드의 넓은 구조 개편은 #112 또는 별도 작업에 남긴다.

## 예상 변경 파일

- `mydocs/skills/task-final-report/SKILL.md`
- `mydocs/orders/20260501.md`
- `mydocs/plans/task_m010_113.md`
- `mydocs/working/task_m010_113_stage{N}.md`
- `mydocs/report/task_m010_113_report.md`

## 잠정 단계

1. Stage 1: `task-final-report` 최종 보고서 섹션 규칙 보강
2. Stage 2: 규칙 검증과 최종 보고서 정리
3. Stage 3: PR 게시 준비 상태 점검

각 단계 완료 시 단계별 완료보고서를 작성하고 작업지시자 승인 후 다음 단계로 진행한다.

## 검증 계획

문서 변경 검증:

```bash
rg -n "변경 파일 목록과 영향 범위|Markdown table|파일 \\| 내용|단순 파일 목록" mydocs/skills/task-final-report/SKILL.md
git diff --check
```

최종 정리 검증:

```bash
rg -n "#113|완료:" mydocs/orders/20260501.md
test -f mydocs/report/task_m010_113_report.md
git status --short
```

문서/운영 규칙 변경이므로 Swift, Rust, Xcode 빌드는 수행하지 않는다.

## 리스크

- 규칙을 너무 넓게 쓰면 #112의 PR 본문 구조 개편 범위와 겹칠 수 있다. 이번 작업은 최종 보고서의 변경 파일 섹션에만 한정한다.
- 기존 완료 보고서를 소급 수정하면 과거 작업 기록이 흐려질 수 있다. 기존 보고서는 유지하고, 앞으로의 작성 규칙만 보강한다.
- `영향 범위`라는 표현과 `파일 | 내용` 기본 컬럼 사이에 혼동이 생길 수 있다. 기본 컬럼은 `파일 | 내용`으로 두되, 필요하면 `내용`에 영향 범위까지 요약하도록 안내한다.

## 승인 요청 사항

이 수행 계획서 기준으로 구현 계획서(`mydocs/plans/task_m010_113_impl.md`) 작성을 진행할지 승인 요청한다. 승인 전에는 `task-final-report` Skill 본문 수정에 착수하지 않는다.
