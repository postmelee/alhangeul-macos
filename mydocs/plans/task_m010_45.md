# Issue #45 수행 계획서

## 목적

매 턴 시스템 프롬프트로 적재되는 `AGENTS.md`와 `CLAUDE.md`의 컨텍스트 비용을 줄이고, 절차형 워크플로우는 Codex/Claude Code 양쪽이 인식하는 Agent Skills로 분리한다. 모범 사례(2026)가 권고하는 "AGENTS.md는 항상 필요한 정책·제약·인덱스만, 수직적 워크플로우는 skill"을 본 저장소의 하이퍼-워터폴 절차에 맞게 적용한다.

## 배경

- `AGENTS.md` 263줄, `CLAUDE.md` 257줄. 두 파일은 약 95% 중복 (도구 명칭과 일부 빌드 규칙 차이만 존재).
- 다수의 강제 규칙이 이미 `mydocs/manual/` 매뉴얼(`build_run_guide.md`, `pr_process_guide.md`, `core_submodule_operation_guide.md`, `swift_macos_code_rules_guide.md`)에 상세히 존재하지만 AGENTS.md에 다시 기재되어 있다.
- 폴더 역할 표(L86~98), Git 워크플로우 다이어그램과 메인테이너/컨트리뷰터 스크립트(L162~226), 타스크 진행 15단계 절차(L243~259)는 분량이 커서 매뉴얼로 분리하기에 적합하다.
- 하이퍼-워터폴 절차의 5개 시점(타스크 시작, 단계 보고, 최종 보고+PR, merge 후 정리, 외부 PR 검토)은 명확한 트리거를 가진 절차형이라 SKILL.md로 분리해도 누락이 발생하지 않는다.
- Codex와 Claude Code가 동일한 Agent Skills 표준(SKILL.md frontmatter, scripts/, references/)을 채택했고, 디렉터리 위치(`.agents/skills/` vs `.claude/skills/`)와 명시 호출 prefix만 다르다.
- 모범 사례 자료
  - SmartScope 2026 가이드: AGENTS.md를 점진적 공개 구조로 전환해 토큰 70% 감소 사례
  - Vercel 평가: AGENTS.md(상시 컨텍스트)와 skill(수직 워크플로우)이 상호보완적
  - HumanLayer/Builder.io 가이드: 200줄 미만, 도구가 잡을 일은 도구에게

## 범위

### 포함

- `CLAUDE.md`를 `@AGENTS.md` 임포트 기반 ≤ 30줄로 축소
- `AGENTS.md`를 ≤ 100줄로 압축
  - 매뉴얼에 이미 있는 강제 규칙 중복 제거 (포인터로 대체)
  - 폴더 역할 표·Git 워크플로우·타스크 진행 절차를 신규 매뉴얼로 이전
  - 핵심: 프로젝트 개요, 하이퍼-워터폴 핵심 규칙, 명명 규칙, 매뉴얼 인덱스만 남김
- 신규 매뉴얼 작성
  - `mydocs/manual/document_structure_guide.md` — 폴더 역할 표, 문서 파일명 강제 규칙, 외부 PR 처리 폴더 정책
  - `mydocs/manual/git_workflow_guide.md` — 브랜치 표, 다이어그램, 메인테이너/컨트리뷰터 워크플로우 스크립트
  - `mydocs/manual/task_workflow_guide.md` — 타스크 진행 15단계 절차, 커밋 메시지 규칙, 승인 간주 조건
- Agent Skills 5종 작성 (도구 비종속)
  - `task-start` — 이슈 확인 → 브랜치 생성 → orders 갱신 → 수행계획서 템플릿
  - `task-stage-report` — `_stage{N}.md` 작성 + Stage 커밋 규칙
  - `task-final-report` — `_report.md` + orders 갱신 + publish push + draft PR
  - `pr-merge-cleanup` — 이슈 close + publish 원격 삭제 + local 정리 + devel 복귀
  - `external-pr-review` — `pr_{N}_review.md` → `pr_{N}_report.md` → archives
- 진실 원천: `mydocs/skills/{skill}/SKILL.md`
- 도구별 디렉터리: `.agents/skills`와 `.claude/skills`를 `mydocs/skills`로 향하는 심볼릭 링크로 구성, 두 링크 모두 저장소에 커밋
- 모든 SKILL.md frontmatter에 `allow_implicit_invocation: false` 명시 (Codex 전용 키, Claude Code는 무시) + description을 좁게 작성해 양 도구 모두 명시 호출 위주로 동작

### 제외

- 매뉴얼 본문 내용 재작성·증보 (분리/이동만)
- 신규 강제 규칙 도입 또는 기존 규칙 폐지
- skill 자동화 범위 확장 (예: skill 내부에서 `gh issue create` 자동 호출 등) — 이번 작업은 절차 문서화에 한정
- 외부 도구(MCP server, plugin packaging) 도입
- README.md 본문 변경 (사전 합의에 따라 별도 처리)
- core/Rust/Swift/Xcode 변경

## 설계 방향

### 1. CLAUDE.md는 차이 표면만 명시

`@AGENTS.md` 임포트로 전체 내용을 상속하고, "클로드 코드"와 "Codex" 표기 차이가 동작에 영향을 주지 않도록 AGENTS.md를 도구 중립적 표현("에이전트")으로 정리한다. CLAUDE.md에는 Claude Code 전용 보강(있는 경우)만 짧게 둔다.

### 2. AGENTS.md는 인덱스 + 강제 핵심만

남길 항목:

- 프로젝트 개요 (목표, 소유 경계 4~5줄)
- 하이퍼-워터폴 핵심 규칙 요약 + 승인 간주 조건
- 필수 참조 문서 인덱스 (기존 + 신규 3개 추가)
- 명명 규칙 통합 블록 (브랜치/커밋/문서 파일명, 마일스톤 표기)
- 작업 시간 규칙 1줄

옮기는 항목:

| 현재 위치(L) | 이전처 |
|--------------|--------|
| 84~98 폴더 역할 표 | `document_structure_guide.md` |
| 100~119 외부 PR 처리 절차 | `document_structure_guide.md` 내 폴더 정책 + 기존 `pr_process_guide.md` 링크 |
| 121~135 빌드 강제 규칙 7개 상세 | `build_run_guide.md` (이미 존재, AGENTS.md엔 1~2줄 요약 + 링크) |
| 162~226 Git 워크플로우 + 메인테이너/컨트리뷰터 워크플로우 | `git_workflow_guide.md` |
| 243~259 타스크 진행 15단계 | `task_workflow_guide.md` |

### 3. Skill 진실 원천 + 심볼릭 링크

- `mydocs/skills/{skill}/SKILL.md` 한 곳에 본문 작성
- `.agents/skills`와 `.claude/skills`를 `mydocs/skills`를 가리키는 심볼릭 링크로 만들고 두 링크를 커밋
- 심볼릭 링크 사용으로 양 도구가 동일 본문을 자동 인식, 유지보수 한곳에서 가능

### 4. SKILL.md 표준 형식

```markdown

name: task-start
description: |

  GitHub 이슈 확인, local/task{N} 브랜치 생성, orders 항목 추가,
```

## 구현 계획서 작성 시 나눌 단계 (잠정 5단계)

1. 신규 매뉴얼 3종 작성 (`document_structure_guide.md`, `git_workflow_guide.md`, `task_workflow_guide.md`) — AGENTS.md 본문에서 그대로 발췌·정렬
2. AGENTS.md 압축 + CLAUDE.md `@AGENTS.md` 임포트 전환 + 매뉴얼 인덱스 갱신
3. Skill 5종 SKILL.md 작성 + `mydocs/skills/` 디렉터리 구성
4. `.agents/skills`/`.claude/skills` 심볼릭 링크 생성 + 양 도구에서 인식되는지 점검
5. 최종 검증, 매뉴얼 링크 무결성 확인, 최종 보고서 작성

## 검증 계획

계획 문서 단계:

```bash
git diff --check -- mydocs/orders/20260425.md mydocs/plans/task_m010_45.md
```

각 단계 공통 검증:

```bash
# 매뉴얼/스킬 내부 상대 링크 점검 (수동 grep + 확인)
rg -n '\(mydocs/' AGENTS.md CLAUDE.md mydocs/manual/*.md
rg -n '\.\./' mydocs/manual/*.md mydocs/skills/*/SKILL.md

# 라인 수 점검
wc -l AGENTS.md CLAUDE.md mydocs/manual/document_structure_guide.md mydocs/manual/git_workflow_guide.md mydocs/manual/task_workflow_guide.md

# 심볼릭 링크 무결성
ls -la .agents/skills .claude/skills
readlink .agents/skills .claude/skills
test -f .agents/skills/task-start/SKILL.md && echo "ok"
test -f .claude/skills/task-start/SKILL.md && echo "ok"

# 작업 트리 잔여 변화 없음
git status --short
```

본 작업은 코드 변경이 아니므로 빌드/Xcode 검증은 수행하지 않는다 (`build_run_guide.md` 변경 유형별 최소 검증 정책상 문서 전용 변경에 빌드 강제 없음).

## 리스크

1. AGENTS.md 압축 시 정보 손실 우려 — 매뉴얼로 옮긴 항목이 누락 없이 모두 인용·연결되는지 grep로 검증해야 한다.
2. 심볼릭 링크가 OS·CI 환경에서 차단될 가능성 — macOS 로컬·GitHub Actions·Codex/Claude Code CLI 모두 심볼릭 링크를 인식하지만, 사전 점검이 필요하다.
3. SKILL.md가 양 도구 모두에서 명시 호출만 트리거되도록 description을 좁게 작성해야 함 — 너무 일반적이면 묵시적 invocation이 발생해 하이퍼-워터폴 승인 절차를 우회할 수 있다.
4. `@AGENTS.md` import 문법 — Claude Code의 import 문법(`@경로`)이 현재 버전에서 지원되는지 확인이 필요하다. 미지원이면 짧은 본문 + 매뉴얼 인덱스 형태로 폴백한다.
5. `.agents/skills` 디렉터리 점(.) prefix가 일부 자동 도구에서 무시될 수 있음 — Codex 공식 문서가 명시한 경로이므로 변경하지 않고 유지한다.

## 승인 요청 사항

이 수행 계획서 기준으로 구현 계획서(`task_m010_45_impl.md`)를 작성하고, 이후 단계별 구현으로 진행할지 승인 요청한다.
