# Task #82 수행계획서 - 렌더링 디버깅 도구 문서 체계 정리

## 목적

`validate-stage3-render.sh`와 `render-debug-compare.sh`의 역할, 사용 시점, 산출물 해석, PR 검증 기록 방법을 저장소 문서 전반에서 일관되게 안내한다. native renderer 개선 기여자가 core 기준 렌더와 Swift native 렌더 차이를 직접 좁힐 수 있도록 디버깅 매뉴얼을 정리한다.

## 배경

PR #68에서 core SVG, render tree JSON, native PNG, 선택적 pixel diff를 생성하는 디버깅 도구가 추가되었다. 그러나 현재 문서에서는 `validate-stage3-render.sh`가 무엇을 자동 검증하는지, `render-debug-compare.sh`가 언제 필요한지, 두 도구가 서로 어떻게 이어지는지가 분산되어 있다.

또한 `render_core_native_compare.md`는 반복 사용 가능한 절차 문서에 가깝지만 `troubleshootings/` 아래에 있고, README/CONTRIBUTING/PR 템플릿의 안내도 실제 renderer 기여자가 필요한 수준까지 연결되지 않는다.

## 범위

포함:

- `README.md`의 디버깅 프로토콜을 렌더링 문제와 Finder/Quick Look 통합 문제로 구분
- `CONTRIBUTING.md`의 버그 리포트, PR 전 체크리스트, 디버깅 가이드 보강
- `mydocs/manual/build_run_guide.md`에 `validate-stage3-render.sh` 검증 항목, 한계, custom sample 사용법, `render-debug-compare.sh`로 넘어가는 조건 추가
- `mydocs/manual/swift_macos_code_rules_guide.md`에 renderer 변경 시 smoke/debug compare 검증 기준 추가
- `render_core_native_compare.md`를 manual 문서로 이동하거나, 이동하지 않을 경우 그 이유와 링크 구조 정리
- `table-in-tbox` core/native 산출물 사례를 짧은 실제 활용 예시로 추가
- `.github/pull_request_template.md`의 `build.noindex/DerivedData` 경로 정합성 및 renderer 변경 선택 검증 안내 보정
- 필요 시 `validate-stage3-render.sh`의 help/usage처럼 self-documenting 개선

제외:

- native renderer 동작 수정
- Rust core dependency 갱신
- 새 HWP/HWPX fixture 추가
- Finder/Quick Look 등록 로직 수정
- 릴리스, 서명, 공증, Homebrew Cask 작업

## 설계 방향

문서의 역할을 분리한다.

- README/CONTRIBUTING: 기여자가 처음 보는 짧은 진입 안내
- `build_run_guide.md`: 표준 빌드/검증 명령과 각 명령의 성공/실패 의미
- renderer 비교 매뉴얼: 특정 문서의 core/native 차이를 좁히는 상세 절차와 사례
- 코드 규칙/PR 템플릿: renderer 변경자가 남겨야 하는 검증 증거

기존 정보를 중복 복제하지 않고, 상세 판단 흐름은 renderer 비교 매뉴얼로 모은다. `validate-stage3-render.sh`는 자동 smoke 관문, `render-debug-compare.sh`는 원인 분해용 진단 도구로 구분한다.

## 잠정 단계

1. Stage 1: 현재 문서 링크와 위치 정리, renderer 비교 문서의 manual 이동 여부 확정
2. Stage 2: `build_run_guide.md`와 renderer 비교 매뉴얼 보강
3. Stage 3: README, CONTRIBUTING, Swift/macOS 코드 규칙, PR 템플릿 안내 보정
4. Stage 4: 필요 시 `validate-stage3-render.sh` usage/help 보강 및 링크 정합성 점검
5. Stage 5: 문서 검증, 최종 점검, 보고서 작성

## 검증 계획

- `git diff --check`
- `rg -n "render_core_native_compare|validate-stage3-render|render-debug-compare|build/DerivedData|build.noindex/DerivedData" README.md CONTRIBUTING.md .github mydocs scripts`
- `./scripts/validate-stage3-render.sh --help` 또는 usage 보강 시 동작 확인
- 문서 링크가 이동 후에도 상대 경로 기준으로 맞는지 `rg`로 확인
- 실제 renderer 동작은 수정하지 않으므로 full Xcode build는 필요 여부를 단계 보고에서 판단

## 리스크

- `render_core_native_compare.md`를 이동하면 기존 PR/문서 링크가 깨질 수 있다. 필요한 경우 troubleshooting 경로에 짧은 forwarding 문서를 남긴다.
- 실제 산출물 예시가 특정 로컬 경로에 묶이면 외부 기여자에게 재현성이 떨어질 수 있다. 예시는 명령과 판단 방식 중심으로 작성하고, 로컬 output 경로는 참고로만 둔다.
- PR 템플릿에 선택 검증 항목을 너무 많이 넣으면 모든 PR에 과한 부담이 될 수 있다. renderer 관련 변경에서만 쓰는 조건부 안내로 둔다.

## 승인 요청 사항

위 범위와 5단계 계획으로 Task #82를 진행해도 되는지 승인 요청한다. 승인 전에는 구현계획서 작성이나 실제 문서 본문 변경을 진행하지 않는다.
