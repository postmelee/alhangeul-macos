# Task M018 #206 수행계획서

## 목적

`v0.1.1` public release 전에 GitHub Pages와 Sparkle appcast 게시 경로를 재검토한다. 현재 `main` 브랜치의 `docs/`를 Pages source로 쓰는 branch publishing 구조를 유지할지, GitHub Actions `deploy-pages` 기반 배포로 전환할지 결정하고, 선택한 구조에서 release workflow 결과만으로 Pages/appcast 게시 성공 여부를 판단할 수 있게 한다.

## 배경

현재 `Release Publish DMG` workflow는 official stable release에서 signed/notarized DMG를 GitHub Release asset으로 게시한 뒤 Sparkle appcast XML을 생성한다. 생성된 appcast는 별도 Pages worktree를 clone한 다음 `docs/appcast.xml`만 커밋하고 Pages source branch로 push하는 방식으로 반영한다.

이 구조는 Pages source가 `main` 브랜치의 `/docs`인 legacy branch publishing 모델에 의존한다. GitHub Pages branch publishing 문서 기준으로 `GITHUB_TOKEN`을 사용하는 workflow가 push한 commit이 Pages build를 항상 trigger하지 않을 수 있으므로, appcast commit/push 성공과 실제 Pages deployment 성공이 같은 workflow summary에서 닫히지 않는 위험이 있다.

#185, #186, #198, #208에서 release note, CI, action runtime, universal DMG 안내 기준은 정리했지만, Pages/appcast deployment model은 #206 범위로 남겼다. #188 public release 실행 전에 이 판단과 검증 기준을 확정해야 한다.

## 범위(포함·제외)

포함:

- 현재 GitHub Pages 설정, `main`/`docs` source, release workflow의 `docs/appcast.xml` push 경로 확인
- GitHub Actions Pages 배포 방식으로 전환할 경우 필요한 `pages: write`, `id-token: write`, `github-pages` environment, Pages source 설정 검토
- `actions/upload-pages-artifact`, `actions/deploy-pages` 적용 가능성과 현재 official major version 확인
- `docs/` 정적 사이트와 release workflow가 생성한 `appcast.xml`을 하나의 Pages artifact로 배포하는 구조 설계
- 전환 시 `release-publish.yml`의 appcast 생성, Pages artifact assembly, deployment URL summary, artifact retention, 실패 조건 정리
- 전환하지 않을 경우 branch publishing 유지 시 필요한 manual/automatic verification 기준 정리
- #188 public release handoff 문서와 release 매뉴얼에 Pages/appcast 검증 기준 연결

제외:

- v0.1.1 public release 실행, tag 생성, GitHub Release publish
- Developer ID signing, notarization, Sparkle EdDSA signing 방식 변경
- Homebrew Cask publish 또는 tap repository 변경
- Pages site redesign 또는 사용자-facing copy 전면 수정
- appcast XML schema나 Sparkle channel 정책 변경
- GitHub repository Pages setting의 실제 전환 실행은 별도 승인 없이 수행하지 않음

## 설계 방향

먼저 공식 문서와 저장소 설정을 기준으로 branch publishing 유지와 Actions Pages deployment 전환의 장단점을 비교한다. 전환이 적절하면 release workflow 안에서 생성한 stable appcast를 `docs/appcast.xml`에 overlay한 Pages artifact를 만들고, `deploy-pages`가 반환하는 deployment URL을 workflow summary에 남기는 방향을 우선 검토한다.

전환안은 official stable release에서만 appcast와 Pages deployment가 실행되어야 한다. `draft=true` 또는 `prerelease=true` 실행에서는 지금처럼 stable appcast 배포를 건너뛰고 summary에 skip 사유를 남긴다. GitHub Release asset upload, Sparkle EdDSA signature 생성, release notes URL, universal DMG URL 기준은 유지한다.

deployment job 분리는 GitHub Pages environment와 release environment 보호 조건을 함께 고려해 결정한다. `release` environment에 묶인 signing/notarization secret 접근과 `github-pages` deployment environment를 같은 job에서 다룰 수 없다면, appcast artifact를 넘기는 별도 deploy job을 둔다. 이때 Pages artifact에는 추적 중인 `docs/` 내용과 방금 생성한 `appcast.xml`이 모두 포함되어야 하며, branch push fallback은 기본 경로에서 제거하거나 명시적 비상 절차로만 남긴다.

전환하지 않는 결론이 나오면 현 branch publishing 구조의 한계를 문서화하고, #188에서 `docs/appcast.xml` commit 존재 여부와 public Pages URL 반영 여부를 별도 검증하는 기준을 강화한다.

## 잠정 단계(3~6단계)

1. 현황 확인과 공식 기준 조사
   - repository Pages 설정과 현재 workflow의 Pages branch push 경로를 확인한다.
   - GitHub Pages publishing source, `upload-pages-artifact`, `deploy-pages`, required permissions/environment 기준을 공식 문서로 확인한다.
   - branch publishing 유지와 Actions deployment 전환의 release 운영상 장단점을 정리한다.
2. 배포 모델 결정과 구현 계획 확정
   - 전환 여부를 결정하고, 필요한 경우 `release-publish.yml` job 구조와 Pages artifact assembly 방식을 확정한다.
   - repository Pages setting 변경이 필요한 경우 작업지시자 승인 항목으로 별도 분리한다.
   - 승인 후 구현계획서(`task_m018_206_impl.md`)를 작성한다.
3. workflow/helper 반영
   - 전환 시 appcast 생성 결과를 Pages artifact에 포함하고 `deploy-pages` deployment URL을 summary에 기록한다.
   - 유지 시 branch push 이후 public Pages URL 확인 기준과 실패 시 중단/복구 기준을 workflow 또는 문서에 반영한다.
   - shell helper가 필요하면 `scripts/ci/` 아래에 분리하고 PR CI에서 문법 또는 dry-run 검증을 연결한다.
4. release 문서와 #188 handoff 갱신
   - release/CI/Pages-Sparkle 매뉴얼에 선택한 배포 모델, 권한, 검증 명령, 실패 처리 기준을 반영한다.
   - `mydocs/release/v0.1.1.md`에 #188에서 확인해야 할 Pages/appcast deployment 기준을 연결한다.
5. 통합 검증과 보고
   - workflow YAML parse, helper syntax/dry-run, release helper check, `git diff --check`를 수행한다.
   - 실제 Pages deployment 실행이 이번 범위에서 불가능하거나 별도 승인 대상이면 #188 handoff에 미실행 사유와 확인 명령을 남긴다.

## 리스크

- repository Pages source를 GitHub Actions로 전환해야 한다면 GitHub UI/API 설정 변경이 필요하며, 이는 별도 승인 없이는 실행할 수 없다.
- `release` environment와 `github-pages` environment 보호 조건이 같은 workflow에서 충돌할 수 있다.
- `deploy-pages`로 전환하면 `docs/appcast.xml`이 더 이상 branch commit으로 남지 않을 수 있어, 장기 추적 기준을 artifact/deployment 기록 또는 release record로 옮겨야 한다.
- release workflow가 tag ref에서 실행되므로 Pages artifact에 포함할 `docs/` 내용이 release tag의 상태와 일치해야 한다.
- 실제 official release와 Pages deployment는 signing/notarization secret, Sparkle private key, public release 승인에 의존하므로 로컬에서 끝까지 재현할 수 없다.

## 승인 요청 사항

1. 위 범위와 5단계 진행 구조 승인
2. Stage 1에서 GitHub Pages 공식 문서와 repository Pages 설정을 확인하고, branch publishing 유지와 `deploy-pages` 전환 중 하나를 결정하는 방향 승인
3. 전환이 적절하다고 판단되면 `release-publish.yml`에서 `docs/` + generated `appcast.xml` Pages artifact를 `deploy-pages`로 배포하는 구현 방향 승인
4. repository Pages source 설정 변경이 필요할 경우, 구현과 별도로 작업지시자 승인을 다시 받는 절차 승인
5. 승인 후 구현계획서(`mydocs/plans/task_m018_206_impl.md`) 작성
