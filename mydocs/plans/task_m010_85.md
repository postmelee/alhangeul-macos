# Task #85 수행 계획서

## 목적

Quick Look preview에서 HWP/HWPX 문서의 첫 페이지만 보이는 현재 동작을 전체 페이지 미리보기로 확장한다. 사용자가 Finder에서 Space 또는 Quick Look 패널을 열었을 때 HostApp을 실행하지 않아도 다중 페이지 문서의 흐름을 확인할 수 있게 한다.

## 배경

현재 Quick Look preview 경로는 `HwpPreviewProvider`가 `HwpPageImageRenderer.renderFirstPage(fileURL:)`를 호출하고, 공통 렌더러가 `renderPageTree(at: 0)`와 `pageSize(at: 0)`만 사용해 단일 PNG reply를 반환한다.

반면 `RhwpDocument`는 이미 `pageCount`, `pageSize(at:)`, `renderPageTree(at:)`를 제공하고, HostApp viewer는 이 API를 사용해 전체 페이지를 지연 렌더링한다. 따라서 Rust core ABI를 바꾸지 않고 Quick Look preview 응답 계층을 다중 페이지 표시용으로 확장할 수 있다.

중요한 전제는 HWP/HWPX 원본을 PDF로 직접 변환하지 않는다는 점이다. 이번 작업에서 검토할 PDF는 Quick Look이 다중 페이지 preview를 표시하도록 하기 위한 임시 preview 컨테이너이며, 실제 렌더링 기준은 기존 rhwp render tree와 `CGTreeRenderer` 경로를 유지한다.

## 범위

### 포함

- Quick Look preview가 전체 페이지를 표시할 수 있는 응답 형식 적용
- 기존 첫 페이지 bitmap 렌더링 경로를 페이지 번호 기반으로 일반화
- rhwp render tree 결과를 Quick Look 표시용 다중 페이지 PDF preview에 담는 구현
- Thumbnail extension의 첫 페이지 기반 동작 유지
- 50 MB 초과 파일 fallback 정책 유지
- extension 메모리와 초기 표시 지연을 고려한 제한 또는 fallback 정책 검토
- 다중 페이지 샘플 기준 render smoke와 Quick Look preview 동작 검증

### 제외

- 사용자용 PDF export 기능
- HWP/HWPX 원본 저장, 변환, 수정 기능
- Rust core ABI 변경
- `edwardkim/rhwp` core 코드 변경
- Thumbnail extension의 다중 페이지 표시
- HostApp viewer UI 구조 변경
- 서명, 공증, 배포 패키징 정책 변경

## 설계 방향

- `Sources/RhwpCoreBridge`에는 AppKit/UIKit 의존을 추가하지 않는다.
- 공통 렌더러는 기존 `renderFirstPage` 사용자를 깨지 않도록 유지하고, 내부 구현을 페이지 번호 기반 helper로 분리한다.
- Quick Look preview는 단일 PNG reply 대신 다중 페이지 PDF reply를 반환하는 방향을 우선 검토한다.
- PDF는 HWP 구조 변환 산출물이 아니라 rhwp native renderer가 만든 페이지 bitmap을 담는 Quick Look 표시용 임시 포맷으로 취급한다.
- 1차 구현은 보수적으로 각 페이지를 bitmap으로 렌더링한 뒤 PDF 각 페이지에 삽입한다. PDF CGContext에 render tree를 직접 그리는 벡터 PDF 방식은 품질과 좌표계 차이를 별도 검증한 뒤 후속 개선 후보로 둔다.
- 문서가 매우 길거나 렌더링 비용이 과도한 경우 Quick Look extension 안정성을 우선해 기존 fallback 또는 페이지 수 제한 정책을 적용할 수 있다.

## 잠정 단계

초기 계획은 총 4단계로 진행하고 각 단계 완료 후 작업지시자 승인을 받는 구조였다. Stage 4 이후 작업지시자가 제공한 Finder Quick Look 영상에서 목록 전환 중 preview 응답 지연이 확인되어, PR 게시 전 성능 보정 단계를 Stage 5로 추가한다.

- **Stage 1 — Quick Look 다중 페이지 preview 설계 확정**
  - SDK의 `QLPreviewReply` PDF 지원 방식과 현재 extension 경로를 확인한다.
  - 단일 PNG, PDF data reply, PDFDocument reply 후보를 비교하고 구현 방식을 확정한다.
  - extension 메모리, 파일 크기 fallback, 페이지 수 제한 필요성을 정리한다.
- **Stage 2 — 페이지 번호 기반 렌더링 helper 분리**
  - `HwpPageImageRenderer`의 첫 페이지 전용 구현을 내부 페이지 렌더 helper로 분리한다.
  - 기존 Thumbnail extension과 첫 페이지 preview 경로가 깨지지 않도록 호환 API를 유지한다.
- **Stage 3 — Quick Look 전체 페이지 PDF preview 구현**
  - Quick Look provider가 전체 페이지를 표시할 수 있는 PDF preview reply를 생성하도록 변경한다.
  - PDF는 기존 native renderer가 만든 페이지 결과를 담는 표시용 컨테이너로 구현한다.
  - 오류와 큰 파일 fallback 동작을 유지한다.
- **Stage 4 — 검증과 문서 정리**
  - Swift/renderer 검증, HostApp build, Quick Look preview smoke를 수행한다.
  - 구현 결과와 제한 사항을 운영 문서 또는 기술 문서에 필요한 만큼 반영한다.
  - 최종 보고서에 PDF preview 방식이 PDF export가 아님을 명확히 기록한다.
- **Stage 5 — Quick Look preview 응답 지연 보정**
  - `providePreview`에서 전체 PDF 렌더링을 완료한 뒤 reply를 반환하던 구조를 피한다.
  - 파일 크기, page count, 첫 페이지 크기만 먼저 검사하고, 실제 preview data 생성은 `QLPreviewReply` data creation block으로 지연한다.
  - 단일 페이지 문서는 PDF container 대신 PNG reply를 사용해 기존 빠른 preview 특성에 가깝게 되돌린다.
  - 다중 페이지 문서는 첫 화면만 즉시 표시한 뒤 나머지를 append하는 true lazy 구조가 현재 data reply 방식으로는 어렵다는 제한을 기록하고, 필요 시 별도 spike 후보로 둔다.

## 리스크

- **긴 문서 초기 지연 (중)**: Quick Look reply 생성 시 모든 페이지를 렌더링하면 큰 다중 페이지 문서에서 표시가 늦어질 수 있다. Stage 1에서 페이지 수 제한 또는 fallback 기준을 검토한다.
- **extension 메모리 사용량 증가 (중)**: 페이지별 bitmap을 PDF에 담으면 메모리와 임시 data 크기가 증가한다. 렌더링 중간 산출물을 오래 보관하지 않는 구조로 구현한다.
- **PDF 의미 혼동 (중)**: 이번 작업의 PDF는 Quick Look 표시용 preview 컨테이너이지 사용자용 PDF export가 아니다. 문서와 보고서에 이 경계를 명확히 남긴다.
- **PDF CGContext 직접 렌더 품질 차이 (저)**: 1차 구현은 bitmap 삽입 방식으로 제한해 CoreGraphics PDF 좌표계와 텍스트/이미지 렌더 차이를 줄인다.
- **기존 thumbnail 회귀 (중)**: `HwpPageImageRenderer`를 일반화하면서 Thumbnail extension의 첫 페이지 cache 경로가 영향을 받을 수 있다. Stage 2와 Stage 4에서 기존 thumbnail 경로를 별도 확인한다.
