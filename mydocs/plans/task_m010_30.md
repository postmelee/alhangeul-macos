# Issue #30 수행 계획서

## 작업명

hwpql 장점 반영 3: RustBridge를 git-rev dependency로 전환하고 Vendor/rhwp submodule 제거

## 이슈

- GitHub Issue: #30
- Milestone: v0.1.0
- 문서 prefix: `task_m010_30`
- 이슈 URL: https://github.com/postmelee/alhangeul-macos/issues/30

## 목적

앱 저장소가 `Vendor/rhwp` submodule을 직접 포함하지 않도록 정리하고, `RustBridge`가 `edwardkim/rhwp`를 git dependency로 가져오게 전환한다.

완료 후 Demo/Preview 경로는 필요한 bridge API가 포함된 resolved commit을 `rev`로 고정하고, Stable 경로는 compatibility gate를 통과한 release tag와 resolved commit을 함께 고정하는 기준을 갖는다. core provenance는 `Cargo.lock`과 `rhwp-core.lock`에 남기며, fresh checkout 또는 `Vendor/rhwp`가 없는 상태에서도 build와 render 검증이 가능해야 한다.

## 배경

현재 저장소는 앱, Quick Look/Thumbnail 확장, Swift bridge, packaging 정책을 소유하고, Rust core는 `edwardkim/rhwp` 기준으로 고정해야 한다. 기존 submodule 방식은 앱 저장소 checkout에 core worktree를 포함하므로 fresh checkout, lock 검증, release provenance 기록이 분산된다.

Issue #55에서 정리한 compatibility 기준에 따르면 2026-04-26 확인 당시 최신 release `v0.7.3`의 resolved commit `c2e8a3461de800a02f76127ff4797bade1d4e532`에는 현재 native render tree 경로가 요구하는 `DocumentCore::build_page_render_tree`와 `DocumentCore::get_bin_data`가 없다. 반면 현재 lock commit `1e9d78a1d40c71779d81c6ec6870cd301d912626`에는 필요한 API가 포함되어 Demo/Preview용 commit-pinned git dependency 후보가 될 수 있다.

따라서 이번 작업은 Stable release tag 전환을 무리하게 완료 처리하지 않고, Demo/Preview commit pin과 Stable compatibility gate를 분리해 구현한다. 구현 단계에서는 `edwardkim/rhwp`의 실제 최신 release와 대상 commit을 다시 확인하고, branch나 floating ref는 배포 기준으로 사용하지 않는다.

## 범위

포함:

- `RustBridge/Cargo.toml`의 `rhwp` dependency를 submodule path에서 `edwardkim/rhwp` git dependency로 전환
- Demo/Preview용 `git` + `rev` commit pin 적용
- Stable용 `git` + `tag` release tag 경로와 compatibility gate 기준 유지
- `RustBridge/Cargo.lock` 갱신과 resolved commit 확인
- `.gitmodules`와 git index에서 `Vendor/rhwp` gitlink 제거
- `scripts/build-rust-macos.sh`의 submodule 전제 제거와 `Cargo.lock`/`rhwp-core.lock` 정합성 검증 보강
- `scripts/update-rhwp-core.sh`를 git dependency update gate로 재정의
- `rhwp-core.lock` v2 필드 정리와 산출물 hash/size 기록 유지
- README, core 운영 매뉴얼, build/run 매뉴얼, architecture 문서의 submodule 기준 설명 보정
- fresh checkout 또는 `Vendor/rhwp` 없는 상태의 build/render 검증

제외:

- upstream `edwardkim/rhwp` core API 변경
- `Rhwp.xcframework` 구조 변경
- Swift native render tree 구조의 대규모 변경
- HTML/WKWebView preview 도입
- release signing, notarization, Homebrew Cask 자동화
- Stable release tag가 required core API를 포함하지 않는 상태에서 Stable 전환을 완료로 표시

## 설계 방향

기준 경로는 native render tree를 유지한다. Swift/macOS 계층은 기존 C ABI와 `Sources/RhwpCoreBridge` 모델을 계속 사용하고, 이번 작업의 중심은 core 소싱 방식과 provenance 검증을 submodule 기준에서 git dependency 기준으로 바꾸는 것이다.

`rhwp-core.lock`은 `Cargo.lock`을 보조하는 앱 저장소 관점의 provenance 문서로 유지한다. Demo/Preview에서는 `rhwp_ref_kind = "commit"`과 `rhwp_release_transition_status = "demo-commit-pin"`을 기록하고, Stable에서는 `rhwp_ref_kind = "release-tag"`, `rhwp_release_tag`, `rhwp_commit`을 함께 기록한다. 두 lock의 repo, ref kind, resolved commit이 불일치하면 build script가 명확한 오류로 중단해야 한다.

`scripts/update-rhwp-core.sh`는 다음 interface를 기준으로 재정의한다.

```bash
./scripts/update-rhwp-core.sh --channel demo --rev <commit-sha>
./scripts/update-rhwp-core.sh --channel stable --tag <release-tag>
./scripts/update-rhwp-core.sh --check --channel stable --tag <release-tag>
```

실패 유형은 `release lookup failure`, `missing core API`, `dependency fetch failure`, `Cargo.lock mismatch`, `artifact hash mismatch`, `FFI symbol diff`, `render smoke failure`처럼 분리해 단계 보고서에 남긴다.

검색 결과에 남는 `Vendor/rhwp`, `git submodule`, `submodule` 항목은 역사 설명 또는 명시적 legacy 문맥이어야 한다. `git submodule status`에는 `Vendor/rhwp`가 나오지 않아야 한다.

스크립트 변경이 포함되므로 다음 문법 검증도 수행한다.

```bash
bash -n scripts/update-rhwp-core.sh
bash -n scripts/build-rust-macos.sh
```

`shellcheck`가 로컬에 있으면 추가로 실행하고, 없으면 미실행 사유를 단계 보고서에 기록한다.

## 리스크

- 첫 git dependency build에는 네트워크가 필요하므로 dependency fetch 실패를 core compatibility 실패와 분리해야 한다.
- Demo/Preview commit pin은 release tag 안정 기준이 아니므로 Stable 또는 정식 release 기준으로 표시하면 안 된다.
- upstream 최신 release가 구현 단계에 바뀔 수 있다. 구현 시작 시 실제 latest release와 resolved commit을 다시 확인한다.
- submodule 제거 후에도 문서나 스크립트에 남은 `Vendor/rhwp` 전제가 있으면 fresh checkout 검증에서 누락이 발생할 수 있다.
- `Cargo.lock`과 `rhwp-core.lock`을 함께 유지하므로 commit/source 불일치를 자동 검증하지 않으면 provenance가 깨질 수 있다.
- `update-rhwp-core.sh`를 크게 재정의하므로 기존 submodule 기반 운용 문서와 script 호출자의 기대를 함께 보정해야 한다.
