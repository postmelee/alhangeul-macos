# Issue #57 수행 계획서

## 작업명

이슈 미등록 작업을 위한 task-register Skill 추가

## 이슈

- GitHub Issue: #57
- Milestone: v0.1.0
- 문서 prefix: `task_m010_57`
- 이슈 URL: https://github.com/postmelee/alhangeul-macos/issues/57

## 목적

이슈가 아직 생성되지 않은 작업을 하이퍼-워터폴 절차에 맞게 GitHub Issue로 등록하는 전용 Skill을 추가한다.

완료 후 에이전트는 작업지시자가 "이 작업 이슈부터 만들어줘" 또는 동등한 명시 지시를 했을 때 열린 milestone과 label 목록을 확인하고, 적합한 milestone/label을 고른 뒤 이슈 생성 전 승인 지점을 분명히 둘 수 있어야 한다. 생성된 이슈는 기존 `task-start` Skill로 자연스럽게 이어져야 한다.

## 배경

PR #46에서 AGENTS.md/CLAUDE.md를 압축하고 하이퍼-워터폴 절차 5종을 Skill로 분리했다. 이때 `task-start`는 "승인된 이슈 번호와 마일스톤이 존재"한다는 전제로 설계되었고, `gh issue create`를 포함한 자동화 범위 확장은 제외했다.

PR #53은 신규 매뉴얼 3종의 자체 완결성을 보강했지만, 이슈가 없는 작업을 등록하는 절차는 여전히 `task_workflow_guide.md`의 한 줄 예시에 머물러 있다. 따라서 작업지시자가 번호 없는 작업을 맡길 때 이슈 등록, milestone 선택, label 선택, `task-start` 진입 경계가 누락될 수 있다.

## 범위

포함:

- 신규 Skill `task-register` 작성
- 열린 GitHub milestone 목록 조회와 선택 기준 문서화
- 저장소 label 목록 조회와 선택 기준 문서화
- 이슈 생성 전 작업지시자 확인을 요구하는 절차 명시
- 이슈 생성 후 `task-start`로 넘어가는 책임 경계 명시
- 관련 매뉴얼의 타스크 번호 관리와 SKILL 호출 표시 안내 최소 보강

제외:

- `task-start`가 이슈 생성까지 수행하도록 책임 확대
- 이슈 생성 후 브랜치/오늘할일/수행계획서 작성을 `task-register`에 포함
- GitHub milestone 또는 label 자체 생성/수정
- PR 템플릿, 릴리스, 배포, 서명, 공증 절차 변경
- Rust/Swift/Xcode 소스 변경

## 설계 방향

`task-register`는 `task-start`보다 앞에 있는 선행 Skill로 둔다. 두 Skill의 책임 경계는 다음과 같이 나눈다.

- `task-register`: 번호 없는 작업을 GitHub Issue로 등록하고 issue number, milestone, label을 확정한다.
- `task-start`: 이미 존재하는 issue number를 기준으로 `local/taskN` 브랜치, 오늘할일, 수행계획서를 만든다.

원격 GitHub 상태를 바꾸는 작업이므로 `task-register`는 이슈 생성 전 제목, 본문, milestone, label 초안을 작업지시자에게 보여주고 승인을 받아야 한다. milestone 또는 label 판단이 모호하면 임의 생성하지 않고 질문한다.

Skill 본문은 기존 `mydocs/skills/*/SKILL.md`와 같은 형식을 사용한다. `allow_implicit_invocation: false`를 유지하고, description은 번호 없는 작업의 이슈 등록에만 좁게 맞춘다.

## 잠정 단계

1. Stage 1: 기존 Skill과 매뉴얼의 책임 경계 조사, `task-register` 절차 세부 설계
2. Stage 2: `task-register/SKILL.md` 작성
3. Stage 3: `task_workflow_guide.md`와 필요 시 문서 구조 매뉴얼에 이슈 등록 선행 절차 반영
4. Stage 4: 통합 검증, 최종 보고서 작성, 오늘할일 완료 처리

각 단계 완료 시 단계별 완료보고서를 작성하고 작업지시자 승인 후 다음 단계로 진행한다.

## 리스크

- `task-register`가 `task-start`와 책임을 겹치면 이슈 생성 후 브랜치/문서 작성 순서가 중복될 수 있다. 생성된 이슈 번호를 반환하고 멈추는 경계를 명확히 둔다.
- milestone 자동 선택 기준이 과도하게 단정적이면 잘못된 release bucket에 이슈가 들어갈 수 있다. 작업 성격이 모호하면 작업지시자 확인을 요구한다.
- label 선택 기준이 현재 GitHub 기본 label 목록에 강하게 묶일 수 있다. 현재 목록을 조회한 뒤 기존 label만 사용하고, 새 label 생성은 범위에서 제외한다.
- 이슈 생성은 원격 상태 변경이므로 승인 없는 자동 생성으로 읽히지 않도록 절차에 승인 지점을 명시한다.

## 승인 요청 사항

이 수행 계획서 기준으로 구현 계획서(`mydocs/plans/task_m010_57_impl.md`) 작성을 진행할지 승인 요청한다. 승인 전에는 `task-register` Skill과 매뉴얼 구현 변경을 진행하지 않는다.
