# Task M016 #147 수행계획서

## 목적

v0.1 공개 배포 전에 앱 bundle과 release artifact에 포함되는 third-party code, bundled `rhwp-studio` 정적 자산, WOFF2 font의 provenance와 라이선스 고지 위치를 정리한다.

완료 후 `THIRD_PARTY_LICENSES.md`, README 또는 release note 계열 문서에서 `rhwp` core, `rhwp-studio` asset snapshot, bundled font, proprietary font 비포함 정책을 일관되게 확인할 수 있어야 한다.

## 배경

현재 v0.1 artifact에는 Rust `rhwp` core 산출물, Swift bridge, HostApp의 WKWebView viewer가 사용하는 `rhwp-studio` 정적 asset, WOFF2 font가 함께 포함된다. `rhwp-core.lock`은 core release tag와 Rust bridge artifact hash/size를 기록하고, `Sources/HostApp/Resources/rhwp-studio/manifest.json`은 bundled `rhwp-studio` asset의 source repository, release tag, resolved commit, entrypoint hash를 기록한다.

폰트는 `Sources/HostApp/Resources/rhwp-studio/fonts/FONTS.md`가 오픈 라이선스 WOFF2와 Git 미포함 proprietary font 정책을 설명하고, native renderer fallback 정책은 `mydocs/tech/font_fallback_strategy.md`가 소유한다. 하지만 공개 배포 사용자가 가장 먼저 볼 license 고지 문서인 `THIRD_PARTY_LICENSES.md`는 현재 `rhwp` MIT license만 간단히 적고 있어, bundled `rhwp-studio` asset과 font attribution을 release artifact 기준으로 설명하기에 부족하다.

## 범위(포함·제외)

포함:

- 현재 bundled `rhwp-studio` asset의 기준 release tag, resolved commit, manifest hash와 `rhwp-core.lock` 정합성 확인
- `Sources/HostApp/Resources/rhwp-studio/fonts/FONTS.md`의 오픈 라이선스 font 목록과 proprietary font 비포함 정책 검토
- native renderer font fallback 정책 문서와 bundled font 고지의 책임 경계 정리
- `THIRD_PARTY_LICENSES.md` 중심의 third-party license/attribution 고지 구조 보강
- README 또는 release note 위치에 release artifact provenance와 license 고지 진입점 추가
- release artifact에 포함되는 `rhwp` core, Rust crate 산출물, generated framework/header, `rhwp-studio` asset, bundled font attribution 범위 점검

제외:

- 새 font 파일 추가 또는 기존 WOFF2 교체
- 한컴/HY/HCR/Microsoft 같은 proprietary font 번들
- `edwardkim/rhwp` core source 변경 또는 dependency pin 변경
- release signing, notarization, DMG, Homebrew Cask 작업
- 법률 자문 수준의 license 해석
- Xcode project 직접 수정

## 설계 방향

고지의 진실 원천은 사용자가 release artifact와 함께 확인할 수 있는 루트 `THIRD_PARTY_LICENSES.md`로 둔다. `FONTS.md`는 bundled font 목록과 font별 license/source 세부 표를 계속 소유하고, `THIRD_PARTY_LICENSES.md`는 그 내용을 release artifact 기준으로 요약하고 상세 위치로 연결한다.

`rhwp-core.lock`과 `rhwp-studio/manifest.json`은 provenance의 기계 판독 가능한 기준으로 유지한다. README에는 배포 전 확인자가 찾기 쉽도록 license/provenance 고지 위치를 짧게 연결하되, 동일한 license 표를 중복 복제하지 않는다.

proprietary font 정책은 명확하게 분리한다. 저장소와 release artifact에는 proprietary 한컴/HY/HCR/Microsoft font 파일을 포함하지 않으며, bundled 오픈 라이선스 WOFF2와 macOS 시스템 fallback으로 대체한다는 점을 license 고지와 font fallback 문서 양쪽에서 충돌 없이 설명한다.

## 잠정 단계(3~6단계)

1. Stage 1: 현재 release artifact 구성과 provenance inventory 확인
   - `rhwp-core.lock`, `rhwp-studio/manifest.json`, bundled font 목록, existing license 문서를 대조한다.
   - 실제 resource tree에 포함된 font/asset 목록과 proprietary font 비포함 여부를 확인한다.
2. Stage 2: license/attribution 고지 구조 설계
   - `THIRD_PARTY_LICENSES.md`, README, `FONTS.md`, font fallback 문서의 책임 경계를 정한다.
   - `rhwp`, `rhwp-studio`, bundled font, generated artifact provenance를 어느 문서에 어느 수준으로 남길지 확정한다.
3. Stage 3: third-party license와 font 고지 문서 보강
   - `THIRD_PARTY_LICENSES.md`를 release artifact 기준으로 보강한다.
   - 필요한 경우 `FONTS.md`와 font fallback 문서에 proprietary font 비포함 정책과 상세 고지 연결을 보강한다.
4. Stage 4: README/release 고지 진입점과 검증 정리
   - README에 license/provenance 고지 위치를 추가하거나 기존 문구를 정리한다.
   - asset 검증 스크립트, font 목록 대조, `git diff --check` 결과를 단계 보고서와 최종 보고서에 남긴다.

## 리스크

- font별 license/source 표가 upstream `FONTS.md`와 실제 bundled WOFF2 목록 사이에서 어긋나 있을 수 있다.
- `rhwp-studio` build 산출물의 JS/CSS/WASM dependency attribution 범위를 어디까지 적을지 판단이 필요할 수 있다.
- `THIRD_PARTY_LICENSES.md`와 README에 같은 정보를 중복하면 이후 core pin 또는 bundled asset 갱신 때 불일치가 생길 수 있다.
- proprietary font 비포함 정책을 모호하게 쓰면 사용자가 한컴/HY/HCR font가 bundle에 포함된 것으로 오해할 수 있다.
- 법률 자문 수준의 license 해석은 이번 범위에서 제외하므로, 배포 전 별도 법무 검토가 필요할 수 있다.
