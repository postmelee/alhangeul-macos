# Task M010 #267 수행계획서

## 목적

upstream `rhwp v0.7.12` stable release를 앱 저장소의 Rust core, bundled `rhwp-studio`, release provenance 기준에 반영하고, 현재까지 `devel`에 누적된 앱 변경을 포함해 `Alhangeul v0.1.3` public patch release를 준비한다.

완료 후에는 `rhwp-core.lock`과 bundled `rhwp-studio` manifest가 같은 upstream release 기준을 가리키고, Debug/Release build와 render/Finder smoke 검증 결과, release note, release record, GitHub Release/Sparkle/Pages 배포 결과가 추적 가능해야 한다.

## 배경

현재 public app release는 `v0.1.2`이고, 저장소의 `rhwp-core.lock` 및 `Sources/HostApp/Resources/rhwp-studio/manifest.json`은 upstream `rhwp v0.7.11` commit `a9dcdee32b17a7f9a20c609a5ed547e62fb8ebae` 기준이다.

2026-05-17에 upstream `edwardkim/rhwp`의 최신 stable release `v0.7.12`가 게시되었고, annotated tag는 resolved commit `1899ef9bc2dfd1c6c0c4d18b192d253a2d0a1fb5`를 가리킨다. 이번 작업은 branch/floating ref가 아니라 release tag와 resolved commit을 기준으로 core/studio provenance를 갱신한다.

프로젝트 운영 기준상 public release는 `devel`의 검증된 변경을 `main`에 반영한 뒤 tag 기준으로 수행한다. 따라서 upstream 반영, 앱 적응, 통합 PR, release PR, protected release workflow 실행을 한 흐름으로 추적하되, 각 단계 완료 후 작업지시자 승인을 받아 다음 단계로 넘어간다.

## 범위(포함·제외)

포함:

- upstream `rhwp v0.7.12` release tag와 resolved commit 확인
- `RustBridge/Cargo.toml`, `RustBridge/Cargo.lock`, `rhwp-core.lock` 갱신
- `Frameworks/generated_rhwp.h`, `Frameworks/universal/librhwp.a`, `rhwp-ffi-symbols.txt` 변화 검토와 lock metadata 갱신
- bundled `rhwp-studio` resource sync 필요 여부 판단과 필요한 경우 manifest/resource 갱신
- Swift `RhwpCoreBridge`, HostApp, Quick Look, Thumbnail 영향 검토와 필요한 적응
- 앱 버전/build 갱신 후보: `0.1.3` / build `9`
- Debug/Release build, lock 검증, studio asset 검증, render smoke, Finder Quick Look/Thumbnail smoke
- `mydocs/release/v0.1.3.md`, release note, delta checklist, 최종 보고서 작성
- `devel` 대상 작업 PR과 `main` 대상 release PR 준비
- 승인된 경우 release rehearsal/publish workflow 실행 결과 기록

제외:

- upstream `edwardkim/rhwp` 자체 코드 수정

- 작업지시자 별도 승인 없는 Homebrew Cask 공개 반영
- 작업지시자 별도 승인 없는 GitHub Issue close, rollback, 기존 public release 수정
- 서명/공증 credential 값, private key, token 등 민감 정보 기록

## 설계 방향

core dependency는 Stable 기준을 유지한다. `scripts/update-rhwp-core.sh --channel stable --tag v0.7.12`로 release tag pin을 갱신하고, `scripts/build-rust-macos.sh --update-lock`과 `--verify-lock`로 Cargo lock, generated header, FFI symbol, reference artifact metadata 정합성을 확인한다. ABI 변화가 있으면 `rhwp-ffi-symbols.txt`와 Swift bridge 영향 범위를 먼저 기록한 뒤 적응한다.

bundled viewer asset은 `rhwp-core.lock`과 별도 provenance다. `Sources/HostApp/Resources/rhwp-studio/manifest.json`의 release tag/commit, copied asset, WASM/JS/CSS entrypoint hash가 `v0.7.12` 기준과 맞는지 확인하고, sync가 필요하면 `scripts/sync-rhwp-studio.sh`와 `scripts/verify-rhwp-studio-assets.sh` 기준으로 반영한다.

릴리즈 identity는 앱 버전 `Alhangeul v0.1.3`을 기준으로 유지한다. upstream `rhwp` 버전은 GitHub Release title에 병기할지 별도 판단하되, release body와 `mydocs/release/v0.1.3.md`에는 `rhwp-core.lock` 및 studio manifest metadata로 명확히 기록한다.

public 배포는 `devel` 통합 PR merge 후 `main` release PR과 tag를 거쳐 진행한다. signed/notarized DMG, GitHub Release, Sparkle appcast, Pages deployment는 작업지시자의 단계별 승인과 보호 workflow 상태를 확인한 뒤 실행한다.

## 잠정 단계(3~6단계)

1. Stage 1: upstream release inventory와 compatibility gate 확인
   - `rhwp v0.7.12` release tag, resolved commit, release note, diff 영향 영역을 확인한다.
   - `scripts/update-rhwp-core.sh --check --channel stable --tag v0.7.12`로 target release compatibility를 조회한다.
   - bundled `rhwp-studio` sync 필요성과 기존 자동 sync PR 후보 유무를 확인한다.
2. Stage 2: core/studio provenance 갱신과 bridge 적응
   - RustBridge dependency, Cargo lock, `rhwp-core.lock`, generated header/staticlib metadata를 갱신한다.
   - bundled `rhwp-studio` asset이 필요한 경우 sync하고 manifest를 갱신한다.
   - FFI ABI 또는 render tree JSON 변화가 있으면 Swift bridge와 renderer 적응을 수행한다.
3. Stage 3: 앱 버전, release record, release communication 준비
   - HostApp/extension version/build를 `0.1.3` / build `9` 후보 기준으로 맞춘다.
   - `mydocs/release/v0.1.3.md`, release note 초안, delta checklist 기준을 작성한다.
   - GitHub Release body의 `전체 요약`, `포함된 rhwp 변화`, `알한글 앱 변화` 구분에 실제 변경사항을 채운다.
4. Stage 4: 통합 검증과 설치본 smoke
   - lock verify, AppKit boundary, studio asset verify, Debug/Release build, universal slice, render smoke를 수행한다.
   - 가능한 범위에서 Finder Quick Look/Thumbnail smoke와 Sparkle update smoke 준비 상태를 확인한다.
   - 실패한 검증은 같은 단계 안에서 회복하고, 미실행 수동 검증은 사유를 기록한다.
5. Stage 5: 최종 보고와 통합 PR
   - 최종 결과보고서와 오늘할일 상태를 정리한다.
   - `publish/task267` PR을 `devel` 대상으로 만들고 PR CI 결과를 확인한다.
   - merge 확인 후 `pr-merge-cleanup` 절차를 수행한다.
6. Stage 6: main release PR과 public 배포
   - `devel`의 검증 commit을 `main`으로 반영하는 release PR을 준비한다.
   - tag `v0.1.3`, rehearsal workflow, protected publish workflow, GitHub Release, Sparkle appcast, Pages deployment를 승인받아 수행한다.
   - public DMG SHA256, notarization/Gatekeeper, Pages/appcast 결과를 release record에 기록한다.

## 리스크

- `rhwp v0.7.12`의 FFI 또는 render tree JSON 변화가 Swift bridge와 native render smoke에 영향을 줄 수 있다.
- `rhwp-studio` asset sync는 WASM/JS/CSS entrypoint hash와 WKWebView resource loading 경로를 동시에 바꾸므로 HostApp smoke가 필요하다.
- Release build와 public DMG는 signing/notarization credential, protected environment, repository secrets에 의존하므로 로컬에서 전부 재현하지 못할 수 있다.
- 앱 version/build를 갱신한 뒤 public publish가 지연되면 release record, Pages, Sparkle, Cask 안내가 stale 상태가 될 수 있다.
- Homebrew Cask 공개 반영은 public DMG URL과 SHA256 확정 이후에만 가능하므로 이번 task 안에서 진행할지 별도 승인 경계가 필요하다.
- Intel Mac 실기기 smoke는 사용 가능한 하드웨어에 따라 미실행 사유를 남겨야 할 수 있다.
