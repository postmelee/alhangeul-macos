# Task M016 #151 수행계획서

## 목적

v0.1 배포 전 Quick Look preview와 Finder Thumbnail extension이 실제 설치본 기준으로 등록되고 동작하는지 반복 가능하게 판정하는 smoke gate를 정리한다.

완료 후에는 `scripts/package-release.sh`가 만든 Release package staging app을 `$HOME/Applications/Alhangeul.app`에 설치한 뒤, PlugInKit 등록 상태, bundle identifier와 principal class 정합성, `qlmanage -t` thumbnail smoke, `qlmanage -p` preview 확인의 자동화 가능 범위와 수동 확인 한계를 같은 절차에서 확인할 수 있어야 한다.

## 배경

Debug build 통과는 Finder Quick Look preview와 Thumbnail extension의 시스템 등록 상태를 보장하지 않는다. `CODE_SIGNING_ALLOWED=NO` 산출물은 compile/link와 bundle resource 확인에는 유효하지만, LaunchServices/PlugInKit 등록 및 ExtensionKit 실행 확인의 진실 원천으로 쓰지 않는다. 기존 `build_run_guide.md`와 `release_distribution_guide.md`는 Release package 산출물을 기준으로 `$HOME/Applications/Alhangeul.app`에 설치해 검증하도록 안내하고 있다.

#33과 #40 기록에서는 non-ASCII `.app` 경로, 이전 이름의 설치본, `qlmanage -m plugins` 판정 오해, Quick Look cache 영향이 반복 실패 원인이었다. 현재 제품명은 `Alhangeul.app`, Preview appex는 `AlhangeulPreview.appex`, Thumbnail appex는 `AlhangeulThumbnail.appex`이며 bundle identifier는 각각 `com.postmelee.alhangeul.QLExtension`, `com.postmelee.alhangeul.ThumbnailExtension`이다. #145 Stage 4에서는 Release package staging app 안에 두 appex와 `rhwp-studio` asset이 포함되고 local signing/sealed resources 검증이 통과했다.

이번 task는 제품 렌더러 변경보다 설치본 smoke gate의 실행 절차, 판정 기준, 실패 로그 수집 경로를 정리하는 작업이다. Quick Look preview의 GUI 확인은 완전 headless gate로 판정하기 어렵기 때문에 `qlmanage -t` thumbnail smoke와 분리해 명시해야 한다.

## 범위(포함·제외)

포함:

- `package-release` 산출물 기준 설치, LaunchServices 등록, PlugInKit 등록 확인 절차 정리
- `Alhangeul.app`, `AlhangeulPreview.appex`, `AlhangeulThumbnail.appex`의 ASCII path와 bundle identifier, principal class 정합 확인
- `qlmanage -t -x` thumbnail smoke를 자동 판정 가능한 gate로 정리
- `qlmanage -p` 또는 Finder Space preview의 자동화 한계와 수동 확인 기준 분리
- HWP/HWPX 대표 샘플, 손상/대용량 fallback 입력, output directory, 실패 시 로그 수집 명령 정리
- 기존 `build_run_guide.md`, `release_distribution_guide.md`, troubleshooting 문서의 오래된 이름이나 중복 절차 보정
- 필요한 경우 smoke gate helper script 또는 문서화된 command block 추가

제외:

- Quick Look lazy preview 구조 변경
- native renderer visual parity 개선
- 손상/대용량 문서 fallback 구현 변경 (#149 범위)
- WKWebView viewer asset fallback 구현 변경 (#150 범위)
- Developer ID signing, notarization, public DMG 생성, Homebrew Cask 배포
- 이전 설치본 무조건 삭제. 충돌이 확인되거나 의심될 때만 작업지시자 승인 후 처리한다.

## 설계 방향

검증 절차는 세 계층으로 분리한다.

| 계층 | 입력 산출물 | 판정 기준 |
|------|-------------|-----------|
| bundle 정합성 | `build.noindex/release/Alhangeul.app` | app/appex 존재, `Info.plist`, bundle id, principal class, codesign verify |
| 시스템 등록 | `$HOME/Applications/Alhangeul.app` | `lsregister`, `pluginkit -a`, `pluginkit -mAvvv` 결과 |
| 실행 smoke | 설치본에 등록된 extension | `qlmanage -t -x` output 생성, 필요 시 `qlmanage -p`/Finder 수동 preview |

자동화 가능한 gate는 thumbnail 중심으로 둔다. `qlmanage -t -x -s 512 -o <output>`는 headless로 산출물을 만들 수 있으므로 HWP/HWPX 대표 샘플과 fallback 입력을 대상으로 pass/fail 판정 후보가 된다. 반면 `qlmanage -p`와 Finder Space preview는 GUI/사용자 세션/캐시 영향을 받기 때문에 자동 판정 대신 수동 확인 절차와 실패 시 확인 로그로 분리한다.

설치 경로는 `$HOME/Applications/Alhangeul.app` 하나로 고정한다. `.app` directory 이름과 `.appex` 이름은 ASCII를 유지하고, 사용자 표시명은 localized `InfoPlist.strings`와 `LSHasLocalizedDisplayName`으로 확인한다. 이전 이름(`RhwpMac.app`, `AlhangeulMac.app`, `알한글.app`)은 discovery 충돌이 확인될 때만 보고 후 제거한다.

실패 진단은 등록 실패와 렌더 실패를 나누어 기록한다. 등록 실패는 `pluginkit`, `lsregister`, `codesign`, `plutil`을 우선 보고, 실행 실패는 `qlmanage` stderr/output directory, Quick Look cache reset, `log show` 또는 `log stream` 후보를 기록한다. smoke 결과는 단계 보고서와 최종 보고서에 샘플별로 남긴다.

## 잠정 단계(3~6단계)

1. Stage 1: 현행 Finder 통합 smoke 절차 inventory
   - `build_run_guide.md`, `release_distribution_guide.md`, troubleshooting, #33/#40/#145 결과를 대조한다.
   - 현재 app/appex 이름, bundle identifier, principal class, supported content type, package 산출물 위치를 확인한다.
2. Stage 2: smoke gate 판정 기준 설계
   - bundle 정합성, PlugInKit 등록, thumbnail headless smoke, preview 수동 확인, 실패 로그 수집을 표준 흐름으로 나눈다.
   - HWP/HWPX 대표 샘플과 손상/대용량 fallback 입력의 최소 세트를 확정한다.
3. Stage 3: 문서와 필요 시 helper script 보강
   - 기존 문서의 오래된 이름과 중복 명령을 현재 `Alhangeul` 기준으로 보정한다.
   - gate를 문서 명령 블록으로 충분히 표현할지, `scripts/smoke-finder-integration.sh`로 묶을지 구현계획서에서 확정한 뒤 적용한다.
4. Stage 4: 설치본 smoke gate 리허설
   - `scripts/package-release.sh 0.1.0`으로 Release package를 생성한다.
   - `$HOME/Applications/Alhangeul.app` 설치, registration, `qlmanage -t` smoke, preview 수동 확인 가능성을 수행하고 결과를 기록한다.
5. Stage 5: 최종 보고와 release gate 연결
   - 자동 gate 통과 조건, 수동 확인 항목, 실패 시 로그 수집 경로, 남은 limitation을 최종 보고서에 정리한다.
   - #146 known limitations 문서화와 release note에 넘길 항목을 분리한다.

## 리스크

- `qlmanage -p`는 GUI 세션과 사용자 조작이 필요해 headless CI gate로 쓰기 어렵다. 자동 판정은 `qlmanage -t -x` 중심으로 제한해야 한다.
- Quick Look cache와 LaunchServices database 상태에 따라 같은 설치본도 재시도 결과가 달라질 수 있다. cache reset과 설치 경로 고정을 절차에 포함해야 한다.
- 이전 이름의 설치본이 남아 있으면 PlugInKit discovery가 충돌할 수 있지만, 무조건 삭제는 사용자 환경을 건드리는 작업이다. 확인 후 승인받아 처리한다.
- `scripts/package-release.sh`는 public Developer ID signing/notarization을 수행하지 않는다. 이번 smoke는 local signed/sealed Release package 기준이며 public 배포 통과를 의미하지 않는다.
- `qlmanage -t`가 thumbnail을 만들더라도 preview PDF/PNG 표시의 시각 품질을 완전히 보장하지 않는다. preview 수동 확인과 #146 known limitations에 남길 항목을 분리해야 한다.
- 문서에 남은 예전 `AlhangeulMac`/`alhangeulmac` 이름은 과거 기록과 현재 운영 지침이 섞여 있을 수 있다. 과거 보고서 원문은 보존하고 현재 운영 문서만 보정한다.
