## 작업명

HostApp Finder drag/drop 원본 URL 확보로 Finder에서 보기 활성화

## 목적

Finder에서 HostApp WKWebView viewer 영역으로 HWP/HWPX 파일을 끌어넣었을 때 native AppKit 경로에서 원본 file URL을 확보하고, 원본 URL이 검증된 문서에 대해 `Finder에서 보기` toolbar 동작을 활성화한다.

완료 후 Finder drag/drop으로 연 문서는 `공유하기`, `PDF로 내보내기`, `Finder에서 보기`가 모두 의도한 조건에서 활성화되어야 한다. 반면 원본 URL을 확보할 수 없는 JS/File 기반 source-less drop은 Task #144에서 정한 정책대로 `공유하기`와 `PDF로 내보내기`만 활성화하고 `Finder에서 보기`는 비활성으로 유지한다.

## 배경

Task #144 / PR #152에서 WKWebView 내부 drag/drop 문서를 Swift `DocumentViewerStore`에 반영해 source-less 문서 상태를 만들었다. 이로써 원본 경로가 없는 drop에서도 문서 있음 상태가 native toolbar에 반영되어 `공유하기`와 `PDF로 내보내기`가 활성화되었다.

다만 JavaScript `File` 객체 기반 payload만으로는 Finder 원본 경로를 신뢰할 수 없어 `sourceDocument`를 비워 두었고, 그 결과 `Finder에서 보기`는 비활성으로 유지했다. Finder에서 실제 파일을 끌어온 경우에는 AppKit drag pasteboard 또는 `NSDraggingDestination` 계층에서 file URL을 확보할 가능성이 있으므로, 해당 native URL을 안전하게 store에 연결하는 후속 작업이 필요하다.

제외:

## 설계 방향

우선 `RhwpStudioNativeCommandWebView` 또는 그 주변 AppKit view 계층에서 Finder drag pasteboard를 읽을 수 있는지 조사한다. 현재 HostApp은 `RhwpStudioWebView` 내부에서 `WKWebView` subclass를 이미 사용하고 있으므로, 가능하면 이 subclass에 좁은 native drop handler와 callback을 추가해 변경 범위를 HostApp viewer로 제한한다.

native 경로에서 지원 문서 file URL을 확보하면 Swift 쪽 callback으로 URL을 전달하고, `DocumentViewerStore`는 기존 `loadDocument(from:)` 또는 동등한 URL 기반 로드 경로를 사용해 `RhwpStudioDocumentPayload`와 `sourceDocument`를 함께 갱신한다. 이렇게 하면 `canRevealInFinder`가 기존 정책대로 `sourceDocument != nil`을 따르며, 최근 문서 기록과 보안 scope/bookmark 생성도 기존 구조를 재사용할 수 있다.

WKWebView 내부 JavaScript drop bridge는 source-less fallback으로 유지한다. native URL drop과 JS source-less drop이 같은 drag/drop 이벤트에서 중복 실행될 수 있는지 Stage 1에서 확인하고, 필요하면 native 처리 시 JS fallback을 무시하거나 최근 native drop과 같은 파일명/시점의 source-less payload를 억제하는 정책을 Stage 2에서 설계한다.

`Sources/RhwpCoreBridge`, Quick Look, Thumbnail extension은 이번 작업의 소유 영역이 아니므로 변경하지 않는다. 새 파일이 필요해지면 `project.yml`을 원본으로 갱신하고 Xcode project 직접 수정은 하지 않는다.

## 리스크

- WKWebView가 drag event를 먼저 소비하면 `WKWebView` subclass만으로는 `NSDraggingDestination` callback을 안정적으로 받지 못할 수 있다. 이 경우 wrapper `NSView` 또는 window-level handling 검토가 필요하다.
- native drop과 JavaScript drop bridge가 같은 이벤트에서 모두 실행되면 문서 revision이 중복 증가하거나 source-less 상태가 source URL 상태를 덮어쓸 수 있다.
- sandbox 또는 보안 scope 처리 방식에 따라 Finder drop URL을 즉시 읽을 수는 있어도 bookmark 생성/최근 문서 기록이 실패할 수 있다.
- iCloud placeholder, file promise, 여러 파일 동시 drop 등 Finder pasteboard 변형은 v0.1 smoke 범위 밖으로 남을 수 있다.
- AppKit 의존은 HostApp 내부에만 제한해야 하며 `Sources/RhwpCoreBridge`로 전파되면 안 된다.
