# Task M016 #145 수행계획서

## 목적

v0.1 공개 배포 전에 사용자가 내려받을 release artifact의 구성, 파일명, checksum, core provenance를 재현 가능한 기준으로 정리한다.

완료 후에는 `scripts/package-release.sh`의 개발/검증용 zip, `scripts/release.sh`와 GitHub Actions의 public DMG, release note skeleton, checksum 파일, `rhwp-core.lock`/`rhwp-studio` manifest/FFI symbol snapshot 사이의 책임 경계가 문서와 검증 절차에서 일관되어야 한다.

## 배경

현재 저장소에는 두 가지 release artifact 경로가 있다.

- `scripts/package-release.sh <version>`: `build.noindex/release/Alhangeul.app`와 `alhangeul-macos-<version>.zip`을 만든다. 이 경로는 개발/검증용 Release package와 Finder 통합 smoke의 입력으로 사용된다.
- `scripts/release.sh <version>`: public release용 `alhangeul-macos-<version>.dmg`와 `.sha256`을 만든다. public mode는 Developer ID 서명, app/DMG notarization, staple, Gatekeeper assessment를 포함한다. `--skip-notarize`는 `*-rehearsal.dmg`만 만들며 public release asset이 아니다.

`rhwp-core.lock`은 현재 `edwardkim/rhwp` `v0.7.9`와 resolved commit `0fb3e6758b8ad11d2f3c3849c83b914684e83863`, `Frameworks/universal/librhwp.a`, `Frameworks/generated_rhwp.h`의 hash/size를 기록한다. `Sources/HostApp/Resources/rhwp-studio/manifest.json`도 같은 release tag와 resolved commit의 bundled `rhwp-studio` asset provenance를 기록한다. #147에서는 third-party license와 bundled asset 고지 위치를 정리했지만, #145에서는 실제 release artifact와 checksum 공개 항목, release note/provenance 검증 절차를 별도로 확정해야 한다.

또한 `.github/workflows/release-publish.yml`의 기본 `expected_rhwp_tag`는 현재 `v0.7.10`이고, 현재 `rhwp-core.lock`은 `v0.7.9`다. 이번 작업에서 core pin을 갱신하지 않는다면 public publish workflow의 기본 입력과 current artifact provenance의 불일치 처리 방식을 명확히 남겨야 한다.

## 설계 방향

artifact 기준은 사용 목적별로 분리한다. `package-release` zip은 로컬/설치본 smoke를 위한 Release package이고, public 사용자 배포 기준은 `release.sh` public mode와 `release-publish.yml`이 만드는 signed/notarized DMG로 둔다. `release.sh --skip-notarize` 산출물은 layout/checksum rehearsal로만 다루며 public release나 Cask digest의 근거로 사용하지 않는다.

provenance는 새 중복 표를 만들기보다 기존 진실 원천을 연결한다. core와 Rust bridge artifact는 `rhwp-core.lock`, FFI ABI 표면은 `rhwp-ffi-symbols.txt`, bundled `rhwp-studio` asset은 `Sources/HostApp/Resources/rhwp-studio/manifest.json`, third-party 고지는 `THIRD_PARTY_LICENSES.md`와 `FONTS.md`를 기준으로 삼는다.

release note skeleton은 실제 public DMG digest를 인자로 받아 생성되는 `scripts/ci/write-release-notes.sh`를 중심으로 점검한다. 현재 release workflow 기본값과 lock provenance가 충돌하는 경우에는 이번 task 보고서에서 “현재 artifact 기준 유지”와 “publish 전 core update 또는 workflow 입력 override 필요”를 분리해 기록한다.

## 잠정 단계(3~6단계)

1. Stage 1: release artifact inventory와 provenance 기준 확인
   - `package-release`, `release.sh`, release workflows, release note writer, current plist version, `rhwp-core.lock`, `rhwp-studio` manifest, FFI symbol snapshot을 대조한다.
   - current `v0.7.9` lock과 publish workflow 기본 `v0.7.10` 기대값의 영향 범위를 분리한다.
2. Stage 2: artifact 구성과 공개 항목 설계
   - zip, rehearsal DMG, public DMG의 파일명, 위치, 사용 금지/허용 범위, checksum 공개 위치를 표로 확정한다.
   - release note에 포함할 core provenance, checksum, 설치 안내, smoke/known limitation 연결 방식을 정한다.
3. Stage 3: 스크립트/워크플로우/문서 정합성 보강
   - Stage 2 설계에 따라 필요한 최소 파일만 수정한다.
   - public signing/notarization 실행 없이도 검증 가능한 preflight, checksum, provenance 출력 또는 문서 기준을 정리한다.
4. Stage 4: artifact 생성 리허설과 bundle 포함 검증
   - 승인된 범위에서 `scripts/package-release.sh 0.1.0` 또는 `scripts/release.sh --skip-notarize 0.1.0`을 실행해 산출물과 checksum을 확인한다.
   - `Alhangeul.app` 내부 HostApp, Quick Look, Thumbnail, `rhwp-studio` 필수 asset 포함 여부를 기록한다.
5. Stage 5: 최종 보고와 후속 gate 연결
   - checksum/provenance 공개 초안, 검증 명령, 남은 public release 조건을 최종 보고서에 정리한다.
   - #151 설치본 smoke gate와 #146 known limitations 문서화에 넘길 입력을 명시한다.

## 리스크

- 현재 `release-publish.yml` 기본값은 `expected_rhwp_tag: v0.7.10`이고 `require_latest_rhwp: true`지만, 현재 artifact provenance는 `v0.7.9`다. core update 없이 public workflow를 기본값 그대로 실행하면 실패할 수 있다.
- `scripts/package-release.sh`는 zip checksum을 stdout에만 출력하고 별도 `.sha256` 파일을 만들지 않는다. 공개할 checksum 기준이 zip인지 DMG인지 혼동될 수 있다.
- `scripts/release.sh --skip-notarize` 산출물은 public release에 사용하면 안 되지만, 파일명과 checksum이 실제 산출물과 비슷해 운영 문서가 불명확하면 오용될 수 있다.
- 실제 public notarization, Gatekeeper 검증, GitHub Release upload는 credential과 release commit/tag 확정이 필요하므로 이번 task에서 실행하지 않는다.
- artifact 생성은 Rust/Xcode 빌드와 `build.noindex/` 산출물을 만들기 때문에 시간이 오래 걸리거나 로컬 환경 의존 실패가 날 수 있다.
- #150, #149, #151 결과가 아직 반영되지 않았으므로 release note의 fallback/smoke/known limitation 문구는 후속 task 결과를 연결할 여지를 남겨야 한다.
