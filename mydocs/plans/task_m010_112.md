# Issue #112 수행 계획서

## 작업명

## 이슈

- GitHub Issue: #112
- GitHub Milestone: 하이퍼-워터폴 작업환경 조성
- 문서 prefix: `task_m010_112`
- 이슈 URL: https://github.com/postmelee/alhangeul-macos/issues/112

## 목적

PR 본문이 리뷰어에게 "무엇/왜/이슈/테스트/문서/리스크"를 짧고 일관되게 전달하도록 PR 템플릿, PR 처리 가이드, `task-final-report` 절차를 정리한다.

완료 후 PR 작성자는 다음 기준을 따를 수 있어야 한다.

- 현재 PR이 직접 수행하는 이슈는 `대상 타스크`로 분리한다.
- `관련 이슈`는 선행, 후속, Epic, upstream, 참고 PR/issue처럼 PR 이해에 필요한 맥락을 하단에 정리한다.
- Stage별 요약, 주요 파일/영역 요약, 작업 문서 링크는 `변경 내역` 안에 짧게 둔다.
- 검증은 실제 실행한 항목만 남기고, 시각 변경이 있을 때만 스크린샷을 유지한다.
- 관련 문서와 Skill은 기존 맥락을 최대한 보존하되 중복 설명은 덜어낸다.

## 배경

Task #24에서 PR 템플릿과 PR 생성 규격을 처음 표준화했고, Task #61에서 PR 본문 문서 링크의 commit SHA 고정 URL 규칙을 보강했다. 현재 템플릿은 `요약`, `변경 내역`, `검증`, `문서`, `관련 이슈`, `남은 리스크`를 갖고 있어 기본 요소는 충족한다.

다만 리뷰어 관점에서 보면 몇 가지 문제가 남아 있다.

- `왜`를 설명하는 기준이 약해 변경 동기와 판단 근거가 `요약`에 섞이기 쉽다.
- `관련 이슈`가 현재 PR의 직접 task issue를 닫는 섹션처럼 해석될 수 있다.
- `문서` 섹션이 별도 대단원이라 Stage 요약과 문서 링크가 분리된다.
- 체크리스트가 길어질수록 "실제로 실행한 검증만 남긴다"는 원칙이 약해질 수 있다.
- 템플릿과 매뉴얼에 같은 설명을 계속 추가하면 PR 작성자가 실제로 읽어야 할 핵심이 흐려진다.

이번 작업은 edwardkim/rhwp PR #316의 `관련 이슈` 사용처럼 PR의 맥락 이슈를 하단에 정리하는 방식을 채택하되, 이 저장소의 하이퍼-워터폴 추적성을 해치지 않도록 `대상 타스크`를 별도 표기한다.

## 설계 방향

기본 방향은 "새 내용을 덧붙이기보다 기존 내용을 줄이고 재배치한다"이다. 기존 파일의 맥락은 유지하되, 같은 의미의 설명이 템플릿, 매뉴얼, Skill에 반복되면 한곳에는 기준만 남기고 다른 곳은 짧은 참조나 검증 항목으로 축약한다.

PR 템플릿은 다음 대단원을 유지한다.

- `요약`: 대상 타스크, 왜, 무엇, 리뷰 포인트를 최대 4개 bullet로 압축
- `변경 내역`: Stage별 요약, 주요 파일/영역 요약, 작업 문서 링크를 한곳에 배치
- `핵심 리뷰 포인트`: 필요한 경우만 1~3개, 짧은 코드 발췌 허용
- `검증`: 실제 실행한 명령만 유지
- `스크린샷`: 시각 변경이 있을 때만 유지
- `관련 이슈`: 현재 task issue가 아니라 PR 이해에 필요한 선행/후속/Epic/upstream/참고 이슈
- `후속 이슈 제안`: 아직 이슈가 없지만 분리할 후보
- `남은 리스크`: 검증 한계와 운영상 주의사항

짧게 강제하는 기준은 다음처럼 둔다.

- `요약`: 최대 4 bullet
- Stage별 요약: Stage당 1줄
- 주요 파일/영역 표: 최대 5행
- 핵심 리뷰 포인트: 최대 3개, 코드 블록은 각 20줄 이하
- 검증: 실제 실행한 명령만 남김
- 해당 없는 선택 섹션은 삭제하거나 `없음` 한 줄로 정리

## 리스크

- `관련 이슈` 의미를 바꾸면서 기존 "이슈 close" 안내가 사라지면 하이퍼-워터폴 추적성이 약해질 수 있다. 이를 막기 위해 `대상 타스크`를 별도 표기한다.
- 템플릿이 지나치게 자세하면 작성자가 지우지 않고 그대로 남길 수 있다. 예시는 짧게 유지하고 세부 규칙은 PR 처리 가이드로 옮긴다.
- 문서 링크 규칙을 `변경 내역`으로 옮길 때 Task #61의 commit SHA 고정 URL 원칙이 약해질 수 있다. `task-final-report` 검증 항목에는 계속 남긴다.
- 기존 파일을 덜어내는 과정에서 외부 기여 PR 검토 절차와 내부 task PR 작성 절차가 섞일 수 있다. 이번 변경은 내부 task PR 작성 규칙에 한정한다.
