# Task M010 #166 수행계획서

## 목적

M16 `v0.1 출시 전 보강`과 #177 Sparkle/GitHub Pages appcast 준비가 완료된 상태를 기준으로, 알한글 v0.1.0 첫 public release를 실제 사용자가 설치할 수 있는 공식 배포 상태로 만든다.

완료 후에는 `v0.1.0` GitHub Release가 draft와 prerelease가 아닌 공식 release로 공개되어야 하며, signed/notarized DMG, SHA256, Sparkle appcast, Pages 직접 다운로드 URL, Homebrew Cask 배포 판단, release note의 core provenance와 알려진 한계가 서로 일치해야 한다.

## 배경

Issue #166은 v0.1 public release 실행만 추적한다. M16에서 release artifact, signing/notarization 판단, fallback, 설치본 smoke, known limitations, core provenance를 정리했고, #167에서 `rhwp` core와 bundled `rhwp-studio` 기준을 `v0.7.10`으로 맞췄다. #177에서는 Sparkle feed URL, appcast skeleton, Pages 업데이트 안내와 release workflow appcast 생성 경로를 준비했다.

따라서 이번 작업은 core bump, Rust bridge artifact 재생성, bundled `rhwp-studio` asset 재동기화, Sparkle 구조 재구현을 반복하지 않는다. #166은 선행 작업 결과가 현재 release candidate에 반영되어 있는지 확인하고, release candidate 검증, `main` 반영, `v0.1.0` tag, GitHub Actions release workflow, public URL/appcast 검증, Homebrew Cask 판단을 순서대로 실행한다.

## 설계 방향

릴리스 실행은 되돌리기 어려운 외부 상태 변경을 포함하므로 각 단계의 go/no-go 지점을 분리한다. 수행계획 승인만으로 public release, `main` 반영, tag 생성, workflow 실행, Homebrew Cask 반영을 자동 진행하지 않고, 구현계획과 단계별 보고에서 다시 승인을 받는다.

release candidate 기준은 `devel-webview`의 검증된 commit이다. v0.1.x public release는 `devel-webview`를 준비 기준 브랜치로 사용하고, release commit 확정 후 `main`에 반영한 뒤 `main` 기준으로 `v0.1.0` tag와 GitHub Release를 만든다.

공식 release workflow 입력은 기본적으로 `version=0.1.0`, `expected_rhwp_tag=v0.7.10`, `require_latest_rhwp=true`, `draft=false`, `prerelease=false`를 목표로 한다. 작업지시자가 검수용 draft lane을 명시하면 `draft=true` 실행을 별도 검수 단계로 둘 수 있지만, stable Sparkle appcast는 draft/prerelease가 아닌 공식 release에서만 갱신된다.

Homebrew Cask는 public DMG와 `.sha256`이 실제로 게시된 뒤에만 최종 digest를 고정한다. 배포 여부가 미결이면 최종 보고서에 URL/SHA256/version과 미배포 판단을 남기고, Cask 파일 변경은 별도 승인 범위로 분리한다.

## 잠정 단계(3~6단계)

1. Stage 1: release readiness preflight
   - M16/#177 선행 이슈 완료, merge 상태, release-critical 문서와 workflow 입력값을 확인한다.
   - `rhwp-core.lock`, `rhwp-studio/manifest.json`, plist version, Sparkle 설정, Pages direct URL, GitHub secret/variable 준비 여부를 점검한다.
2. Stage 2: release candidate 로컬 검증과 rehearsal
   - Rust lock, no-AppKit, Xcodegen, Debug/Release build, render smoke를 실행한다.
   - 승인된 범위에서 package 또는 `--skip-notarize` rehearsal DMG를 만들고 Finder/Quick Look/Thumbnail smoke gate와 release note 입력을 확인한다.
3. Stage 3: release ref 확정
   - release 기준 commit을 확정하고, 작업지시자 승인 후 `devel-webview` 결과를 `main`에 반영한다.
   - `v0.1.0` tag를 `main`의 release commit에 생성하고 push한다.
4. Stage 4: official GitHub Release publish
   - 작업지시자 승인 후 `Release Publish DMG` workflow를 official release 입력으로 실행한다.
   - signed/notarized DMG, SHA256, release note, draft/prerelease/latest 상태, direct download URL을 검증한다.
5. Stage 5: appcast, 설치본, Homebrew Cask, 최종 보고
   - Sparkle appcast item과 Pages 업데이트 안내를 확인하고, 설치 앱의 업데이트 확인 smoke를 수행한다.
   - Homebrew Cask 배포 여부를 결정해 필요한 경우 URL/SHA256/version을 반영한다.
   - 최종 보고서와 오늘할일 완료 처리 후 PR 준비 상태로 정리한다.

## 리스크

- release 작업은 `main`, tag, GitHub Release, Pages appcast 같은 외부 상태를 바꾸므로 각 단계 승인과 실행 로그가 필요하다.
- `require_latest_rhwp=true`는 upstream `edwardkim/rhwp` 최신 release가 `v0.7.10`이 아닐 경우 publish workflow를 중단시킨다. 예외 실행은 별도 승인 없이는 하지 않는다.
- Developer ID certificate, notarization credential, Sparkle EdDSA private key, GitHub Actions secret/variable 누락은 release workflow 실패로 이어질 수 있다.
- `draft=true` 검수 lane을 사용하면 stable appcast가 갱신되지 않는다. official release 상태 검증은 `draft=false`, `prerelease=false` 이후에만 완료로 본다.
- public DMG SHA256을 확인하기 전에는 Homebrew Cask digest를 확정할 수 없다.
- `releases/latest`는 draft/prerelease가 아닌 latest release 판정에 의존하므로 기존 release 상태가 있으면 검증이 필요하다.
- 설치본 Quick Look/Thumbnail smoke는 사용자 세션, LaunchServices/PlugInKit cache, 이전 설치본 영향을 받을 수 있다.
