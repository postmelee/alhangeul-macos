# Task M019 #227 수행계획서

## 목적

Rust bridge staticlib artifact 검증 정책을 재정의해 PR CI와 release workflow가 같은 provenance 기준을 공유하도록 정리한다.

완료 후에는 `rhwp-core.lock`의 artifact hash/size가 무엇을 보장하는지, GitHub-hosted macOS runner에서 `librhwp.a` byte hash 검증을 언제 건너뛸 수 있는지, 대신 어떤 source lock/header/ABI 검증을 반드시 유지하는지 명확해야 한다. 관련 이슈 #220의 release workflow staticlib hash 예외 정책도 본 작업 범위에서 함께 정리한다.

## 배경

#224 PR CI에서 `RustBridge/examples/*` benchmark helper 변경만으로 `run_rust_verify=true`가 켜졌고, `./scripts/build-rust-macos.sh --verify-lock`가 `Frameworks/universal/librhwp.a` byte hash mismatch로 실패했다. #221에서는 examples 분류만 단기 보정했다.

이후 #226(#188) PR CI에서도 같은 계열의 mismatch가 재현되었다. release/rehearsal workflow에는 이미 `ALHANGEUL_SKIP_RHWP_STATICLIB_HASH_VERIFY=1`이 적용되어 있었지만, PR CI에는 누락되어 `librhwp.a` byte hash만 실패했다. 단기 보정으로 PR CI에도 같은 환경 변수를 적용했으며, `rhwp` repo/tag/commit, `Cargo.lock`, generated header, FFI symbol 검증은 유지했다.

현재 `rhwp-core.lock`은 `rhwp` release tag/commit뿐 아니라 `Frameworks/universal/librhwp.a`와 `Frameworks/generated_rhwp.h`의 hash/size를 기록한다. generated header는 Swift/Rust ABI 표면 검증에 직접 유용하지만, static archive byte hash는 Rust compiler, macOS runner image, Xcode/linker/ar 도구, build path, dependency build 환경 차이의 영향을 받을 수 있다. 이 차이를 false failure로 볼지, toolchain pinning으로 byte reproducibility를 보장할지 결정해야 한다.

## 범위(포함·제외)

- core provenance, `Cargo.lock`, generated header, `rhwp-ffi-symbols.txt` 검증 기준 문서화
- #220 완료 기준을 #227 정책 결정과 구현 결과에 연결

## 설계 방향

기본 방향은 commit/header/ABI 중심 검증으로 전환하고 staticlib byte hash는 provenance record로 유지하는 hybrid 정책을 검토한다. 즉 PR CI와 GitHub-hosted release workflow에서는 `librhwp.a` byte-for-byte 동일성을 필수 gate로 보지 않고, `rhwp` repo/ref/tag/commit, `RustBridge/Cargo.lock`, generated header hash/size, FFI symbol set, 앱 build/smoke를 핵심 gate로 둔다.

`rhwp-core.lock`의 `Frameworks/universal/librhwp.a` hash/size는 "기준 환경에서 생성된 reference artifact 식별자"로 남긴다. 다만 GitHub-hosted macOS runner에서 이 값이 다르다는 이유만으로 source provenance나 Swift ABI가 깨졌다고 판단하지 않는다. strict staticlib byte hash 검증을 다시 release gate로 올리려면 Rust toolchain, Xcode, macOS runner image, archive tool, build path 영향까지 고정하거나 lock 생성 환경을 CI 기준으로 재정의해야 한다.

`ALHANGEUL_SKIP_RHWP_STATICLIB_HASH_VERIFY=1`은 hidden workaround가 아니라 명시 정책으로 둔다. 이 값이 켜져도 source lock, Cargo lock, generated header, FFI symbol 검증은 계속 실패할 수 있어야 하며, script warning과 manual 문구가 그 보장 범위를 정확히 설명해야 한다.

PR CI 분류는 제품 artifact 검증이 필요한 Rust bridge/core 변경과 benchmark/helper 변경을 구분한다. `RustBridge/examples/*` 같은 helper 변경은 macOS build는 요구할 수 있지만 lock-level Rust verify를 불필요하게 켜지 않는 방향을 유지한다.

## 리스크

- staticlib byte hash를 필수 gate에서 낮추면 artifact 내부 byte-level 동일성 보장은 약해진다.
- 반대로 strict hash 검증을 유지하면 GitHub-hosted macOS runner 또는 Xcode 업데이트 때 false failure가 반복될 수 있다.
- `--verify-lock` 이름이 계속 "모든 artifact byte 동일성"처럼 읽히면 작업자가 skip 상태의 보장 범위를 오해할 수 있다.
- release workflow에서 skip 환경 변수를 문서화하더라도, source lock/header/ABI 검증이 우연히 약해지면 release provenance가 실제로 낮아질 수 있다.
- #220과 #227을 한 PR에서 함께 닫을 경우 PR 설명과 최종 보고서가 두 이슈의 완료 기준을 각각 명확히 대조해야 한다.
