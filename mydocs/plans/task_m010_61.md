# Issue #61 수행 계획서

## 작업명

PR 문서 링크 전수 보정과 작성 규격 강화

## 이슈

- GitHub Issue: #61
- Milestone: v0.1.0
- 문서 prefix: `task_m010_61`
- 이슈 URL: https://github.com/postmelee/alhangeul-macos/issues/61

## 목적

merged PR 본문에 남아 있는 문서 링크 문제를 전수 조사해 실제 클릭 가능한 형태로 보정하고, 이후 PR 작성 시 같은 문제가 반복되지 않도록 링크 작성 형식과 검증 절차를 표준화한다.

완료 후 PR 본문의 문서 링크는 다음 기준을 만족해야 한다.

- `github.com/postmelee/alhangeul-macos/blob/{40자 SHA}/mydocs/...` 형식의 commit SHA 고정 URL을 사용한다.
- 링크 대상 SHA는 실제 commit이어야 한다.
- 문서 섹션의 표시 텍스트는 raw URL이 아니라 `[파일명](URL)` 형식을 우선 사용한다.
- `task-final-report` 절차와 PR 템플릿이 위 기준을 명시한다.

## 배경

PR #59의 문서 섹션은 존재하지 않는 commit SHA를 사용해 클릭 시 파일이 조회되지 않는다. 실제 마지막 head commit은 `6f57ccc178438bb45ba7df85f6e278af4b428af0`인데, PR 본문에는 `6f57cccda6110abe999a54eec159aa91efa3b646`가 들어가 있다.

PR #60의 문서 섹션은 commit SHA 고정 URL을 사용했지만 raw URL 전체가 노출되어 가독성이 낮다. 이는 Task #49와 Task #52에서 남긴 잔여 위험, 즉 "정책은 있지만 자동화와 템플릿 강제가 부족하다"는 문제가 실제로 반복된 사례다.

## 설계 방향

전수조사는 "본문 링크 형식"과 "실제 접근 가능성"을 분리해 판단한다. 로컬 `git cat-file`은 빠른 1차 검증으로 사용하되, 원격 PR 본문 수정 대상 확정에는 GitHub에서 조회 가능한 URL과 commit 존재 여부를 함께 확인한다.

보정 기준은 다음과 같이 둔다.

- 문서 링크는 작업 산출물을 가장 정확히 가리키는 PR head commit SHA를 우선 사용한다.
- PR head commit이 문서 파일을 포함하지 않는 예외가 있으면 merge commit SHA를 후보로 삼고, 선택 근거를 단계 보고서에 기록한다.
- 문서 섹션의 표시 텍스트는 파일 단위 링크면 `[task_m010_61.md](URL)`, 범위 링크면 `[stage1](URL) ~ [stage4](URL)`처럼 짧게 작성한다.
- 검증 명령이나 변경 내역 설명에 포함된 파일 경로는 문서 섹션 링크가 아니면 무리하게 URL로 바꾸지 않는다.

재발 방지는 문서 한 줄 안내가 아니라 PR 생성 절차의 체크리스트로 둔다. `task-final-report`에는 PR 생성 전 head SHA 확인, 문서 링크 작성 형식, PR 본문 조회 후 raw URL/깨진 SHA 확인을 포함한다. PR 템플릿에는 문서 섹션의 예시를 직접 제공한다.

## 잠정 단계

1. Stage 1: merged PR 본문 문서 링크 전수 조사와 문제 유형 분류
2. Stage 2: 보정 대상별 기준 SHA와 수정안 확정
3. Stage 3: 승인된 PR 본문 원격 보정과 접근 검증
4. Stage 4: `task-final-report` SKILL, PR 템플릿, 관련 매뉴얼 규격 보강
5. Stage 5: 통합 검증, 최종 보고서 작성, 오늘할일 완료 처리

각 단계 완료 시 단계별 완료보고서를 작성하고 작업지시자 승인 후 다음 단계로 진행한다.

## 검증 계획

계획 문서 단계:

```bash
git diff --check -- mydocs/orders/20260426.md mydocs/plans/task_m010_61.md
```

구현 단계 진입 후 최소 검증:

```bash
git diff --check
gh pr view 59 --repo postmelee/alhangeul-macos --json body
gh pr view 60 --repo postmelee/alhangeul-macos --json body
gh pr list --repo postmelee/alhangeul-macos --state merged --limit 100 --json number,title,body,mergeCommit
rg -n "PR 본문|문서 링크|파일명|blob/\\{sha\\}|git rev-parse HEAD" .github/pull_request_template.md mydocs/skills/task-final-report/SKILL.md mydocs/manual
```

PR 본문 보정 후에는 다음을 확인한다.

- PR #59 문서 링크 SHA가 실제 commit으로 교체되었는가
- PR #60 문서 섹션이 raw URL 노출 대신 `[파일명](URL)` 형식인가
- 전수 조사 대상에서 존재하지 않는 40자 SHA가 남지 않았는가
- 문서 섹션의 `mydocs/` 링크가 commit SHA 고정 URL인가
- 검증 명령의 파일 경로처럼 의도적으로 제외한 항목이 보고서에 기록되었는가

문서/운영 규격 변경이므로 Xcode/Rust 빌드는 수행하지 않는다.

## 리스크

- GitHub PR 본문 수정은 원격 상태 변경이라 로컬 diff에 남지 않는다. 수정 전후 본문과 기준 SHA를 단계 보고서에 기록해야 한다.
- PR 본문 Markdown을 기계적으로 치환하면 검증 명령, 일반 설명, 외부 링크까지 잘못 바꿀 수 있다. 문서 섹션과 `mydocs/` 링크만 좁게 보정한다.
- PR head SHA와 merge commit SHA 중 어느 쪽이 더 적절한지 애매한 PR이 있을 수 있다. 문서 파일이 실제로 존재하는 기준 SHA를 확인하고 선택 근거를 남긴다.
- PR 템플릿 예시가 지나치게 장황하면 작성자가 지울 가능성이 크다. 짧은 예시와 검증 체크만 남긴다.
- 로컬 git object 검증과 GitHub 원격 접근 검증 결과가 다를 수 있다. 최종 판단에는 원격 조회 결과를 함께 사용한다.
