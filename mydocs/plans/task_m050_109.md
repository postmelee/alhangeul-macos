# Task M050 #109 수행계획서

## 목적

Swift native renderer의 도형과 텍스트 스타일 누락을 1차로 줄인다.

완료 후 `rhwp_render_page_tree` 기반 Swift renderer가 도형 shadow, line arrow, dash, pattern fill, path arcTo, text shadow, rotation, vertical text, superscript/subscript, emphasis dot, tab leader 중 CoreGraphics/CoreText로 안전하게 처리 가능한 항목을 더 명확히 지원하거나 문서화된 fallback으로 남겨야 한다.

## 배경

Issue #109는 v0.5 milestone의 native viewer renderer 품질 보강 작업이다. 작업지시자가 현재 `devel-webview`는 배포 준비 브랜치, `devel`은 Swift native 전환 작업 브랜치라고 명확히 했으므로, 이번 작업 기준 통합 브랜치는 `devel`로 둔다.

현재 `devel-webview`의 HostApp viewer는 `rhwp-studio` WKWebView 경로를 사용하므로 #109 구현이 배포용 앱 viewer 화면에 직접 반영되지는 않는다. 다만 Quick Look preview와 Finder thumbnail은 `Sources/Shared/HwpPageImageRenderer.swift`와 `Sources/RhwpCoreBridge/CGTreeRenderer.swift` native bitmap 경로를 공유한다. 따라서 #109를 `devel-webview`에 별도 백포트하거나 양 브랜치 공통 renderer 변경으로 옮기면 Quick Look/Thumbnail 시각 결과에는 영향을 줄 수 있다.

#119에서 font registration과 fallback 정책이 정리되었고, #123에서 body/table clip 정책이 보강되었다. 이번 작업은 그 위에서 font 누락이나 clip 정책 문제가 아니라 도형/텍스트 style 해석 자체의 1차 parity를 좁히는 데 집중한다.

## 범위(포함·제외)

포함:

- rhwp-studio의 `WebCanvasRenderer`/view/engine 계층에서 style 처리 기준 조사
- Swift `RenderTree.swift`가 이미 디코딩하는 style 필드와 `CGTreeRenderer.swift`가 실제 사용하는 필드 차이 정리
- 도형 shadow, line arrow, dash, pattern fill, path arcTo의 구현 가능 범위와 fallback 정책 확정
- text shadow, rotation, vertical text, superscript/subscript, emphasis dot, tab leader 중 안전한 1차 구현 항목 선정
- 기존 render output을 크게 흔드는 항목은 샘플 기반으로 단계적 적용
- HostApp native viewer, Quick Look, Thumbnail 공통 renderer 경로의 대표 샘플 검증
- `devel-webview` 배포 브랜치에 직접 반영하지 않는 변경과 백포트 시 영향을 문서화

## 설계 방향

reference implementation은 issue 본문 기준대로 `rhwp-studio`의 현재 렌더 결과와 관련 `WebCanvasRenderer`/view/engine 계층으로 삼는다. 다만 DOM/Canvas/TypeScript 구조를 Swift에 그대로 복제하지 않고, Swift/CoreGraphics/CoreText/CoreImage에 맞는 native 구현으로 재해석한다.

도형은 `ShapeStyle`, `LineStyle`, `PathCommand`, `ShapeTransform`의 Swift 모델이 이미 가진 정보부터 사용한다. 이미 처리 중인 dash나 단순 pattern fill은 회귀 위험을 먼저 확인하고, 부족한 dash 종류나 pattern 방향/색 처리만 보강한다. `arcTo`는 현재 직선 fallback으로 그려질 수 있으므로 CoreGraphics arc 변환을 별도 helper로 두고, 수학적으로 신뢰하기 어려운 경우 기존 fallback을 유지한다. shadow와 arrow는 draw order, clipping, transform 영향을 많이 받으므로 shape별 공통 helper를 만든 뒤 샘플 기준으로 적용 대상을 좁힌다.

텍스트는 CoreText 좌표계와 기존 cluster drawing/advance 보정이 민감하다. rotation과 vertical text는 run 단위 변환과 bbox 기준이 기존 baseline 계산을 흔들 수 있으므로 먼저 현행 `TextRunNode.rotation`, `isVertical`, `baseline` 필드의 의미를 샘플로 확인한다. superscript/subscript와 shadow처럼 run 내부에서 비교적 좁게 처리 가능한 항목을 우선 검토하고, emphasis dot과 tab leader는 render tree가 제공하는 `emphasisDot`, `tabLeaders` 정보를 기존 tab stop/decoration helper와 연결할 수 있는 범위에서 진행한다.

작업 산출물은 `devel`의 native renderer 품질 향상으로 제한한다. `devel-webview` 배포 흐름에는 직접 merge하지 않고, 필요한 경우 별도 작업지시자 승인으로 Quick Look/Thumbnail 백포트 범위를 분리한다.

## 잠정 단계(3~6단계)

1. Stage 1: rhwp-studio 기준과 Swift 현행 style 처리 조사
   - issue 범위의 각 style 항목을 Web renderer 기준, render tree 필드, Swift 구현 여부로 분류한다.
   - `devel-webview` 배포 브랜치에 직접 영향이 있는 경로와 없는 경로를 문서로 분리한다.
2. Stage 2: 도형 style 1차 보강
   - shadow, arrow, dash, pattern fill, path arcTo 중 샘플과 필드가 충분한 항목부터 구현한다.
   - transform, clip, fill/stroke draw order 회귀를 확인한다.
3. Stage 3: 텍스트 style 1차 보강
   - text shadow, rotation, vertical text, superscript/subscript, emphasis dot, tab leader 중 안전한 항목을 구현한다.
   - baseline, underline/strike, cluster advance 보정이 흔들리지 않는지 확인한다.
4. Stage 4: 대표 샘플 비교와 회귀 보정
   - core SVG/rhwp-studio 기준 출력과 Swift native PNG를 비교하고, 눈에 띄는 누락 감소와 회귀를 기록한다.
   - 과도한 변화가 있는 항목은 fallback 또는 후속 이슈로 분리한다.
5. Stage 5: 문서화와 최종 검증 정리
   - 지원 항목, fallback 항목, `devel-webview` 백포트 영향 여부를 최종 보고서에 정리한다.
   - 구현계획서, 단계 보고서, 최종 보고서와 오늘할일 상태를 정리한다.

## 리스크

- render tree가 style 정보를 이미 잃은 항목은 Swift renderer만으로 parity를 맞출 수 없다.
- shadow, arrow, arcTo는 transform과 clip 상태에 민감해 기존 도형 위치나 z-order를 흔들 수 있다.
- text rotation/vertical text는 CoreText 좌표계, baseline, bbox 계산을 크게 바꿀 수 있다.
- superscript/subscript, emphasis dot, tab leader를 잘못 적용하면 기존 underline/strike, tab stop, cluster advance 보정이 회귀할 수 있다.
- rhwp-studio와 core SVG 기준이 서로 다르게 보이는 샘플에서는 기준 선택을 단계 보고서에 명시해야 한다.
- `devel-webview`에 백포트하면 HostApp WKWebView viewer에는 영향이 없더라도 Quick Look/Thumbnail 출력이 바뀔 수 있다.
