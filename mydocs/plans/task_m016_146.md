# Task M016 #146 수행계획서

## 목적

v0.1 공개 배포 전에 HostApp viewer, PDF/인쇄, Quick Look preview, Finder thumbnail의 렌더링 경로와 한계를 사용자와 기여자가 오해하지 않도록 문서화한다.

완료 후에는 README와 release note 초안에서 "앱에서 열리는 viewer", "Finder/Quick Look 미리보기", "PDF 내보내기/인쇄"가 어느 renderer를 쓰는지 일관되게 설명되어야 한다. 또한 v0.1은 완전한 HWP/HWPX 호환 viewer가 아니라 `rhwp-studio` WKWebView 경로와 Swift native render tree 경로를 함께 쓰는 첫 공개 배포라는 점, native renderer parity 개선은 v0.5 이후 범위라는 점을 명확히 남긴다.

## 배경

현재 HostApp 화면 viewer는 bundled `rhwp-studio`를 WKWebView로 로드한다. 정적 asset은 `alhangeul-studio://app` custom scheme으로 제공되고, 현재 문서 bytes는 `alhangeul-document://current` scheme으로 전달된다. #150은 이 WKWebView asset/document delivery 실패에서 빈 화면 대신 fatal fallback과 diagnostics를 보이도록 정리했다.

Quick Look preview와 Finder thumbnail은 `Sources/Shared/HwpPageImageRenderer`와 `HwpPreviewPDFRenderer`를 통해 Rust bridge render tree JSON을 Swift `CGTreeRenderer`로 그리는 native bitmap/PDF 경로를 사용한다. #151은 설치본 기준 smoke gate와 자동/수동 확인 한계를 정리했고, #149는 손상·대용량·미지원 입력 fallback을 HostApp과 extension 경로에 연결했다.

문서에는 아직 경로별 책임이 충분히 선명하지 않다. README의 "Rendering Paths"는 큰 방향은 맞지만, 코드와 아키텍처 문서 기준으로 PDF export, print, Quick Look/Thumbnail, 장기 native viewer 전환의 경계를 다시 확인해야 한다. release note skeleton도 현재 provenance와 smoke 결과는 담지만 known limitations를 별도 섹션으로 드러내지 않는다.

이번 작업은 renderer를 고치지 않고, 이미 완료된 M016 작업(#145, #147, #148, #149, #150, #151, #167)과 기존 native renderer 조사 결과(#90, #106, #118, #120 등)를 사용자-facing 문구와 운영 문서로 연결하는 문서 작업이다.

## 범위(포함·제외)

포함:

- HostApp viewer가 WKWebView `rhwp-studio v0.7.10` 경로를 사용한다는 설명 정리
- Quick Look preview와 Finder thumbnail이 Rust bridge + Swift native renderer 경로를 사용한다는 설명 정리
- PDF 내보내기와 인쇄 경로가 현재 코드에서 어떤 입력과 renderer를 쓰는지 inventory 후 문서 정합성 보정
- native renderer style parity, image fill/effect, body overflow, RawSvg/OLE, watermark 계열 등 알려진 제한 사항을 사용자에게 과장 없이 설명
- 손상/대용량/미지원 입력 fallback은 복구가 아니라 crash/hang/raw error 방지라는 점 정리
- Quick Look/Thumbnail smoke gate 통과와 시각 품질 parity 보장이 다른 문제라는 점 정리
- README, release guide, release note skeleton에서 완전 호환으로 읽힐 수 있는 문구 보정
- 후속 milestone v0.2/v0.5/v0.6과의 분리 기준 연결

제외:

- native renderer parity 구현 또는 visual diff corpus 확장
- WKWebView viewer, `rhwp-studio` upstream asset, Rust core parser/renderer 수정
- Quick Look/Thumbnail extension 동작 변경
- release artifact 생성, notarization, GitHub Release 게시, Homebrew Cask 배포
- 기존 renderer 이슈의 milestone 이동 또는 close
- 법률 고지나 third-party license 본문 변경 (#147 범위)

## 설계 방향

문서는 제품 표면별 책임 경계를 먼저 보여주는 방식으로 정리한다.

| 표면 | v0.1 기준 경로 | 문서화 기준 |
|------|----------------|-------------|
| HostApp viewer/editor 화면 | `rhwp-studio` Web/WASM rendering in WKWebView | 첫 배포 기준선, native parity 전 fallback/비교 기준 |
| HostApp PDF 내보내기 | 현재 문서 bytes를 native render tree PDF 경로로 변환 | viewer 화면과 같은 renderer라고 표현하지 않음 |
| HostApp 인쇄 | `rhwp-studio`가 제공하는 page HTML/SVG를 별도 WKWebView/PDFKit/AppKit print operation으로 처리 | PDF export와 print 경로를 혼동하지 않음 |
| Quick Look preview | Rust bridge render tree + Swift native bitmap/PDF | 설치본 smoke와 시각 parity를 분리 |
| Finder thumbnail | Rust bridge render tree + Swift native first-page bitmap/cache | thumbnail 생성 성공과 문서 전체 호환성을 분리 |

known limitations는 "현재 한계", "사용자가 보는 영향", "후속 범위"를 함께 적는다. 예를 들어 native renderer style parity는 v0.5 native viewer 개선, 문서 정보/본문 추출은 v0.2 Mac 통합, native editor 기능은 v0.6/v1.0 범위로 분리한다.

README는 public 사용자 기대치를 조정하는 짧은 설명을 맡고, release note skeleton은 실제 배포 시 known limitations 섹션을 빠뜨리지 않도록 기본 구조를 제공한다. 세부 운영 절차는 `release_distribution_guide.md`에 남기되, 같은 표를 중복 복제해 이후 불일치를 만들지 않는다.

## 잠정 단계(3~6단계)

1. Stage 1: 현재 렌더 경로와 handoff inventory
   - HostApp viewer, PDF export, print, Quick Look, Thumbnail의 현재 코드 경로를 대조한다.
   - #145/#149/#150/#151/#167 최종 보고서에서 release note와 known limitations에 넘긴 항목을 수집한다.
2. Stage 2: known limitations와 milestone 분리 기준 설계
   - 사용자에게 보일 한계 문구를 "경로 차이", "렌더 품질", "fallback", "검증 한계", "후속 milestone"으로 나눈다.
   - README, release guide, release note skeleton 중 어느 문서가 어느 수준의 정보를 소유할지 구현계획서에 확정한다.
3. Stage 3: README와 운영 문서 보정
   - Rendering Paths, Release/Install, Roadmap 문구에서 완전 호환으로 읽히는 표현을 줄인다.
   - `project_architecture.md`와 release guide가 현재 코드 경로와 충돌하지 않도록 필요한 최소 문구만 보정한다.
4. Stage 4: release note skeleton과 검증 정리
   - `scripts/ci/write-release-notes.sh`가 known limitations 섹션 또는 최종 보고서 연결 문구를 포함하도록 보강한다.
   - dummy checksum으로 release note skeleton을 생성해 사용자-facing 문구와 provenance/limitation 위치를 확인한다.
5. Stage 5: 최종 보고와 PR 준비
   - 변경 문서, release note 초안 예시, 남은 renderer parity 이슈, 후속 milestone 연결을 최종 보고서에 정리한다.
   - 문서 전용 검증 결과와 미수행 범위를 기록한다.

## 리스크

- HostApp viewer, PDF export, print가 모두 "앱에서 보는 문서"라는 사용자 표면으로 묶이지만 실제 renderer가 다르다. 문구를 단순화하다가 다시 경로를 섞어 쓸 위험이 있다.
- known limitations를 너무 길게 쓰면 README가 문제 목록처럼 보이고, 너무 짧게 쓰면 사용자가 완전 호환으로 오해할 수 있다. README는 요약, release note는 배포 시점 한계, 최종 보고서는 상세 근거로 나눈다.
- native renderer parity 한계 중 일부는 이미 개선되었거나 다른 milestone에서 다뤄진 항목이 섞여 있다. 기존 보고서 기준으로 현재 남은 한계와 완료된 보강을 구분해야 한다.
- `scripts/ci/write-release-notes.sh`를 바꾸면 release workflow 출력 형식에 영향을 준다. syntax와 dummy output 검증을 반드시 수행한다.
- 문서 작업 중 renderer 이슈의 milestone 이동이나 scope 재분류가 필요해 보일 수 있지만, 이번 task 범위에서는 이동하지 않는다.
