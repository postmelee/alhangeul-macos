## 목적

MVP 출시 우선순위를 앞당기기 위해 HostApp Viewer를 현재 Swift native renderer 중심 경로에서 `rhwp-studio`를 `WKWebView`로 표시하는 경로로 전환한다.

완료 후 사용자는 macOS 앱에서 HWP/HWPX 파일을 열었을 때 `rhwp-studio` 기반 viewer 화면을 통해 문서를 확인할 수 있어야 한다. 이번 작업은 `postmelee/alhangeul-macos` 저장소의 `devel-webview` 브랜치에서 진행하며, MVP 배포 판단에 필요한 build와 샘플 열기 smoke 검증 결과를 남긴다.

## 배경

기존 계획은 첫 출시 전 Swift native renderer의 렌더 품질 보강과 PageLayerTree 전환을 단계적으로 진행하는 흐름이었다. 그러나 MVP 출시를 먼저 진행하기 위해 Viewer의 렌더링 구현 리스크를 낮추고, upstream `rhwp` 쪽의 `rhwp-studio` web viewer 흐름을 macOS HostApp 안에 포함하는 방식으로 우선순위를 조정한다.

작업지시자는 `edwardkim/rhwp` 저장소가 아니라 `postmelee/alhangeul-macos` 저장소에 `devel-webview` 브랜치를 만들고 이 브랜치에서 작업하라고 정정했다. 따라서 이번 타스크는 일반 `local/task134` 브랜치 대신 작업지시자가 지정한 `devel-webview` 브랜치를 사용한다.

## 설계 방향

HostApp은 macOS SwiftUI/AppKit 경계에서 `WKWebView`를 안전하게 감싸는 viewer component를 제공한다. SwiftUI 화면에서는 문서 선택, 열기 상태, 오류 상태를 관리하고, 실제 web content 표시와 navigation policy는 AppKit/WKWebView wrapper가 담당한다.

`rhwp-studio`는 앱 bundle 내부의 정적 web asset으로 포함하는 방식을 우선 검토한다. 개발 중에는 재현 가능한 build command와 산출물 위치를 명확히 하고, 최종 앱에서는 네트워크 의존 없이 bundle resource에서 viewer가 로드되는 구성을 목표로 한다. HWP/HWPX 파일 전달은 query parameter, JavaScript bridge, temporary copied file, custom URL scheme 중 sandbox와 file permission이 가장 명확한 방식을 비교한 뒤 구현 계획서에서 확정한다.

기존 native viewer 코드는 Quick Look/Thumbnail과 공유되는 경계가 있을 수 있으므로 무리하게 삭제하지 않는다. MVP HostApp 경로에서 충돌하는 부분만 비활성화하거나 우회하고, 공유 renderer/bridge code는 후속 native renderer 작업을 위해 보존한다.

`Sources/RhwpCoreBridge`에는 AppKit/WebKit 의존을 추가하지 않는다. WKWebView 관련 코드는 HostApp 소유 영역에 한정하고, `project.yml`을 Xcode project의 원본으로 유지한다.

## 리스크

- `rhwp-studio`가 앱 bundle의 file URL에서 바로 동작하지 않으면 custom URL scheme, local server, temporary directory copy 중 하나를 선택해야 할 수 있다.
- macOS sandbox에서 원본 문서 file access가 WKWebView JavaScript runtime까지 자연스럽게 전달되지 않을 수 있다.
- web asset build가 Node toolchain에 의존하면 MVP build 재현성과 CI/CD 영향이 생길 수 있다.
- 기존 HostApp viewer와 Quick Look/Thumbnail이 공유하는 renderer code를 과하게 제거하면 확장 렌더 경로가 깨질 수 있다.
- `project.yml` resource 설정이 잘못되면 Xcode project 재생성 후 asset bundle 위치가 달라질 수 있다.
- WKWebView 경로가 MVP에는 유리하지만, 후속 native renderer/PageLayerTree 작업과 병행할 때 소유 경계를 문서화하지 않으면 장기 구조가 불명확해질 수 있다.
