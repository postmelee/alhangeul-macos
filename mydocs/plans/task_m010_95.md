# Issue #95 수행 계획서

## 작업명

rhwp v0.7.8 stable tag 승격

## 이슈

- GitHub Issue: #95
- Milestone: v0.1
- 문서 prefix: `task_m010_95`
- 이슈 URL: https://github.com/postmelee/alhangeul-macos/issues/95

## 목적

현재 Demo/Preview commit pin 상태인 `edwardkim/rhwp` dependency를 정식 release tag `v0.7.8` 기반 Stable pin으로 승격한다.

완료 후 앱 저장소는 `RustBridge/Cargo.toml`, `RustBridge/Cargo.lock`, `rhwp-core.lock`, Rust bridge 산출물이 모두 `v0.7.8` release tag와 해당 resolved commit 기준으로 일치해야 한다. 또한 HostApp, Quick Look, Thumbnail의 기존 PageRenderTree 기반 render smoke가 유지됨을 검증 결과로 설명할 수 있어야 한다.

## 배경

Task #76에서는 `edwardkim/rhwp` PR #385 merge commit `e91ecea3174a0da0ad7a1ea495cacc4f8772c31d`를 Demo/Preview commit pin으로 적용했다. 당시 확인한 최신 release `v0.7.7`에는 `RustBridge`가 요구하는 `DocumentCore::build_page_render_tree`, `DocumentCore::get_bin_data`가 없어 Stable 전환은 blocked 상태로 남겼다.

Issue #95에 따르면 `rhwp v0.7.8` release tag에는 PR #385의 PageRenderTree API와 PR #419의 PageLayerTree API가 함께 포함되어 있다. 따라서 이번 작업은 현재 Demo/Preview commit pin을 release tag 안정 기준인 `tag + resolved commit`으로 승격할 수 있는지 검증하고, 가능하면 Stable 기준으로 lock과 artifact를 갱신한다.

이번 작업에서 PageLayerTree 기반 Swift renderer 전환은 하지 않는다. `v0.7.8`에 PageLayerTree API가 포함되어 있어도 앱의 실제 rendering path는 기존 PageRenderTree 기반 HostApp/Quick Look/Thumbnail 경로를 유지한다.

## 범위

포함:

- upstream `edwardkim/rhwp` release tag `v0.7.8` 존재와 resolved commit 확인
- `v0.7.8` 기준 required core API 확인
- `RustBridge/Cargo.toml`의 `rhwp` dependency를 `tag = "v0.7.8"` 기준으로 갱신
- `RustBridge/Cargo.lock`의 `rhwp` source를 release tag dependency resolved commit으로 갱신
- `rhwp-core.lock`의 ref kind, release tag, resolved commit, transition status, artifact hash/size 갱신
- Rust static library와 `Rhwp.xcframework` 산출물 재생성
- C ABI symbol set 유지 여부 확인
- 기존 PageRenderTree 기반 HostApp/Quick Look/Thumbnail render smoke 검증
- #76 이후 문서에 남은 stale latest release 또는 Stable blocked 표현 보정

제외:

- PageLayerTree 기반 Swift renderer 전환
- `rhwp_render_page_layer_tree` 신규 ABI 추가
- upstream `rhwp` API 설계 변경
- renderer 시각 품질 개선
- release 배포, 서명, 공증, Homebrew Cask 작업

## 설계 방향

Stable 기준은 `release tag + resolved commit`으로만 판단한다. `RustBridge/Cargo.toml`은 branch나 floating ref를 사용하지 않고 `tag = "v0.7.8"`로 고정한다. `RustBridge/Cargo.lock`에는 Cargo가 해석한 실제 source commit을 남기고, `rhwp-core.lock`에는 `rhwp_ref_kind = "release-tag"`, `rhwp_release_tag = "v0.7.8"`, resolved commit, artifact hash/size를 함께 기록한다.

앱 쪽 bridge contract는 기존과 동일하게 유지한다. Rust bridge는 `DocumentCore::build_page_render_tree` 결과를 JSON으로 직렬화해 `rhwp_render_page_tree`로 반환하고, Swift는 기존 `RenderTree` decode와 `rhwp_image_data` 이미지 조회 경로를 사용한다. `v0.7.8`에 포함된 PageLayerTree API는 후속 전환 후보로만 기록하고 이번 변경의 ABI 표면에는 추가하지 않는다.

문서 보정은 stale 표현 제거에 한정한다. #76 문서와 core dependency 관련 문서에서 `v0.7.7` latest release 또는 Stable blocked를 현재 상태처럼 말하는 부분을 `v0.7.8` 전환 결과 기준으로 갱신하되, 과거 단계 보고서처럼 당시 시점 기록으로 보존해야 하는 문서는 수정하지 않는다.

## 리스크

- `v0.7.8` tag가 존재하더라도 Cargo가 해석한 resolved commit과 `rhwp-core.lock` 기록이 불일치하면 Stable provenance가 깨질 수 있다.
- release tag dependency 전환 후 transitive dependency 해석이 달라져 Rust bridge build 또는 generated artifact hash/size가 바뀔 수 있다.
- PR #419의 PageLayerTree API가 포함되어도 이번 앱 경로는 PageRenderTree를 유지하므로, 새 API 존재를 이유로 ABI를 확장하면 범위를 초과한다.
- render tree JSON shape나 이미지 조회 동작이 #76 pin과 달라진 경우 HostApp/Quick Look/Thumbnail 렌더가 실패할 수 있다.
- `v0.7.7` latest release 표현 중 과거 보고서의 당시 기록은 보존해야 하므로, stale 문서 보정 범위를 잘못 잡으면 이력 문서의 의미를 훼손할 수 있다.
- Rust bridge 산출물 재생성은 시간이 오래 걸리고 네트워크 dependency fetch가 필요할 수 있다. 실패 시 release tag 검증, Cargo dependency 해석, artifact 검증 실패를 분리해 보고한다.
