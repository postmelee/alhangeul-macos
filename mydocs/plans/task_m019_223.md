# Task M019 #223 수행계획서

## 목적

그림 또는 개체 선택 상태에서 Space 입력 시 WKWebView viewer가 전체 runtime error 화면으로 전환되는 문제를 재현하고, 사용자 조작 중 발생하는 recoverable viewer 예외와 실제 fatal load/runtime failure를 구분해 앱이 정상 보기 상태를 유지하도록 보강한다.

완료 후에는 문제 샘플에서 Space 입력으로 앱 전체 오류 화면이 표시되지 않아야 하며, 필요한 경우 진단 정보는 남기되 사용자가 문서를 계속 볼 수 있어야 한다.

## 배경

작업지시자는 `samples/exam_social.hwp`를 열고 첨부한 사진에 커서가 간 뒤 Space를 누르자 오류 화면이 발생했다고 보고했다. 첨부 스크린샷의 상태바에는 `exam_science.hwp - 4페이지`가 표시되어 있어, 실제 재현 샘플과 페이지를 먼저 확정해야 한다.

오류 화면의 진단 정보는 `message=렌더링 오류: 지정된 컨트롤이 표, 글상자 또는 그림이 아닙니다`, `sourceURL=alhangeul-studio://app/assets/index-BN69C-Lp.js`, `column=30942`이다. 번들 JS에서 해당 위치는 `insertTextInCell` 또는 `insertTextInCellByPath` WASM glue 주변이며, upstream `rhwp` core에서는 지정된 control index가 표, 글상자, 그림이 아닐 때 이 오류를 던진다.

현재 HostApp은 `window.error`를 native bridge로 전달하고, `runtime-error` 메시지를 fatal fallback으로 표시한다. 따라서 같은 JS/WASM 예외가 rhwp web demo에서는 콘솔 오류 또는 recoverable 상태로 남더라도, 앱에서는 전체 오류 화면으로 전환될 수 있다.

## 범위(포함·제외)

포함:

- `samples/exam_social.hwp`와 스크린샷에 보인 `exam_science.hwp` 기준 재현 샘플과 페이지 확정
- 그림/개체 선택 후 Space 입력 시 발생하는 JS/WASM 예외 경로 확인
- HostApp `runtime-error` bridge의 fatal 처리 기준 검토
- 사용자 상호작용 중 recoverable viewer 예외를 전체 오류 화면으로 전환하지 않는 최소 보강
- 필요 시 번들 `rhwp-studio` 자산의 입력 처리 또는 예외 처리 보강 가능성 검토
- 재현 샘플 기반 수동 smoke와 관련 자동 검증 범위 정리
- 단계 보고서, 최종 보고서, 오늘할일 갱신

제외:

- Swift native viewer/editor 구현
- HWP core 편집 기능 확장
- upstream `edwardkim/rhwp` 저장소 자체 수정 또는 upstream PR

- `Alhangeul.xcodeproj` 직접 수정

## 설계 방향

먼저 재현 대상을 확정한다. 요청 파일명과 스크린샷 파일명이 다르므로 `exam_social.hwp`와 `exam_science.hwp`를 모두 확인하고, 문제가 발생하는 페이지와 개체 선택 상태를 기록한다.

원인 분리는 세 층으로 나눈다. 첫째, bundled `rhwp-studio` 입력 처리에서 Space가 picture/object selection을 해제한 뒤 text insertion으로 fall-through하는지 확인한다. 둘째, `DocumentPosition`의 `parentParaIndex`와 `controlIndex`가 잘못 남아 `insertTextInCell`로 라우팅되는지 확인한다. 셋째, HostApp이 이 예외를 fatal viewer failure로 분류하는 것이 사용자 경험상 적절한지 판단한다.

수정은 가능한 한 앱 소유 범위에서 시작한다. 문서 로드 실패, asset 로드 실패, WASM 초기화 실패는 fatal fallback으로 유지하되, 이미 문서가 표시된 뒤 사용자 입력 중 발생한 특정 recoverable runtime error는 문서를 닫지 않는 방식으로 처리하는 방안을 우선 검토한다. 단, 실제 viewer state가 계속 손상되는 경우에는 번들 `rhwp-studio` 입력 처리 보강 또는 upstream release 반영 경로를 별도 단계에서 검토한다.

번들 viewer 자산을 변경해야 한다면 provenance와 재생성 절차를 확인한 뒤 최소 변경만 적용한다. 임의로 minified JS를 직접 고치는 방식은 최후 수단으로 두고, 가능하면 source 기준 patch 또는 upstream release update 경로를 우선한다.

## 리스크

- HostApp에서 특정 runtime error를 recoverable로 분류하면 실제 viewer state 손상을 숨길 수 있다.
- 반대로 모든 runtime error를 fatal로 유지하면 사용자 입력 하나로 열린 문서 전체가 닫히는 현재 문제가 반복된다.
- 번들 `rhwp-studio` asset을 직접 수정하면 upstream provenance와 재현 빌드 절차가 흐려질 수 있다.
- upstream 최신 viewer로 갱신하는 방식은 #223 범위를 넘어 dependency update와 회귀 검증 범위가 커질 수 있다.
- 요청 파일명과 스크린샷 파일명이 달라 재현 기준을 잘못 잡으면 수정 검증이 빗나갈 수 있다.
- WebView console 예외와 native bridge payload가 같은 이벤트를 중복 보고할 수 있어 로그 해석에 주의가 필요하다.
