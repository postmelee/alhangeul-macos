# Issue #48 수행 계획서

## 타스크

- GitHub Issue: #48
- 마일스톤: M010 (v0.1.0)
- 제목: Task #45 후속: Codex/Claude Code SKILL 인식 실측과 description 튜닝
- 작업 브랜치: `local/task48`
- 기준 브랜치: `origin/devel` `0a3443d`

## 목적

Task #45에서 추가한 하이퍼-워터폴 절차 SKILL 5종이 Codex와 Claude Code 양쪽에서 실제 세션 단위로 노출되는지 확인하고, `allow_implicit_invocation: false`와 좁은 description이 의도하지 않은 묵시 호출을 막는지 관찰한다. 현재 세션에서는 먼저 Codex 측에서 가능한 노출·묵시 호출 회피 측정을 수행하고, Claude Code 측 측정은 같은 브랜치에서 이어받을 수 있도록 기록 형식을 정리한다.

## 배경

- Task #45 (PR #46)에서 `mydocs/skills/` 아래에 `task-start`, `task-stage-report`, `task-final-report`, `pr-merge-cleanup`, `external-pr-review` 5종 SKILL을 작성했다.
- `.agents/skills`와 `.claude/skills`는 `mydocs/skills`를 가리키는 심볼릭 링크로 구성되어 파일 시스템·git 단위 검증은 통과했다.
- Task #45 최종 보고서의 "잔여 위험 2번"은 실제 Codex/Claude Code 세션에서 user-invocable skill 목록 노출과 묵시 호출 회피를 사람이 측정해야 한다고 기록했다.
- Issue #48은 이 잔여 위험을 별도 운영 작업으로 추적한다.

## 범위

### 포함

- Codex 현재 세션에서 5종 SKILL 노출 여부를 확인한다.
- Codex 현재 세션에서 일반 작업 지시가 묵시적 SKILL 호출로 처리되지 않는지 관찰 가능한 범위 안에서 기록한다.
- Claude Code 측이 같은 브랜치에서 이어서 측정할 수 있도록 측정 항목과 판정 기준을 `mydocs/troubleshootings/` 문서에 남긴다.
- Claude Code 측 측정 결과가 추가된 뒤, 필요하면 5종 SKILL의 description 또는 트리거 문구를 더 좁힌다.
- 최종 결과를 `mydocs/troubleshootings/` 또는 `mydocs/feedback/`에 기록한다.

### 제외

- Rust/Swift/Xcode 소스 변경.
- 빌드, 서명, 배포, 릴리스 작업.
- 하이퍼-워터폴 절차 자체의 정책 변경.
- Task #45에서 만든 심볼릭 링크 구조의 대체 설계.
- 1주일 전체 관찰을 본 타스크 안에서 강제로 완료한 것으로 판정하는 일. 장기 관찰은 문서에 시작 시점과 후속 확인 방식으로 남긴다.

## 설계 방향

- 측정 기록은 단일 문서 `mydocs/troubleshootings/task_m010_48_skill_exposure.md`에 도구별 섹션으로 누적한다.
- Codex 측은 현재 세션의 Available skills 목록에 5종이 나타나는지, 각 description이 "명시 호출 시에만"이라는 조건을 포함하는지 확인한다.
- Claude Code 측은 같은 문서에 새 세션의 user-invocable skills 목록과 일반 대화에서의 묵시 트리거 여부를 이어서 기록한다.
- description 튜닝은 실제 오동작이 관찰된 경우에만 수행한다. 오동작이 없으면 SKILL 본문은 변경하지 않고 측정 기록과 보고서만 작성한다.
- 명시 호출 동작 자체는 각 SKILL이 문서/브랜치/PR 작업을 실제 수행할 수 있으므로, 실측 시에는 사용자 승인 범위 안에서 드라이런 또는 목록 노출 확인 중심으로 제한한다.

- `mydocs/orders/20260425.md` (오늘할일 갱신)
- `mydocs/plans/task_m010_48.md` (본 수행계획서)
- `mydocs/plans/task_m010_48_impl.md` (승인 후 구현계획서)

- 오동작 관찰 시에만 `mydocs/skills/*/SKILL.md` 5종 중 필요한 파일

## 잠정 단계

1. **Stage 1 — Codex 측 노출·묵시 호출 회피 측정**: 현재 Codex 세션의 Available skills 목록과 `allow_implicit_invocation: false` 설명 노출을 기록하고, 본 타스크 시작 과정에서 명시 호출 조건이 어떻게 판정되었는지 관찰 결과를 문서화한다.
2. **Stage 2 — Claude Code 측 측정 인계와 결과 수집**: Claude Code 새 세션에서 5종 노출 여부와 일반 대화 중 묵시 호출 여부를 같은 문서에 기록한다. 본 단계는 작업지시자 또는 Claude 측 측정 결과가 브랜치에 반영된 뒤 진행한다.
3. **Stage 3 — description 튜닝 여부 결정**: 양 도구 측정에서 오동작이 없으면 변경 없음으로 확정한다. 오동작이 있으면 해당 SKILL description과 트리거 문구만 최소 수정한다.
4. **Stage 4 — 검증과 최종 보고**: 측정 기록, 필요한 경우 SKILL 문구 변경 diff, 장기 관찰 항목을 최종 보고서에 정리하고 PR 게시 준비를 완료한다.

각 단계 완료 시 단계별 완료보고서(`mydocs/working/task_m010_48_stage{N}.md`)를 작성하고 승인 후 다음 단계로 진행한다.

## 검증 계획

- `rg --files mydocs/skills .agents/skills .claude/skills`로 5종 SKILL 원본과 양 도구 심볼릭 링크 접근을 확인한다.
- Codex 현재 세션의 Available skills 목록에 5종이 모두 포함되는지 기록한다.
- Claude Code 측 측정 결과가 추가되면 5종 이름, description, 명시 호출 경로(`/task-start` 등)가 문서에 모두 기록되었는지 확인한다.
- description 변경 시 `git diff -- mydocs/skills`로 변경 범위가 문구 튜닝에 한정되는지 확인한다.
- 문서 변경 후 `git diff --check`를 실행한다.

## 리스크

- 현재 세션에서 관찰 가능한 Codex 정보는 이미 적재된 시스템 컨텍스트와 도구 목록에 한정된다. 별도 Codex CLI UI의 `/skills` 메뉴는 이 환경에서 접근성이 제한될 수 있다.
- Claude Code 측 측정은 같은 브랜치에서 이어서 수행되어야 하므로, Stage 2 이후 일정은 외부 측정 결과에 의존한다.
- Issue 수용 기준의 "1주일 사용 동안 0건"은 즉시 완료 가능한 검증이 아니라 운영 관찰 기준이다. 본 타스크에서는 관찰 시작 기록과 후속 확인 방식을 남긴다.
- description을 과도하게 좁히면 명시 호출 검색성이 낮아질 수 있다. 실제 오동작이 있는 항목만 최소 수정한다.
