# Task M013 #263 수행계획서

## 목적

`native-viewer-editor` 장기 라인의 의미를 기존 “Swift native viewer/editor” 중심에서 “Swift native macOS 앱 셸 + Rust/rhwp Skia renderer + Swift 편집 UI/오버레이” 중심으로 재정의한다.

이 작업의 목적은 실제 구현을 시작하기 전에 로드맵, 아키텍처, 브랜치 정책, PR base 안내가 같은 방향을 말하게 만드는 것이다. `native-viewer-editor`는 제거하지 않고, 제품 `devel` 라인과 분리된 장기 native macOS shell/editor integration 라인으로 유지한다.

## 배경

upstream `rhwp`는 `native-skia` 기반 PNG raster backend와 `PageLayerTree` 계열 렌더링 경로를 제공하기 시작했다. 현재 앱 저장소도 #255-#259에서 Quick Look/Thumbnail의 Skia optional backend를 단계적으로 검증하는 방향을 문서화했다.

기존 문서에는 아직 “Swift native viewer/editor”, “CoreGraphics/CoreText 기반 native renderer parity” 표현이 넓게 남아 있다. 이 표현은 Swift가 HWP renderer 전체를 장기적으로 재구현하는 것처럼 읽힐 수 있다. 하지만 editor까지 고려하면 더 안전한 경계는 문서 구조, layout, render, 저장 안정성을 `rhwp` core가 소유하고, Swift/macOS 계층은 앱 셸, 이벤트, 오버레이, IME, 접근성, 메뉴/단축키와 fallback routing을 소유하는 구조다.

또한 `devel`과 `native-viewer-editor`가 이미 분리되어 있으므로, Skia 기반 공통 infrastructure와 장기 native editor shell 작업의 PR base 기준을 다시 명확히 해야 한다.

## 설계 방향

브랜치 정책은 `native-viewer-editor` 유지 쪽으로 정리한다. 제거하지 않는 이유는 장기 native editor shell과 Swift overlay 작업이 release 제품 라인에 섞이면 `devel`의 public release 안정성 판단이 흐려지기 때문이다.

역할 분리는 다음 기준으로 둔다.

- `devel`: 제품 공통 기반, WKWebView viewer/editor 유지, Quick Look/Thumbnail, Skia FFI와 Shared renderer backend, provenance, packaging, release 문서
- `native-viewer-editor`: HostApp native macOS viewer/editor shell, Skia 기반 page/tile viewport, zoom/cache/sidebar, Swift caret/selection/IME/ruler/object overlay, native editor command routing 실험
- `rhwp` core: 문서 parse/model/layout/render, edit transaction, hit-test/selection anchor, dirty page/rect, save/round-trip 안정성에 필요한 core API

문서 표현은 “Swift renderer 완성”이 아니라 “Mac-native interaction shell과 overlay를 Swift가 소유하고, 문서 의미와 rendering은 `rhwp` core/Skia 경로를 우선한다”로 맞춘다. CoreGraphics/CoreText renderer는 당장 제거하지 않고 Quick Look/Thumbnail fallback, diagnostic, 비교 기준으로 남긴다.

1. Stage 1: 현재 문서와 workflow에서 `Swift native viewer/editor`, `native-viewer-editor`, `CoreGraphics/CoreText renderer`, `Skia` 참조를 inventory하고 변경 범위를 확정한다.

5. Stage 5: 문서 일관성 검증, stage report, 최종 보고서, 오늘할일 상태 갱신, PR 게시 준비를 수행한다.

## 검증 계획

- `rg -n "Swift native viewer|Swift native editor|native-viewer-editor|CoreGraphics|CoreText|Skia|skia|PR base|통합 브랜치"`로 변경 전후 참조를 확인한다.
- `git diff --check`로 문서 whitespace 문제를 확인한다.
- Markdown 표와 링크가 깨지지 않았는지 변경 파일을 직접 검토한다.
- 코드와 workflow 실행 조건을 바꾸지 않는 한 Xcode build와 render smoke는 필수 검증에서 제외한다.
- `.github/workflows` 내용을 수정하지 않으면 YAML 검증은 생략하고, 수정하는 경우 해당 YAML 문법 확인을 추가한다.
- 최종 보고서에는 `devel`과 `native-viewer-editor`의 새 PR base 기준, 제외한 구현 범위, 후속 이슈 후보를 요약한다.

## 리스크

- `native-viewer-editor`를 유지하되 역할을 바꾸면 기존 “Swift native viewer/editor” 작업의 일부 문맥이 혼란스러울 수 있다. 문서에서는 기존 라인을 보존하되 목표 표현을 새 구조로 바꾼다는 점을 명확히 해야 한다.
- Skia 기반 HostApp renderer가 아직 제품 기본 경로가 아니므로, 문서가 구현 완료처럼 읽히면 안 된다. 로드맵과 제한 사항에는 단계적 검증과 fallback 유지가 드러나야 한다.
- CoreGraphics/CoreText renderer를 fallback/diagnostic으로 낮춰 설명하더라도 Quick Look/Thumbnail 현재 기본 경로와 충돌하면 안 된다.
- `devel`과 `native-viewer-editor`의 기준이 너무 복잡해지면 contributor가 PR base를 다시 잘못 고를 수 있다. 표와 예시를 단순하게 유지해야 한다.
- 실제 구현 없이 문서만 바꾸는 작업이므로 후속 구현 이슈와 readiness gate를 혼동하지 않게 해야 한다.

## 승인 요청 사항

이 수행계획 기준으로 Stage 1을 진행해도 되는지 승인 요청한다. Stage 1에서는 문서와 workflow 참조 inventory를 정리하고, 코드 변경, RustBridge ABI 변경, 원격 브랜치 조작은 수행하지 않는다.
