# Task M020 #87 수행계획서

## 목적

Quick Look preview에서 `QLPreviewReply(forPDFWithPageSize:documentCreationBlock:)`와 custom `PDFDocument`/`PDFPage` 조합이 실제 visible page 중심 lazy rendering으로 동작하는지 검증한다.

완료 후에는 현재 data-based PDF preview 유지, PDFKit 기반 lazy PDF reply 채택, 또는 #88 view-based Quick Look preview 전환 중 어느 방향이 맞는지 판단할 수 있어야 한다.

## 배경

Task #85에서 Quick Look preview는 다중 페이지 문서를 Quick Look 표시용 PDF로 반환하도록 확장됐지만, 현재 data-based reply는 완성된 PDF data blob을 반환해야 한다. 따라서 첫 페이지만 먼저 표시하고 후속 page를 visible page 중심으로 나중에 그리는 true lazy pagination은 제공하지 못한다.

Task #221은 rhwp core SVG 기반 PDF 생성 경로를 비교했고, 단기적으로는 현재 Swift native bitmap PDF/PNG fast path를 유지한다는 결론을 냈다. 이 결론은 SVG PDF 전환 여부에 대한 판단이며, Quick Look이 page draw 시점에 lazy하게 page를 요구하는지 여부를 해결하지는 않는다.

현재 별도 진행 중인 M020/#254는 Quick Look/Thumbnail raster backend로 Skia를 옵션화할지 설계하는 작업이다. #254는 page를 어떤 backend로 rasterize할지의 문제이고, #87은 Quick Look preview container가 lazy page 요청을 허용하는지의 문제라서 경계가 다르다. 다만 #88의 view-based 구현 범위는 #87 결론과 #254 backend 정책 이후에 확정하는 편이 안전하다.

## 범위

포함:

- 현재 `HwpPreviewProvider`와 `HwpPreviewPDFRenderer`의 data-based PDF reply 경로 재확인
- macOS SDK의 `QLPreviewReply(forPDFWithPageSize:documentCreationBlock:)`와 `PDFDocument`/`PDFPage` subclass 적용 가능성 확인
- custom `PDFPage.draw(with:to:)` 호출 시점, 호출 횟수, page materialization 여부를 로그로 관측하는 최소 prototype 작성
- 샘플 HWP/HWPX 기준 first preview latency, page draw 호출 순서, 전체 render 여부 측정
- PDFKit 기반 경로가 가능/불가능/부분 가능 중 어디에 해당하는지 결론 기록
- #88 view-based Quick Look preview 전환 필요성과 선행 조건 재판단

제외:

- 사용자용 PDF export 구현 또는 품질 개선
- #88 view-based Quick Look preview 전환 구현
- HostApp viewer 구조 변경
- Rust core ABI 변경
- Skia optional backend 구현
- signed/notarized public release 실행

## 설계 방향

이 작업은 제품 기본 동작 전환이 아니라 Quick Look API 동작 검증 spike로 진행한다. 먼저 현재 data-based 경로와 PDFKit reply API의 compile/runtime 적용 가능성을 확인하고, 그 다음 최소 prototype을 통해 Quick Look이 실제 표시 page만 `PDFPage.draw(with:to:)`로 요청하는지 관측한다.

prototype은 제품 경로를 오염시키지 않도록 가능한 한 격리한다. 제품 코드에 임시 진단 경로가 필요하면 명확한 debug/probe gate를 두고, 최종 단계에서 채택 또는 제거 여부를 결정한다. 실제 Quick Look 등록과 smoke가 필요한 경우 `build_run_guide.md`의 Release package 기준 smoke 절차를 따른다.

판단 기준은 다음과 같이 둔다.

- 가능: Quick Look이 visible page 중심으로 `PDFPage.draw(with:to:)`를 호출하고, 전체 PDF data materialization을 요구하지 않는다.
- 부분 가능: page draw hook은 호출되지만 초기 표시 전에 많은 page를 선행 materialize하거나, cache/cancellation 제약이 커서 제품화 전에 추가 구조 설계가 필요하다.
- 불가능: Quick Look이 `dataRepresentation` 또는 전체 page materialization을 요구해 현재 data-based PDF와 실질 차이가 없다.

## 잠정 단계

1. Stage 1: 현행 Quick Look PDF reply와 PDFKit lazy 후보 inventory
   - 현재 data-based PDF reply 흐름과 #85/#221 결론을 코드와 문서 기준으로 정리한다.
   - `QLPreviewReply(forPDFWithPageSize:documentCreationBlock:)` compile probe와 runtime 제약을 확인한다.
2. Stage 2: 최소 PDFKit lazy probe 설계와 구현계획 고정
   - prototype 위치, debug/probe gate, 로그 항목, 검증 샘플을 확정한다.
   - 구현계획서(`task_m020_87_impl.md`)에 단계별 변경과 검증 명령을 고정한다.
3. Stage 3: custom `PDFDocument`/`PDFPage` prototype 작성
   - `PDFPage.draw(with:to:)` 호출 순서와 page index를 관측할 수 있는 최소 prototype을 작성한다.
   - 제품 기본 preview 경로를 바꾸지 않도록 gate와 cleanup 방침을 함께 둔다.
4. Stage 4: Quick Look smoke와 latency/page draw 관측
   - 다중 페이지 HWP/HWPX 샘플로 first preview latency, visible page 이동, 전체 page render 여부를 기록한다.
   - 빠른 selection 전환에서 이전 render 결과가 남는지 관찰한다.
5. Stage 5: 결론과 #88 handoff 정리
   - 가능/불가능/부분 가능 결론을 최종 보고서에 남긴다.
   - #88을 즉시 진행할지, #254 backend gate 이후로 미룰지, 또는 data-based 경로를 유지할지 제안한다.

## 리스크

- Quick Look이 `PDFDocument` 객체를 받더라도 내부에서 전체 `dataRepresentation`을 요구하면 lazy 효과가 없을 수 있다.
- `PDFPage.draw(with:to:)` 호출이 visible page와 1:1로 대응하지 않고, Quick Look 내부 cache/preload 정책에 따라 선행 호출될 수 있다.
- Quick Look extension 환경에서는 stdout/log 수집과 timing 측정이 불안정할 수 있다.
- prototype gate가 불충분하면 제품 preview 경로에 실험 코드가 남을 수 있다.
- #254 Skia backend 설계와 병렬로 진행되므로, #88 구현 범위를 성급히 확정하면 renderer backend 정책과 충돌할 수 있다.
