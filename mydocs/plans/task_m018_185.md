# Task M018 #185 수행계획서

## 목적

`v0.1.1`부터 GitHub Release 본문, GitHub Pages 업데이트 페이지, Sparkle appcast, README 최신 릴리즈 요약, `mydocs/release/` 릴리즈 상세 문서가 같은 기준으로 작성되도록 release communication template과 생성/검증 흐름을 고도화한다.

완료 후에는 다음 릴리즈부터 사용자가 먼저 확인해야 하는 설치·첫 실행·업데이트·다운로드 정보를 상단에 두고, 운영자가 검증해야 하는 checksum, provenance, known limitations, release delta 기반 추가 확인 항목을 빠뜨리지 않는 기준을 갖춰야 한다. 이 작업은 실제 `v0.1.1` public release 게시가 아니라 #188 최종 배포 실행에서 사용할 기준과 후보 산출물을 준비하는 선행 작업이다.

## 배경

`v0.1.0` release note는 `scripts/ci/write-release-notes.sh`가 skeleton을 만들 수 있지만, 현재 구조는 설치 후 첫 실행 안내, Quick Look/Thumbnail 활성화, Sparkle 업데이트 확인, Pages 업데이트 페이지 갱신, delta 기반 릴리즈 검증 체크리스트를 일관되게 강제하기 어렵다.

현재 `docs/updates/`에는 `index.html`과 `v0.1.0.html`이 있고, `docs/appcast.xml`은 Sparkle stable feed로 사용된다. `README.md`는 로드맵과 v0.1 세부 항목이 커지고 있어 앞으로 릴리즈가 누적될수록 최신 정보와 과거 상세 기록이 섞일 위험이 있다. #185는 `v0.1.1` 마일스톤 실행 순서 3/6 단계로, #184의 설치 UX 기준을 받은 뒤 #198 CI 보강과 #188 public 배포 실행에 릴리즈 본문/페이지/체크리스트 기준을 넘기는 역할을 한다.

- `mydocs/release/` 폴더 역할, `index.md`, 릴리즈별 상세 문서 형식 정리

## 설계 방향

릴리즈 communication은 사람이 읽는 문서와 자동화 산출물이 함께 쓰이므로, 먼저 섹션 이름과 정보 우선순위를 안정적인 계약으로 둔다. GitHub Release 본문은 사용자용 요약, 설치 방법, 첫 실행/Quick Look 활성화, 업데이트 확인, 주요 변경, 산출물/SHA256, provenance, 검증 결과, release delta 추가 확인 항목, known limitations, third-party notices 순서를 기본으로 한다.

Pages 업데이트 페이지는 GitHub Release 본문의 복사본이 아니라 공개 사용자 안내 페이지로 유지한다. 다만 version, DMG URL, SHA256, Sparkle 안내, 주요 변경, known limitations가 GitHub Release 및 `docs/appcast.xml`과 충돌하지 않게 같은 입력 값에서 생성하거나 같은 체크리스트로 검증한다. 장기적으로 자동화를 확장하기 쉽도록 version, URL, checksum, provenance, 검증 결과, delta 요약 같은 입력을 명확히 분리하고, template 자체에 하드코딩된 릴리즈별 서술은 최소화한다.

README는 제품 개요와 현재 작업 축을 보여주는 문서로 좁히고, 과거 릴리즈 상세 기록은 `mydocs/release/`로 넘긴다. `mydocs/release/v<version>.md`는 운영 기록의 진실 원천에 가깝게 두되, public GitHub Release와 Pages URL로 연결한다. PR별 검증과 release candidate 검증의 경계도 이 문서에 남겨 #188의 final smoke 입력으로 재사용한다.

release delta 수집은 완전 자동 승인 장치가 아니라 초안 생성 도구로 본다. `previous public release tag -> current release candidate commit` 범위에서 PR/Issue/commit/file 목록을 모으고 HostApp viewer, Quick Look preview, Finder thumbnail, 저장/PDF/공유, Sparkle/Pages/appcast, DMG/signing/notarization, Homebrew Cask, core/viewer provenance, 문서 전용 변경 같은 영향 영역으로 분류한다. 자동 분류가 틀릴 수 있음을 전제로 release owner가 누락/과잉 항목을 보정하는 흐름을 문서화한다.

## 잠정 단계(3~6단계)

1. Stage 1: 현재 릴리즈 communication 표면과 기준 확정
   - `write-release-notes.sh`, `release_distribution_guide.md`, `docs/updates/`, `docs/appcast.xml`, README의 현재 중복·누락 지점을 정리한다.
   - `edwardkim/rhwp` `v0.7.10` 릴리즈 본문에서 참고할 구조를 확인하되, 알한글의 설치/첫 실행/provenance 우선순위에 맞게 조정한다.
   - GitHub Release, Pages, README, `mydocs/release/` 사이의 정보 소유 경계를 확정한다.
2. Stage 2: GitHub Release 본문 template과 필수 섹션 검증 구현
   - `scripts/ci/write-release-notes.sh`가 제안 template 순서와 필수 안내를 생성하도록 보강한다.
   - version, DMG filename, SHA256, rhwp core, viewer asset provenance, third-party notices 입력 검증을 유지한다.
   - template 누락을 dry-run 또는 shell-level 검증으로 확인할 수 있게 한다.
3. Stage 3: Pages 업데이트 페이지와 README 최신 릴리즈 기준 정리
   - `docs/updates/v0.1.1.html` 후보와 `docs/updates/index.html` 최신 항목/다운로드 링크 갱신 기준을 마련한다.
   - README는 현재 작업 축과 최신 공개 릴리즈 1개 요약 중심으로 줄이고, 상세는 릴리즈 문서로 연결한다.
   - Pages, Release body, appcast URL/DMG URL/SHA256이 서로 충돌하지 않는지 확인한다.
4. Stage 4: `mydocs/release/` 장기 기록 구조와 delta 검증 체크리스트 기준 구현
   - `mydocs/release/index.md`와 `mydocs/release/v0.1.1.md` 초안을 만든다.
   - 이전 공개 릴리즈 대비 merged PR/Issue/commit/file delta 수집 규칙과 영향 영역 분류 기준을 정리하거나 최소 helper를 추가한다.
   - PR별 검증과 release candidate 검증의 경계를 릴리즈 문서에 명확히 남긴다.
5. Stage 5: 매뉴얼 통합과 #188 handoff 검증
   - `release_distribution_guide.md`와 `document_structure_guide.md`에 최종 기준을 반영한다.
   - `write-release-notes.sh` dry-run, Pages 후보, README 최신 릴리즈 블록, `mydocs/release/v0.1.1.md`가 같은 version/DMG/SHA256/provenance 기준을 쓰는지 검증한다.
   - #188에서 public release 게시 전 반복할 smoke와 공개 URL 확인 항목을 최종 보고서에 넘긴다.

## 리스크

- Release body, Pages, README, `mydocs/release/`에 같은 정보를 과도하게 복제하면 다음 릴리즈에서 stale 상태가 생길 수 있다.
- 반대로 자동화 입력을 너무 추상화하면 작은 patch release의 수동 보정 비용이 커질 수 있다.
- HTML update page 생성 자동화를 도입할 경우 기존 사이트 스타일과 접근성을 해칠 수 있다.
- delta 영향 영역 자동 분류는 파일 경로만으로 의미를 완전히 판단할 수 없어 release owner 보정이 필요하다.
- `v0.1.1`의 실제 SHA256과 public DMG URL은 #188에서 확정되므로 이번 작업의 후보 산출물에는 placeholder와 handoff 기준이 섞일 수 있다.
- GitHub Release, Pages, appcast는 게시 시점과 branch가 다르므로 최종 공개 상태는 #188에서 실제 URL smoke로 확인해야 한다.
