# Task M018 #212 수행계획서

## 목적

GitHub Pages 홍보 페이지의 footer 문구와 업데이트 안내를 일반 사용자 기준으로 정리한다. 홈, `/updates/`, 릴리즈 노트 페이지의 footer 설명을 통일하고, `/updates/` footer 정렬과 중간 반응형 구간의 링크 배치를 보정한다.

## 배경

#208에서 v0.1.1 단일 universal DMG 안내가 보강됐지만, 이후 화면 확인에서 홈 화면과 `/updates/` 화면의 footer 문구가 서로 다르고 `/updates/`의 footer 중앙 정렬이 어긋나 보이는 문제가 확인됐다.

또한 `/updates/`의 “앱에서 확인” 섹션은 “수동 확인”, “자동 확인”, “Sparkle appcast URL”을 구현 관점으로 설명한다. 해당 페이지는 일반 사용자를 위한 안내 페이지이므로 앱 메뉴에서 업데이트를 확인하는 흐름과 설치 선택권을 쉽게 설명하고, appcast URL은 앱이 사용하는 고정 업데이트 feed라는 보조 정보로 다루는 편이 적절하다.

현재 #206은 다른 브랜치에서 Pages/appcast 배포 방식 전환 검토로 진행 중이다. 이 작업은 #206의 workflow, Pages source, appcast 게시 방식과 충돌하지 않도록 정적 HTML/CSS의 사용자-facing 문구와 레이아웃에만 집중한다.

## 범위

포함:

- 홈, `/updates/`, 릴리즈 노트 페이지 footer 문구를 다음 문구로 통일
  - Mac을 위한 HWP/HWPX 문서 미리보기 및 편집 앱입니다. 한글 파일이 더 이상 낯선 파일로 남지 않도록 만듭니다.
- `/updates/` footer 문구의 중앙 정렬과 responsive layout 보정
- 중간 화면 폭에서 footer 우측 링크가 설명 아래로 내려가지 않도록 CSS grid/flex 기준 조정
- `/updates/`의 “앱에서 확인” 섹션을 일반 사용자 안내 중심으로 단순화
- Sparkle appcast URL 섹션은 유지하되 노출 위계와 설명 문구를 보조 정보에 맞게 조정
- desktop, 중간 폭, mobile viewport에서 홈과 `/updates/` footer 렌더링 확인

제외:

- `docs/appcast.xml` release item, EdDSA signature, enclosure URL 변경
- GitHub Pages 배포 방식, Pages source branch, deploy-pages workflow 변경
- GitHub Release 게시, public DMG 생성, appcast publish 실행
- macOS 앱의 Sparkle 설정 또는 `SUFeedURL` 변경
- Homebrew Cask 또는 release workflow 변경

## 설계 방향

Footer는 페이지별 철학 문구가 갈라지지 않도록 하나의 제품 설명으로 통일한다. “뷰어”보다 실제 공개 범위에 맞는 “미리보기 및 편집 앱” 표현을 사용하되, 문장 길이가 길어지는 만큼 가운데 설명 영역이 자연스럽게 줄바꿈되도록 CSS를 보정한다.

넓은 화면과 중간 화면에서는 브랜드, 설명, 링크를 같은 행의 3영역으로 유지한다. 설명 문구만 중앙 칼럼에서 줄바꿈되게 하고, 링크 묶음은 우측에 고정한다. 모바일 폭에서는 기존처럼 1열 stack을 허용하되 링크가 과도하게 벌어지거나 중앙 정렬이 흐트러지지 않게 한다.

`/updates/`의 업데이트 확인 안내는 Sparkle 내부 동작을 자세히 설명하지 않는다. 기본 흐름은 “앱 메뉴에서 업데이트 확인을 선택하면 새 버전을 확인할 수 있고, 새 버전이 있어도 설치는 사용자가 선택한다”로 정리한다. appcast URL은 앱과 release 운영자가 쓰는 고정 feed 주소로 남기되, 일반 사용자가 반드시 이해해야 하는 필수 절차처럼 보이지 않게 문구와 배치를 조정한다.

#206과의 충돌을 줄이기 위해 이번 작업은 정적 페이지 콘텐츠와 CSS만 다룬다. 배포 workflow, Pages branch, appcast 생성/게시 자동화 관련 파일은 수정하지 않는다.

## 리스크

- #206이 Pages/appcast 배포 방식 파일을 수정 중이므로, 이번 작업이 `.github/`, release workflow, appcast publish script를 건드리면 병합 충돌과 책임 경계 혼선이 생긴다.
- footer 문구가 길어 중간 폭에서 nav가 밀리거나 브랜드와 설명의 시각 중심이 어긋날 수 있다. 실제 viewport 확인이 필요하다.
- appcast URL을 너무 숨기면 release 운영 정보 확인성이 떨어지고, 너무 강조하면 일반 사용자에게 구현 세부사항처럼 보일 수 있다.
- 정적 페이지는 별도 빌드 파이프라인 없이 서비스되므로 cache-busting query와 기존 링크 경로를 유지하는지 확인해야 한다.

## 승인 요청 사항

- footer 문구는 작업지시자가 지정한 “Mac을 위한 HWP/HWPX 문서 미리보기 및 편집 앱입니다. 한글 파일이 더 이상 낯선 파일로 남지 않도록 만듭니다.”로 홈, `/updates/`, 릴리즈 노트 페이지에 통일하는 방향으로 진행해도 되는지 승인 요청한다.
- `/updates/`의 “수동 확인/자동 확인” 카드는 제거하거나 단일 안내 문단으로 합치고, appcast URL은 보조 정보 섹션으로 낮추는 방향으로 진행해도 되는지 승인 요청한다.
- #206과의 충돌을 피하기 위해 이번 작업에서는 `docs/` HTML/CSS와 task 문서만 수정하고, Pages/appcast 배포 workflow와 appcast XML은 변경하지 않는 방향으로 진행해도 되는지 승인 요청한다.
