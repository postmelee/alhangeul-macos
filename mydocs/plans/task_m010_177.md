# Task M010 #177 수행계획서

## 목적

첫 공개 배포 전에 알한글 앱이 Sparkle 기반 업데이트 확인과 자동 설치 경로를 가질 수 있도록 앱 설정, GitHub Pages appcast 고정 URL, release workflow의 appcast 생성 경로를 정리한다.

완료 후에는 v0.1 첫 배포 앱에 업데이트 feed URL과 Sparkle public key가 포함되고, GitHub Release DMG를 참조하는 appcast를 GitHub Pages에서 안정적으로 제공할 수 있어야 한다.

## 배경

PR #176으로 GitHub Pages 소개 페이지가 `docs/` 기준으로 준비되었고, 공개 URL은 `https://postmelee.github.io/alhangeul-macos/`이다. 첫 출시 전에 업데이트 feed의 고정 URL을 확정하면 앱의 `SUFeedURL`에 해당 URL을 넣을 수 있다.

#166은 M16 완료 후 v0.1 public release를 실행하는 이슈이며, 새 updater 또는 auto-update 체계는 제외 범위다. 따라서 Sparkle 통합과 appcast 생성은 #166 전에 별도 작업으로 완료해야 한다. 첫 배포 후에 이 작업을 하면 이미 설치된 v0.1 앱은 업데이트 feed를 알지 못하므로, 첫 배포 바이너리에 최소 업데이트 확인 기반을 포함하는 것이 중요하다.

현재 release publish workflow는 signed/notarized DMG를 만들고 GitHub Release asset으로 업로드한다. 이번 작업은 그 DMG를 Sparkle appcast의 enclosure로 참조하되, 실제 public release publish와 Homebrew Cask 배포 결정은 #166에 남긴다.

## 범위(포함·제외)

포함:

- Sparkle 2 도입 방식 조사와 `project.yml` 기반 의존성 추가 방향 확정

- sandboxed macOS app에서 Sparkle 자동 설치에 필요한 entitlement, XPC service, copy phase 구성 확인과 반영
- 앱 메뉴에 “업데이트 확인...” 명령 추가
- GitHub Pages `docs/` 아래 사용자용 업데이트 안내 페이지와 Sparkle appcast 파일 위치 확정
- release workflow 또는 보조 script가 DMG signature와 appcast item을 생성/갱신하는 구조 구현
- Sparkle signing key 생성/보관/CI secret 운영 문서화

## 설계 방향

업데이트 경로는 Sparkle을 단일 기준으로 둔다. 별도 GitHub Releases API polling을 먼저 구현하지 않고, Sparkle appcast가 수동 업데이트 확인과 자동 업데이트 안내/설치를 모두 담당하게 한다.

GitHub Pages는 사용자가 보는 소개 페이지와 Sparkle이 읽는 appcast를 같은 `docs/` 정적 사이트 안에서 제공한다. 첫 v0.1 배포는 prerelease가 아닌 공식 release로 게시하는 것을 기준으로 두며, 앱의 stable feed는 `https://postmelee.github.io/alhangeul-macos/appcast.xml`로 고정한다. 별도 beta channel UI가 없으므로 이번 범위에서는 prerelease feed를 만들지 않는다.

release publish workflow는 GitHub Release DMG asset을 계속 배포 기준으로 삼는다. appcast XML은 DMG를 저장소에 넣지 않고 GitHub Release asset URL, version, minimum system version, sparkle signature, release notes URL을 기록하는 정적 feed로 만든다.

Sparkle private key는 저장소에 두지 않는다. 로컬 또는 GitHub Actions secret에서만 사용하고, 앱에는 public key만 기록한다. release automation은 secret이 없을 때 fail-fast로 중단하거나 appcast 갱신을 명확히 건너뛰도록 설계한다.

## 잠정 단계(3~6단계)

1. Stage 1: Sparkle/GitHub Pages/release workflow 요구사항 확정
   - Sparkle 2의 macOS sandboxed app 요구사항, appcast 형식, EdDSA signing 흐름을 공식 문서 기준으로 확인한다.
   - 공식 release 기준 feed 정책과 Pages URL을 확정한다.
2. Stage 2: HostApp Sparkle 통합
   - `project.yml`에 Sparkle dependency와 필요한 framework/XPC embedding을 반영한다.
   - `Info.plist`, entitlement, 앱 메뉴 명령을 추가한다.
3. Stage 3: GitHub Pages 업데이트 페이지와 appcast skeleton 추가
   - `docs/`에 사용자용 업데이트 안내 페이지와 초기 appcast 파일을 둔다.
   - landing page에서 다운로드/업데이트 안내 경로가 자연스럽게 연결되는지 확인한다.
4. Stage 4: appcast 생성 script와 release workflow 연동
   - release publish 결과의 version, DMG URL, SHA256, Sparkle signature, release note URL을 appcast로 기록하는 script/workflow 경로를 만든다.
   - secret 미설정 시 실패 모드와 rehearsal 동작을 분리한다.
5. Stage 5: 빌드, 정적 파일, workflow 검증과 최종 보고
   - Xcode project 생성, HostApp build, appcast XML 검증, Pages 정적 파일 경로 확인, release workflow 문법 검토를 수행한다.

## 리스크

- Sparkle private key를 저장소나 로그에 노출하면 안 된다. key 생성, 보관, GitHub Actions secret 등록 절차를 문서와 workflow에서 분리해야 한다.
- sandboxed app에서 Sparkle 자동 설치를 지원하려면 framework 추가만으로 끝나지 않을 수 있다. XPC service embed, entitlement, code signing 검증을 Stage 1에서 먼저 확인해야 한다.
- GitHub Pages가 `docs/`를 그대로 서빙하므로 release workflow가 appcast만 갱신하려면 Pages 소스 브랜치와 release branch 운영이 충돌할 수 있다.
- GitHub Pages 다운로드 버튼을 `latest/download`로 두려면 #166에서 `prerelease=false`로 실행하고, 검수 후 non-draft 공식 release로 publish해야 한다.
- Sparkle dependency 추가는 네트워크와 Xcode package resolution에 의존한다. 로컬 검증에서 package fetch가 실패할 수 있으며, 실패 시 네트워크/권한 문제와 코드 문제를 구분해 기록한다.
- `project.yml`이 Xcode project 원본이므로 `Alhangeul.xcodeproj` 직접 수정은 금지한다. 필요한 Xcode project 변경은 모두 `project.yml`에 반영한다.
