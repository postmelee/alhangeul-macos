# Issue #130 수행 계획서

## 목적

프로젝트 작업 중 누적되는 빌드 산출물, 렌더 검증 출력, `/private/tmp` 임시 파일을 안전하게 분류하고 정리할 수 있도록 `project-artifact-cleanup` Skill을 추가한다. 기본 동작은 삭제가 아니라 dry-run 후보 보고로 제한하고, 실제 삭제는 작업지시자 명시 승인 후에만 수행하도록 절차를 고정한다.

## 배경

- `build.noindex/`, `output/`, `RustBridge/target/`, `Frameworks/`는 재생성 가능한 산출물이지만 빌드 시간, 검증 증거, 설치본 갱신 여부에 따라 삭제 판단이 달라진다.
- `/private/tmp/rhwp*`에는 렌더 비교 결과, Quick Look thumbnail 출력, PR/Issue 임시 본문, Swift/Clang module cache가 섞인다.
- 같은 `/private/tmp/rhwp*` 패턴 안에 `/private/tmp/rhwp-mac-task{N}` 형태의 실제 git worktree도 존재할 수 있으므로 이름 패턴만으로 삭제하면 작업 손실 위험이 있다.
- Debug 빌드는 compile/link 확인용이어서 정리 가능하지만, Quick Look/Thumbnail/Viewer 테스트에는 `$HOME/Applications/AlhangeulMac.app`에 설치된 Release package 산출물을 기준으로 판단해야 한다.
- 기존 `pr-merge-cleanup` Skill은 PR merge 후 이슈, branch, worktree를 정리하는 절차이므로 이번 빌드/임시 산출물 정리 Skill과 역할을 분리해야 한다.

## 범위

### 포함

- `mydocs/skills/project-artifact-cleanup/SKILL.md` 신규 작성
- 부산물 후보를 `safe`, `approval-required`, `never-delete`로 분류하는 절차 정의
- `/private/tmp/rhwp*`, `/private/tmp/alhangeul*`, `output/`, `build.noindex/`, `RustBridge/target/`, `Frameworks/`의 삭제 판단 기준 정리
- git worktree, git repository, 현재 진행 task 검증 증거 보호 규칙 명시
- Debug 빌드 삭제 전 Release 설치본 갱신 및 LaunchServices/PlugInKit/Quick Look 확인 기준 명시
- 기존 `pr-merge-cleanup`과 역할 경계 명시

### 제외

- 실제 부산물 삭제 실행
- 정리 스크립트 구현
- PR merge 후 branch/worktree 정리 절차 변경
- `$HOME/Applications/AlhangeulMac.app` 무단 삭제
- 이전 이름 설치본(`RhwpMac.app`, `알한글.app`) 무단 삭제
- Rust/Swift/Xcode 소스 변경

## 설계 방향

- Skill frontmatter는 기존 저장소 Skill과 동일하게 `allow_implicit_invocation: false`를 명시한다.
- Skill 이름은 `project-artifact-cleanup`으로 한다.
- 본문은 도구 비종속 절차 중심으로 작성하고, Codex/Claude Code 호출 방법만 말미에 분리한다.
- 삭제 전에는 반드시 후보 경로, 분류, 용량, 삭제 이유, 보호 이유를 보고하도록 한다.
- 실제 삭제 명령은 승인 후에만 실행하며, `rm -rf` 대상은 개별 절대 경로 목록으로 제한한다.
- `/private/tmp` 전체, 저장소 루트, `$HOME`, git worktree로 등록된 경로, `.git`이 있는 경로는 삭제 금지 대상으로 둔다.
- Debug build cleanup은 Release package를 `$HOME/Applications/AlhangeulMac.app`로 갱신하고 `lsregister`, `pluginkit`, `qlmanage -t` 기준을 확인한 뒤 허용한다.

## 잠정 단계

1. **Stage 1 — 정리 대상과 보호 규칙 확정**: 관련 매뉴얼, 기존 Skill, 현재 부산물 사례를 기준으로 삭제 가능/승인 필요/삭제 금지 분류 기준을 확정한다.
2. **Stage 2 — `project-artifact-cleanup` Skill 작성**: `SKILL.md` 본문을 작성하고 dry-run, 승인, 삭제 금지, Debug build cleanup 판단 절차를 반영한다.
3. **Stage 3 — 검증과 문서 정리**: Skill frontmatter, 링크, 호출 안내, 예상 사용 흐름을 검증하고 단계 보고서와 최종 보고서로 정리한다.

각 단계 완료 시 단계별 완료보고서(`mydocs/working/task_m010_130_stage{N}.md`)를 작성하고 승인 후 다음 단계로 진행한다.

## 리스크

- 삭제 절차가 과도하게 넓으면 작업지시자 또는 다른 작업자의 worktree와 검증 증거를 잃을 수 있다.
- 반대로 보호 규칙이 지나치게 강하면 실질적인 정리 Skill로 쓰기 어렵다.
- Debug 산출물 정리 기준을 잘못 쓰면 Finder Quick Look/Thumbnail 검증에 필요한 설치본과 개발 산출물을 혼동할 수 있다.
- `/private/tmp`의 파일명은 작업자가 자유롭게 만들 수 있으므로 패턴 기반 분류에는 항상 예외가 있다.
