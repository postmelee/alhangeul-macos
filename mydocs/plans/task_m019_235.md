# Task M019 #235 수행계획서

## 목적

`samples/exam_social.hwp`의 성명 필드 입력 중 upstream `rhwp` editor/runtime 오류가 발생해도 HostApp Web viewer가 전체 fatal 오류 화면으로 전환되지 않게 한다.

완료 후에는 이미 문서를 표시한 뒤 발생하는 JavaScript console/runtime 오류가 문서 화면을 가리지 않는 상단 banner 또는 toast로 표시되어야 한다. 사용자는 오류를 인지하되 기존 문서 화면을 계속 볼 수 있어야 하며, 초기 load/resource/process failure처럼 문서 표시 자체가 불가능한 오류는 기존 fatal fallback을 유지해야 한다.

## 배경

Issue #235는 `exam_social.hwp` 성명 필드 입력 중 다음 runtime 오류가 발생하면 현재 HostApp이 viewer 전체를 fatal 화면으로 바꾸는 문제를 기록한다.

```text
message=렌더링 오류: 컨트롤 인덱스 0 범위 초과
sourceURL=alhangeul-studio://app/assets/index-CRsGAVvx.js
line=1
column=32679
reason=렌더링 오류: 컨트롤 인덱스 0 범위 초과
```

근본 렌더링/편집 오류는 upstream `edwardkim/rhwp` 이슈 #850에서 별도 추적 중이다. 본 저장소의 책임 범위는 HostApp WKWebView viewer가 post-load runtime 오류를 사용자 작업을 막는 fatal 화면으로 과잉 승격하지 않도록 하는 것이다.

선행 #223에서는 문서 표시 이후 발생한 `지정된 컨트롤이 표, 글상자 또는 그림이 아닙니다` runtime 오류를 nonfatal로 분류하고, 상단 banner와 5초 자동 dismiss/수동 닫기 UI를 추가했다. 이번 작업은 같은 UX 정책을 `컨트롤 인덱스 0 범위 초과` 계열과 반복 오류 제어까지 확장하되, fatal fallback 기준을 약하게 만들지 않는 것이 핵심이다.

## 범위(포함·제외)

포함:

- `samples/exam_social.hwp` 성명 필드 입력 재현 경로와 runtime payload 확인
- HostApp WKWebView `runtime-error` 수신 시 fatal/nonfatal 분류 조건 재검토
- 이미 문서가 표시된 뒤 발생한 `컨트롤 인덱스 0 범위 초과` 계열 runtime 오류의 nonfatal banner 처리
- 같은 오류가 반복 발생할 때 banner가 과도하게 누적되거나 계속 reset되지 않도록 dedupe 또는 throttle 정책 적용
- 필요할 때 진단 정보를 확인할 수 있는 현행 failure/banner 구조의 보강 가능성 검토
- 초기 asset load 실패, document load 실패, WebContent process crash, resource 누락, timeout의 fatal fallback 회귀 확인
- `v0.1.2` release blocker 처리 결과를 작업 보고서와 release 기록에 남기는 범위 검토
- 수행계획서, 구현계획서, 단계 보고서, 최종 보고서, 오늘할일 갱신

제외:

- HWP control index 범위 초과 원인 분석과 parser/editor 수정
- Swift native viewer/editor 구현
- Quick Look/Thumbnail native renderer 오류 정책 변경
- public release 실행, signing, notarization, Homebrew Cask 반영
- `Alhangeul.xcodeproj` 직접 수정

## 설계 방향

수정은 HostApp이 소유한 WKWebView failure routing에서 시작한다. `RhwpStudioWebView.Coordinator`는 현재 문서 표시 완료 여부와 runtime payload를 바탕으로 recoverable 후보를 판별하고, `DocumentViewerStore`는 `RhwpStudioWebViewFailure.isFatal == false`일 때 전체 fallback 대신 `webViewErrorMessage` banner를 표시한다.

이번 작업에서는 #223의 invalid-control 전용 판별을 그대로 넓히기보다, post-load bundled viewer runtime 오류를 안정적으로 분류하는 작은 정책 계층을 만든다. 최소 조건은 current document 존재, WebView load 완료, source가 bundled `rhwp-studio` asset 또는 그 stack인 점, 메시지가 upstream 편집/runtime 계열로 확인된 점이다. 이 조건을 만족하지 않는 navigation/resource/document/process/timeout failure는 계속 fatal로 둔다.

반복 오류 제어는 banner 상태 소유자인 `DocumentViewerStore` 쪽을 우선 검토한다. 같은 runtime diagnostic이 짧은 시간 안에 반복될 때 사용자에게 같은 banner를 계속 재표시하지 않거나, 메시지와 마지막 표시 시각 기준으로 throttle한다. 다만 서로 다른 fatal failure나 사용자 명령 오류까지 숨기지 않도록 nonfatal runtime banner 범위에 한정한다.

진단 정보는 기본 화면을 방해하지 않아야 한다. 기존 fatal fallback의 diagnostic detail은 유지하고, nonfatal banner에서는 짧은 사용자 문구를 우선한다. 더 자세한 payload가 필요하면 단계 조사에서 로그, 단계 보고서, 또는 별도 진단 표시 후보를 검토하되, MVP는 문서 유지와 과도한 반복 방지에 둔다.

## 잠정 단계(3~6단계)

1. Stage 1: 재현과 현행 오류 정책 inventory
   - `exam_social.hwp` 성명 필드 입력 경로를 재현하고 runtime payload, source asset, load 완료 상태를 기록한다.
   - #223에서 추가된 invalid-control recoverable 분류와 banner dismiss 동작을 현행 기준으로 정리한다.
   - fatal로 유지해야 할 WebView failure 경로를 목록화한다.
2. Stage 2: 구현계획 작성과 분류 정책 확정
   - post-load runtime 오류 중 nonfatal로 처리할 메시지/source 조건과 반복 오류 dedupe/throttle 기준을 확정한다.
   - 구현계획서(`task_m019_235_impl.md`)에 단계별 변경 파일, 검증 명령, smoke 절차를 고정한다.
3. Stage 3: nonfatal runtime 분류와 banner 반복 제어 구현
   - `컨트롤 인덱스 0 범위 초과` 계열 runtime 오류가 문서 화면을 유지하고 banner로 표시되도록 보강한다.
   - 같은 오류 반복 시 banner가 과도하게 누적 또는 재표시되지 않도록 dedupe/throttle 처리를 적용한다.
4. Stage 4: fatal fallback 회귀 검증과 GUI smoke
   - `exam_social.hwp` 성명 필드 입력 smoke에서 전체 fatal 화면 미전환과 banner 표시를 확인한다.
   - asset/resource/document/process/timeout failure가 기존 fatal fallback을 유지하는지 가능한 범위에서 회귀 확인한다.
5. Stage 5: release 기록과 최종 보고
   - `v0.1.2` release blocker 처리 결과, 검증 결과, upstream 잔여 이슈 #850 연결을 보고서와 release 기록 후보에 정리한다.
   - 최종 결과 보고서와 오늘할일 완료 갱신을 수행한다.

## 리스크

- runtime 오류를 넓게 nonfatal로 분류하면 실제 viewer state 손상이나 문서 표시 불능 상태를 숨길 수 있다.
- 분류 조건을 메시지 문자열에 강하게 의존하면 upstream asset 업데이트 후 같은 UX 문제가 다시 나타날 수 있다.
- 반복 오류 throttle이 너무 강하면 사용자가 새 오류 발생을 인지하지 못할 수 있다.
- 반대로 throttle이 약하면 입력 반복 중 banner가 계속 reset되어 문서 사용을 방해할 수 있다.
- `exam_social.hwp`의 수동 재현성이 낮으면 코드 검증과 실제 완료 기준 확인이 분리될 수 있다.
- upstream #850이 해결되기 전까지 오류 자체는 남을 수 있으므로, 본 작업은 HostApp UX 완화로 범위를 제한해야 한다.

## 승인 요청 사항

1. #235를 #223 후속으로 보고, HostApp post-load runtime 오류 정책과 banner 반복 제어를 보강하는 방향 승인
2. `컨트롤 인덱스 0 범위 초과`는 upstream 원인 수정 없이 HostApp에서 nonfatal UX로 처리하는 범위 승인
3. 다음 단계: 승인 후 구현계획서(`mydocs/plans/task_m019_235_impl.md`) 작성
