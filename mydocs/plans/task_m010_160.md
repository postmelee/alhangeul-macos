# Task #160 수행계획서: 브랜치 전략 정리 및 README/자동화 기준 업데이트

## 목적

HostApp viewer 첫 출시는 native renderer 대신 `rhwp-studio`를 WKWebView로 로드하는 방식으로 확정했다. 이에 맞춰 첫 출시 기준 브랜치, 기여 PR base, `main` 승격 흐름, 배포 자동화 기준, native renderer 실험 브랜치의 역할을 한 문서로 정리한다.

완료 후에는 `main`, `devel-webview`, `devel`의 역할이 README, CONTRIBUTING, 운영 매뉴얼, GitHub review instruction에서 서로 모순 없이 설명되어야 한다. 실제 브랜치 rename이나 GitHub 설정 변경은 이번 task에서 수행하지 않고, 필요 시 후속 이슈로 분리한다.

## 배경

현재 이전 native renderer 작업은 `devel`에 기록되어 있고, WKWebView 전환 및 첫 출시 준비 작업은 `devel-webview`에서 진행 중이다. 작업지시자는 향후 `devel-webview` 기준으로 `main`을 업데이트한 뒤, 계속 `devel-webview`를 주 작업 트리로 사용하고 native renderer 관련 실험적 기여는 `devel`로 받고 싶다는 방향을 제시했다.

검토 결과 README와 CONTRIBUTING에는 이미 `devel-webview`/`devel` 분리 정책이 대체로 반영되어 있었다. 다만 `.github/copilot-instructions.md`, `mydocs/manual/release_distribution_guide.md`, `mydocs/manual/document_structure_guide.md` 등 일부 운영 문서에는 `devel` 단일 기준으로 읽히는 문구가 남아 있다.

최신 remote-tracking 기준의 브랜치 상태는 다음과 같다.

- `origin/main...origin/devel-webview`: main 전용 6 commits, devel-webview 전용 232 commits
- `origin/devel...origin/devel-webview`: devel 전용 22 commits, devel-webview 전용 69 commits

따라서 첫 출시 전에는 브랜치 rename보다 정책 정합화와 `devel-webview -> main` 승격 주의사항 문서화가 우선이다.

## 범위

포함:

- `mydocs/tech/branch_strategy_webview_native.md` 신규 작성
- `main`, `devel-webview`, `devel`, `local/task{N}`, `publish/task{N}`의 역할 정리
- 첫 출시 전 단기안과 출시 후 장기안을 분리해 기록
- `devel-webview -> main` 출시 승격 시 `main` 전용 README/banner 커밋 보존 여부 점검 항목 정리
- `devel`을 native renderer 실험/장기 개발 브랜치로 유지할 때의 이름 혼동 리스크 정리
- README, CONTRIBUTING, GitHub review instruction, release/workflow/document structure 문서의 브랜치 정책 불일치 문구 보정
- 외부 기여 PR base와 메인테이너 `publish/task{N}` PR base 기준 재확인

제외:

- 실제 브랜치 rename
- GitHub default branch 또는 branch protection 설정 변경
- `devel-webview -> main` release PR 생성
- release artifact 생성, signing, notarization, Homebrew Cask 배포
- 배포 자동화 script 동작 변경
- native renderer 코드 또는 WKWebView viewer 코드 변경

## 설계 방향

- 이번 task의 진실 원천은 `mydocs/tech/branch_strategy_webview_native.md`로 둔다. README와 CONTRIBUTING은 외부 독자가 바로 봐야 하는 요약만 유지하고, 세부 판단 근거는 tech 문서로 연결한다.
- 첫 출시 전에는 `devel-webview`를 출시 우선 통합 브랜치로 유지한다. 출시 전 rename은 PR base, branch protection, 자동화, 문서 링크를 동시에 흔들 수 있으므로 권장하지 않는 방향으로 정리한다.
- `devel`은 native viewer renderer와 장기 native viewer 실험 브랜치로 유지하되, 일반적인 `devel` 명칭이 주 개발 브랜치로 오해될 수 있음을 명시한다.
- 출시 후 장기안은 두 선택지로 남긴다. 하나는 현재 이름 유지, 다른 하나는 `devel-webview`를 `devel` 또는 `develop`으로 승격하고 기존 `devel`을 `native-renderer` 계열로 rename하는 안이다.
- release/rollback 문서는 특정 브랜치명을 하드코딩하기보다 "출시 대상 통합 브랜치, 현재는 `devel-webview`"처럼 현재 정책과 향후 rename 가능성을 동시에 견디는 표현으로 보정한다.
- 문서 변경은 기존 정책을 뒤집지 않고, 현재 결정 사항과 남은 판단 지점을 명확히 하는 범위로 제한한다.

## 잠정 단계

### Stage 1: 브랜치 현황과 문서 불일치 재조사

- `origin/main`, `origin/devel-webview`, `origin/devel`의 ahead/behind와 주요 전용 커밋을 재확인한다.
- README, CONTRIBUTING, `.github`, `mydocs/manual`, `mydocs/tech`에서 브랜치 정책 표현을 전수 검색한다.
- 수정 필수, 요약 보정, 참고 유지로 분류한다.

### Stage 2: tech 문서 작성

- `mydocs/tech/branch_strategy_webview_native.md`를 작성한다.
- 단기 운영안, 장기 rename 후보, 첫 출시 `main` 승격 체크리스트, 기여 PR base 기준, 자동화 점검 항목을 정리한다.
- 실제 rename과 GitHub 설정 변경은 후속 이슈 후보로 분리한다.

### Stage 3: README/CONTRIBUTING/운영 문서 정합화

- README와 CONTRIBUTING의 요약 정책이 tech 문서와 같은 기준을 바라보도록 보정한다.
- Copilot review instruction, release rollback, document structure 관련 잔여 `devel` 단일 기준 문구를 수정한다.
- `git_workflow_guide.md`와 `pr_process_guide.md`의 예시 명령이 현재 기준과 일치하는지 확인한다.

### Stage 4: 문서 검증과 최종 정리

- 브랜치명 검색 결과를 재확인하고 의도적으로 남긴 legacy/reference 문맥을 구분한다.
- Markdown link, heading, 명령 예시, 한국어 문서 형식을 점검한다.
- 단계 보고서와 최종 보고서에 실제 수정 범위와 남은 후속 이슈 후보를 정리한다.

## 검증 계획

계획 문서 단계:

```bash
git diff --check -- mydocs/orders/20260506.md mydocs/plans/task_m010_160.md
```

구현 단계 진입 후 최소 검증:

```bash
rg -n "devel-webview|devel|main|PR base|통합 브랜치|릴리즈 기준|출시 대상" \
  README.md CONTRIBUTING.md .github mydocs/manual mydocs/tech
git diff --check
```

문서 전용 작업이므로 Xcode build, Rust bridge build, Finder/Quick Look smoke test는 기본 검증 범위에 포함하지 않는다. 단, 변경 중 build/run/release 명령 예시를 실제로 바꾸게 되면 해당 명령의 syntax 검증 가능 여부를 구현계획서에서 별도로 정한다.

## 리스크

- `devel-webview`를 계속 주 작업 트리로 두면 브랜치 이름만 보고는 "임시 WebView 브랜치"로 오해할 수 있다.
- `devel`을 native renderer 실험 브랜치로 두면 일반적인 개발 브랜치 관례와 어긋나 외부 기여자가 잘못된 base로 PR을 만들 수 있다.
- 출시 전 브랜치 rename을 하면 branch protection, PR base, 문서 링크, 자동화 기준을 한꺼번에 바꿔야 하므로 v0.1 일정 리스크가 커진다.
- `main`에는 `devel-webview`에 없는 README/banner 커밋이 있으므로 release PR에서 보존/대체 판단을 명확히 하지 않으면 문서가 의도치 않게 되돌아갈 수 있다.
- 운영 문서가 중복으로 브랜치 정책을 설명하므로 한 문서만 수정하면 다시 불일치가 생길 수 있다.
