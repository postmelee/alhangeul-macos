# Task #77 수행 계획서

## 목적

v0.1.0 배포물에 포함될 HostApp 앱 아이콘과 앱 내 로고 표시 자산을 신규 이미지로 교체한다. 사용자가 Dock, Finder, About 화면에서 보는 앱 대표 이미지를 배포 준비 상태에 맞춘다.

## 배경

신규 아이콘 PNG 세트가 `/Users/melee/Documents/projects/Icon Exports`에 준비되어 있다. 현재 HostApp은 `Sources/HostApp/Assets.xcassets/AppIcon.appiconset`을 앱 아이콘으로 사용하며, `Sources/HostApp/Views/AboutView.swift`의 About 헤더는 `NSApp.applicationIconImage`를 표시하므로 AppIcon 교체 결과가 앱 내 로고 표시에도 반영된다.

GitHub Issue #77은 v0.1 milestone에 할당했다. 작업 성격만 보면 장기 배포 품질 범위인 v0.5에 가까운 부분이 있지만, 실제 v0.1.0 배포물에 포함되어야 하는 필수 자산이므로 v0.1 릴리스 범위에서 추적한다.

## 범위

### 포함

- `Sources/HostApp/Assets.xcassets/AppIcon.appiconset`의 macOS AppIcon PNG 10개 교체
- 신규 PNG 해상도와 `Contents.json` 슬롯의 size/scale 정합성 확인
- About 화면이 신규 앱 아이콘을 로고처럼 표시하는 구조 확인
- 빌드 산출물에 신규 앱 아이콘이 포함되는지 검증

### 제외

- 앱 이름, bundle identifier, 표시명 변경
- Quick Look preview / Thumbnail extension 렌더링 로직 변경
- 문서 렌더링, viewer 동작, RustBridge 변경
- 릴리스 서명, 공증, DMG, Homebrew Cask 작업
- README 문서용 `assets/logo-256@2x.png`, `assets/home_banner.png` 교체. 단, 구현계획서 승인 시 작업지시자가 명시하면 별도 단계로 추가 판단한다.

## 설계 방향

- `project.yml`은 이미 `ASSETCATALOG_COMPILER_APPICON_NAME: AppIcon`을 지정하므로 프로젝트 설정은 변경하지 않는다.
- `Contents.json`의 파일명은 유지하고 PNG 파일 내용만 교체해 Xcode asset catalog 참조 변경 위험을 줄인다.
- 입력 PNG와 AppIcon 슬롯은 다음 대응을 기본으로 한다.
  - `icon_16x16.png` ← `Icon-iOS-Default-16x16@1x.png`
  - `icon_16x16@2x.png` ← `Icon-iOS-Default-16x16@2x.png`
  - `icon_32x32.png` ← `Icon-iOS-Default-32x32@1x.png`
  - `icon_32x32@2x.png` ← `Icon-iOS-Default-32x32@2x.png`
  - `icon_128x128.png` ← `Icon-iOS-Default-128x128@1x.png`
  - `icon_128x128@2x.png` ← `Icon-iOS-Default-128x128@2x.png`
  - `icon_256x256.png` ← `Icon-iOS-Default-256x256@1x.png`
  - `icon_256x256@2x.png` ← `Icon-iOS-Default-256x256@2x.png`
  - `icon_512x512.png` ← `Icon-iOS-Default-512x512@1x.png`
  - `icon_512x512@2x.png` ← `Icon-iOS-Default-1024x1024@1x.png`
- About 화면은 앱 아이콘을 직접 참조하므로 Swift 코드 변경 없이 정합성을 확인한다. 코드 변경이 필요해지는 경우 즉시 보고하고 구현계획서에서 범위를 보정한다.

## 잠정 단계

총 3단계로 진행하고 각 단계 완료 후 작업지시자 승인을 받는다.

- **Stage 1 — 입력 자산과 현재 AppIcon 정합성 확인**
  - 외부 PNG 세트의 파일 존재, 해상도, AppIcon 슬롯 대응표를 검증한다.
  - `Contents.json`, `project.yml`, About 화면의 아이콘 사용 경로를 재확인한다.
- **Stage 2 — AppIcon PNG 교체**
  - 기존 AppIcon 파일명은 유지하고 PNG 내용만 신규 파일로 교체한다.
  - 교체 후 각 파일 해상도와 `Contents.json` 슬롯 정합성을 다시 확인한다.
- **Stage 3 — 빌드 산출물 검증과 보고**
  - XcodeGen 필요 여부를 확인하고 HostApp 빌드를 수행한다.
  - 앱 번들의 asset catalog 및 About 화면 아이콘 경로가 신규 AppIcon을 사용함을 확인한다.
  - 최종 검증 결과와 남은 리스크를 보고한다.

## 검증 계획

- `sips -g pixelWidth -g pixelHeight`로 외부 입력 PNG와 교체 후 AppIcon PNG 해상도 확인
- `git diff --check`
- `plutil -lint Sources/HostApp/Assets.xcassets/AppIcon.appiconset/Contents.json`
- `xcodegen generate`가 필요한 변경인지 확인. `project.yml`을 변경하지 않으면 생략
- 최소 빌드 검증: `xcodebuild -project AlhangeulMac.xcodeproj -scheme HostApp -configuration Debug -derivedDataPath build.noindex/DerivedData build`
- 필요 시 산출물 앱의 `Assets.car` 또는 앱 아이콘 표시를 추가 확인

## 리스크

- **입력 자산 명명 차이 (저)**: 외부 폴더 파일명은 iOS 기본 export 명칭이지만 해상도는 macOS AppIcon 슬롯과 대응 가능하다. Stage 1에서 실제 pixel size로 확인한다.
- **아이콘 캐시로 인한 수동 확인 혼선 (중)**: macOS Dock/Finder는 앱 아이콘 캐시를 사용할 수 있어 빌드 직후 화면 확인이 이전 아이콘처럼 보일 수 있다. 파일 해상도와 빌드 산출물 포함 여부를 우선 검증한다.
- **README 자산과 앱 자산 혼동 (저)**: `assets/logo-256@2x.png`는 README 문서 표시용으로 보이며 앱 번들에서 직접 참조하지 않는다. 본 작업에서는 앱 사용 자산만 교체한다.
- **배포용 서명 검증 미포함 (저)**: 이 작업은 v0.1.0 배포에 들어갈 아이콘 교체지만 서명/공증 자체는 제외 범위다. 배포 파이프라인 작업에서 별도 확인한다.
