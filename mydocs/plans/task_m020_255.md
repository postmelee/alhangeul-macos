# Task M020 #255 수행계획서

## 목적

`rhwp v0.7.11`의 `native-skia` PNG renderer를 macOS 앱 저장소의 RustBridge에서 안전하게 노출할 수 있도록 C ABI, build feature, provenance gate를 추가한다.

완료 후에는 Swift 계층이 후속 #256에서 Skia PNG backend를 선택적으로 호출할 수 있는 최소 ABI와 검증 기준을 갖춰야 한다. Quick Look/Thumbnail 제품 동작 변경은 이 이슈에서 직접 수행하지 않는다.

## 배경

현재 `RustBridge/Cargo.toml`은 `rhwp = { git = "https://github.com/edwardkim/rhwp.git", tag = "v0.7.11" }`만 선언하고 `native-skia` feature를 활성화하지 않는다. FFI 표면도 `rhwp_render_page_tree`, `rhwp_render_page_svg`, `rhwp_page_size`, `rhwp_image_data`, `rhwp_free_bytes` 중심이며, Swift가 Skia PNG bytes를 받을 ABI는 없다.

M020 #254에서는 Quick Look/Thumbnail Skia 도입을 바로 default로 전환하지 않고 `Skia opt-in + CoreGraphics fallback` 순서로 진행하기로 정리했다. 그 첫 단계가 #255이며, 이 작업은 Swift renderer abstraction(#256)보다 먼저 RustBridge ABI와 generated header, symbol lock, provenance 검증을 고정해야 한다.

## 범위

포함:

- `RustBridge/Cargo.toml`에서 `rhwp/native-skia` feature 활성화 방식 확정
- `RustBridge/src/lib.rs`에 Skia PNG rendering C ABI 추가
- page index, scale, max-dimension, failure를 표현하는 ABI contract 정의
- PNG byte buffer 소유권과 `rhwp_free_bytes` 해제 규칙 정합성 확인
- #255 이슈 본문의 failure taxonomy와 ABI 오류 결과 표 반영
- `rhwp-ffi-symbols.txt`, cbindgen generated header, `Rhwp.xcframework`, `rhwp-core.lock` 정합성 갱신
- staticlib/header/package 크기 변화 기록
- Rust bridge build, symbol check, lock verification 결과 기록

제외:

- `Sources/RhwpCoreBridge/RhwpDocument.swift` Swift wrapper 추가는 #256에서 처리한다.
- `Sources/Shared/HwpPageImageRenderer.swift` backend abstraction 추가는 #256에서 처리한다.
- Quick Look preview 적용은 #257에서 처리한다.
- Finder thumbnail 적용은 #258에서 처리한다.
- Skia default 전환 및 release readiness 판단은 #259에서 처리한다.
- upstream `edwardkim/rhwp` API 수정은 포함하지 않는다.
- HostApp `rhwp-studio` renderer 선택 변경은 포함하지 않는다.

## 설계 방향

Rust FFI 경계는 기존 패턴을 따른다. 입력 포인터와 output 포인터는 null을 먼저 검사하고, panic은 `catch_unwind`로 격리하며, 실패 시 Swift가 CoreGraphics fallback을 선택할 수 있도록 buffer 없는 실패 결과를 반환한다.

초기 ABI는 Swift에서 안정적으로 호출할 수 있는 단순한 형태를 우선한다. PNG bytes는 Rust가 `Vec<u8>` 또는 boxed slice 소유권을 caller에게 넘기고, Swift 쪽은 기존 `rhwp_free_bytes(ptr, len)`로 해제하는 구조를 유지한다.

오류는 Swift wrapper가 의미 있는 `fallbackReason`으로 매핑할 수 있게 명확히 나눈다. #255에서 최소 고정할 명칭은 `ffiUnavailable`, `invalidDocumentHandle`, `invalidPageIndex`, `invalidPageSize`, `skiaRenderFailure`이며, PNG decode 실패는 Rust가 아니라 #256 Swift decode 단계에서 다룬다.

build/provenance gate는 `rhwp-core.lock`의 release tag와 resolved commit을 유지하면서 generated header, FFI symbol set, staticlib size/hash metadata를 갱신하는 방향으로 설계한다. staticlib byte hash는 환경 차이에 민감하므로 기존 `build-rust-macos.sh`의 strict/local 검증과 CI skip 정책을 유지한다.

## 잠정 단계

1. Stage 1: ABI와 upstream API inventory
   - `rhwp v0.7.11`의 `native-skia` API signature와 option 구조를 현재 lock 기준으로 확인한다.
   - 현재 RustBridge FFI 패턴, generated header, symbol lock, lock verification 흐름을 정리한다.
   - 구현계획서(`task_m020_255_impl.md`)에 ABI 후보와 검증 순서를 고정한다.
2. Stage 2: RustBridge `native-skia` feature와 PNG FFI 구현
   - `Cargo.toml` feature 설정과 `Cargo.lock` 변경을 적용한다.
   - Skia PNG render ABI, output buffer, error/failure contract를 구현한다.
   - null pointer, invalid option, render failure 경로를 기존 FFI guard 패턴에 맞춘다.
3. Stage 3: generated artifacts와 provenance gate 갱신
   - `build-rust-macos.sh`로 header, symbol set, universal staticlib, xcframework를 재생성한다.
   - `rhwp-ffi-symbols.txt`와 `rhwp-core.lock`을 의도된 ABI/artifact 변경으로 갱신한다.
   - staticlib/header size 변화와 lock verification 결과를 기록한다.
4. Stage 4: Rust bridge 검증과 실패 경로 점검
   - Rust build, generated symbol diff, lock verification을 실행한다.
   - 가능하면 대표 샘플 1개로 ABI smoke를 수행하고, Swift 적용 전 남은 제약을 정리한다.
5. Stage 5: 최종 보고서와 PR 준비
   - stage 산출물과 검증 결과를 보고서로 정리한다.
   - 오늘할일 완료 처리, 최종 커밋, PR 생성 준비를 수행한다.

## 리스크

- `native-skia` feature 활성화로 Rust dependency graph, build time, staticlib/package 크기가 크게 증가할 수 있다.
- upstream `render_page_png_native*` API가 현재 lock의 실제 signature와 문서화된 기대가 다르면 ABI 설계를 조정해야 한다.
- PNG render options의 scale/max-dimension 해석이 Quick Look과 Thumbnail 요구를 동시에 만족하지 못할 수 있다.
- FFI에서 오류 코드를 너무 단순화하면 #256 Swift fallback 진단이 부족해질 수 있다.
- generated `Rhwp.xcframework`와 staticlib는 큰 binary artifact라 diff와 PR 크기가 커질 수 있다.
- `librhwp.a` hash는 toolchain/build path 영향을 받을 수 있어 lock update와 verification 결과를 구분해 기록해야 한다.
- 네트워크 또는 Cargo cache 상태에 따라 `rhwp`/crate fetch가 실패할 수 있으며, 이 경우 compatibility 실패와 dependency fetch 실패를 분리해 보고해야 한다.

## 승인 요청 사항

1. #255를 RustBridge `native-skia` PNG FFI와 build/provenance gate 추가 작업으로 진행하는 범위 승인
2. Swift wrapper, Shared renderer, Quick Look/Thumbnail 제품 적용은 각각 #256-#258로 넘기는 경계 승인
3. 다음 단계: 승인 후 구현계획서(`mydocs/plans/task_m020_255_impl.md`) 작성과 Stage 1 ABI/upstream inventory 진행

승인 전에는 RustBridge source, generated artifact, lock 파일 변경을 진행하지 않는다.
