# Task M018 #184 수행계획서

## 목적

`v0.1.1` public DMG를 여는 사용자가 DMG 창만 보고도 `Alhangeul.app`을 `/Applications`로 드래그해 설치해야 한다는 점과, 설치 후 앱을 한 번 실행해야 Quick Look preview와 Finder thumbnail extension이 등록될 수 있다는 점을 바로 이해하도록 DMG 설치 창 안내를 개선한다.

완료 후에는 rehearsal DMG와 public release DMG가 같은 layout 생성 경로를 사용해야 하며, #188의 실제 `v0.1.1` 배포 실행 전 smoke 기준으로 재검증할 수 있어야 한다.

## 배경

현재 `scripts/release.sh`의 `create_dmg`는 staging directory에 `Alhangeul.app`과 `/Applications` symlink를 넣고 `hdiutil create -srcfolder`로 바로 UDZO DMG를 만든다. 이 방식은 단순하고 안정적이지만 Finder 창 배경, 아이콘 위치, 드래그 방향, 첫 실행 안내를 사용자에게 보여주지 않는다.

#184는 `v0.1.1` 마일스톤 실행 순서 2/6 단계다. 선행 이슈 #183은 GitHub 기준 닫힌 상태이며, 이번 작업은 release artifact의 사용자 설치 UX와 smoke 기준을 보강한 뒤 #185 release note/template 작업과 #188 public 배포 실행으로 넘기는 역할을 한다.

## 범위(포함·제외)

포함:

- DMG background image, 안내 이미지, README 파일, Finder window layout 중 유지보수가 쉬운 방식 선정
- `Alhangeul.app` 아이콘, `Applications` symlink, 설치 방향 안내, 첫 실행 안내 문구 배치
- 한국어 우선 안내와 필요한 짧은 영어 병기 검토
- rehearsal DMG와 public release DMG가 같은 layout 생성 함수를 통과하도록 구성
- mounted DMG에서 Finder 창 크기, 아이콘 위치, 배경/안내 문구 표시 smoke
- `mydocs/manual/release_distribution_guide.md`의 DMG layout smoke 기준 보강

- Homebrew tap 배포 경로 확정

## 설계 방향

먼저 현재 `create_dmg`가 단순 `-srcfolder` 압축 이미지 생성에 머무르는 구조를 확인하고, macOS 기본 도구만으로 유지보수 가능한 layout 방식을 고른다. 새 외부 의존성은 release machine과 CI/rehearsal 환경의 변수를 늘리므로 우선 피하고, `hdiutil`, Finder metadata, AppleScript 또는 생성된 정적 asset 조합으로 해결 가능한지 검토한다.

선호 방향은 DMG 안에 `.background` asset을 두고 Finder 창 크기와 icon 위치를 명시적으로 지정해 `Alhangeul.app -> Applications` 드래그 흐름을 시각화하는 것이다. 이 방식이 sandbox/headless 환경에서 불안정하면, DMG 내부 README/안내 이미지와 안정적인 icon 배치의 조합으로 낮춘다. 어떤 방식을 택하든 public mode와 `--skip-notarize` rehearsal mode가 같은 `create_dmg` 경로를 쓰게 해 두 산출물의 layout 차이를 줄인다.

안내 문구는 DMG 창 안에서 짧게 유지한다. 핵심 문장은 "Alhangeul.app을 Applications로 드래그해 설치"와 "설치 후 앱을 한 번 실행하면 Quick Look/Thumbnail이 활성화"로 둔다. 영어 병기는 Finder/Applications 같은 시스템 용어와 다운로드 사용자 혼선을 줄이는 범위에서만 짧게 검토한다.

## 잠정 단계(3~6단계)

1. Stage 1: DMG layout 방식 조사와 기준 확정
   - 현재 `scripts/release.sh`의 DMG 생성 흐름과 mount 결과를 확인한다.
   - background image, README, Finder window metadata, AppleScript 방식의 유지보수성과 release 환경 리스크를 비교한다.
   - 선택한 layout 방식, 안내 문구 초안, 창 크기/아이콘 위치 기준을 확정한다.
2. Stage 2: DMG layout asset과 release script 구현
   - 선택한 방식에 맞춰 DMG staging, background/안내 asset, Finder metadata 설정 로직을 구현한다.
   - rehearsal/public mode가 같은 layout path를 사용하도록 중복을 줄인다.
   - public release credential, signing, notarization, checksum 흐름과 충돌하지 않도록 기존 단계 순서를 유지한다.
3. Stage 3: rehearsal DMG 생성과 mounted layout smoke
   - `--skip-notarize` rehearsal DMG를 생성하고 mount한다.
   - DMG 창에서 앱, Applications target, 설치 안내, 첫 실행 안내가 보이는지 확인한다.
   - checksum, `hdiutil verify`, bundle 구조, 기존 release script fail-fast 동작을 확인한다.
4. Stage 4: public release 호환성과 배포 가이드 보강
   - 실제 `v0.1.1` public release 실행은 하지 않고, public mode와 공유되는 코드 경로 및 credential guard를 검토한다.
   - `release_distribution_guide.md`에 DMG layout smoke 기준과 #188에서 반복할 public DMG 확인 항목을 추가한다.
   - 안내 문구가 release note/README 설치 안내와 모순되지 않는지 점검한다.
5. Stage 5: 최종 검증과 #188 handoff 정리
   - 최종 검증 명령과 수동 smoke 결과를 정리한다.
   - #188에서 signed/notarized public DMG 생성 후 반드시 다시 확인할 항목을 최종 보고서에 남긴다.

## 검증 계획

계획 문서 단계:

```bash
git diff --check -- mydocs/orders/20260509.md mydocs/plans/task_m018_184.md
```

구현 단계 진입 후 기본 검증:

```bash
git status --short --branch
bash -n scripts/release.sh
./scripts/release.sh --help
env -u ALHANGEUL_DEVELOPER_ID_APPLICATION -u ALHANGEUL_NOTARY_PROFILE ./scripts/release.sh 0.1.1
```

rehearsal DMG 후보 검증:

```bash
./scripts/release.sh --skip-notarize 0.1.1
hdiutil verify build.noindex/release/alhangeul-macos-0.1.1-rehearsal.dmg
hdiutil attach build.noindex/release/alhangeul-macos-0.1.1-rehearsal.dmg
shasum -a 256 -c build.noindex/release/alhangeul-macos-0.1.1-rehearsal.dmg.sha256
```

수동 smoke 후보:

- mounted DMG Finder 창에서 `Alhangeul.app`과 `Applications` target이 보이는지 확인
- 앱을 Applications로 드래그하라는 안내가 DMG 창 안에서 보이는지 확인
- 설치 후 앱을 한 번 실행해야 Quick Look/Thumbnail이 활성화된다는 안내가 보이는지 확인
- 창 크기와 아이콘 위치가 작은 화면에서도 겹치지 않는지 확인

최종 단계 후보 검증:

```bash
git diff --check
git status --short --branch
```

실제 signed/notarized public DMG 생성, staple, Gatekeeper 검증은 #188의 명시 승인 범위에서 반복한다.

## 리스크

- Finder window layout metadata 설정은 macOS Finder/AppleScript 실행 가능 여부에 영향을 받을 수 있다.
- headless 또는 sandbox 환경에서 Finder 기반 layout 설정이 불안정하면 더 단순한 README/안내 이미지 방식으로 낮춰야 할 수 있다.
- DMG background asset을 새로 만들 경우 retina/non-retina 표시 품질과 파일 크기를 함께 확인해야 한다.
- 한글 안내 문구가 background image에만 들어가면 접근성·복사 가능성이 낮아질 수 있다.
- public mode에서 signing/notarization 전후로 DMG metadata가 보존되는지 #188에서 재확인해야 한다.
- `v0.1.1` plist version이 아직 준비되지 않았다면 rehearsal 명령의 version 값은 현재 소스 version에 맞춰 조정해야 한다.
