# Issue #24 수행 계획서

## 목적

`edwardkim/rhwp` 저장소의 PR 템플릿과 실제 PR 본문 사례를 참고해, 이 저장소의 draft PR 본문 형식을 규격화한다.

- `.github/pull_request_template.md`를 이 저장소 규칙에 맞게 작성한다.
- PR 본문이 단순 템플릿 복붙이 아니라 `계획서 -> 단계별 보고 -> 최종 보고서` 흐름과 연결되도록 정리한다.
- `mydocs/manual/pr_process_guide.md`에 PR 본문이 해당 템플릿을 따른다고 명시한다.
- `gh pr create`를 사용할 때도 같은 형식을 재사용할 수 있도록 운영 규칙을 남긴다.

## 배경

현재 저장소는 `publish/task{N}` 기반 draft PR 워크플로우를 이미 갖추고 있지만, PR 본문 형식은 아직 작업마다 수동으로 작성되고 있다.

비교 대상으로 확인한 `edwardkim/rhwp` 저장소는 다음 특징이 있다.

1. 기본 템플릿은 얇고 단순하다.
   - `변경 요약`
   - `관련 이슈`
   - `테스트`
   - `스크린샷`
2. 실제 좋은 PR 본문은 템플릿보다 더 구체적이다.
   - `요약`
   - `원인` 또는 `아키텍처 차이`
   - `변경 내역`
   - `파일 변경 목록`
   - 상세 `검증`
3. 즉, 좋은 PR 품질은 템플릿 파일 하나보다도 템플릿 + 팀 규약 + 실제 작성 습관에서 나온다.

이 저장소는 하이퍼-워터폴 산출물(`plans`, `working`, `report`)이 이미 존재하므로, PR 본문도 그 구조를 반영하는 쪽이 맞다.

## 범위

### 포함

- `.github/pull_request_template.md` 추가
- `mydocs/manual/pr_process_guide.md`에 PR 본문 규약 반영
- `gh pr create` 시 body-file 또는 템플릿 기반 재사용 규칙 문서화
- 필요 시 예시 PR 본문 구조를 가이드에 포함

### 제외

- 과거 모든 PR 본문의 일괄 수정
- GitHub Actions를 이용한 PR 본문 lint 자동화
- release PR, hotfix PR 등 다중 템플릿의 즉시 도입

## 목표 구조

이번 작업의 목표는 다음과 같다.

1. GitHub가 자동으로 채워주는 기본 PR 템플릿이 존재한다.
2. 이 템플릿은 `rhwp` 템플릿처럼 단순한 시작점을 유지하되, 우리 저장소에 필요한 핵심 섹션을 포함한다.
3. `mydocs/manual/pr_process_guide.md`는 아래 원칙을 명시한다.
   - PR 본문은 템플릿을 따른다.
   - 최종 보고서의 압축본으로 작성한다.
   - `관련`에는 `Closes #번호`를 명시한다.
   - 검증은 실제 실행한 명령만 적는다.

## 설계 방향

### 1. upstream 참고 원칙

`rhwp`의 `.github/pull_request_template.md`는 좋은 출발점이다. 다만 그대로 가져오면 이 저장소의 작업 흐름에는 부족하다.

이유:

- 현재 저장소는 stage 문서와 최종 보고서가 이미 존재한다.
- 따라서 PR은 단순 체크리스트보다 `무엇을 왜 바꿨는지`, `어떤 stage를 거쳤는지`, `무엇을 검증했는지`를 더 잘 드러내야 한다.

### 2. 우리 저장소용 기본 섹션

기본 템플릿에는 다음 섹션을 둔다.

- `## 요약`
- `## 변경 내역`
- `## 검증`
- `## 문서`
- `## 관련 이슈`

선택 섹션은 가이드에서 설명한다.

- `## 아키텍처 차이`
- `## 파일 변경 목록`
- `## 남은 리스크`
- `## 스크린샷`

### 3. 운영 문서와의 연결

PR 템플릿 파일만 두면 실제 품질은 들쭉날쭉할 수 있다. 그래서 `mydocs/manual/pr_process_guide.md`에 다음을 같이 적는다.

1. 어떤 작업에서 어떤 섹션을 반드시 채워야 하는지
2. stage가 있는 task는 `변경 내역`을 stage 기준으로 적는지 여부
3. `gh pr create --body-file` 사용 예시
4. 최종 보고서와 PR 본문이 어떤 관계인지

## 예상 효과

1. draft PR 본문 형식이 일관돼 self-review와 외부 리뷰가 쉬워진다.
2. PR만 봐도 stage 진행과 검증 결과를 빠르게 파악할 수 있다.
3. 이후 task에서도 `gh pr create` 시 재사용 가능한 형식이 생긴다.

## 리스크

1. 템플릿을 너무 무겁게 만들면 작은 작업에서 작성 부담이 커질 수 있다.
2. 너무 가볍게 만들면 `rhwp` 템플릿 복사 이상의 효과가 없어진다.
3. guide와 실제 PR 작성 방식이 어긋나면 문서만 남고 실사용이 흐트러질 수 있다.
