# Issue #29 수행 계획서

## 목적

`hwpql`의 `libs/rhwp.lock` 장점을 우리 저장소의 Rust bridge / `Rhwp.xcframework` 생성 흐름에 맞게 반영한다.

현재 `rhwp-core.lock`은 core 저장소, branch, commit, 예상 산출물 경로만 기록한다. 하지만 실제 `Frameworks/universal/librhwp.a`와 generated header가 그 commit에서 생성된 산출물인지 sha256/size 기준으로 검증하지 못한다.

이번 작업의 목적은 `rhwp-core.lock`을 산출물 provenance lock으로 확장하고, build/package 단계에서 lock 불일치를 조기에 차단하는 것이다.

## 배경

`hwpql`은 다음 정보를 `libs/rhwp.lock`에 기록한다.

- core repo
- core commit
- built_at
- artifact path
- sha256
- size

그리고 release script가 정적 라이브러리 sha256을 lock과 비교해, 누군가 `.a` 파일만 교체하고 lock을 갱신하지 않은 상태를 차단한다.

우리 저장소는 다음 차이가 있다.

1. `Frameworks/`는 `.gitignore` 대상이므로 산출물을 repository에 커밋하지 않는다.
2. `Rhwp.xcframework`는 directory artifact라 파일 순서, metadata, nested path를 어떻게 hash할지 별도 정책이 필요하다.
3. `scripts/build-rust-macos.sh`는 이미 FFI symbol set 검증을 수행하지만, artifact provenance 검증은 하지 않는다.
4. `scripts/package-release.sh`는 zip sha256만 출력하고, Rust bridge 산출물 lock 검증은 하지 않는다.

따라서 `hwpql` 구조를 그대로 복사하지 않고, 우리 빌드 산출물 생성 방식에 맞는 lock update/verify 모드를 설계한다.

## 설계 방향

### 1. lock update와 verify를 분리

일반 `./scripts/build-rust-macos.sh` 실행은 기존처럼 산출물을 재생성하고 FFI symbol set을 검증한다. lock 파일은 명시적 옵션에서만 변경한다.

- `--update-lock`: 빌드 후 현재 산출물의 sha256/size/built_at을 `rhwp-core.lock`에 기록한다.
- `--verify-lock`: 빌드 후 현재 산출물과 `rhwp-core.lock` 기록을 비교한다.

이렇게 해야 일반 build가 임의로 lock 파일을 바꾸지 않는다.

### 2. 첫 단계 검증 대상은 file artifact로 제한

`Rhwp.xcframework` 전체 directory hash는 tar 생성 방식, file ordering, metadata 제외 정책을 먼저 정해야 한다. 이번 작업에서는 deterministic성이 명확한 file artifact를 우선 대상으로 삼는다.

- `Frameworks/universal/librhwp.a`
- `Frameworks/generated_rhwp.h`

필요하면 후속 작업에서 `Rhwp.xcframework` directory manifest hash를 추가한다.

### 3. staticlib deterministic 여부 확인

`librhwp.a`가 같은 commit과 같은 toolchain에서 반복 빌드 시 같은 sha256을 내는지 확인한다. 만약 불안정하다면 lock 정책은 release-candidate artifact 검증으로 제한한다.

### 4. package 전 검증

`scripts/package-release.sh`는 Rust bridge를 재생성한 뒤 `--verify-lock`을 수행하거나, build script를 verify mode로 호출한다. lock 불일치가 있으면 package 생성을 중단한다.

## 구현 계획서 작성 시 나눌 단계

구현 계획서는 승인 후 별도 문서로 작성한다. 현재 예상 단계는 다음과 같다.

1. 현재 build 산출물과 lock 구조 분석, deterministic 검증 방법 확정

4. `package-release.sh` lock verify 연동
5. 문서 갱신과 전체 검증

```bash
shasum -a 256 Frameworks/universal/librhwp.a
stat -f%z Frameworks/universal/librhwp.a
shasum -a 256 Frameworks/generated_rhwp.h
stat -f%z Frameworks/generated_rhwp.h
```

## 리스크

1. `librhwp.a`가 반복 빌드에서 byte-for-byte 동일하지 않을 수 있다.
2. `built_at`을 lock에 포함하면 일반 재생성 검증과 충돌할 수 있다.
3. `Frameworks/`가 gitignore 대상이므로 lock과 실제 artifact의 관계를 문서로 명확히 설명해야 한다.
4. release candidate artifact와 developer local artifact를 같은 정책으로 다루면 개발 경험이 나빠질 수 있다.
5. shell script에 TOML parser를 과하게 구현하면 유지보수가 어려워질 수 있다.
