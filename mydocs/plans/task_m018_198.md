# Task M018 #198 수행계획서

## 목적

Pull Request 생성/갱신 시 항상 실행되는 기본 CI gate를 마련하고, 기존 release rehearsal/publish workflow를 #185에서 정리한 release delta 기반 검증 체크리스트와 연결한다.

완료 후에는 문서 전용 변경, 앱/Swift 변경, RustBridge/core lock 변경, release workflow/script 변경이 구분되어 불필요한 macOS build 비용을 줄이면서도 PR 병합 전 필요한 최소 검증이 자동으로 실행되어야 한다. release publish는 signing, notarization, GitHub Release 게시, Sparkle appcast 갱신이 포함되므로 계속 보호된 수동 실행으로 유지한다.

## 배경

현재 저장소에는 release 관련 workflow가 있다.

- `.github/workflows/release-rehearsal.yml`
- `.github/workflows/release-publish.yml`
- `.github/workflows/rhwp-upstream-check.yml`

반면 PR 생성 시 공통으로 실행되는 기본 CI workflow는 아직 정리되어 있지 않다. 개별 PR에서는 빠르게 실패를 발견해야 하고, release CI는 여러 PR이 합쳐진 release candidate와 실제 배포 산출물의 사용자 경로를 확인해야 한다.

#198은 M018 `v0.1.1` 실행 순서에서 #185 다음 작업이다. #185가 release note template, delta checklist helper, release manual 분리를 정리했으므로, 이번 작업은 그 기준을 GitHub Actions와 문서화된 CI 운영 기준에 연결한다.

## 범위(포함·제외)

포함:

- PR 생성/갱신 시 실행되는 기본 CI workflow 추가
- 대상 브랜치: `main`, `devel-webview`, `devel`
- `concurrency`로 같은 PR의 이전 run 취소
- PR 변경 파일을 기준으로 docs-only, app/Swift, RustBridge/core lock, renderer/fixture, release script/workflow 변경 범위 분류
- 기본 script syntax check와 release helper syntax check
- `./scripts/check-no-appkit.sh`, `xcodegen generate`, HostApp Debug build 기준 정리
- RustBridge/core lock 관련 변경 시 `./scripts/build-rust-macos.sh --verify-lock` 실행
- renderer/fixture 관련 변경 시 `./scripts/validate-stage3-render.sh` 실행
- release script/workflow 변경 시 `scripts/release.sh --help`, release note/appcast/delta helper syntax/preflight 확인
- docs-only PR에서 macOS build를 생략하거나 생략 사유를 summary에 명확히 남김
- 기존 `release-rehearsal.yml`, `release-publish.yml`에 release delta checklist 생성/summary 연결 보강
- CI 역할과 release CI 경계를 `mydocs/manual/`에 문서화

제외:

- GitHub Release 게시 실행
- release secrets 또는 Apple credential 생성/교체
- release publish를 tag push만으로 자동 실행하는 정책
- Homebrew tap push 자동화
- 모든 PR을 release 시점에 처음부터 재검증하는 대형 통합 테스트 매트릭스
- 외부 PR에서 secrets가 필요한 job 실행

## 설계 방향

PR CI와 release CI를 같은 workflow로 합치지 않는다. PR CI는 빠른 피드백과 병합 전 기본 안전성 확인을 담당하고, release CI는 release candidate와 public artifact 검증을 담당한다.

PR CI는 `pull_request` 이벤트에서만 동작한다. `pull_request_target`은 사용하지 않는다. 외부 PR에서는 repository secrets에 접근하지 않고, signing/notarization/GitHub Release 게시와 관련된 job을 실행하지 않는다.

변경 범위 분류는 GitHub Actions workflow 안에서 `git diff --name-only`와 작은 shell helper로 처리하는 방향을 우선한다. 외부 action 의존을 추가하기보다 저장소 안의 `scripts/ci/` helper로 분류 기준을 version-control하는 편이 이 저장소의 하이퍼-워터폴 문서화 흐름과 맞다.

docs-only PR은 workflow 자체는 실행하되 비싼 macOS build job을 생략하고, 어떤 기준으로 생략됐는지 `GITHUB_STEP_SUMMARY`에 남긴다. 앱/Swift/bridge/release 변경이 있으면 macOS runner에서 `xcodegen generate`, `HostApp` Debug build, 필요한 조건부 검증을 수행한다.

release rehearsal/publish는 기존 manual dispatch 정책을 유지한다. 다만 #185에서 만든 `scripts/ci/write-release-delta-checklist.sh`를 활용해 직전 public release tag부터 candidate ref까지의 변경 영향 영역 초안을 생성하고, workflow summary 또는 artifact로 남긴다. release publish는 계속 `workflow_dispatch` + `environment: release` 보호 규칙과 tag `v<version>` 검증을 유지한다.

## 잠정 단계

1. Stage 1: 기존 workflow와 검증 비용 경계 조사
   - release rehearsal/publish/upstream workflow의 현재 trigger, permission, secrets 사용, summary, artifact 경로를 확인한다.
   - PR에서 항상 실행할 검증과 조건부 실행할 검증을 분류한다.
   - docs-only/app/RustBridge/renderer/release 변경 파일 기준을 확정한다.
2. Stage 2: PR CI workflow와 변경 범위 분류 구현
   - `pull_request` 대상 브랜치와 concurrency를 설정한다.
   - 변경 파일 분류 output을 만들고 docs-only 여부를 summary에 남긴다.
   - shell syntax, no-AppKit, xcodegen, HostApp Debug build, 조건부 Rust/render/release helper 검증을 연결한다.
3. Stage 3: release rehearsal/publish workflow에 delta checklist 연결
   - `previous_release_ref` 또는 동등한 입력을 검토한다.
   - rehearsal/publish 전에 release delta checklist를 생성하고 summary/artifact로 남긴다.
   - release publish의 보호된 수동 실행, tag 검증, signing/notarization, checksum, GitHub Release asset, appcast 경로가 유지되는지 확인한다.
4. Stage 4: CI 역할과 경계 문서화
   - PR CI, release rehearsal, release publish, rhwp upstream check의 역할을 문서화한다.
   - docs-only 생략 기준과 수동 재현 명령을 매뉴얼에 반영한다.
   - #188 handoff에서 확인할 CI summary와 release delta 출력 위치를 정리한다.
5. Stage 5: 최종 dry-run, 보고, PR 준비
   - workflow YAML parse, shell syntax, helper dry-run, 기존 release workflow 경로 보존 여부를 검증한다.
   - 최종 보고서에 실행한 검증, 미실행 GitHub-hosted CI 한계, #188 handoff를 기록한다.

GitHub Actions 실제 실행 확인은 PR 게시 후 GitHub-hosted runner에서 별도로 확인한다. 로컬에서는 workflow syntax, helper syntax, dry-run, 문서 대조까지만 수행한다.

## 리스크

- macOS runner 비용을 줄이려다 필요한 앱/bridge 검증이 빠질 수 있다.
- 반대로 path filter가 너무 보수적이면 docs-only PR에서도 비싼 build가 자주 실행될 수 있다.
- GitHub Actions expression과 shell helper 사이의 변경 범위 기준이 어긋나면 summary와 실제 실행 job이 달라질 수 있다.
- 외부 PR에서 secrets가 필요한 workflow를 잘못 열면 보안 리스크가 생긴다.
- release publish workflow는 실제 signing/notarization과 GitHub Release 게시를 포함하므로 로컬 검증으로 전체 동작을 대체할 수 없다.
- #185의 delta checklist는 path 기반 초안이므로 release owner 보정이 계속 필요하다.

## 승인 요청 사항

1. 위 범위와 5단계 진행 구조 승인
2. PR CI는 `pull_request` 이벤트만 사용하고 `pull_request_target`을 사용하지 않는 방향 승인
3. docs-only PR은 macOS build를 생략하되 생략 사유를 summary에 남기는 방향 승인
4. release publish는 계속 `workflow_dispatch` + `environment: release` 보호 규칙과 tag 검증을 유지하는 방향 승인
5. 다음 단계: 승인 후 구현계획서(`mydocs/plans/task_m018_198_impl.md`) 작성
