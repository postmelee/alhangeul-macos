## 목적

HostApp WKWebView viewer 영역에 HWP/HWPX 파일을 끌어넣어 연 뒤에도 macOS titlebar toolbar의 문서 동작 상태가 실제 viewer 상태와 일치하도록 보정한다.

완료 후 드래그앤드롭으로 문서가 로드된 상태에서 `공유`, `PDF로 내보내기`는 문서가 있는 상태로 활성화되어야 한다. 원본 파일 경로를 안전하게 확보할 수 있는 경우 `Finder에서 보기`도 활성화되어야 하며, 경로를 확보할 수 없는 로드 경로라면 비활성 또는 명확한 fallback 정책을 유지해야 한다.

## 배경

Release 앱 수동 테스트 중, 앱을 실행한 뒤 WKWebView viewer 영역에 파일을 끌어넣어 열면 문서는 화면에 표시되지만 titlebar toolbar의 `공유`, `Finder에서 보기`, `PDF로 내보내기`가 계속 비활성화되는 문제가 확인됐다.

현재 toolbar 활성 조건은 Swift `DocumentViewerStore`의 `hasDocument`와 `canRevealInFinder`를 기준으로 판단한다. 기존 native open panel, Finder open request, 최근 문서 경로는 `DocumentViewerStore.loadDocument(from:)`으로 들어와 `rhwpStudioDocument`와 `sourceDocument`를 갱신한다. 반면 WKWebView 내부 drag/drop은 bundled `rhwp-studio` JavaScript가 `File.arrayBuffer()`로 파일 바이트를 읽고 Web/WASM 문서 상태만 갱신한다. 이 경로는 Swift store를 통과하지 않으므로 native toolbar 기준으로는 문서가 없는 상태로 남는다.

## 범위

포함:

- WKWebView 내부 drag/drop과 HostApp native document state 사이의 ownership 재확인
- 드래그앤드롭 로드 후 `DocumentViewerStore` 또는 동등한 native 상태가 갱신되도록 bridge 경로 보정
- `공유`, `PDF로 내보내기`, `Finder에서 보기` 활성 조건 정리
- native open panel, Finder open request, 최근 문서 재열기 경로의 기존 동작 회귀 확인
- HWP 샘플 기준 app smoke 검증

제외:

- upstream `rhwp-studio` 저장소 수정
- `rhwp-studio` 전체 drag/drop UX 재설계
- PDF export 품질 자체 개선
- 공유 picker 위치 또는 표시 방식 재설계
- Quick Look/Thumbnail extension 변경
- native renderer 품질 개선

## 설계 방향

가장 우선 검토할 방향은 WKWebView drag/drop을 HostApp bridge가 인지할 수 있게 해, 파일명과 파일 바이트를 Swift `DocumentViewerStore`에 전달하는 것이다. 이 경우 문서 로드의 native truth source를 기존 `RhwpStudioDocumentPayload`로 유지할 수 있고, toolbar 활성 조건도 현재 구조를 크게 바꾸지 않아도 된다.

드롭 파일의 실제 filesystem URL을 안전하게 확보할 수 있는지 별도로 확인한다. JavaScript `File` 객체만으로는 원본 경로를 얻지 못할 수 있으므로, `Finder에서 보기`는 원본 URL을 얻은 경로에서만 활성화하는 정책을 유지할 수 있다. 만약 AppKit/WKWebView 쪽에서 `NSDraggingInfo` 기반 URL을 안정적으로 받을 수 있다면, 해당 URL을 `RecentDocumentItem`으로 기록해 최근 문서와 Finder reveal까지 연결한다.

`공유`와 `PDF로 내보내기`는 현재 viewer의 export bridge를 사용하므로 원본 URL이 없어도 문서가 로드되어 있으면 동작 가능해야 한다. 따라서 Stage 1에서 원본 URL이 없는 문서 상태와 export 가능한 문서 상태를 분리할지, 또는 store에 source-less document 상태를 명시할지 결정한다.

`Sources/RhwpCoreBridge`와 Quick Look/Thumbnail 경로는 이번 작업의 소유 영역이 아니므로 변경하지 않는다.

수동 smoke:

- drag/drop 로드 후 `PDF로 내보내기` toolbar 항목이 활성화되고 save panel 또는 export 경로에 진입하는지 확인

- native open panel 또는 `/usr/bin/open -a` 경로로 문서를 열었을 때 기존 toolbar 기능과 최근 문서 기록이 유지되는지 확인

## 리스크

- WKWebView JavaScript drag/drop의 `File` 객체는 원본 filesystem URL을 제공하지 않을 수 있다. 이 경우 `Finder에서 보기`까지 활성화하려면 별도 AppKit drop handling이 필요할 수 있다.
- WebView 내부 로드 상태와 Swift store 상태를 이중으로 만들면 문서 revision, reload timing, export 대상이 어긋날 수 있다.
- `rhwp-studio` bundled asset을 직접 수정하면 upstream sync 시 덮어써질 수 있으므로 HostApp injection 또는 native wrapper 쪽 변경을 우선해야 한다.
- toolbar validation은 AppKit `NSToolbarItemValidation` 시점에 따라 즉시 갱신되지 않을 수 있으므로 state 변경 후 validation refresh가 필요할 수 있다.
- HWPX 문서는 upstream 베타 정책으로 저장 제한이 있으므로 drag/drop 상태 보정이 저장 정책을 우회하지 않아야 한다.
