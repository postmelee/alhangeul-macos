# Task M019 #204 수행계획서

## 목적

`edwardkim/rhwp` upstream release를 사람이 매번 확인하지 않아도 감지하고, bundled `rhwp-studio` 또는 WASM/core viewer 영향 변경이 있을 때 유지보수자가 검토할 수 있는 자동 업데이트 PR 후보를 만드는 파이프라인을 추가한다.

완료 후에는 새 upstream release가 없을 때는 read-only summary만 남고, 새 release가 있으며 viewer 영향 변경이 있으면 현재 통합 브랜치 정책에 맞는 PR이 생성되어야 한다. Public release, signed/notarized DMG, Sparkle stable appcast, Homebrew 배포는 기존 보호된 수동 승인 흐름으로 남긴다.

## 배경

현재 HostApp viewer/editor는 `Sources/HostApp/Resources/rhwp-studio/manifest.json`에 기록된 `edwardkim/rhwp` release tag와 resolved commit의 `rhwp-studio` build artifact를 WKWebView에서 실행한다. Core provenance는 `rhwp-core.lock`, bundled studio asset provenance는 `manifest.json`, local sync 절차는 `scripts/sync-rhwp-studio.sh`, asset 검증은 `scripts/verify-rhwp-studio-assets.sh`가 담당한다.

현재 `.github/workflows/rhwp-upstream-check.yml`와 `scripts/ci/check-rhwp-upstream-release.sh`는 upstream latest release와 `rhwp-core.lock`을 비교하는 read-only 감시 역할을 한다. 반면 #204의 목표는 release 감지 뒤 자동 branch/PR 생성까지 확장하는 것이다. 이 단계는 repository write 권한과 PR 생성 권한이 필요하므로 read-only check와 write-capable sync PR 생성 workflow를 분리하는 방향이 안전하다.

이슈 본문에는 자동 PR base로 `devel-webview`가 적혀 있으나, 현재 `README.md`, `mydocs/manual/git_workflow_guide.md`, `mydocs/manual/ci_workflow_guide.md`에서는 `devel-webview`를 퇴역한 legacy alias로 규정하고 신규 PR base나 자동화 기준으로 쓰지 않는다. 따라서 이번 계획의 기본 PR base는 `devel`로 둔다. 다른 base가 필요하면 구현계획서 승인 전에 별도 작업지시가 필요하다.

## 범위(포함·제외)

포함:

- upstream `edwardkim/rhwp` latest release를 scheduled workflow와 수동 dispatch에서 조회하는 흐름 정리
- 현재 bundled `rhwp-studio` manifest의 `source_release_tag`/`source_resolved_commit`과 upstream target release 비교
- viewer 영향 변경 감지 기준 설계와 helper 추가 또는 확장
- `scripts/sync-rhwp-studio.sh`의 tag/commit/upstream-dir 인자화와 검증 모드 검토
- viewer 영향 변경이 있을 때 자동 업데이트 branch와 PR을 생성하는 write-capable workflow 추가
- 자동 PR body에 previous/new tag, commit, upstream release URL, 변경 감지 결과, 검증 결과, maintainer 확인 항목, public release 분리 문구 기록
- 자동 PR 생성 시 maintainer 알림을 위해 assignee, reviewer request, body mention 중 하나 이상 설정
- PR CI 변경 범위 분류가 bundled studio/core/provenance 변경에서 필요한 검증을 켜는지 확인하고 필요한 보강 반영
- 관련 운영 매뉴얼 갱신

제외:

- `main` 대상 자동 PR 생성
- 퇴역한 `devel-webview` 대상 자동 PR 생성
- tag push만으로 public release 자동 실행
- signed/notarized DMG, GitHub Release, Sparkle stable appcast, Homebrew 배포의 무인 실행
- upstream `edwardkim/rhwp` source 수정
- `rhwp-studio` 자체 기능 수정
- 이번 작업 중 실제 upstream 새 release를 강제로 bundled asset에 반영하는 것

## 설계 방향

read-only 감시와 write-capable PR 생성을 분리한다. 기존 `rhwp Upstream Release Check`는 upstream release와 현재 lock/provenance 비교 summary를 남기는 경로로 유지하고, 자동 PR 생성은 별도 workflow 후보인 `.github/workflows/rhwp-upstream-sync-pr.yml`에서 처리한다. 이 workflow는 `schedule`과 `workflow_dispatch`를 지원하되, repository write와 PR 권한이 필요한 단계는 조건을 명확히 둔다.

자동 PR 생성의 기본 흐름은 다음 후보로 검토한다.

1. 최신 upstream release tag, release URL, resolved commit을 조회한다.
2. `Sources/HostApp/Resources/rhwp-studio/manifest.json`의 current tag/commit과 비교한다.
3. current commit과 target commit 사이의 upstream diff에서 viewer 영향 path를 판정한다.
4. 영향이 없으면 PR을 만들지 않고 skipped/current summary를 남긴다.
5. 영향이 있으면 `automation/rhwp-<tag>-studio-sync` 형식의 branch를 만들고 sync helper로 bundled studio asset 후보를 갱신한다.
6. 검증 helper를 실행하고 변경 요약과 검증 결과를 PR body에 기록한다.
7. base `devel` 대상 PR을 만들고 maintainer 알림 설정을 적용한다.

viewer 영향 path는 `rhwp-studio/**`만으로 제한하지 않는다. WASM/core viewer output에 영향을 줄 수 있는 upstream source, `pkg/rhwp.js`, `pkg/rhwp_bg.wasm`, package/Vite/TypeScript build input, fonts/license/provenance 파일까지 포함하는 보수적 기준을 구현계획서에서 확정한다.

외부 GitHub Action 추가는 최소화한다. 우선 GitHub-hosted runner의 `gh` CLI와 repository-local script를 사용하고, third-party action이 꼭 필요하면 필요성, 권한, pinning 기준을 구현계획서 승인 항목으로 분리한다.

## 잠정 단계(3~6단계)

1. 현황 확인과 base 정책 확정
   - 현재 upstream check, sync script, manifest, PR CI 분류 기준을 조사한다.
   - 이슈 본문의 `devel-webview` 기준을 현재 퇴역 정책과 대조하고 자동 PR base를 `devel`로 확정할지 승인받는다.
   - read-only check와 write-capable sync PR workflow 분리 원칙을 확정한다.
2. 구현계획서 작성
   - workflow 권한, trigger, branch naming, PR body, maintainer 알림 방식, viewer 영향 path, 실패 정책을 확정한다.
   - `sync-rhwp-studio.sh` 인자화와 신규 helper 경계를 정한다.
   - 승인 후 구현계획서(`task_m019_204_impl.md`)를 작성한다.
3. 감지 helper와 sync script 개선
   - upstream release/current manifest 비교 helper를 보강한다.
   - viewer 영향 path 감지 helper와 summary output을 추가한다.
   - `sync-rhwp-studio.sh`가 `--tag`, `--commit`, `--upstream-dir`, 검증 모드 후보를 처리하도록 개선한다.
4. 자동 PR 생성 workflow 추가
   - write-capable workflow를 추가하고 `contents: write`, `pull-requests: write` 등 필요한 최소 권한을 설정한다.
   - 새 release 없음, viewer 영향 없음, PR 생성 필요 상태를 분기한다.
   - 자동 branch/PR 생성, body 작성, assignee/reviewer/mention 설정을 구현한다.
5. PR CI와 운영 문서 갱신
   - 자동 생성 PR이 bundled studio/core/provenance 변경에 필요한 macOS build, Rust verify, release helper 검증을 실행하는지 분류 기준을 보강한다.
   - CI workflow, core dependency, release distribution 문서에 upstream sync PR 운영 기준을 기록한다.
6. 통합 검증과 보고
   - shell syntax, workflow YAML parse, helper help/dry-run, 변경 범위 분류, asset verify를 실행한다.
   - 실제 PR 생성은 GitHub-hosted runner와 repository 권한에 의존하므로 로컬 한계와 merge 후 확인 항목을 단계 보고서와 최종 보고서에 남긴다.

- viewer/WASM/core 영향 변경이 있으면 `devel` 대상 자동 PR 후보가 생성된다.

## 리스크

- 이슈 본문의 `devel-webview` 기준은 현재 브랜치 정책과 충돌한다. 이번 작업은 `devel` 기준으로 계획하되, 다른 base가 필요하면 구현 전 별도 승인과 문서 정리가 필요하다.
- GitHub Actions에서 branch push와 PR 생성 권한을 다루므로 workflow permission과 token scope를 과하게 열지 않도록 주의해야 한다.
- viewer 영향 path를 너무 좁게 잡으면 WASM/core 변경이 누락될 수 있고, 너무 넓게 잡으면 불필요한 PR이 생성될 수 있다.
- 자동 sync가 실제 upstream build artifact를 재현하려면 WASM build와 `rhwp-studio` build 전제 도구가 필요하다. GitHub-hosted runner 시간과 의존성 설치 비용을 확인해야 한다.
- 자동 PR이 기존 열린 sync PR과 중복될 수 있으므로 branch naming과 existing PR 탐색 기준이 필요하다.
- PR 생성/assignee/reviewer request는 로컬에서 완전히 재현하기 어렵고 GitHub-hosted runner 권한과 repository 설정 확인이 필요하다.

6. 승인 후 구현계획서(`mydocs/plans/task_m019_204_impl.md`) 작성
