## 목적

GitHub Pages 홍보 페이지의 세 번째 기능 설명 섹션을 현재의 긴 스크롤 기반 이미지 전환에서 기능 항목 hover/focus 기반 동영상 쇼케이스로 바꾼다.

완료 후 방문자는 각 기능 항목에 마우스를 올리거나 키보드 포커스를 이동했을 때 해당 기능 영상이 즉시 처음부터 재생되는 것을 볼 수 있어야 한다. 모바일과 좁은 화면에서는 기능별 동영상 카드를 좌우로 넘기는 scroll snap 흐름과 pagination dots로 같은 정보를 확인할 수 있어야 한다.

## 배경

현재 `docs/`의 기능 섹션은 `docs/script.js`가 스크롤 위치를 계산해 `docs/styles.css`의 CSS 변수와 클래스 상태를 바꾸고, `docs/index.html` 안의 여러 이미지 레이어를 crossfade하는 방식이다. 이 구조는 Finder/Quick Look/viewer/share 상태를 세밀하게 보여주지만, 섹션 높이가 길고 progress checkpoint UI가 커서 기능별 실제 사용 영상의 가시성이 낮다.

작업지시자는 `https://nexters.co.kr/`의 기능 소개 구간처럼 텍스트 항목에 반응해 대표 미디어가 전환되는 구성을 레퍼런스로 제시했다. 알한글 홍보 페이지에서는 이미지 대신 실제 동영상을 자동 재생해 기능 동작을 더 직접적으로 보여주는 방향이 맞다.

관련 기존 작업으로는 GitHub Pages 랜딩페이지를 만든 #135와 두 번째 섹션을 보강한 #178이 있다. 이번 작업은 #178의 두 번째 섹션 보강과 분리하고, 세 번째 기능 설명 섹션만 대상으로 한다.

## 범위

- `docs/index.html`, `docs/styles.css`, `docs/script.js`의 세 번째 기능 섹션 구조 개편
- 데스크톱/넓은 화면에서 큰 동영상 영역과 기능 항목 리스트를 함께 보여주는 레이아웃 구성
- 기능 항목 hover/focus/click 시 active 상태 전환
- 같은 기능을 다시 hover/focus/click하면 해당 동영상이 `currentTime = 0`에서 다시 재생되는 동작 구현
- 기존 progress bar/checkpoint 박스 제거
- `Finder에서 썸네일로 찾기`용 동영상 자산 생성 또는 반영
- 제공된 `quicklook.mp4`, `share.mp4`, `edit_and_save.mp4`를 홍보 페이지 자산으로 반영
- 필요 시 GitHub Pages 로딩 성능을 위해 무음 웹용 압축본 생성
- 모바일/좁은 화면에서 video carousel, CSS scroll snap, pagination dots 구현
- 데스크톱/모바일 브라우저 시각 검증과 미디어 재생 검증

## 설계 방향

섹션 상단 문구는 작업지시자 요청대로 유지한다.

```text
미리보기부터 뷰어까지, Mac 방식으로
Finder와 Quick Look, 앱 뷰어까지 HWP/HWPX 문서를 Mac 안에서 자연스럽게 열고 공유합니다.
```

데스크톱 기본안은 레퍼런스와 달리 이미지만 바꾸는 방식이 아니라, 왼쪽에 큰 동영상 프레임을 두고 오른쪽에 기능 항목 리스트를 배치한다. 제공 영상이 16:9 화면 기록 중심이므로 미디어를 크게 보여주는 쪽이 가시성에 유리하다. 기능 항목은 버튼 성격의 카드 또는 리스트 항목으로 구성해 hover뿐 아니라 keyboard focus와 click에서도 동일하게 작동하게 한다.

동영상은 `muted`, `playsinline`, `preload="metadata"`를 기본으로 두고 active 항목만 재생한다. 재생 실패 promise는 조용히 처리하되, 영상이 정지해도 poster/첫 프레임이 보이도록 구성한다. `prefers-reduced-motion: reduce` 환경에서는 자동 재생을 억제하거나 최소화하고 active 영상의 정적인 첫 프레임이 보이도록 한다.

모바일과 좁은 화면에서는 데스크톱 hover 모델을 그대로 축소하지 않는다. 각 기능을 동영상 카드로 만들고 horizontal scroll snap을 적용한다. pagination dots는 현재 snap된 카드와 동기화하고, dot click으로도 해당 카드로 이동할 수 있게 한다.

자산은 원본 파일명을 그대로 노출하기보다 `docs/assets/feature-*.mp4`처럼 URL 안전한 이름으로 정리한다. 원본 MP4 합계가 크므로 구현 단계에서 무음 웹용 압축본 크기와 화질을 확인하고, 브라우저에서 실제 가독성이 떨어지면 압축 조건을 조정한다.

## 검증 계획

계획 문서 단계:

```bash
git diff --check -- mydocs/orders/20260509.md mydocs/plans/task_m010_182.md
rg -n "#182|호버 재생|동영상|scroll snap|pagination" mydocs/orders/20260509.md mydocs/plans/task_m010_182.md
```

구현 단계 진입 후 최소 검증:

```bash
node --check docs/script.js
rg -n "feature-.*\\.mp4|data-feature|scroll-snap|pagination|playsinline|muted|currentTime" docs
find docs/assets -maxdepth 1 -type f | sort
git diff --check -- docs mydocs
```

브라우저 검증:

```bash
python3 -m http.server 8080 --directory docs
```

확인 항목:

- 데스크톱에서 기능 hover/focus/click 시 active 영상이 처음부터 재생된다.
- 같은 기능에 다시 진입하면 영상이 처음부터 다시 재생된다.
- inactive 영상은 동시에 재생되지 않는다.
- progress bar/checkpoint 박스가 제거되어 있다.
- 기존 섹션 상단 문구가 유지된다.
- 모바일에서 좌우 scroll snap으로 기능 영상이 전환된다.
- pagination dots가 현재 카드와 동기화된다.
- 텍스트와 영상이 320px 이상 폭에서 겹치거나 잘리지 않는다.
- Browser console error가 없다.

## 리스크

- 제공된 세 MP4 파일의 합계가 커서 GitHub Pages 초기 로딩이 느려질 수 있다. 구현 단계에서 무음 압축본과 `preload="metadata"`를 기본으로 검토한다.
- Finder 썸네일 기능은 현재 페이지의 스크롤 기반 이미지 전환을 동영상으로 만들어야 하므로, 원본 화면 기록을 그대로 쓸지 기존 이미지 레이어를 재녹화할지 Stage 1에서 확정해야 한다.
- hover 기반 UI는 터치 환경과 맞지 않으므로 모바일은 별도 carousel 상호작용으로 구현해야 한다.
- 동영상 자동 재생은 브라우저 정책상 muted/playsinline 조건을 지켜야 안정적으로 동작한다.
- #178에서 들어온 두 번째 섹션 보강과 같은 `docs/` 파일을 공유하므로, 최신 `devel-webview` 기준으로 충돌 없이 진행해야 한다.
