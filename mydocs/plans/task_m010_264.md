# Task M010 #264 수행계획서

## 목적

릴리즈 안내에서 사용자가 체감할 변경사항을 `전체 요약`, `포함된 rhwp 변화`, `알한글 앱 변화`로 구분하는 기준을 도입한다.

완료 후에는 GitHub Release body, GitHub Pages 릴리즈 노트, 내부 릴리즈 기록, release note 생성 스크립트가 같은 정보 구조를 따르며, 다음 upstream `rhwp` 반영 릴리즈 전에 실제 변경사항을 자리표시자 없이 작성할 수 있어야 한다.

## 배경

현재 release communication 기준은 #185와 #205에서 상당 부분 정리되어 있다. `mydocs/release/`는 릴리즈별 장기 기록을 남기고, Pages 릴리즈 노트는 사용자용 짧은 안내 표면을 맡으며, GitHub Release body는 설치/검증/provenance까지 포함하는 public 배포 본문을 맡는다.

다만 공개된 `v0.1.2` GitHub Release body의 `이번 버전의 주요 변경 사항`은 실제 변경 목록 대신 release delta checklist를 참조하라는 템플릿 문구가 남아 있다. 반면 `mydocs/release/v0.1.2.md`와 `docs/updates/v0.1.2.html`에는 `rhwp v0.7.11` 반영, Finder thumbnail refresh, UTI 정책, About provenance, Web viewer runtime banner 같은 실제 사용자-facing 변화가 비교적 잘 정리되어 있다.

다음 릴리즈는 upstream `rhwp` 새 릴리즈 반영과 앱 현재 변경 누적분을 함께 안내해야 하므로, release owner가 매번 수동 판단으로 보정하더라도 기본 template과 manual이 `전체/rhwp/앱` 구분을 요구하게 만드는 것이 필요하다.

## 범위(포함·제외)

포함:

- GitHub Release body의 사용자-facing 변경사항 구분 기준 문서화
- Pages 릴리즈 노트에서 같은 구분을 짧게 적용하는 기준 문서화
- `scripts/ci/write-release-notes.sh`의 변경사항 섹션 구조 보강
- `scripts/ci/check-release-notes-template.sh`의 필수 heading 기준 보강
- release owner가 실제 변경점을 채워야 하는 placeholder 문구와 검증 기준 정리
- `mydocs/release/index.md` 또는 release manual의 정보 소유 기준 보정
- 수행계획서, 구현계획서, 단계 보고서, 최종 보고서, 오늘할일 갱신

제외:

- 실제 upstream `rhwp` dependency 갱신
- `rhwp-core.lock`, Rust bridge artifact, bundled `rhwp-studio` asset 변경
- signed/notarized DMG 생성
- GitHub Release 게시 또는 기존 public release body 수정
- Sparkle appcast 갱신, Pages public deployment 실행
- Homebrew Cask 변경

## 설계 방향

GitHub Release body는 사용자가 다운로드 전에 가장 자세한 public 안내를 보는 표면이다. 따라서 `사용자용 요약` 아래 또는 `이번 버전의 주요 변경 사항` 안에 다음 세 구분을 둔다.

- `전체 요약`: 이번 릴리즈를 설치해야 하는 이유를 3~5개 bullet로 요약한다.
- `포함된 rhwp 변화`: upstream `rhwp` core 또는 bundled `rhwp-studio` 변경이 문서 열기, 렌더링, HWP/HWPX 호환성, viewer/editor 동작에 미치는 사용자-facing 영향만 적는다.
- `알한글 앱 변화`: HostApp, Quick Look, Finder thumbnail, 저장/공유/PDF/인쇄, 설치, 업데이트, About, DMG, Homebrew, Pages/Sparkle 등 앱 저장소가 소유한 변화를 적는다.

`rhwp` 버전이 바뀌지 않은 릴리즈에서도 구분은 유지하되, `포함된 rhwp 변화`에는 "이번 릴리즈에서 bundled rhwp 버전 변경은 없습니다"처럼 짧게 쓴다. `rhwp` 반영이 중심인 릴리즈라면 기존 정책대로 GitHub Release title의 `(rhwp vX.Y.Z)` 병기 여부를 검토하고, `포함된 rhwp 변화`에 upstream release 링크와 실제 앱 영향만 남긴다.

Pages는 GitHub Release body를 그대로 복제하지 않는다. 다만 사용자가 빠르게 이해할 수 있도록 `주요 변경`을 너무 큰 혼합 목록으로 두지 않고, 필요한 경우 `포함된 rhwp`와 `알한글 앱 변화` 또는 이에 준하는 짧은 구분을 둔다. commit, manifest, checksum 같은 긴 provenance는 GitHub Release body와 `mydocs/release/v<version>.md`로 연결한다.

## 잠정 단계(3~6단계)

1. Stage 1: 현황 inventory와 정보 구조 확정
   - 현재 manual, release note generator, checker, Pages 최신 릴리즈 문서, 내부 릴리즈 기록의 변경사항 구조를 정리한다.
   - `전체 요약`, `포함된 rhwp 변화`, `알한글 앱 변화`의 각 표면별 상세도를 확정한다.
2. Stage 2: release manual과 내부 기록 기준 보강
   - GitHub Release body와 Pages 릴리즈 노트의 구분 작성 기준을 manual에 반영한다.
   - `mydocs/release/` 정보 소유 기준과 release checklist가 새 구조를 참조하도록 보정한다.
3. Stage 3: release note 생성/검증 스크립트 보강
   - `write-release-notes.sh`가 새 변경사항 구분 heading과 작성 지침 placeholder를 생성하도록 바꾼다.
   - `check-release-notes-template.sh`가 새 필수 heading을 검증하도록 바꾼다.
4. Stage 4: 통합 검증과 보고
   - shell syntax, release note dry-run, template check, `rg` 기반 heading 대조, `git diff --check`를 수행한다.
   - 단계 보고서와 최종 보고서에 실제 public release 실행이 제외 범위임을 기록한다.

검증에서 public GitHub Release 게시, Pages deployment, appcast 갱신, notarized DMG 생성은 실행하지 않는다.

## 리스크

- 세 구분을 항상 노출하면 작은 patch release에서 본문이 장황해질 수 있다.
- upstream `rhwp` release note를 그대로 옮기면 앱 사용자에게 실제 영향이 없는 내부 변경까지 과장될 수 있다.
- Pages에 GitHub Release body 수준의 검증/provenance를 복제하면 사용자용 업데이트 페이지가 무거워지고 stale 위험이 커진다.
- `write-release-notes.sh`는 현재 실제 delta 내용을 자동 수집하지 않으므로, 자동 생성 결과는 여전히 release owner의 수동 보정을 요구한다.
- 기존 public release body를 소급 수정하면 이미 배포된 안내 표면에 불필요한 churn이 생길 수 있으므로 이번 작업에서는 수행하지 않는다.

## 승인 요청 사항

1. 위 범위와 4단계 진행 구조 승인
2. GitHub Release body에는 `전체 요약`, `포함된 rhwp 변화`, `알한글 앱 변화`를 필수 구분으로 두고, Pages에는 같은 기준을 짧게 적용하는 방향 승인
3. 기존 public GitHub Release body 수정, public Pages deployment, appcast 갱신은 이번 작업에서 제외하는 방향 승인
4. 다음 단계: 승인 후 구현계획서(`mydocs/plans/task_m010_264_impl.md`) 작성
