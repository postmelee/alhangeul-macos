## 목적

알한글 macOS 앱을 소개하고 다운로드로 이어지는 GitHub Pages용 랜딩페이지를 만든다.

완료 후 방문자는 첫 화면에서 "Mac에서 한글 파일은 더 이상 이방인이 아닙니다."라는 제품 메시지를 바로 보고, Finder Quick Look preview, Finder thumbnail, WKWebView 기반 HWP/HWPX viewer, 향후 편집 방향을 짧은 흐름으로 이해할 수 있어야 한다. 이번 작업은 `devel-webview` 기준으로 진행하며, README의 제품 설명을 바탕으로 한국어 카피와 정적 웹 페이지를 구성한다.

## 배경

현재 README는 개발자와 기여자를 위한 프로젝트 설명, 로드맵, 빌드 절차를 중심으로 구성되어 있다. 반면 일반 사용자가 GitHub Pages에서 앱의 가치를 빠르게 이해하고 다운로드 버튼을 찾을 수 있는 공개 홍보 페이지는 별도로 없다.

작업지시자는 `https://openscreen.vercel.app/`를 디자인 레퍼런스로 지정했다. 레퍼런스는 상단 플로팅 헤더, 중앙 대형 히어로, 플랫폼 다운로드 버튼, 실행 영상/프리뷰, Feature 섹션, FAQ, 짧은 푸터 흐름으로 구성되어 있다. 알한글 랜딩페이지도 이 리듬을 따르되, 색상은 흰색과 검정색을 중심으로 두고 앱 로고의 파란색만 하이라이트로 사용한다.

## 설계 방향

정적 사이트는 GitHub Pages에서 별도 빌드 파이프라인 없이 서빙 가능한 구조를 우선한다. 기존 저장소에 웹 프레임워크가 없으므로, 특별한 이유가 없으면 `docs/` 하위에 `index.html`, `styles.css`, `script.js`, `assets/`를 두는 단순 정적 페이지로 구현한다. GitHub Pages 설정 자체는 작업 범위에서 제외하고, Pages source를 `docs/`로 지정할 수 있는 구조만 만든다.

시각 방향은 레퍼런스의 섹션 흐름과 첫 화면 균형을 따른다. 다만 전체 팔레트는 true white, black, near-black 텍스트, 얇은 gray border, 앱 로고 blue accent로 제한한다. 히어로에는 불필요한 eyebrow, badge, 과한 gradient orb를 넣지 않는다. 상단 헤더는 로고/이름/GitHub 링크 중심의 얇은 floating bar로 구성한다.

Build Web Apps 지침에 따라 구현 전 디자인 콘셉트를 먼저 만든다. 하이퍼-워터폴 승인 절차와 충돌하지 않도록, 수행계획서 승인 후 구현계획서 또는 Stage 1에서 레퍼런스 분석과 디자인 콘셉트/토큰을 확정하고, 그 다음 정적 페이지 구현으로 넘어간다.

사용자 제공 자산인 `thumbnail2.mov`와 `Group 1_4x.png`는 현재 메인 worktree 루트에 미추적 파일로 존재한다. 구현 단계에서 이 파일을 `docs/assets/`로 복사해 페이지가 GitHub Pages에서 직접 참조할 수 있게 한다. 기존 `assets/logo-256@2x.png`는 앱 로고로 사용하되, 필요하면 `docs/assets/`에 사본을 둔다.

## 리스크

- 현재 제공된 `thumbnail2.mov`와 `Group 1_4x.png`는 메인 worktree의 미추적 파일이라 분리 worktree에는 없다. 구현 단계에서 파일 복사 위치와 파일명 정규화가 필요하다.
- 실제 다운로드 산출물 URL이 아직 없으면 Mac 다운로드 버튼은 GitHub Releases 최신 페이지 또는 placeholder 정책을 결정해야 한다.
- `Group 1_4x.png` 원본이 6144 x 4132로 커서 GitHub Pages 로딩 성능에 영향을 줄 수 있다. 필요하면 Stage 2에서 웹용 압축 사본을 생성한다.
- `thumbnail2.mov`는 브라우저 autoplay 정책의 영향을 받으므로 muted, playsinline, controls/fallback 처리를 확인해야 한다.
- 레퍼런스가 어두운 녹색 하이라이트와 배경 grid를 사용하지만, 이번 작업은 흰색/검정 중심을 요구하므로 레퍼런스의 구조만 따르고 색감은 제품 요구에 맞춰 조정해야 한다.
- FAQ 문구가 제품 현재 구현 상태를 과장하지 않도록 README의 체크된 기능과 로드맵을 구분해야 한다.
