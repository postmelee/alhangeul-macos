## 목적

HostApp WKWebView viewer의 저장 단축키를 macOS 관례에 맞게 정리한다.

완료 후 `Command+S`는 현재 문서의 원본 위치에 즉시 저장하고, 기존 저장 패널 기반 동작은 `Command+Shift+S`에서 실행되어야 한다. 또한 WKWebView 내부 메뉴와 tooltip에서 Windows/Linux 기준 `Ctrl` shortcut 표기가 macOS 사용자에게 맞는 `Command` 표기로 보여야 한다.

## 배경

현재 HostApp의 macOS 메뉴는 `Command+S`를 `file:save`에 연결한다. 그러나 `file:save`의 실제 구현은 항상 `NSSavePanel`을 먼저 띄우는 "저장 위치 선택" 흐름이다. WKWebView 내부 bridge script와 AppKit key equivalent fallback도 같은 command로 이어져 있어, 사용자가 일반 macOS 앱처럼 `Command+S`로 즉시 저장할 수 없다.

`rhwp-studio` bundled HTML에는 `Ctrl+S`, `Ctrl+P`, `Ctrl+Z` 등 shortcut label이 하드코딩되어 있다. 현재 HostApp은 macOS 앱이므로 표시 문자열은 `Command` 기준으로 보정하는 것이 사용자 기대에 맞다. 단, `view:ctrl-mark`나 `icon-ctrl-mark`의 `ctrl`은 조판 부호/control mark 의미이므로 shortcut 표기 보정 대상이 아니다.

## 범위

포함:

- HostApp `CommandGroup(replacing: .saveItem)`에서 저장과 다른 이름으로 저장 command 분리
- WKWebView injected bridge script의 native command set과 shortcut routing 확장
- AppKit fallback key handler에서 `Command+S`와 `Command+Shift+S` 구분
- 현재 문서 원본 URL/security-scoped bookmark를 사용한 즉시 저장 경로 추가
- 원본 URL이 없거나 저장 권한을 확보할 수 없는 경우 save panel fallback 처리
- save panel 완료 후 이후 `Command+S`가 새 저장 위치를 대상으로 하도록 source document 상태 갱신
- WKWebView 내부 shortcut label/tooltip의 `Ctrl+`을 macOS용 `Command+`로 표시 보정

제외:

- upstream `rhwp-studio` source repository 수정
- autosave, dirty-state 추적, 저장 완료 toast 등 추가 UX
- HWPX 저장 지원 확대
- native renderer 저장 기능
- 저장 전후 round-trip compatibility 검증 체계 구축

## 설계 방향

저장 command를 `file:save`와 `file:save-as`로 분리한다. `file:save`는 원본 문서가 있을 때 바로 export payload를 받아 기존 URL에 쓴다. 원본 URL이 없거나 security-scoped access/write가 실패하면 사용자 데이터를 잃지 않도록 `file:save-as` 흐름으로 fallback한다.

`file:save-as`는 현재 `requestSaveDocument`가 담당하는 save panel 흐름을 유지하되, 저장이 완료되면 현재 window의 source document를 새 URL로 갱신한다. 이렇게 해야 이후 `Command+S`가 방금 선택한 위치를 대상으로 즉시 저장된다.

shortcut label 보정은 bundled HTML을 대량 수정하지 않고 HostApp 주입 script에서 DOM이 준비된 뒤 수행한다. `Ctrl+` 문자열을 가진 `.md-shortcut`, `.tb-split-shortcut`, `title` attribute를 대상으로 하며, command id/class name의 `ctrl` 의미는 변경하지 않는다.

`Sources/RhwpCoreBridge`는 이 작업의 소유 영역이 아니므로 변경하지 않는다.

## 리스크

- HWP 저장은 원본 파일을 덮어쓰는 동작이므로 security-scoped write 실패나 export 실패 시 원본을 손상시키지 않는 실패 경로가 중요하다.
- 현재 dirty-state 추적이 없으므로 `Command+S`는 변경 여부와 무관하게 export/write를 수행할 가능성이 있다.
- `rhwp-studio`의 HWPX 저장은 별도 제한이 있으므로 HWPX 문서에서는 기존 경고/제한과 충돌하지 않게 처리해야 한다.
- shortcut label 보정이 단순 문자열 치환이면 `Ctrl`이 shortcut 의미가 아닌 텍스트까지 바꿀 수 있으므로 대상 selector를 좁혀야 한다.
- SwiftUI command와 WKWebView/AppKit fallback이 같은 shortcut을 중복 처리하면 저장이 두 번 실행될 수 있다.
