# Task M018 #183 수행계획서

## 목적

`v0.1.0` GitHub Release DMG 설치본에서 HWP/HWPX 문서를 연 뒤 창 title bar 영역 더블 클릭 또는 창 확대 동작을 수행하면 `웹 viewer 실행 중 오류가 발생했습니다` fallback 화면이 표시되는 문제를 재현, 진단, 수정한다.

완료 후에는 창 확대, 창 크기 변경, 원래 크기 복귀를 반복해도 현재 문서 viewer 상태가 유지되고, 실제 WKWebView asset/document/runtime failure fallback은 #150에서 정리한 의미를 잃지 않아야 한다.

## 배경

#150에서 HostApp WKWebView viewer의 asset/document/navigation/runtime failure가 fatal fallback으로 분리되었다. 해당 fallback은 빈 화면이나 무응답을 방지하는 보호 장치지만, `v0.1.0` 설치본 smoke 중 창 확대만으로 runtime fallback이 표시되면서 정상 resize 흐름을 fatal failure로 오판하거나, `rhwp-studio` resize/layout 경로가 실제로 예외를 발생시키는 가능성이 생겼다.

현재 HostApp viewer는 `DocumentViewerStore`가 문서 bytes와 revision을 보관하고, `RhwpStudioWebView`가 bundled `rhwp-studio`를 WKWebView로 로드한다. `RhwpStudioHostBridgeScript.runtimeErrorSource`는 `window.error`와 `unhandledrejection`을 native failure로 전달하며, `DocumentViewerView`는 `store.webViewFailure`가 생기면 viewer를 fallback 화면으로 대체한다. 따라서 이번 작업은 창 확대 이벤트가 어떤 JS/WASM 오류, WebKit 상태 변화, host bridge 메시지로 이어지는지 먼저 확인해야 한다.

## 설계 방향

먼저 `v0.1.0` 설치본에서 표시되는 fallback의 진단 정보를 확보하고, 같은 동작이 현재 `devel-webview` build에서도 재현되는지 비교한다. 이미 수정된 상태라면 #150 이후 또는 release 이후 변경 중 어떤 차이가 영향을 줬는지 확인하고, 아직 재현된다면 창 확대 이벤트가 `rhwp-studio` 내부 resize/layout 코드, service worker/PWA 부산물, WKWebView process lifecycle, host bridge runtime error 포착 중 어느 경로로 들어오는지 분리한다.

수정은 실제 오류를 숨기지 않는 방향을 우선한다. 창 확대 시 문서가 정상 렌더링된 상태인데 특정 known benign rejection만 올라오는 경우에는 `RhwpStudioHostBridgeScript.runtimeErrorSource`의 필터를 좁게 보강한다. 반대로 resize/layout 코드가 실제로 viewer 상태를 깨뜨린다면 HostApp injection 또는 `alhangeul-wkwebview-overrides.css` 같은 저장소 소유 override로 안정화한다. Bundled upstream asset을 직접 수정해야 할 경우에는 hash/provenance와 `scripts/verify-rhwp-studio-assets.sh` 영향을 별도로 기록한다.

Viewer document identity는 `documentRevision`과 `webViewReloadToken`으로 관리되므로, 창 크기 변경만으로 문서 payload가 재로드되거나 `webViewFailure`가 남는 구조는 피한다. 재시도 fallback은 계속 복구 경로로 유지하되, 정상 resize는 fallback으로 들어가지 않는 것을 수용 기준으로 둔다.

## 잠정 단계(3~6단계)

1. Stage 1: 설치본 재현과 진단 정보 수집
   - `v0.1.0` 설치본에서 HWP/HWPX 문서 열기 후 title bar 더블 클릭, green zoom button, 수동 resize를 확인한다.
   - fallback 진단 정보, WebKit/runtime 로그, viewer URL, 문서 상태를 기록한다.
   - 현재 `devel-webview` build에서 같은 절차를 비교한다.
2. Stage 2: 원인 경로 분리와 수정안 확정
   - runtime bridge, WKWebView lifecycle, `rhwp-studio` resize/layout 코드, override CSS/JS 경계를 대조한다.
   - benign runtime issue 필터 보강인지, resize 안정화 patch인지, fallback 상태 처리 보정인지 결정한다.
3. Stage 3: 최소 수정 구현
   - 확정한 경계에 맞춰 HostApp bridge 또는 bundled viewer override를 수정한다.
   - 실제 runtime failure를 과하게 숨기지 않도록 필터 조건과 진단 정보를 좁게 유지한다.
4. Stage 4: 창 확대/resize smoke와 회귀 검증
   - HWP/HWPX 샘플에서 창 확대, 원복, 수동 resize 반복 시 viewer 상태 유지 여부를 확인한다.
   - #150의 asset/runtime failure fallback이 의도대로 남아 있는지 필요한 범위에서 negative smoke를 수행한다.
5. Stage 5: v0.1.1 patch candidate handoff 정리
   - 최종 보고서에 설치본 재현 여부, 수정 내용, smoke 결과, #188 release 실행에서 재검증할 항목을 기록한다.
   - 오늘할일을 완료 처리하고 PR 게시 전 산출물을 정리한다.

## 리스크

- `/Applications/Alhangeul.app`에 `v0.1.0` 설치본이 없거나 release DMG 재설치가 필요할 수 있다.
- 문제가 Debug/Release unsigned build에서는 재현되지 않고 signed/notarized 설치본에서만 재현될 수 있다.
- bundled `rhwp-studio` asset은 minified/hash 기반 파일을 포함하므로 원인 stack이 불완전할 수 있다.
- runtime error 필터를 넓게 잡으면 실제 JS/WASM 실패를 숨길 위험이 있다.
- macOS 버전, 디스플레이 scale, window zoom 정책에 따라 재현성이 달라질 수 있다.
- signed/notarized DMG 생성과 public release 실행은 별도 승인과 #188 범위가 필요하다.
