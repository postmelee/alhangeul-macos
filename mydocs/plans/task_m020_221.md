# Task M020 #221 수행계획서

## 목적

rhwp core SVG 기반 PDF 생성 경로가 현재 Swift native bitmap PDF 경로보다 Quick Look preview에 적합한지 성능, 안정성, 표시 품질 관점에서 비교한다.

완료 후 작업자는 Quick Look의 PDF reply와 우측 페이지 정보 UI를 유지하면서 렌더링 기준을 rhwp core SVG 쪽으로 옮길 수 있는지, 또는 현재 native bitmap PDF 경로를 유지해야 하는지 판단할 수 있어야 한다.

## 배경

현재 Quick Look preview와 Finder thumbnail은 rhwp core에서 page info/render tree를 받은 뒤 Swift native renderer가 CoreGraphics bitmap을 만들고, 단일 페이지는 PNG reply, 다중 페이지는 bitmap을 담은 PDF reply로 반환한다.

HostApp은 rhwp-studio/WASM 렌더링을 사용하므로 Quick Look/Thumbnail과 시각 차이가 발생할 수 있다. 사용자 경험상 Finder preview, thumbnail, HostApp viewer의 렌더링 기준은 가능한 한 일치하는 편이 좋다.

hulryung/hwpql은 rhwp core가 생성한 SVG 기반 HTML을 Quick Look preview로 반환한다. 다만 HTML reply는 macOS PDF preview처럼 우측 페이지 정보와 페이지 썸네일 UI를 유지하기 어렵다. 본 작업은 WebView 방식이 아니라 rhwp core SVG를 PDF reply로 연결할 수 있는지 비교하는 spike다.

## 범위

포함:

- hwpql의 rhwp SVG/HTML Quick Look 구조 조사 결과 정리
- 현재 `HwpPreviewPDFRenderer` native bitmap PDF 경로 성능 측정
- `RhwpDocument.renderPageSVG(at:)` 기반 SVG 생성 성능 측정
- SVG를 Quick Look PDF reply로 연결할 수 있는 후보 방식 조사
- 단일/다중 페이지 샘플에서 native bitmap PDF와 SVG 기반 후보의 산출물 크기, 생성 시간, 실패 양상 비교
- thumbnail fast path에 SVG 기반 경로를 적용할지 여부 판단
- 후속 구현 이슈 또는 보류 판단 근거 정리

제외:

- WKWebView/WebView를 Quick Look extension 내부 PDF 생성 경로로 채택하지 않는다.
- 본 spike에서 제품 기본 렌더러를 곧바로 교체하지 않는다.
- 사용자용 PDF export semantics 개선은 다루지 않는다.
- Swift native renderer 전체 제거는 별도 아키텍처 결정 후 진행한다.

## 설계 방향

비교의 기준선은 현재 제품 경로인 `HwpPreviewPDFRenderer`와 `HwpPageImageRenderer`로 둔다. 측정 스크립트는 제품 코드를 우회하지 않고 현재 Quick Look preview가 사용하는 경로와 같은 helper를 호출한다.

SVG 후보는 먼저 기존 FFI 표면인 `RhwpDocument.renderPageSVG(at:)`를 사용한다. 이 경로가 충분히 빠르고 안정적인지 확인한 뒤, Swift 쪽에서 SVG를 PDF page로 변환하는 방식과 Rust FFI에서 PDF bytes를 직접 반환하는 방식의 장단점을 나눈다.

WebView 기반 PDF 생성은 cold start, 비동기 settle, extension 메모리, sandbox 리스크가 커서 후보에서 제외한다. Quick Look 우측 페이지 정보 UI를 유지하려면 최종 reply는 `.pdf`여야 한다.

## 검증 계획

- `git diff --check`
- 신규 shell script가 있으면 `bash -n`
- 신규 Swift helper가 있으면 실행 가능한 샘플 기준 smoke
- `samples/basic` 또는 Desktop sample 중 단일 페이지, 다중 페이지 각각 1개 이상 측정
- native bitmap PDF 산출물의 page count 확인
- core SVG 산출물 생성 성공 여부와 생성 시간 확인
- 가능한 경우 SVG 기반 PDF 후보 산출물의 page count와 파일 크기 확인

## 리스크

- SVG -> PDF 변환 도구가 macOS 기본 환경만으로 충분하지 않을 수 있다.
- SVG 렌더링과 HostApp rhwp-studio 화면이 완전히 같다는 보장은 별도 검증이 필요하다.
- SVG 기반 PDF가 vector 품질을 유지하지 못하고 내부적으로 bitmap화될 수 있다.
- 다중 페이지 문서에서는 전체 페이지 생성 비용이 여전히 크며 true lazy pagination을 제공하지 못할 수 있다.
- thumbnail에 SVG 경로를 쓰면 현재 fast path보다 느려질 수 있다.
