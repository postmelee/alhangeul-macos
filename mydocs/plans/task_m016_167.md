# Task M016 #167 수행계획서

## 목적

M16 완료 후 바로 v0.1 public release를 진행할 수 있도록 `rhwp` core와 release artifact provenance 기준을 upstream 최신 stable release인 `v0.7.10`으로 맞춘다.

완료 후 `rhwp-core.lock`, `RustBridge/Cargo.lock`, Rust bridge artifact hash/size, FFI symbol snapshot, bundled `rhwp-studio` manifest가 서로 충돌하지 않아야 한다. #145 release artifact/provenance 정리, #151 설치본 smoke gate, #146 known limitations 문서화는 이 작업의 결과를 기준으로 이어받는다.

## 배경

#145 Stage 1-2에서 현재 앱 저장소의 release 기준과 public publish workflow 기준이 어긋나 있음을 확인했다. 현재 `rhwp-core.lock`과 bundled `rhwp-studio` manifest는 `edwardkim/rhwp` `v0.7.9`를 가리키지만, upstream 최신 release와 `release-publish.yml` 기본 guard는 `v0.7.10` 기준이다.

M16 완료 후 곧바로 public release를 진행할 예정이라면 #166에서 core를 다시 올리는 방식은 중복 검증과 provenance 불일치를 만든다. 따라서 M16 안에서 `v0.7.10` core/studio 기준을 먼저 확정하고, #166은 이 작업의 결과를 확인만 하는 배포 실행 이슈로 유지한다.

#166 본문은 이미 #167을 선행 조건으로 참조하도록 갱신했다. #166에서는 `rhwp-core.lock` 갱신, generated artifact 재생성, bundled `rhwp-studio` asset 갱신을 반복하지 않는다.

- #145 이후 단계가 `v0.7.10` 기준 release artifact/provenance를 이어받을 수 있도록 문서 연결

## 설계 방향

core 기준은 Stable release 원칙에 따라 `release tag + resolved commit`으로 고정한다. branch나 floating ref는 배포 기준으로 사용하지 않는다. 먼저 `scripts/update-rhwp-core.sh --check --channel stable --tag v0.7.10`으로 required API와 compatibility gate를 확인하고, 통과할 때만 실제 dependency/lock 갱신으로 넘어간다.

Rust bridge artifact는 `scripts/build-rust-macos.sh --update-lock`로 재생성해 `rhwp-core.lock`에 hash/size를 기록한 뒤, `--verify-lock`으로 다시 확인한다. FFI symbol snapshot은 변화 여부를 명시적으로 비교하고, 의도된 변화일 때만 갱신한다.

HostApp viewer는 bundled `rhwp-studio`를 WKWebView로 사용하므로 core lock만 `v0.7.10`으로 올리고 `rhwp-studio/manifest.json`이 `v0.7.9`에 남으면 release artifact provenance가 깨진다. 따라서 Stage 1에서 `v0.7.10`의 `rhwp-studio` build/sync 가능 여부를 먼저 확인하고, 가능하면 asset과 manifest도 같은 tag/commit 기준으로 맞춘다. 불가능하면 release artifact 기준을 어떻게 설명할지 작업지시자에게 별도 판단을 요청한다.

작업은 현재 `local/task145` worktree에 사용자 `README.md` 미커밋 변경이 남아 있어 `/private/tmp/rhwp-mac-task167` 분리 worktree에서 진행한다.

## 잠정 단계(3~6단계)

1. Stage 1: `v0.7.10` compatibility와 sync 가능성 조사
   - upstream release tag/commit, `update-rhwp-core --check`, 현재 scripts, `rhwp-studio` sync 절차를 확인한다.
   - `v0.7.10`으로 core와 studio를 함께 맞출 수 있는지 판단한다.
2. Stage 2: core dependency와 Rust bridge artifact 갱신
   - `RustBridge/Cargo.toml`, `Cargo.lock`, `rhwp-core.lock`을 `v0.7.10` 기준으로 갱신한다.
   - Rust bridge artifact를 재생성하고 hash/size를 lock에 반영한다.
3. Stage 3: bundled `rhwp-studio` asset/manifest 기준 정합화
   - `rhwp-studio` resource tree와 manifest가 `v0.7.10` 기준이 되도록 갱신하거나, 불가능할 경우 명확한 분리 기준을 보고한다.
   - `scripts/verify-rhwp-studio-assets.sh`로 asset 구조를 검증한다.
4. Stage 4: build/render 검증과 release 기준 문서 연결
   - no-AppKit check, Xcodegen, Debug/Release build, render smoke를 수행한다.
   - #145/#151/#146/#166으로 이어지는 release 기준 문구를 필요한 문서에 반영한다.
5. Stage 5: 최종 보고와 handoff
   - core tag/commit, artifact hash/size, FFI symbol 변화, studio manifest, 검증 결과를 최종 보고서에 정리한다.
   - #145 재개 시 반영할 release artifact 기준을 명시한다.

## 리스크

- `v0.7.10`이 Rust bridge required API는 만족해도 render output이나 generated artifact hash가 예상보다 크게 바뀔 수 있다.
- `rhwp-studio` asset을 `v0.7.10` 기준으로 재빌드하려면 Node/Docker/upstream build 환경이 필요할 수 있다.
- core는 `v0.7.10`인데 bundled studio asset은 `v0.7.9`로 남으면 release artifact provenance가 애매해진다.
- `rhwp-ffi-symbols.txt` 변화가 발생하면 Swift bridge와 extension 검증 범위를 넓혀야 한다.
- 이 작업은 네트워크 dependency fetch와 Rust/Xcode build를 포함하므로 로컬 환경 실패를 compatibility 실패와 구분해야 한다.
- #145가 이미 `local/task145`에서 진행 중이므로, 최종적으로 M16 문서와 오늘할일 merge 시 충돌이 날 수 있다.
- 현재 메인 worktree의 `README.md` 미커밋 변경은 이 작업 범위 밖이다.
