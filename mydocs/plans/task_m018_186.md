# Task M018 #186 수행계획서

## 목적

GitHub Actions의 Node.js 20 deprecation warning이 release workflow와 PR/release 검증 판단을 흐리지 않도록, 저장소 workflow에서 사용하는 공식 action version을 Node.js 24 대응 major로 갱신하거나 처리 방침을 명확히 한다.

## 배경

이슈 #186은 `v0.1.0` release workflow와 Pages 관련 run에서 관측된 Node.js 20 action deprecation annotation을 후속 정리하는 작업이다. #198에서 PR CI와 release workflow 검증 기준이 추가되었으므로, 이번 작업은 그 workflow들이 다음 public release 전에 Node.js 24 전환 경고를 만들지 않도록 정리한다.

2026-05-10 기준 공식 확인 내용:

- GitHub Changelog: Node.js 20은 2026년 4월 EOL이며, GitHub-hosted runner는 2026년 6월 2일부터 Node.js 24를 기본으로 사용하기 시작한다. Node.js 20은 2026년 가을 runner에서 제거될 예정이다. 사용자는 Node.js 24에서 실행되는 최신 action version으로 workflow를 갱신해야 한다.
- `actions/checkout`: 최신 release는 `v6.0.2`이며, `actions/checkout@v6`의 `action.yml`은 `runs.using: node24`를 선언한다. `v6`는 credential storage가 `$RUNNER_TEMP` 쪽으로 바뀌는 변경이 있으나, 이 저장소 workflow는 Docker container action 내부에서 checkout credential을 재사용하지 않는다.
- `actions/upload-artifact`: 최신 release는 `v7.0.1`이며, `actions/upload-artifact@v7`의 `action.yml`은 `runs.using: node24`를 선언한다. 현재 workflow에서 쓰는 `name`, `path`, `if-no-files-found`, `retention-days` 입력은 `v7`에서도 유지된다.
- 현재 `.github/workflows/`에는 GitHub Pages 전용 `actions/deploy-pages` 또는 `actions/upload-pages-artifact` workflow가 없다. Pages appcast 갱신은 `release-publish.yml`에서 `git clone`/`git push`로 수행한다.

참고한 공식 URL:

- https://github.blog/changelog/2025-09-19-deprecation-of-node-20-on-github-actions-runners/
- https://github.com/actions/checkout
- https://github.com/actions/upload-artifact

## 범위(포함·제외)

포함:

- `.github/workflows/`의 `uses:` action version 전수 점검
- `actions/checkout@v4`를 Node.js 24 runtime major로 갱신
- `actions/upload-artifact@v4`를 Node.js 24 runtime major로 갱신
- release rehearsal/publish, PR CI, rhwp upstream check의 checkout/artifact upload 경로 보존 확인
- `mydocs/manual/ci_workflow_guide.md`와 release 관련 guide에 Node.js action runtime 대응 기준 반영
- PR CI 또는 실제 PR run에서 Node.js 20 deprecation annotation이 사라졌는지 확인할 handoff 기록

제외:

- release workflow 전체 재작성
- GitHub Pages 배포 방식을 branch push에서 `actions/deploy-pages`로 전환
- self-hosted runner 전환
- signing/notarization 로직 변경
- GitHub Release publish 또는 public release 실행

## 설계 방향

이번 작업은 최소 변경으로 official action major를 갱신한다. workflow 구조, job 분리, release publish 보호 조건, appcast push 방식은 바꾸지 않는다.

후보 변경:

- `actions/checkout@v4` -> `actions/checkout@v6`
- `actions/upload-artifact@v4` -> `actions/upload-artifact@v7`

주요 판단:

- GitHub-hosted runner는 필요한 runner version을 충족한다고 보고, 별도 runner pinning은 추가하지 않는다.
- checkout `v6`의 credential storage 변경은 이 저장소 workflow의 일반 checkout, release tag checkout, `gh` CLI 사용, 별도 token clone 기반 Pages branch push와 충돌하지 않는지 Stage 1에서 확인한다.
- artifact action `v7`은 기존 입력 호환성을 확인하되, hidden files 포함 정책이나 artifact immutability 관련 동작이 현 workflow 산출물과 충돌하지 않는지 점검한다.
- Node.js 24 전환을 강제하는 임시 env(`FORCE_JAVASCRIPT_ACTIONS_TO_NODE24`)나 Node.js 20 opt-out env(`ACTIONS_ALLOW_USE_UNSECURE_NODE_VERSION`)는 기본 대응책으로 사용하지 않는다. action version update로 해결하는 것을 우선한다.

## 잠정 단계

1. Stage 1: 공식 action/runtime 기준과 현 workflow 영향 분석
   - official changelog와 action repo의 Node.js 24 runtime 기준을 정리한다.
   - 현재 workflow의 `uses:` 목록을 전수 확인한다.
   - checkout `v6`, upload-artifact `v7` 갱신 시 workflow별 위험을 정리한다.
2. Stage 2: workflow action version 갱신
   - checkout과 upload-artifact major tag를 갱신한다.
   - workflow YAML parse와 action reference 검색으로 누락을 확인한다.
   - PR CI helper 분류 결과가 release checks를 켜는지 확인한다.
3. Stage 3: CI/runtime 대응 기준 문서화
   - CI guide와 release guide에 Node.js action runtime 점검 기준, 공식 action major 갱신 원칙, PR run 확인 항목을 반영한다.
4. Stage 4: 최종 dry-run, 보고, PR 준비
   - workflow parse, helper syntax/dry-run, action reference 검색, 최종 보고서, PR 게시를 수행한다.


확인 기준:

- `PR CI`가 통과한다.
- `actions/checkout@v4`, `actions/upload-artifact@v4` reference가 남아 있지 않다.
- PR run annotation에 Node.js 20 deprecation warning이 남지 않는다.
- 이번 변경 범위가 workflow/manual 중심이면 `macOS validation` skip과 `release helper checks` 실행 여부가 의도와 맞다.

## 리스크

- `actions/checkout@v6` credential storage 변경이 예상치 못한 authenticated git command 경로에 영향을 줄 수 있다.
- `actions/upload-artifact@v7`의 artifact 동작 변경이 release artifact upload 경로와 충돌할 수 있다.
- Node.js 20 warning 해소 여부는 GitHub-hosted runner 실제 run annotation으로만 최종 확인할 수 있다.
- GitHub 공식 action latest major가 계속 갱신될 수 있으므로, 문서에는 특정 버전만 고정하기보다 Node.js runtime 확인 절차를 함께 남겨야 한다.

## 승인 요청 사항

1. 위 범위와 4단계 진행 구조 승인
2. `actions/checkout@v4` -> `actions/checkout@v6` 갱신 방향 승인
3. `actions/upload-artifact@v4` -> `actions/upload-artifact@v7` 갱신 방향 승인
4. GitHub Pages 배포 방식은 이번 작업에서 바꾸지 않는 방향 승인
5. 승인 후 구현계획서(`mydocs/plans/task_m018_186_impl.md`) 작성
