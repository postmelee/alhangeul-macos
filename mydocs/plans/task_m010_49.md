# Issue #49 수행 계획서

## 타스크

- GitHub Issue: #49
- 마일스톤: M010 (v0.1.0)
- 제목: Task #45 후속: 신규 매뉴얼 3종 자체 완결성 보강
- 작업 브랜치: `local/task49`
- 기준 브랜치: `origin/devel` `e21fcb8`

## 목적

Task #45에서 AGENTS.md 본문을 무손실 이전해 만든 신규 매뉴얼 3종이 단독 문서로도 충분히 읽히도록 도입부, 핵심 용어, FAQ/흔한 실수, 상호 참조를 보강한다. 추가로, Issue #48 후속 대화에서 확인한 "하이퍼-워터폴 SKILL을 적용할 때 Codex도 Claude Code처럼 스킬명 호출 문장을 명시한다"는 운영 안내를 관련 매뉴얼에 포함한다.

## 배경

- Task #45 (PR #46)에서 `AGENTS.md` 축소를 위해 기존 본문을 다음 3개 매뉴얼로 이전했다.
  - `mydocs/manual/document_structure_guide.md`
  - `mydocs/manual/git_workflow_guide.md`
  - `mydocs/manual/task_workflow_guide.md`
- 당시 작업은 본문 무손실 이전이 목적이어서, 각 매뉴얼 자체의 도입부·맥락·예시·FAQ가 부족하다.
- 신규 에이전트나 사람이 `AGENTS.md` 링크를 따라 매뉴얼만 읽을 때 운영 의도, 폴더 선택 기준, 충돌 회복 방법, 단계 실패 회복 방법을 빠르게 판단할 수 있어야 한다.
- Issue #48에서 SKILL 노출과 묵시 호출 회피를 확인한 뒤, Codex도 SKILL 절차를 적용할 때 "task-stage-report 스킬을 호출합니다"처럼 사용자에게 명시적으로 알리는 것이 운영 투명성에 도움이 된다는 요구가 추가되었다.

## 범위

### 포함

- `document_structure_guide.md` 보강:
  - 다루는 범위/제외 범위/읽는 시점 도입부
  - 문서 폴더, 내부 타스크, 외부 PR, 마일스톤 등 핵심 용어 정의
  - 폴더 선택 실수, 마일스톤 미정, 외부 PR/내부 타스크 경계 모호 상황 FAQ
  - 관련 매뉴얼 상호 참조
- `git_workflow_guide.md` 보강:
  - 브랜치 정책과 PR 게시 절차의 목적 도입부
  - `devel`, `local/taskN`, `publish/taskN`, 분리 worktree 등 핵심 용어 정의
  - 메인 worktree 충돌, devel rebase 필요 상황, 잘못된 브랜치 push 사고 회복 FAQ
  - 관련 매뉴얼 상호 참조
- `task_workflow_guide.md` 보강:
  - 하이퍼-워터폴 단계 진행의 목적과 읽는 시점 도입부
  - 수행계획서, 구현계획서, 단계 보고서, 최종 보고서, 승인 간주 조건 등 핵심 용어 정의
  - 단계 실패 후 회복, 단계 분할 기준, `working/` vs `report/` 위치 혼동 FAQ
  - 하이퍼-워터폴 SKILL 적용 시 사용자에게 스킬명 호출 문장을 한 줄로 알리는 안내
  - 관련 매뉴얼 상호 참조

### 제외

- 새 강제 규칙 도입 또는 기존 강제 규칙 폐지.
- `AGENTS.md`/`CLAUDE.md` 본문 구조 변경.
- SKILL 동작 자체 변경, `allow_implicit_invocation` 정책 변경.
- Rust/Swift/Xcode 소스 변경.
- 릴리스, 배포, 서명, 공증 관련 변경.

## 설계 방향

- 기존 강제 규칙은 유지하고, 새 문장은 "안내", "권장 처리", "예시" 성격으로 작성한다.
- 각 매뉴얼은 개별 200줄 이하를 목표로 하되, 읽는 흐름을 해치지 않도록 FAQ는 짧은 문답 중심으로 둔다.
- 상호 참조는 문서 하단 또는 관련 섹션에 두어 중복 설명을 줄인다.
- SKILL 호출 표시 안내는 `task_workflow_guide.md`에 우선 배치한다. 이는 하이퍼-워터폴 절차 운영 투명성 안내이며, `AGENTS.md` 상시 컨텍스트를 늘리지 않는다.
- Codex와 Claude Code 양쪽 표현을 모두 포함하되, 도구별 내부 구현 차이 대신 사용자가 관찰할 수 있는 메시지 형식에 초점을 둔다.

## 잠정 단계

1. **Stage 1 — 문서 구조 매뉴얼 보강**: `document_structure_guide.md`에 도입부, 핵심 용어, FAQ, 상호 참조를 추가한다.
2. **Stage 2 — Git 워크플로우 매뉴얼 보강**: `git_workflow_guide.md`에 도입부, 핵심 용어, 충돌/브랜치 사고 회복 FAQ, 상호 참조를 추가한다.
3. **Stage 3 — 타스크 진행 매뉴얼 보강과 SKILL 호출 표시 안내 추가**: `task_workflow_guide.md`에 도입부, 핵심 용어, 단계 실패/분할/보고서 위치 FAQ, SKILL 호출 표시 안내, 상호 참조를 추가한다.
4. **Stage 4 — 통합 검증과 최종 보고**: 3개 매뉴얼 줄 수, 새 강제 규칙 미도입 여부, 링크/문구 일관성을 검증하고 최종 보고서를 작성한다.

각 단계 완료 시 단계별 완료보고서(`mydocs/working/task_m010_49_stage{N}.md`)를 작성하고 승인 후 다음 단계로 진행한다.

## 검증 계획

- 3개 매뉴얼 각각에 도입부, 핵심 용어, FAQ 또는 흔한 실수, 관련 매뉴얼 상호 참조가 존재하는지 `rg`로 확인한다.
- 각 매뉴얼 줄 수가 200줄 이하인지 `wc -l`로 확인한다.
- `task_workflow_guide.md`에 SKILL 호출 표시 예시(`task-stage-report 스킬을 호출합니다`, `task-final-report 스킬로 진행합니다`, `pr-merge-cleanup 스킬을 호출합니다`)가 포함되는지 확인한다.
- 새 강제 규칙을 만들지 않았는지 `git diff`로 "강제", "반드시", "금지" 등 표현을 점검한다.
- 문서 변경 후 `git diff --check`를 실행한다.

## 리스크

- 보강 문구가 사실상 새 규칙처럼 읽힐 수 있다. 안내·권장 처리와 기존 강제 규칙을 명확히 구분한다.
- FAQ가 길어지면 200줄 권장 기준을 넘을 수 있다. 긴 설명은 관련 매뉴얼 링크로 넘긴다.
- SKILL 호출 표시 안내가 `allow_implicit_invocation: false` 정책과 혼동될 수 있다. "자동 호출 허용"이 아니라 "승인된 SKILL 절차 적용 시 사용자에게 명시적으로 알림"임을 분명히 적는다.
