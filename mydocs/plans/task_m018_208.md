# Task M018 #208 수행계획서

## 목적

`v0.1.1` public release 전에 알한글 설치본이 Intel Mac과 Apple Silicon Mac을 모두 지원하는 단일 universal DMG 기준을 만족하게 한다. 앱 본체, Quick Look extension, Thumbnail extension 실행 파일을 `arm64 + x86_64` universal로 빌드하고, release/package 산출물 검증 gate와 GitHub Pages direct download 안내를 같은 정책으로 맞춘다.

## 배경

#208 이슈 등록 전 조사에서 Rust bridge 산출물(`Rhwp.xcframework`, `Frameworks/universal/librhwp.a`)과 Sparkle framework는 universal 기준을 만족하지만, release app bundle의 실행 파일이 `arm64` 단일 slice로 만들어질 수 있음이 확인됐다.

현재 `project.yml`, `scripts/package-release.sh`, `scripts/release.sh`의 `xcodebuild` 호출은 release 산출물용 `ARCHS`와 `ONLY_ACTIVE_ARCH`를 명시하지 않는다. GitHub Pages의 다운로드 버튼은 direct DMG URL을 바로 가리키는 현재 방식을 유지하되, v0.1.1부터 이 DMG가 Intel Mac과 Apple Silicon Mac을 모두 지원하는 단일 universal DMG임을 사용자가 알 수 있어야 한다.

이 작업은 #188 public release 실행 전에 완료되어야 한다. 실제 signed/notarized public DMG 게시, Sparkle appcast 게시, Homebrew tap 공개 배포는 각각 #188/#209 범위로 남긴다.

## 범위

포함:

- Release/package build가 `Alhangeul`, `AlhangeulPreview`, `AlhangeulThumbnail`을 `arm64 + x86_64` universal로 만들도록 build setting 또는 `xcodebuild` 인자 보정
- app bundle 내부 실행 파일과 extension 실행 파일을 `lipo`로 검증하는 release gate 추가
- `scripts/package-release.sh`, `scripts/release.sh`에서 universal architecture 검증 실패 시 중단하도록 보강
- GitHub Pages의 최신 다운로드 버튼은 direct DMG 다운로드로 유지하고, 단일 universal DMG 지원 안내를 보강
- `docs/index.html`, `docs/updates/index.html`, `docs/updates/v0.1.1.html`의 다운로드 안내 보정
- `README.md`, `scripts/ci/write-release-notes.sh`, `mydocs/release/v0.1.1.md`, release 관련 매뉴얼의 지원 아키텍처/단일 universal DMG 안내 보강
- Homebrew Cask와 Sparkle appcast가 단일 universal DMG URL을 유지한다는 전제 확인과 기록

제외:

- v0.1.1 public release 실행, tag 생성, GitHub Release 게시
- Developer ID credential, notarization credential, Sparkle private key 변경
- Intel 전용 DMG와 Apple Silicon 전용 DMG를 따로 만드는 배포 정책 변경
- Homebrew tap 생성, Cask SHA256 고정, tap publish
- `Homebrew/homebrew-cask` official 제출

## 설계 방향

기본 배포 산출물은 단일 universal DMG로 유지한다. 아키텍처별로 다른 DMG를 제공하지 않으므로, Pages에는 Intel Mac / Apple Silicon 선택 UI를 만들지 않는다. 다운로드 버튼은 현재처럼 최신 DMG를 바로 내려받게 하며, 주변 안내와 FAQ에서 “같은 DMG가 Intel Mac과 Apple Silicon Mac을 모두 지원한다”는 점을 명확히 한다.

Release build의 universal 보장은 Xcode project 전역 설정과 release/package script gate를 함께 사용한다. `project.yml`은 Xcode project 원본이므로 필요한 build setting은 `project.yml`에 반영하고, release/package script에는 명시적 `ARCHS="arm64 x86_64"`와 `ONLY_ACTIVE_ARCH=NO` 또는 동등한 검증 방식을 둔다. 산출물 검증은 실제 bundle 내부 binary를 대상으로 해야 하며, `.app` 본체뿐 아니라 `Contents/PlugIns/*.appex/Contents/MacOS/*`까지 확인한다.

검증 helper는 release workflow와 로컬 script에서 재사용 가능하게 분리하는 방향을 우선 검토한다. GitHub-hosted runner 아키텍처와 Intel runner 사용 가능 여부는 Stage 1에서 공식 문서와 workflow 환경을 다시 확인하고, 실제 Intel 하드웨어 smoke가 어려우면 `lipo -verify_arch x86_64 arm64`를 release gate로 둔 뒤 #188에서 설치본 smoke 한계를 기록한다.

## 잠정 단계

1. 현황 재검증과 설계 확정
   - 현재 build setting, release/package script, workflow, Pages direct download 위치를 대조한다.
   - 기존 산출물이 있으면 app/extension binary slice를 `lipo` 또는 `file`로 확인한다.
   - GitHub runner 아키텍처 제약과 Intel 검증 가능성을 확인한다.
2. universal build와 release gate 구현
   - release/package build가 universal app/extension binary를 만들도록 설정한다.
   - app bundle 내부 실행 파일을 검증하는 helper 또는 script 함수를 추가한다.
   - package/release script와 release workflow에서 검증 실패 시 중단하도록 연결한다.
3. Pages direct download 안내 보강
   - 최신/릴리즈 페이지의 direct DMG 버튼을 유지한다.
   - Intel Mac과 Apple Silicon Mac이 같은 universal DMG를 받는다는 안내를 보강한다.
   - 선택 UI 관련 이슈/마일스톤/계획 문구를 단일 universal DMG 정책으로 정정한다.
4. release 문서와 template 보정
   - README, release note template, v0.1.1 record, release manuals에 Intel Mac + Apple Silicon 지원과 단일 universal DMG 기준을 반영한다.
   - Homebrew Cask와 Sparkle appcast는 단일 universal DMG URL을 유지한다는 전제를 기록한다.
5. 통합 검증과 보고
   - XcodeGen, script syntax, release helper dry-run, 가능 범위의 package/release build, Pages 정적 검증을 수행한다.
   - 실행하지 못한 Intel 실기기 smoke는 이유와 #188 handoff로 기록한다.

## 검증 계획

- 문서/스크립트 정합성:
  - `bash -n scripts/package-release.sh`
  - `bash -n scripts/release.sh`
  - `bash -n scripts/ci/*.sh`
  - `ruby -e 'require "psych"; Psych.parse_file("project.yml")'`
  - `git diff --check`
- build/project 검증:
  - `xcodegen generate`
  - `xcodebuild -project Alhangeul.xcodeproj -scheme HostApp -configuration Release ... build` 또는 release/package script 경로
  - app/extension 실행 파일에 대해 `lipo -verify_arch arm64 x86_64`
  - 가능하면 `scripts/package-release.sh 0.1.1` 또는 rehearsal release script 실행
- release helper 검증:
  - universal 검증 helper에 정상/실패 fixture 또는 실제 bundle 경로 검증
  - `scripts/ci/write-release-notes.sh` sample SHA dry-run
  - `scripts/ci/check-release-notes-template.sh` generated note 검증
- Pages 검증:
  - `rg`로 direct DMG download link가 유지되는지 확인
  - `rg`로 선택 UI 구현 또는 아키텍처별 다운로드 분기 요구가 남아 있지 않은지 확인
  - Pages HTML에 단일 universal DMG 안내가 반영됐는지 확인
- 미실행 가능성이 있는 검증:
  - 실제 Intel Mac 실행 smoke는 현재 장비/runner 접근성에 따라 #188 handoff로 남길 수 있다.
  - signed/notarized public DMG Gatekeeper 검증은 #188에서 수행한다.

## 리스크

- GitHub-hosted runner 아키텍처가 release workflow의 실제 universal build 가능성과 다를 수 있다. build setting과 `lipo` gate를 함께 둬야 한다.
- Swift package dependency 또는 extension embedding이 특정 architecture slice를 누락하면 app 본체만 universal이고 extension이 single-arch인 상태가 될 수 있다.
- Pages에서 direct DMG 버튼만 제공하면 사용자가 Intel Mac 지원 여부를 놓칠 수 있으므로 단일 universal DMG 문구를 다운로드 주변에 명확히 둬야 한다.
- #209 Homebrew Cask는 단일 universal DMG URL을 전제로 하므로, 이번 작업에서 아키텍처별 asset 정책으로 바꾸면 #187/#209 결정과 충돌한다.
- public release 전에는 실제 signed/notarized DMG와 appcast URL을 최종 확인할 수 없으므로 #188 handoff 항목을 명확히 남겨야 한다.

## 승인 요청 사항

- 기본 배포 산출물은 단일 universal DMG로 유지하고, Pages 다운로드 버튼은 direct DMG 방식으로 유지하는 대신 안내 문구로 지원 아키텍처를 명시하는 방향으로 진행해도 되는지 승인 요청한다.
- release/package build에는 `arm64 + x86_64`와 `ONLY_ACTIVE_ARCH=NO`를 명시하고, app/extension binary `lipo` 검증 gate를 추가하는 방향으로 진행해도 되는지 승인 요청한다.
- 실제 Intel Mac 실기기 smoke는 접근 가능한 환경이 없으면 #188 handoff로 남기는 전제로 진행해도 되는지 승인 요청한다.
