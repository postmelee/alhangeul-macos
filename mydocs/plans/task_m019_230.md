# Task M019 #230 수행계획서

## 목적

`v0.1.1` public DMG에서 확인된 앱/DMG 용량 증가 원인을 재확인하고, 아키텍처별 DMG 분리와 Rust core 공유 구조가 실제로 줄일 수 있는 용량과 운영 비용을 비교한다.

완료 후에는 `v0.1.x` release line에서 단일 universal DMG를 유지할지, 아키텍처별 DMG 또는 Rust core 공유 구조를 후속 작업으로 채택할지에 대한 권고와 근거 수치가 문서화되어 있어야 한다.

## 배경

이슈 #230은 `v0.1.0` 대비 `v0.1.1` public release asset과 설치 앱 bundle 크기가 크게 증가한 현상을 출발점으로 한다. 이슈 본문 기준으로 public DMG는 `66,111,087` bytes에서 `91,439,787` bytes로 약 `25.3 MB` 증가했고, mounted `Alhangeul.app` bundle은 `113,644 KiB`에서 `192,680 KiB`로 약 `79.0 MiB` 증가했다.

주요 원인은 `v0.1.1`부터 Intel Mac 지원을 위해 앱 본체와 Quick Look/Thumbnail extension 실행 파일이 `arm64 + x86_64` universal binary가 되었고, Rust `rhwp` staticlib가 HostApp, Quick Look extension, Thumbnail extension에 각각 링크되어 세 실행 파일에 중복 포함되었기 때문이다.

현재 release 정책 문서는 `v0.1.1`부터 Intel Mac과 Apple Silicon Mac을 같은 단일 universal DMG로 지원한다고 명시한다. 따라서 배포 형태를 바꾸려면 단순 빌드 크기뿐 아니라 Sparkle appcast, GitHub Release asset, Pages 다운로드 UX, Homebrew Cask URL/SHA, support matrix, signing/notarization 검증 비용을 함께 비교해야 한다.

## 설계 방향

측정 산출물은 `build.noindex/task230/` 아래에만 만들고 public release 산출물과 혼동하지 않게 한다. signed/notarized public DMG는 이번 작업 범위가 아니므로, local unsigned 또는 rehearsal 성격 산출물은 크기 비교용으로만 기록한다.

첫 단계에서는 현재 정책과 빌드 구조를 고정된 사실로 정리한다. `project.yml`의 HostApp/Preview/Thumbnail target dependency, `scripts/package-release.sh`, `scripts/release.sh`, `scripts/ci/verify-universal-macos-app.sh`, release policy 문서, Sparkle/Homebrew 문서를 조사해 어떤 경로가 단일 universal DMG를 전제로 하는지 표시한다.

두 번째로 같은 commit과 같은 측정 명령에서 universal, arm64-only, x86_64-only 후보를 비교한다. 가능하면 실제 app bundle과 DMG 후보를 만들고, 불가능하거나 signing/packaging 조건이 막히면 app bundle과 실행 파일 크기 기반 시뮬레이션으로 분리해 기록한다. 이때 `v0.1.1` public 수치는 기준선으로만 사용하고, 현 브랜치 산출물과 직접 비교할 때는 commit, Xcode, macOS, code signing 여부를 함께 남긴다.

세 번째로 Rust core 공유 구조는 즉시 구조 변경을 전제로 하지 않고 feasibility와 절감량을 분리해 판단한다. dynamic framework 전환이 extension sandbox, nested framework embedding, hardened runtime, notarization, Sparkle update 후 path/rpath 안정성에 미치는 영향을 먼저 문서화하고, 실제 절감량은 현재 중복 실행 파일에서 Rust staticlib 중복분을 추정하는 방식으로 보수적으로 계산한다.

최종 권고는 다운로드 크기 절감량, 설치 앱 크기 절감량, release 운영 복잡도, 사용자가 잘못된 DMG를 받을 위험, 유지보수 난이도를 함께 놓고 결정한다. `v0.1.x`에서 바로 바꿀 가치가 낮으면 단일 universal DMG 유지와 build setting 최적화만 권고하고, 구조 변경은 `v0.2+` 또는 native viewer 구조 개선과 묶는 방향으로 정리한다.

## 잠정 단계(3~6단계)

1. Stage 1: release 구조 inventory와 측정 프로토콜 확정
   - 단일 universal DMG를 전제로 하는 스크립트, workflow, 문서, Cask, Sparkle 경로를 정리한다.
   - size 측정 명령, output directory, 비교 표 형식을 확정한다.
   - `v0.1.1` public 수치와 현 브랜치 측정 수치를 구분하는 기준을 기록한다.
2. Stage 2: universal/arch별 후보 산출물 측정
   - 현 브랜치 기준 universal Release app/DMG 후보의 크기를 측정한다.
   - arm64-only와 x86_64-only 후보 산출물 또는 시뮬레이션을 만들고 같은 항목으로 측정한다.
   - app 본체, Quick Look extension, Thumbnail extension 실행 파일의 slice와 byte size를 비교한다.
3. Stage 3: 배포 운영 영향 비교
   - 아키텍처별 DMG를 도입할 경우 GitHub Release asset naming, Pages 다운로드, Sparkle appcast enclosure, Homebrew Cask URL/SHA, release note/support matrix 변경을 정리한다.
   - 사용자가 잘못된 DMG를 받는 경우와 Intel Mac/Apple Silicon Mac support 안내 비용을 비교한다.
4. Stage 4: Rust core 공유 구조와 build setting 최적화 검토
   - dynamic/shared Rust core 구조의 signing, notarization, app extension loading, rpath, Sparkle update 리스크를 검토한다.
   - strip/LTO/build setting 조정 후보와 실제 또는 추정 절감량을 분리해 기록한다.
5. Stage 5: 권고안, 최종 검증, 보고
   - 크기 절감량과 운영 비용을 한 표로 비교한다.
   - `v0.1.x` 즉시 적용 여부와 후속 이슈로 분리할 항목을 권고한다.
   - 최종 보고서를 작성하고 PR 게시 전 검증 결과를 정리한다.

## 리스크

- local unsigned 또는 rehearsal 산출물의 DMG 압축률과 public signed/notarized DMG 크기가 완전히 같지 않을 수 있다.
- 현재 machine의 Xcode/macOS/linker 버전 차이가 public `v0.1.1` 산출물 수치와 차이를 만들 수 있다.
- arm64-only와 x86_64-only 후보를 만들기 위해 release script를 임시 우회하면 실제 release workflow와 다른 결과가 나올 수 있다.
- Rust core dynamic/shared 전환은 크기 절감 효과가 있어도 extension embedding, signing, notarization, rpath, update 안정성 검증 범위가 크게 늘어날 수 있다.
- 아키텍처별 DMG는 다운로드 크기를 줄일 수 있지만 Pages/Sparkle/Homebrew/release note 운영 복잡도와 사용자 오선택 가능성을 키운다.
- strip/LTO 설정은 binary size를 줄일 수 있지만 build time, symbolication, crash 조사, Rust/C ABI 경계 디버깅에 영향을 줄 수 있다.
- public release, Sparkle appcast, Homebrew Cask를 실제로 변경하는 작업은 별도 승인 없이는 진행하지 않는다.
