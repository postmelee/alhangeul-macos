## 이슈

- 문서 prefix: `task_m010_54`

## 목적

앱 저장소의 core dependency, lock provenance, 운영 문서, 검증 스크립트를 `edwardkim/rhwp` 기준으로 일관화한다.

완료 후 저장소를 처음 읽는 사람은 이 프로젝트가 `edwardkim/rhwp` core를 macOS 앱, Quick Look preview extension, Finder thumbnail extension으로 연결하는 구조임을 자연스럽게 이해할 수 있어야 한다. 문서는 경위 설명이 아니라 현재 구조 설명으로 정리한다.

## 배경

Issue #28은 render smoke fixture를 앱 저장소 소유 경로로 독립화했고, Issue #29는 `rhwp-core.lock` v2 산출물 provenance 검증을 도입했다. Issue #30은 그 다음 단계로 `RustBridge`를 git-rev dependency로 전환하려 했지만, 먼저 core repository 기준을 통일해야 한다.

현재 확인된 정리 대상은 다음이다.

- `scripts/build-rust-macos.sh`
- `scripts/update-rhwp-core.sh`

현재 lock의 core commit `1e9d78a1d40c71779d81c6ec6870cd301d912626`은 `edwardkim/rhwp`에서도 조회 가능한 commit으로 확인했다. 따라서 우선 commit을 바꾸지 않고 repository 기준과 provenance 표현을 정리한다.

## 설계 방향

이번 작업은 dependency source 전환이 아니라 기준 정합화다. 따라서 `Vendor/rhwp` submodule 구조는 유지하고, URL과 lock provenance만 `edwardkim/rhwp` 기준으로 맞춘다.

`rhwp-core.lock` v2의 의미는 다음처럼 유지한다.

- `rhwp_repo`: core source repository
- `rhwp_branch`: core update 기준 branch
- `rhwp_commit`: reproducible build 기준 commit SHA
- `artifacts`: 해당 commit에서 생성한 Rust bridge 산출물의 sha256/size

스크립트 오류 메시지는 현재 구조에서 수행해야 할 명령을 안내해야 한다. submodule 기준 문서와 스크립트는 이번 작업 범위에서는 유지하되, 후속 Issue #30에서 git-rev dependency 전환 시 다시 갱신한다.

문서 작성 원칙:

- 현재 프로젝트 구조를 설명한다.
- 불필요한 변경 이력 설명을 넣지 않는다.
- `edwardkim/rhwp`가 core source repository임을 직접적이고 일관되게 쓴다.
- 앱 저장소 자체의 GitHub URL과 bundle identifier는 core repository 정리 범위와 분리한다.

## 리스크

- submodule URL 변경 후 로컬 `.git/config`의 submodule remote가 이전 값을 유지할 수 있으므로 `git submodule sync` 검증이 필요하다.
- 현재 lock commit은 유지하지만 산출물 hash 검증이 환경에 따라 실패할 수 있다. 실패 시 commit mismatch와 artifact hash mismatch를 분리해 기록해야 한다.
- 현재 고정 commit이 release tag가 아니라면 안정 기준으로 설명하지 않는다. 최신 release tag로 즉시 전환할 수 없는 경우 lock과 문서에 전환 대기 상태를 명확히 기록한다.
- 과거 완료 보고서를 무리하게 재작성하면 작업 이력이 흐려질 수 있다. 현재 운영 기준 문서와 후속 작업에 영향을 주는 문서를 우선한다.
- Issue #30 계획 문서는 이미 별도 브랜치에 시작 커밋이 있으므로, 이번 작업 완료 후 새 기준으로 재작성하는 편이 안전하다.
