# Task M019 #225 수행계획서

## 목적

알한글 v0.1.2 public release를 위해 upstream `rhwp` 최신 release를 반영하고, 앱 업데이트 후 Finder thumbnail stale cache를 줄이는 maintenance 경로와 사용자-facing provenance 표시를 추가한다.

완료 후에는 앱/Quick Look/Thumbnail extension이 `rhwp v0.7.11` 기준 core와 bundled `rhwp-studio` asset을 사용하고, 새 build 최초 실행 시 LaunchServices/extension registration refresh와 최근 HWP/HWPX 문서 중심의 thumbnail refresh가 수행되어야 한다. About 창에는 앱 버전/build뿐 아니라 bundled `rhwp` release tag와 short commit이 표시되어야 한다.

## 배경

v0.1.1 clean install에서는 Quick Look preview와 Finder thumbnail이 정상 동작하지만, v0.1.0 설치본에서 Sparkle 업데이트로 v0.1.1까지 올라온 환경에서는 Finder가 v0.1.0 thumbnail cache를 계속 보여 깨진 썸네일이 남을 수 있다. 앱 내부에서 전역 `qlmanage -r cache`나 Finder 강제 재실행을 자동 수행하지 않는다는 제약이 있으므로, 앱 업데이트 후 1회 maintenance와 최근 문서 중심의 targeted refresh로 사용자 영향을 줄여야 한다.

#205는 release 표기 정책을 정리했고, #219는 release signing/notarization preflight를 보강했으며, #227은 Rust bridge staticlib 검증 정책을 재정의했다. #225는 이 선행 작업을 v0.1.2 실제 반영과 배포 검증으로 연결하는 release execution 작업이다.

2026-05-11 KST 기준 GitHub release 목록에서 `edwardkim/rhwp` 최신 release는 `v0.7.11`이며, publishedAt은 `2026-05-10T19:50:46Z`이다. 원격 tag `refs/tags/v0.7.11`은 `a9dcdee32b17a7f9a20c609a5ed547e62fb8ebae`로 확인했다. 현재 저장소의 `rhwp-core.lock`과 `RustBridge/Cargo.toml`, bundled legal/provenance 문서는 `v0.7.10`을 기준으로 한다.

## 설계 방향

`rhwp` 업데이트는 기존 진실 원천을 유지한다. Rust core는 `RustBridge/Cargo.toml`의 release tag dependency와 `Cargo.lock`에 반영하고, `rhwp-core.lock`에는 release tag, resolved commit, generated artifact hash/size를 기록한다. bundled Web/WASM viewer/editor는 `Sources/HostApp/Resources/rhwp-studio/manifest.json`을 provenance 기준으로 삼고, asset 검증 script로 필수 파일과 manifest 정합성을 확인한다.

About 창은 #205의 정책에 맞춰 앱 release identity와 bundled `rhwp` provenance를 분리한다. `BuildInfo` 또는 별도 provenance loader가 bundle resource의 manifest/lock-derived metadata를 읽어 `rhwp v0.7.11 (a9dcdee)`처럼 짧게 표시하고, 값이 없을 때는 앱 표시가 깨지지 않게 graceful fallback을 둔다.

업데이트 maintenance는 HostApp 최초 실행 경로에 build-scoped marker를 둔다. 마지막 maintenance build가 현재 `CFBundleShortVersionString`/`CFBundleVersion`과 다르면 1회만 실행하고 marker를 갱신한다. LaunchServices/extension registration refresh는 시스템에 등록된 앱/extension metadata를 새 bundle 기준으로 다시 보게 하는 안전한 경로를 우선하며, 실패해도 앱 실행을 막지 않고 로그로 남긴다.

thumbnail refresh는 전역 cache reset 대신 최근 문서 후보를 제한한다. 후보는 앱이 최근에 연 문서 기록, 보안 범위 bookmark/Recent Documents, 또는 macOS가 허용하는 최근 문서 목록에서 HWP/HWPX 확장자만 골라 구성한다. refresh 동작은 각 파일의 thumbnail 갱신을 유도하는 최소 조작으로 제한하고, 사용자의 파일 내용이나 위치를 변경하지 않는다.

release 실행은 두 단계로 분리한다. 구현과 로컬 검증 단계에서는 v0.1.2 artifact/rehearsal, Sparkle 업데이트 시나리오, Quick Look/Thumbnail smoke까지 확인한다. GitHub Release 게시, stable appcast 갱신, Pages public deploy, 최종 public 배포는 고위험 외부 공개 작업이므로 단계 보고서에서 산출물과 검증 결과를 제시한 뒤 명시 승인을 받아 진행한다.

## 잠정 단계(3~6단계)

1. Stage 1: inventory와 구현계획 확정
   - `rhwp v0.7.11` release/tag, 현재 `v0.7.10` provenance, #205/#219/#227 산출물, release script와 smoke 절차를 대조한다.
   - update maintenance/thumbnail refresh의 안전 경계와 검증 가능한 성공 조건을 확정한다.
2. Stage 2: `rhwp v0.7.11` core/studio 갱신
   - core dependency, Cargo lock, Rust bridge artifact, `rhwp-core.lock`, bundled `rhwp-studio` asset과 manifest를 갱신한다.
   - FFI symbol/header/staticlib 정합성과 asset manifest를 검증한다.
3. Stage 3: About provenance와 update maintenance 구현
   - About 창에 bundled `rhwp` tag/short commit을 표시한다.
   - build-scoped 1회 LaunchServices/extension registration refresh와 targeted thumbnail refresh 경로를 구현한다.
4. Stage 4: v0.1.2 release metadata와 문서 정리
   - app/extension version, README, release record, Pages update 문서, release note/appcast 입력값을 v0.1.2 기준으로 정리한다.
   - public 배포용 외부 공개 변경은 승인 전에는 적용 또는 게시하지 않는다.
5. Stage 5: 통합 빌드와 Finder/Sparkle smoke
   - macOS app/Quick Look/Thumbnail build, Rust bridge 검증, asset 검증, Quick Look/Thumbnail smoke, Sparkle 업데이트 smoke를 수행한다.
   - v0.1.0/v0.1.1에서 v0.1.2로 업데이트한 뒤 stale thumbnail refresh 효과를 확인한다.
6. Stage 6: public release 실행과 최종 보고
   - 승인된 경우 signed/notarized DMG, GitHub Release, Pages, stable appcast, Sparkle update 검증을 수행한다.
   - 산출물 checksum, provenance, 검증 결과, 제한 사항을 최종 보고서에 기록한다.

## 리스크

- `rhwp v0.7.11`의 Rust API, WASM asset, `rhwp-studio` static bundle 구조가 v0.7.10과 달라져 FFI header나 WKWebView resource validation이 깨질 수 있다.
- Rust bridge staticlib 재생성 과정에서 toolchain 또는 target 설치 상태가 부족하면 artifact hash/size 갱신이 막힐 수 있다.
- LaunchServices/extension registration refresh는 macOS 버전별 동작 차이가 있어 성공 로그와 실제 Finder 반영 시점이 다를 수 있다.
- targeted thumbnail refresh는 Finder cache의 내부 정책을 직접 제어하지 못하므로 최근 문서 후보 밖의 파일에는 stale thumbnail이 남을 수 있다.
- Recent document/bookmark 기반 후보 수집은 privacy와 sandbox 경계를 지켜야 하며, 접근 권한이 없는 파일은 조용히 건너뛰어야 한다.
- Sparkle 업데이트 smoke는 이전 public build 설치본, appcast 채널, signing/notary 상태에 의존한다.
- public release 단계는 외부 사용자에게 즉시 영향을 주므로 GitHub Release/appcast/Pages 갱신 전 별도 승인 게이트를 지켜야 한다.
