# Task M015 #118 수행계획서

## 목적

Swift native renderer가 `Equation` 노드를 렌더링하도록 보강해 수식이 포함된 HWP 문서의 기본 미리보기 품질을 높인다.

완료 후 `samples/exam_math_no.hwp` 첫 페이지에서 core render tree에 포함된 수식이 HostApp, Quick Look, Thumbnail이 공유하는 native render 경로에서도 보이는 상태가 되어야 한다.

## 배경

현재 제품 렌더 경로는 `rhwp_render_page_tree`가 반환하는 PageRenderTree JSON을 Swift `RenderTree.swift`가 디코딩하고 `CGTreeRenderer.swift`가 CoreGraphics/CoreText로 그린다.

`RenderTree.swift`는 `EquationNode`를 이미 디코딩하고 `svg_content`, `color`, `font_size`를 받을 수 있다. 하지만 `CGTreeRenderer.swift`의 `.equation` 분기는 현재 렌더링 없이 `break`한다. 그래서 core SVG에는 수식이 포함되지만 앱/Quick Look/Thumbnail 공통 native PNG에서는 해당 수식이 빠진다.

`samples/exam_math_no.hwp` 1페이지 진단 결과 render tree에는 `Equation` 노드 29개가 존재했다. 이는 core 데이터 누락보다는 Swift native renderer의 미구현으로 다루는 것이 맞다.

## 범위(포함·제외)

포함:

- `EquationNode.svgContent`의 실제 SVG fragment 패턴 조사
- 수식 fragment의 기본 텍스트, 선, 위치, 색상, 크기 렌더링 구현
- `Equation` bbox 기준 좌표 변환과 clipping 정책 정리
- 수식 글꼴 fallback과 CoreText 렌더링 경계 확인
- HostApp, Quick Look, Thumbnail 공통 renderer 경로에서 재현 샘플 검증
- `samples/exam_math_no.hwp`와 `samples/eq-01.hwp` 중심 render debug 산출물 기록

제외:

- HWP 수식 문법 자체의 새 파서 구현
- 한컴 viewer와 완전히 동일한 수식 조판 parity
- 임의의 SVG 전체 기능 지원
- WebKit/AppKit 기반 SVG rasterization 도입
- PageLayerTree 기본 경로 전환
- rhwp core upstream 수정

## 설계 방향

우선 `EquationNode.svgContent`를 native renderer 내부에서 처리 가능한 작은 SVG fragment로 보고, 필요한 subset만 파싱해 CoreGraphics/CoreText로 직접 그린다. `Sources/RhwpCoreBridge`는 AppKit/UIKit에 의존할 수 없으므로 WebKit, NSImage, NSAttributedString의 AppKit 기능을 사용하지 않는다.

초기 구현은 `text`, `line`, 좌표, `font-size`, `fill`, `font-style`, `text-anchor` 같은 core 수식 SVG fragment에 실제로 나타나는 속성부터 지원한다. fragment가 XML 문서가 아닐 수 있으므로 안전하게 wrapper를 씌운 뒤 Foundation XML 파서 또는 제한된 구조 파서로 해석하는 방식을 검토한다.

렌더링은 `Equation` 노드의 `bbox`를 기준으로 local SVG 좌표를 page 좌표로 옮겨 그린다. 수식 fragment의 intrinsic size가 bbox와 다를 수 있으므로 조사 결과에 따라 1:1 좌표 사용, bbox scale 보정, baseline 보정 중 가장 보수적인 방식을 선택한다.

지원하지 않는 SVG 요소가 나오면 전체 렌더를 실패시키지 않고 해당 요소만 건너뛰며, 추후 확장 가능하도록 미지원 요소를 좁은 함수 경계 안에 둔다.

## 리스크

- core가 내보내는 수식 SVG fragment의 요소 종류가 문서별로 달라 subset 구현만으로는 일부 수식이 여전히 누락될 수 있다.
- 수식 font fallback이 부족하면 라틴/수학 기호의 폭과 baseline이 core SVG 또는 한컴 viewer와 다르게 보일 수 있다.
- bbox와 SVG local 좌표의 관계를 잘못 해석하면 수식이 작게 보이거나 baseline에서 어긋날 수 있다.
- `Sources/RhwpCoreBridge`의 의존성 경계를 지키기 위해 AppKit/WebKit SVG rasterization을 쓰지 않으므로 구현 범위를 좁게 시작해야 한다.
- Quick Look 환경에서는 별도 rasterization 도구 사용이 제한될 수 있어 native PNG 중심의 검증과 앱 빌드 검증을 병행해야 한다.
