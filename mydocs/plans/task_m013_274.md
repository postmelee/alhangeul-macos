# Task M013 #274 수행계획서

## 목적

PR CI classifier가 `.github/ISSUE_TEMPLATE/*` 변경을 앱 빌드 입력으로 오분류하지 않도록 보정한다.

이 작업의 목적은 Issue Template만 바뀐 PR에서 `macOS validation`이 불필요하게 실행되는 것을 막고, 대신 script/YAML 수준 검증으로 충분히 확인되도록 CI 범위 분류를 정리하는 것이다.

## 배경

PR #273은 `.github/ISSUE_TEMPLATE/*.yml`과 `mydocs/` 문서만 변경했지만 `PR CI`의 `macOS validation` job이 실행되었다.

원인은 `scripts/ci/classify-pr-changes.sh`의 `is_docs_path()`가 현재 `README.md`, `*.md`, `docs/*`, `mydocs/*`만 docs-only 경로로 인정하기 때문이다. `.github/ISSUE_TEMPLATE/*.yml`은 어느 세부 분류에도 매칭되지 않는 non-docs 변경으로 남고, 마지막 fallback에서 `run_macos_build=true`가 된다.

Issue Template은 GitHub metadata이며 앱, RustBridge, renderer, Xcode build input이 아니다. 따라서 `.github/workflows/*` 같은 CI 실행 정의와 구분해 안전하게 docs/metadata 성격으로 분류할 필요가 있다.

## 범위

포함한다.

- `scripts/ci/classify-pr-changes.sh`에서 `.github/ISSUE_TEMPLATE/*`를 docs/metadata 성격으로 분류
- `.github/workflows/*`는 계속 CI/release automation 변경으로 분류되게 유지
- 필요 시 `.github/workflows/pr-ci.yml`의 script checks에 issue template YAML parse 검증 추가
- classifier 로컬 재현 명령으로 #273과 같은 Issue Template-only 변경에서 `run_macos_build=false`가 되는지 확인
- CI 분류 동작과 검증 결과를 단계 보고서와 최종 보고서에 기록

제외한다.

- #273에서 추가한 Issue Template 내용 수정
- `macOS validation` job 자체 변경
- release workflow 변경
- branch protection 또는 required check 설정 변경
- GitHub repository settings 변경
- 제품 Swift/Rust 코드, RustBridge, renderer, Xcode project 변경

## 설계 방향

`.github/*` 전체를 docs-only로 열지 않는다. 이 경로에는 workflow, GitHub Actions 설정, PR template, Copilot instruction처럼 서로 다른 운영 영향 범위가 섞여 있기 때문이다.

이번 작업의 기본 방향은 `.github/ISSUE_TEMPLATE/*`만 명시적으로 docs/metadata 경로로 다루는 것이다. 이렇게 하면 Issue Form 변경은 macOS build fallback을 피하면서도, `.github/workflows/*` 변경은 기존처럼 release/script 검증 범위로 남는다.

YAML 검증은 PR CI의 `script-checks` 안에 추가하는 방향을 우선 검토한다. 이미 workflow YAML syntax check를 Ruby/Psych로 하고 있으므로, Issue Template YAML도 같은 job에서 parse하는 것이 가장 작고 직접적인 검증이다.

## 잠정 단계

1. Stage 1: 현재 classifier 동작을 #273 변경 경로로 재현하고, `.github/ISSUE_TEMPLATE/*` 분류 보정과 YAML 검증 추가 여부를 확정한다.
2. Stage 2: `scripts/ci/classify-pr-changes.sh`를 수정하고, 필요 시 `pr-ci.yml` script checks에 Issue Template YAML parse 검증을 추가한다.
3. Stage 3: classifier case 검증과 YAML/script 검증을 수행하고 단계 보고서를 작성한다.
4. Stage 4: 최종 결과보고서와 오늘할일 상태를 정리하고 PR 게시 준비를 수행한다.

## 검증 계획

- `bash scripts/ci/classify-pr-changes.sh --help`
- `bash -n scripts/ci/classify-pr-changes.sh`
- `ruby -e 'require "psych"; Psych.parse_file(".github/workflows/pr-ci.yml")'`
- `ruby -e 'require "psych"; Dir[".github/ISSUE_TEMPLATE/*.yml"].sort.each { |path| Psych.parse_file(path); puts "Parsed #{path}" }'`
- Issue Template-only 변경 fixture 또는 실제 #273 diff와 같은 경로 목록으로 classifier가 `run_macos_build=false`, `run_release_checks=false`, `docs_only=true`를 내는지 확인
- workflow 변경 자체는 release/script automation 변경으로 계속 분류되는지 확인
- `git diff --check`

## 리스크

- `.github/ISSUE_TEMPLATE/*`를 너무 넓게 docs-only로 보면 실제 automation 영향이 있는 `.github` 변경을 놓칠 수 있다. 경로 예외는 Issue Template 하위로만 제한한다.
- PR CI script checks에 Issue Template YAML parse를 추가하면 기존 PR #273 이전 브랜치에서는 해당 디렉터리가 없을 수 있다. glob이 비어 있을 때 실패하지 않도록 작성해야 한다.
- classifier 검증을 실제 git diff에만 의존하면 future path case를 놓칠 수 있다. 가능하면 fixture성 임시 branch 또는 diff 범위를 만들어 분류 결과를 직접 확인한다.
- macOS validation을 줄이는 변경이므로, 오분류로 필요한 build가 빠지지 않도록 앱/Swift/Rust/workflow 경로의 기존 분류는 건드리지 않는다.

## 승인 요청 사항

이 수행계획 기준으로 구현계획서 작성 후 Stage 1을 진행해도 되는지 승인 요청한다. Stage 1에서는 현재 classifier 동작 재현과 보정 설계를 확정하고, 실제 스크립트/workflow 수정은 Stage 2에서 수행한다.
