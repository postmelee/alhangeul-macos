# Task M015 #120 수행계획서

## 목적

Swift native renderer가 HWP/rhwp core render tree의 텍스트 줄 내 위치와 advance를 더 충실히 반영하도록 개선한다.

완료 후 `samples/20250130-hongbo.hwp` 첫 페이지의 제목 중앙 정렬과 상단 오른쪽 배포일자 간격이 core SVG에 더 가깝게 보이고, HostApp, Quick Look, Thumbnail이 공유하는 native render 경로에서 같은 개선을 얻어야 한다.

## 배경

M015 마일스톤은 첫 출시 전 HostApp, Quick Look, Thumbnail이 공유하는 Swift native renderer의 주요 렌더 갭을 줄이는 작업이다. 이번 이슈는 텍스트 조판 parity 중 글자별 위치와 advance 보존에 집중한다.

현재 `Sources/RhwpCoreBridge/CGTreeRenderer.swift`의 `renderTextRun`은 render tree의 `TextRun` 전체를 하나의 `NSAttributedString`/`CTLine`으로 다시 조판해 그린다. 반면 core SVG는 글자를 개별 x 좌표로 배치한다. 그래서 CoreText가 계산한 폭과 HWP/rhwp core가 계산한 bbox 폭이 다르면 중앙/우측 정렬, 배포일자처럼 run이 나뉜 텍스트, 양끝 정렬, 탭/스페이스가 포함된 줄에서 위치가 어긋날 수 있다.

`Sources/RhwpCoreBridge/RenderTree.swift`에는 `letterSpacing`, `ratio`, `availableWidth`, `lineXOffset`, `extraWordSpacing`, `extraCharSpacing`, `tabStops`, `inlineTabs` 등이 이미 디코딩된다. 하지만 현 renderer는 `letterSpacing`과 `ratio` 일부만 CoreText 속성/폰트 변환으로 반영하고, 줄 내 x 오프셋과 space/tab advance, run bbox 폭을 기준으로 한 위치 보정은 충분히 반영하지 않는다.

## 범위(포함·제외)

포함:

- `samples/20250130-hongbo.hwp`의 제목, 배포일자, 본문 텍스트 위치 차이 원인 재측정
- `samples/re-align-center-hancom.hwp`, `samples/re-align-right-hancom.hwp`, `samples/re-align-justify-hancom.hwp`의 정렬/간격 회귀 확인
- `TextRun` bbox, baseline, `TextStyle`의 `lineXOffset`, `availableWidth`, `extraWordSpacing`, `extraCharSpacing`, `tabStops`, `inlineTabs`, `ratio`, `letterSpacing` 사용 방식 정리
- Swift native renderer에서 run 전체 재조판으로 인한 폭 차이를 줄이는 배치 전략 구현
- 필요 시 render tree 계약 확장 가능성 조사와 Swift decoder 대응
- 기존 render debug 비교 흐름으로 core SVG와 native PNG 차이 기록

## 설계 방향

먼저 현재 render tree만으로 해결 가능한 범위를 분리한다. `TextRun` bbox와 style 값만으로 run 시작 x, baseline, 장평, 자간, 단어/글자 추가 간격, 탭 advance를 보정할 수 있는지 샘플별로 측정한다. 이 범위 안에서는 `CGTreeRenderer.renderTextRun` 내부의 좌표/advance 계산을 보강하고, `Sources/RhwpCoreBridge`의 AppKit/UIKit 직접 의존 금지 규칙을 유지한다.

run 전체를 CoreText 한 줄로 그리는 방식이 bbox 폭을 보존하지 못하는 경우에는 cluster 또는 scalar 단위 drawing을 검토한다. CoreText는 glyph shaping과 fallback에 계속 사용하되, HWP/rhwp core가 준 bbox와 style-derived advance를 우선해 각 글자 또는 cluster의 x 위치를 결정하는 방향을 우선 검토한다. 한글/라틴/공백/탭을 같은 단위로 무리하게 처리하지 않고, 샘플에서 실제로 문제를 만드는 정렬과 space/tab 간격부터 좁게 구현한다.

현재 render tree에 글자별 x/advance 정보가 없어서 보정 한계가 명확하면, `rhwp` core render tree 계약 확장이 필요한지 별도 하위 단계로 정리한다. 이 경우 Swift decoder는 optional 필드로 받아 기존 JSON 호환성을 유지하고, core 계약 변경이 필요하면 단계 보고서에서 upstream/bridge 영향 범위를 명확히 남긴다.

## 잠정 단계(3~6단계)

1. 문제 샘플의 core SVG/render tree/native PNG를 재생성하고 텍스트 run 폭, bbox, baseline, style 값을 비교한다.
2. 현재 render tree 필드만으로 가능한 `renderTextRun` 위치/advance 보정안을 구현한다.
3. 중앙/우측/양끝 정렬, 배포일자 run 분리, space/tab 포함 텍스트 샘플에서 native PNG 개선 여부를 검증한다.
4. 글자별 또는 cluster별 x/advance 계약 확장이 필요한지 판단하고, 필요한 경우 optional decoder/renderer 경계를 설계한다.
5. HostApp/Quick Look/Thumbnail 공통 경로 회귀 검증과 단계 보고서/최종 보고서 정리를 수행한다.

## 리스크

- 현재 render tree에 글자별 x/advance가 없으면 Swift renderer만으로 core SVG와 같은 위치를 완전히 재현하기 어렵다.
- CoreText fallback font의 glyph 폭이 HWP/rhwp core가 사용한 폰트 폭과 다르면 bbox 기반 보정 후에도 일부 텍스트가 미세하게 어긋날 수 있다.
- cluster 단위 drawing을 잘못 나누면 한글 조합, 라틴 ligature, combining mark, emoji 같은 텍스트가 깨질 수 있다.
- bbox 폭에 맞춘 강제 scaling은 정렬은 개선하더라도 글자 모양이나 spacing을 부자연스럽게 만들 수 있다.
- 탭/양끝 정렬/장평/추가 자간이 서로 섞인 문서에서는 보정 우선순위를 잘못 잡으면 다른 샘플 회귀가 생길 수 있다.
- render tree 계약 확장이 필요하면 Rust core, C ABI, Swift decoder, debug script의 변경 범위가 커질 수 있다.
