# Task M019 #214 수행계획서

## 목적

GitHub Pages source를 `workflow` 기반으로 전환한 뒤에도 `main` 브랜치의 `docs/**` 변경이 public Pages에 즉시 반영되도록 docs-only Pages deploy workflow를 추가한다.

완료 후에는 일반 문서 변경이 release workflow를 기다리지 않고 Pages에 배포되어야 하며, docs-only 배포가 release workflow가 게시한 최신 stable Sparkle `appcast.xml`을 repository 안의 오래된 `docs/appcast.xml`로 덮어쓰지 않아야 한다.

## 배경

#206에서 Pages/appcast 배포 경로는 legacy `main/docs` branch publishing이 아니라 GitHub Actions `deploy-pages` 기반으로 전환되었다. 공식 release에서는 `Release Publish DMG` workflow가 signed/notarized DMG를 게시하고, generated `appcast.xml`과 `docs/` 정적 파일을 하나의 Pages artifact로 묶어 배포한다.

이 구조에서는 repository Pages source가 `workflow`이므로, `main/docs`에 문서 변경 commit이 들어가도 branch publishing처럼 Pages가 자동 갱신되지 않는다. 따라서 release와 무관한 `docs/**` 변경을 public Pages에 즉시 반영하려면 별도 docs-only workflow가 필요하다.

단, docs-only workflow가 단순히 `docs/`를 artifact로 업로드하면 `docs/appcast.xml`에 남아 있는 stale feed가 public Sparkle feed를 덮어쓸 수 있다. 이번 작업은 docs-only 배포 경로에서 public 최신 `appcast.xml`을 보존하거나 검증된 최신 feed를 주입하는 기준을 만든다.

## 범위(포함·제외)

포함:

- `main` 브랜치의 `docs/**` push를 감지하는 docs-only Pages deploy workflow 추가
- 필요 시 수동 재실행용 `workflow_dispatch` 입력 추가
- docs-only Pages artifact 생성 시 최신 public `appcast.xml` 보존 전략 구현
- `appcast.xml` 누락, 빈 파일, XML invalid 상태를 배포 전에 실패시키는 검증 추가
- Pages artifact에 `index.html`, `updates/index.html`, `appcast.xml`이 포함되는지 검증
- release workflow의 official stable Pages deployment와 docs-only workflow의 concurrency 정책 정리
- `github-pages` environment와 Pages source=`workflow` 운영 전제 문서화
- PR CI에서 workflow/helper 변경을 release helper check로 검증하도록 필요한 범위 보강
- release/CI/Pages-Sparkle 운영 매뉴얼 갱신

제외:

- v0.1.1 또는 v0.1.2 official release 실행
- Sparkle feed item 생성 정책 변경
- DMG 생성, Developer ID signing, notarization, GitHub Release asset upload 변경
- `main/docs` legacy Pages publishing으로 되돌리기
- public Pages 사이트 디자인 또는 사용자-facing copy 전면 개편
- Homebrew tap 또는 Cask 배포 정책 변경

## 설계 방향

docs-only workflow는 release workflow와 분리한다. release workflow는 official stable release에서 generated appcast를 새로 만들고 Pages를 배포하는 권한 있는 경로로 유지한다. docs-only workflow는 `docs/**` 정적 파일 변경을 반영하되, Sparkle feed는 새로 생성하지 않고 현재 public Pages의 `appcast.xml`을 가져와 artifact root에 보존하는 방향을 우선 검토한다.

기본 후보는 다음 구조다.

- trigger: `push` to `main` with `paths: ["docs/**"]`, 필요 시 `workflow_dispatch`
- permissions: `contents: read`, deploy job에서 `pages: write`, `id-token: write`
- artifact 준비:
  - repository의 `docs/`를 staging directory로 복사
  - public `https://postmelee.github.io/alhangeul-macos/appcast.xml`을 다운로드
  - 다운로드한 appcast가 비어 있지 않고 XML 검증을 통과하는지 확인
  - staging root의 `appcast.xml`을 public appcast로 덮어쓰기
  - `index.html`, `updates/index.html`, `appcast.xml` 존재 확인
- deployment: `actions/upload-pages-artifact@v5`와 `actions/deploy-pages@v5`

public appcast 다운로드가 실패했을 때 stale repository feed를 fallback으로 배포하지 않는 정책을 우선한다. docs-only 배포가 실패하면 문서 반영은 지연되지만 Sparkle stable feed를 손상시키지 않는다. release workflow가 생성한 official appcast가 없거나 public Pages가 아직 초기화되지 않은 경우의 예외는 구현계획서 단계에서 별도 승인 항목으로 분리한다.

concurrency는 release-driven Pages deployment와 docs-only Pages deployment가 같은 Pages 대상에 동시에 쓰지 않도록 같은 group을 쓰는 방향을 검토한다. official release 배포는 취소되면 안 되므로 `cancel-in-progress: false`를 기본 후보로 둔다.

1. 현황 확인과 보존 전략 확정
   - #206 결과물인 release-driven Pages deployment 구조와 현재 helper 동작을 확인한다.
   - repository의 `docs/appcast.xml` 상태와 public appcast 보존 필요성을 대조한다.
   - public appcast 다운로드 보존 방식과 검증된 feed 재생성 방식 중 이번 작업의 구현 방식을 확정한다.
2. 구현계획서 작성
   - docs-only workflow trigger, permission, job 구조, concurrency, 실패 정책을 단계 단위로 확정한다.
   - helper를 기존 `prepare-pages-artifact.sh`에 확장할지 신규 helper로 둘지 결정한다.
   - 승인 후 구현계획서(`task_m019_214_impl.md`)를 작성한다.
3. docs-only workflow와 helper 반영
   - `docs/**` push와 수동 실행 경로를 추가한다.
   - artifact staging에서 public appcast를 보존하고 XML/필수 파일 검증을 수행한다.
   - Pages deployment URL과 appcast 보존 출처를 workflow summary에 남긴다.
4. PR CI와 운영 매뉴얼 갱신
   - workflow/helper 변경이 PR CI의 script/release helper 검증에 포함되는지 확인하고 필요한 dry-run을 보강한다.
   - CI workflow, release distribution, Pages/Sparkle 매뉴얼에 release-driven 배포와 docs-only 배포의 역할 차이와 concurrency 정책을 기록한다.
5. 통합 검증과 보고
   - workflow YAML parse, shell syntax, helper dry-run, XML 검증, `git diff --check`를 수행한다.
   - 실제 Pages deployment는 PR merge 후 GitHub Actions에서 확인해야 하므로, 로컬 검증 한계와 PR 후 확인 항목을 단계 보고서와 최종 보고서에 남긴다.

최종 확인 기준:

- `main`의 `docs/**` 변경으로 docs-only Pages deploy workflow가 실행된다.
- docs-only Pages artifact root의 `appcast.xml`은 public 최신 appcast를 보존하거나 검증된 최신 feed를 사용한다.
- repository의 stale `docs/appcast.xml`이 stable public appcast를 덮어쓰지 않는다.
- release workflow와 docs-only workflow의 Pages deployment concurrency 정책이 문서화되어 있다.
- 관련 release/CI/Pages 운영 매뉴얼이 최신 구조를 설명한다.

## 리스크

- public appcast 다운로드 실패 시 docs-only 배포가 실패할 수 있다. 이 경우 문서 반영 지연을 감수하고 Sparkle feed 손상 방지를 우선한다.
- release workflow와 docs-only workflow가 같은 Pages site를 배포하므로 concurrency가 잘못되면 최신 release appcast가 늦게 끝난 docs-only run에 의해 덮일 수 있다.
- `workflow_dispatch`로 docs-only workflow를 임의 실행할 때도 stale feed fallback을 허용하면 문제가 재발할 수 있다.
- GitHub Pages `github-pages` environment branch/tag policy가 `main` push와 release tag 배포를 모두 허용해야 한다.
- 로컬에서는 `actions/deploy-pages` 실제 deployment를 재현할 수 없으므로 PR/merge 후 GitHub-hosted runner 확인이 필요하다.

## 승인 요청 사항

1. 위 범위와 5단계 진행 구조 승인
2. docs-only workflow를 release workflow와 분리하고 `main`의 `docs/**` push에서 실행하는 방향 승인
3. docs-only 배포에서는 public 최신 `appcast.xml`을 다운로드해 보존하고, 실패 시 stale repository feed로 fallback하지 않는 방향 승인
4. release workflow와 docs-only workflow가 같은 Pages concurrency group을 사용하되 `cancel-in-progress: false`를 우선 검토하는 방향 승인
5. 승인 후 구현계획서(`mydocs/plans/task_m019_214_impl.md`) 작성
