# Task #32 수행 계획서

## 목적

macOS 공개 배포용 release pipeline을 개발용 packaging 흐름과 분리해 설계·구현한다. 목표는 Rust bridge 산출물 검증, Release build, Developer ID 서명, notarization, staple, DMG 생성·서명·공증, Gatekeeper 검증, sha256 산출까지 한 흐름으로 실행 가능한 자동화 경로를 마련하는 것이다.

이번 작업은 실제 배포 실행 자체가 아니라, 저장소 소유자가 명시 지시했을 때 실행할 수 있는 signed release pipeline을 준비하는 작업이다.

## 배경

현재 저장소에는 `scripts/package-release.sh`가 있으며, 이 스크립트는 개발/검증용 Release package를 만드는 역할에 가깝다.

현재 확인된 기준:

- `scripts/package-release.sh`는 Rust bridge build, `xcodegen generate`, Xcode Release build, app zip 생성, sha256 출력까지 수행한다.
- `mydocs/manual/release_distribution_guide.md`는 Developer ID 서명과 notarization 자동화가 아직 없다고 기록한다.
- `project.yml`의 Xcode project 원본은 `AlhangeulMac.xcodeproj`이며, release script도 이 이름을 기준으로 해야 한다.
- `build.noindex/` 아래에 산출물을 둬 Spotlight 검색 혼선을 피해야 한다.
- v0.1.0 현재 목표는 Demo/Preview release이며, Stable release tag 승격은 별도 작업이다.

릴리스/배포/서명/공증/Homebrew Cask는 작업지시자 명시 지시가 있을 때만 수행한다. 따라서 이번 작업의 검증은 credential 없이 가능한 정적 검증, dry-run 또는 missing credential error 확인을 기본으로 한다.

## 설계 방향

개발용 package와 공개 배포용 release를 분리한다.

- `scripts/package-release.sh`
  - 기존 개발/검증용 package script로 유지한다.
  - code signing과 notarization을 필수로 요구하지 않는다.
  - Finder 통합 smoke test용 signed/sealed package가 필요한 경우 release guide에서 한계를 명확히 적는다.

- `scripts/release.sh`
  - 공개 배포용 signed release pipeline을 소유한다.
  - version을 필수 입력으로 받는다.
  - 산출물은 `build.noindex/release/` 아래에 둔다.
  - credential은 환경변수와 keychain profile 이름만 참조한다.
  - 비밀값은 저장소에 기록하지 않는다.
  - 실패 단계와 필요한 사용자 조치를 명확한 error로 출력한다.

환경변수 후보:

- `NOTARY_PROFILE`: `xcrun notarytool` keychain profile 이름
- `DEVELOPER_ID_APPLICATION`: app 서명 identity
- `DEVELOPER_ID_INSTALLER` 또는 `DEVELOPER_ID_DMG`: DMG 서명 identity

script 인터페이스 후보:

```bash
NOTARY_PROFILE=AlhangeulMac \
DEVELOPER_ID_APPLICATION="Developer ID Application: ..." \
DEVELOPER_ID_DMG="Developer ID Application: ..." \
./scripts/release.sh 0.1.0
```

옵션 후보:

- `--skip-notarize`: credential 없는 로컬 rehearsal
- `--output build.noindex/release`: 산출물 경로 지정
- `--keep-staging`: 실패 시 중간 산출물 보존

첫 구현에서는 옵션을 과하게 늘리지 않고, 실패를 명확히 중단하는 안전한 경로를 우선한다.

## 검증 계획

기본 검증:

```bash
git diff --check
bash -n scripts/release.sh
bash -n scripts/package-release.sh
./scripts/build-rust-macos.sh --verify-lock
./scripts/check-no-appkit.sh
xcodegen generate
xcodebuild -project AlhangeulMac.xcodeproj -scheme HostApp -configuration Release -derivedDataPath build.noindex/DerivedDataRelease CODE_SIGNING_ALLOWED=NO build
```

script 동작 검증 후보:

```bash
./scripts/release.sh --help
./scripts/release.sh 0.1.0 --skip-notarize
```

환경에 따라 조정:

- `shellcheck`가 사용 가능하면 `scripts/release.sh`와 변경된 shell script에 적용한다.
- Apple Developer credential이 없는 환경에서는 실제 notarization submit/wait/staple을 실행하지 않는다.
- credential 누락 시 error message가 명확한지 확인한다.

저장소 소유자 별도 승인 후에만 가능한 검증:

- Developer ID signed Release build
- app notarization submit/wait
- app staple
- DMG signing/notarization/staple
- `spctl --assess --type open --context context:primary-signature <dmg>`
- `spctl --assess --type execute <app>`
- Finder Quick Look/Thumbnail smoke test

## 리스크

- 실제 signing/notarization은 로컬 credential과 Apple Developer 계정 상태에 의존한다.
- script가 credential 없는 환경에서도 너무 늦게 실패하면 원인 파악이 어렵다. preflight에서 빠르게 중단해야 한다.
- `package-release.sh`를 공개 배포용으로 과하게 확장하면 개발용 package 경로가 복잡해진다.
- `AlhangeulMac.xcodeproj` 원본이 `project.yml`이라는 규칙을 놓치면 generated project를 직접 수정하는 흐름으로 흐를 수 있다.
- DMG 파일명, app bundle filesystem name, localized display name 정책이 엇갈리면 Finder/ExtensionKit 검증이 흔들릴 수 있다.
- Homebrew Cask URL/sha256 정책을 실제 release 산출물명과 맞추지 않으면 이후 배포 작업이 실패한다.
