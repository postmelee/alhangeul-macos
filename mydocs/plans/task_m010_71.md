## 설계 방향

- **단일 진실 원천 원칙**: 운영 정보는 한 매뉴얼에 본문, 다른 문서는 한 줄 요약 + 링크
  - Demo/Preview vs Stable 채널 표 → `mydocs/tech/core_release_compatibility.md`
  - mydocs 폴더 구조 상세 표 → `mydocs/manual/document_structure_guide.md`
  - Finder smoke test 명령 시퀀스 → `mydocs/manual/build_run_guide.md`
  - 코드 구조 트리 → `mydocs/tech/project_architecture.md` (README는 요약 트리만)
- **가독성**: 한국어 매뉴얼 톤 유지, ASCII 다이어그램은 모노스페이스 정렬에 의존하지 않는 표현으로 교체
- **외부 기여자 친화**: `CONTRIBUTING.md`는 edwardkim/rhwp 포맷을 따라 fork-based PR 워크플로우, 빌드 명령, 폴더/파일명 규칙을 한 문서에서 파악 가능하게 정리
- **링크 경로 일관성**: 절대 경로 대신 상대 경로(매뉴얼 내부 참조), 또는 GitHub blob URL(commit SHA 고정)
- **rename 영향 최소화**: `core_submodule_operation_guide.md` rename은 동일 단계에서 모든 참조를 함께 갱신해 중간 깨짐 방지
- **존재하지 않는 폴더 참조 처리**: `mydocs/feedback/`은 4개 문서가 절차상 가리키므로 폴더만 생성(`.gitkeep`)해 참조를 살리고, 문서 본문은 그대로 유지

## 리스크

- **링크 깨짐 위험 (중)**: C6 rename은 4개 파일에 영향. 단일 커밋 atomic 처리로 완화하지만, 새 참조 누락 시 매뉴얼 탐색이 깨질 수 있음. Stage 3 검증에서 `grep -rln "core_submodule_operation_guide" .` 0건 확인으로 차단.
- **중복 정리 시 정보 손실 (중)**: B1~B4에서 본문을 줄이는 과정에서 5곳/3곳에 흩어진 미세하게 다른 표현이 모두 단일 진실 원천에 반영되지 않을 수 있음. Stage 2 검증에서 정리 전후 핵심 키워드 grep 비교.
- **C4 (`task_m010_28_sample_provenance.md`) 삭제 판단의 영구성 (저)**: 삭제 후 복원이 필요해지면 git history에서 회수 가능하므로 영향 제한적. 단, 작업지시자가 "삭제 또는 다른 폴더 이동" 중 어느 쪽을 원하는지 Stage 3 진입 전 별도 확인.
- **`CONTRIBUTING.md` 톤 일치 (저)**: edwardkim/rhwp의 톤과 본 프로젝트 톤 차이가 있을 수 있음. 본 프로젝트 README/AGENTS의 한국어 톤을 우선 적용.
- **사용자 동시 작업 충돌 (중)**: 작업자가 메인 worktree(`local/task70`)에서 About 윈도우 개발 중. 본 task는 분리 worktree(`/Users/melee/Documents/projects/rhwp-mac-task71`)에서 진행해 격리하지만, 같은 README/AGENTS를 양쪽에서 동시 수정할 위험은 낮음(task70은 코드 중심).
- **devel 진척으로 인한 rebase 필요 (저)**: PR 시점에 devel이 진행되어 있을 수 있음. 그때 `task-final-report` 직전 `git fetch + rebase` 또는 merge 보정 검토.
