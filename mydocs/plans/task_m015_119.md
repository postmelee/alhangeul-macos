# Task M015 #119 수행계획서

## 목적

첫 출시 전 Quick Look preview와 Finder thumbnail의 한글 폰트 fallback 품질을 높인다.

완료 후 Swift native renderer가 `함초롬돋움`, `함초롬바탕`, `한컴돋움`, `한컴바탕`, `돋움`, `바탕`, `굴림`, `궁서`, `HY*` 계열 폰트명을 재배포 가능한 오픈 라이선스 폰트 또는 macOS 시스템 폰트로 일관되게 해석하고, HostApp native renderer, Quick Look, Thumbnail 공통 bitmap 경로에서 같은 결과를 얻어야 한다.

## 배경

이번 연속 작업은 `#119 -> #109 -> #122 -> #123 -> #121 -> #87` 순서로 `devel-webview` 브랜치에서 진행한다. 현재 별도 작업에서 HostApp viewer는 `rhwp-studio` WKWebView 기반으로 전환 중이므로, 이번 #119의 1차 목적은 viewer UI가 아니라 Quick Look/Thumbnail이 공유하는 Swift native render 경로의 첫 화면 품질 강화다.

현재 Swift 경로는 `Sources/RhwpCoreBridge/FontFallback.swift`의 수동 매핑으로 HWP 폰트명을 Apple 기본 폰트에 연결한다. glyph는 나오지만 원문 문서와 폭/metric이 달라져 정렬, 제목 위치, 줄 내 간격이 흔들릴 수 있다. #120에서 글자별 위치와 advance 보정을 추가했더라도, 선택된 fallback 폰트의 metric이 크게 다르면 Quick Look 출력 차이는 계속 남는다.

`devel-webview`에는 `Sources/HostApp/Resources/rhwp-studio/fonts` 아래에 Noto, Nanum, Pretendard, D2Coding, Gowun 등 오픈 라이선스 WOFF2 폰트와 `FONTS.md`가 이미 포함되어 있다. 이 자산은 라이선스/alias 정책의 중요한 reference다. 다만 CoreText native renderer가 WOFF2를 직접 등록해 사용할 수 있는지는 별도 검증이 필요하므로, 구현 단계에서는 WOFF2 직접 재사용 가능성과 TTF/OTF native resource 추가 필요성을 먼저 판정한다.

## 범위(포함·제외)

포함:

- `rhwp-studio/fonts/FONTS.md`와 현재 번들된 WOFF2 목록을 기준으로 재사용 가능한 폰트/라이선스/alias 정책 정리
- CoreText가 현재 bundled WOFF2를 process-local font로 등록할 수 있는지 검증
- WOFF2 직접 등록이 불가능하거나 불안정하면 동일 오픈 라이선스 계열의 native용 TTF/OTF resource를 추가하는 경로 설계
- HostApp, Quick Look extension, Thumbnail extension에서 같은 폰트 등록 정책을 공유하는 구조 구현
- `FontFallback.swift`의 HWP 폰트명 alias를 번들 폰트 우선, 시스템 폰트 보조 fallback 순서로 정리
- 폰트 출처와 라이선스를 `FONTS.md` 또는 별도 third-party license 문서에 기록
- 대표 샘플에서 Quick Look/Thumbnail 공통 native render 결과의 텍스트 위치와 glyph fallback 회귀 확인

제외:

- 한컴/HCR/함초롬/HY/맑은고딕/돋움/굴림 등 권리 확인이 필요한 proprietary font 파일 직접 번들
- 원문 proprietary font와 완전히 같은 metric 보장
- HostApp WKWebView viewer의 CSS/font loading 정책 변경
- rhwp-studio web asset build 파이프라인 재작성
- PageLayerTree 기본 경로 전환
- #109, #122, #123, #121의 도형/이미지/clip/RawSvg parity 구현

## 설계 방향

작업 기준 브랜치는 사용자가 명시한 `devel-webview`로 고정한다. 메인 worktree와 기존 viewer 전환 worktree를 건드리지 않기 위해 `/private/tmp/rhwp-mac-task119`의 `local/task119`에서 작업한다.

첫 단계에서는 `rhwp-studio` 번들 자산을 그대로 native renderer가 쓸 수 있는지 검증한다. 가능하면 중복 폰트 파일을 추가하지 않고, 기존 `Sources/HostApp/Resources/rhwp-studio/fonts`의 오픈 라이선스 WOFF2를 process-local로 등록해 Quick Look/Thumbnail에서도 접근 가능한 resource 배치만 보강한다. 불가능하면 `rhwp-studio`의 `FONTS.md`와 같은 출처/라이선스 기준으로 native용 TTF/OTF subset을 별도 `Shared` resource로 두고, WebView 전용 WOFF2와 native 전용 font resource의 관계를 문서화한다.

폰트 등록은 UI target별 entry point에 흩뿌리지 않고 공통 helper로 만든다. 다만 `Sources/RhwpCoreBridge`에는 AppKit/UIKit 의존을 넣지 않는 규칙을 유지한다. `CoreText`와 `Foundation`만 필요한 process-local registration은 bridge 계층 또는 `Shared` helper 중 어느 위치가 더 안전한지 Stage 1에서 판단하고, Quick Look/Thumbnail extension의 bundle resource lookup 제약을 함께 확인한다.

fallback 정책은 `rhwp-studio`의 alias 목록을 reference implementation으로 삼되, DOM/CSS 구조를 복제하지 않는다. Swift에서는 HWP font family 정규화, bundle font PostScript name 확인, bold/italic trait 적용 가능성, 최종 시스템 fallback 순서를 명시적으로 둔다. 폰트 등록 실패나 resource 누락은 crash가 아니라 기존 시스템 fallback으로 이어져야 한다.

1. `rhwp-studio` 번들 폰트 목록, 라이선스 문서, Swift fallback 매핑, Quick Look/Thumbnail resource 포함 상태를 조사한다.

## 리스크

- CoreText가 WOFF2를 안정적으로 등록하지 못하면 `rhwp-studio` 번들 폰트를 그대로 재사용할 수 없고, native용 TTF/OTF resource를 별도로 관리해야 한다.
- HostApp bundle의 `rhwp-studio/fonts`는 HostApp resource이므로 Quick Look/Thumbnail extension process에서 같은 경로로 접근하지 못할 수 있다.
- 폰트 파일을 extension마다 중복 embed하면 app bundle 크기가 증가한다.
- OFL 또는 무료 배포 폰트라도 이름 사용, 변환, 재배포 조건을 문서화하지 않으면 public release provenance가 약해진다.
- fallback mapping을 크게 바꾸면 #120의 텍스트 위치 개선이 다른 방향으로 보일 수 있어, 전후 비교가 필요하다.
- bold/italic trait가 없는 일부 한글 폰트는 CoreText synthetic trait 결과가 WebView/rhwp-studio와 다르게 보일 수 있다.
- proprietary font명을 오픈 폰트로 대체하는 정책은 한컴/Hancom Viewer와 완전 같은 시각 결과를 보장하지 않는다.
