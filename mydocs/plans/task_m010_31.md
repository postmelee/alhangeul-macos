# Task #31 수행 계획서

## 목적

README와 아키텍처 문서를 처음 읽는 사람이 macOS viewer app, Quick Look preview extension, Thumbnail extension, 공통 Swift 계층, Rust bridge, 생성 산출물, 운영 스크립트의 경계를 빠르게 이해할 수 있도록 재정렬한다.

이번 작업은 Issue #30 이후 상태를 전제로 한다. `Vendor/rhwp` submodule은 제거되었고, 현재 core 기준은 Demo/Preview release를 위한 `edwardkim/rhwp` git `rev` dependency다. Stable release는 upstream release tag가 필수 API를 포함할 때 별도 승격한다.

## 배경

Issue #31은 외부 참고 구현의 명확한 폴더 경계 설명 방식을 참고하되, 앱 저장소의 실제 구조를 바꾸지 않고 문서 구조를 정리하는 작업이다. 현재 저장소도 `Sources/HostApp`, `Sources/QLExtension`, `Sources/ThumbnailExtension`, `Sources/Shared`, `Sources/RhwpCoreBridge`, `RustBridge`로 책임 경계가 이미 나뉘어 있다.

다만 README와 아키텍처 문서에는 제품 타깃, bridge, generated artifact, hyper-waterfall 운영 문서가 함께 노출되어 신규 진입자가 “무엇이 앱 제품 코드이고 무엇이 빌드 산출물/운영 문서인지”를 한눈에 파악하기 어렵다.

또한 #30 완료 후 문서에서 Demo/Preview commit pin과 Stable release tag 기준을 명확히 나눠 설명해야 한다. 현재 v0.1.0 목표는 Demo/Preview release이며, Stable release 표현은 아직 사용하지 않는다.

## 범위

포함:

- `README.md`의 프로젝트 구조, Quick Start, core/release 설명을 타깃별 제품 경계 중심으로 재정렬
- `mydocs/tech/project_architecture.md`의 상위 구조와 소유 경계를 제품 타깃 -> 공통 계층 -> Rust bridge -> generated artifact -> 운영 문서 순서로 보정
- `mydocs/manual/build_run_guide.md`에 신규 진입자가 읽고 실행할 순서가 드러나도록 보강
- 필요 시 `Sources/README.md` 추가
  - HostApp, QLExtension, ThumbnailExtension, Shared, RhwpCoreBridge 역할 요약
- 필요 시 `RustBridge/README.md` 추가
  - Rust FFI crate, generated header/XCFramework, `Cargo.lock`/`rhwp-core.lock` 관계 요약

  - 현재 앱의 native render tree 기반 viewer/preview 방향 유지
  - preview, thumbnail, viewer의 제품 타깃 경계 유지
  - 현재 Rust bridge FFI 경계 유지

제외:

## 설계 방향

문서의 첫 구조 설명은 제품 타깃 중심으로 둔다.

1. `Sources/HostApp`: macOS viewer app
2. `Sources/QLExtension`: Quick Look preview extension
3. `Sources/ThumbnailExtension`: Finder thumbnail extension
4. `Sources/Shared`: app/extension 공통 macOS helper
5. `Sources/RhwpCoreBridge`: AppKit/UIKit 없는 Swift FFI wrapper, render tree decoder, CoreGraphics/CoreText renderer
6. `RustBridge`: Rust C ABI staticlib crate
7. `Frameworks`: generated XCFramework/header/modulemap 산출물
8. `scripts`: build, lock verify, render smoke, release/package 보조 스크립트
9. `mydocs`: hyper-waterfall 작업 문서와 운영 매뉴얼

core 설명은 다음 기준으로 통일한다.

- 현재 v0.1.0 목표는 Demo/Preview release다.
- Demo/Preview는 필수 bridge API가 포함된 resolved commit을 `rev`로 고정한다.
- Stable은 release tag + resolved commit 기준이며 현재 `v0.7.3`은 `build_page_render_tree`, `get_bin_data`가 없어 blocked 상태다.
- branch/floating ref는 배포 기준으로 사용하지 않는다.

렌더링 설명은 native render tree 경로를 기준으로 유지한다. 외부 참고 구현의 다른 preview/viewer 구조는 이 저장소의 장기 방향으로 도입하지 않는다.

## 리스크

- README를 지나치게 운영 문서처럼 만들면 신규 진입자가 제품 구조를 빠르게 읽기 어려워질 수 있다.
- `Shared`와 `RhwpCoreBridge`의 책임을 섞어 쓰면 AppKit/UIKit 금지 경계가 흐려질 수 있다.
- Demo/Preview commit pin을 Stable release처럼 표현하면 #30에서 정리한 배포 기준과 충돌한다.
- 외부 참고 구현 비교 설명이 과해지면 현재 저장소의 native render tree 방향이 약해 보일 수 있다.
- `RhwpMac.xcodeproj` 같은 과거/보조 프로젝트명을 원본처럼 설명하면 `project.yml` 원본 규칙과 충돌한다.
