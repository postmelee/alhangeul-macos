# Task M019 #205 수행계획서

## 목적

알한글 public release에서 앱 버전과 bundled `rhwp` provenance를 어떻게 표기할지 표준 정책으로 정리한다.

완료 후에는 GitHub Release title/body, release note 생성 스크립트, release 문서, README 최신 릴리즈 요약, GitHub Pages 업데이트 문서가 같은 기준으로 앱 버전과 bundled `rhwp` 버전을 안내해야 한다.

## 배경

알한글 release identity는 `Alhangeul v<app-version>` 하나로 유지하지만, 실제 앱에는 `edwardkim/rhwp` core와 `rhwp-studio` asset이 함께 포함된다. 현재 `rhwp-core.lock`과 `Sources/HostApp/Resources/rhwp-studio/manifest.json`은 provenance 진실 원천이고, release note 생성 스크립트도 core/studio tag와 commit을 일부 노출한다.

다만 GitHub Release title의 기본형과 예외형, release body 상단 metadata 구조, README 최신 공개 릴리즈 요약, #204 upstream sync PR/release handoff 문구, GitHub Pages 업데이트 문서의 사용자용 provenance 노출 기준은 아직 하나의 정책으로 묶여 있지 않다.

작업지시자는 추가로 `https://postmelee.github.io/alhangeul-macos/updates/v0.1.1.html`에서 해당 릴리즈가 사용하는 `rhwp` 버전을 안내하고, 관련 release 링크도 추가하기를 요청했다. 현재 `rhwp-core.lock`과 bundled `rhwp-studio` manifest 기준 `v0.1.1` 후보 provenance는 `rhwp v0.7.10`, commit `62a458aa317e962cd3d0eec6096728c172d57110`이다.

- README 최신 공개 릴리즈 요약의 bundled `rhwp` provenance 표기 기준 정리

## 설계 방향

앱 버전은 모든 배포 식별자의 단일 기준으로 유지한다. `Alhangeul v<app-version>`은 Git tag, app bundle version, DMG filename, Sparkle item version, Homebrew Cask version에만 사용하고, bundled `rhwp` 정보는 provenance metadata로 분리한다.

GitHub Release title은 기본적으로 `Alhangeul v<app-version>`을 사용한다. upstream `rhwp` 반영이 release의 중심 변경이고 release note에서 사용자 영향을 설명할 수 있을 때만 `Alhangeul v<app-version> (rhwp v<rhwp-version>)` 형식을 허용한다.

GitHub Release body와 내부 release record는 기계적으로 대조할 수 있도록 app version, `rhwp` core tag/commit, bundled `rhwp-studio` tag/commit, `rhwp-core.lock`, studio manifest 경로를 표준 metadata로 표시한다. 스크립트는 현재 진실 원천인 `rhwp-core.lock`과 `manifest.json`을 읽어 이 값을 생성하고, 수동 release note가 같은 구조를 따르도록 한다.

Pages는 사용자용 안내 표면이므로 GitHub Release body만큼 길게 복제하지 않는다. 대신 릴리즈별 업데이트 문서에는 bundled `rhwp-studio` 또는 `rhwp` 버전을 짧게 노출하고, 필요한 경우 upstream `rhwp` GitHub Release 링크와 알한글 GitHub Release 링크를 함께 둔다. 이번 요청의 `v0.1.1` 페이지는 `rhwp v0.7.10`과 upstream release 링크를 명시한다.

## 잠정 단계(3~6단계)

1. Stage 1: 현황 inventory와 표기 정책 확정
   - release policy, Pages/Sparkle guide, release note script, publish workflow, README, `v0.1.1` Pages 문서의 현재 표기 구조를 정리한다.
   - app version 단독 식별자와 bundled `rhwp` provenance metadata의 경계를 확정한다.
2. Stage 2: release 정책 문서와 사용자-facing 문서 보강
   - release policy와 Pages/Sparkle guide에 표준 metadata, release title 조건, README/Pages 표기 기준을 추가한다.
   - README 최신 릴리즈 요약 기준과 `docs/updates/v0.1.1.html`의 `rhwp v0.7.10`/release 링크 안내를 정리한다.
3. Stage 3: release note 생성 경로와 publish workflow 보강
   - `write-release-notes.sh`가 표준 metadata 구조를 생성하도록 보강한다.
   - 필요 시 template checker와 `release-publish.yml`의 title 생성 입력/조건을 정책과 맞춘다.
4. Stage 4: 통합 검증과 보고
   - release note dry-run, template check, YAML/shell 정적 검증, HTML 구조 확인, diff check를 수행한다.
   - 단계 보고서와 최종 보고서에 변경된 정책과 남은 release 실행 범위를 기록한다.

## 리스크

- GitHub Release title에 `rhwp` 버전을 너무 자주 병기하면 앱 release identity가 두 개처럼 보일 수 있다.
- Pages에 provenance를 너무 길게 넣으면 사용자용 릴리즈 노트가 내부 release record처럼 무거워질 수 있다.
- `rhwp-core.lock`과 bundled `rhwp-studio` manifest가 다른 버전을 가리키는 예외 상황에서 표기 정책이 모호해질 수 있다.
- `release-publish.yml`에 title 조건을 추가할 경우 workflow_dispatch 입력이 늘어나 release 실행자의 실수 가능성이 생길 수 있다.
- `v0.1.1` Pages 문서는 이미 공개된 릴리즈 안내 표면이므로, 사실 정보만 보강하고 release 결과를 새로 확정한 것처럼 쓰지 않아야 한다.
- public release 실행과 Pages deployment는 명시 승인 없이는 수행하지 않는다.
