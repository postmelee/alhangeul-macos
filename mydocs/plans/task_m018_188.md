# Task M018 #188 수행계획서

## 목적

`v0.1.1` patch release 후보를 확정하고, signed/notarized public DMG와 Sparkle appcast를 공식 배포 기준으로 게시한다. `v0.1.0` public 설치본에서 Sparkle 업데이트 감지를 확인한 뒤, 완전 삭제 후 `v0.1.1` public DMG를 다시 설치해 앱, Finder Quick Look, Finder thumbnail smoke를 반복 검증한다.

## 배경

`v0.1.0` public release 이후 설치본 smoke에서 확인된 창 확대/resize WebView runtime error, Finder thumbnail hang, DMG 설치 안내, release communication, CI/release workflow, Pages/appcast deployment, legal notice, Intel Mac/Apple Silicon 단일 universal DMG 안내가 `v0.1.1` milestone 선행 이슈로 정리되었다.

현재 GitHub milestone 기준 #183, #184, #185, #186, #187, #198, #199, #206, #208, #212, #215는 closed 상태이고, #209는 #188에서 public DMG URL과 SHA256을 확정한 뒤 진행할 Homebrew tap 공개 배포 이슈로 남아 있다. #215 comment는 #188 public release 실행 시 signed/notarized DMG 안의 copyright notice와 `Resources/Legal/*` 포함 여부를 반복 확인하라고 handoff했다.

작업지시자는 현재 로컬 Mac에 public release `v0.1.0` 설치본이 있다고 알렸다. 이 설치본은 Sparkle 업데이트 감지 확인 전까지 보존하고, 업데이트 확인이 끝난 뒤 완전 삭제 후 `v0.1.1`을 새로 설치하는 검증 흐름으로 다룬다.

## 설계 방향

먼저 release candidate commit과 선행 이슈 closure를 고정하고, 버전/build number 변경과 release communication 문서의 후보 상태를 실제 `v0.1.1` 기준으로 정리한다. Xcode project는 `project.yml`을 원본으로 취급하고, 필요한 경우 생성 산출물은 기존 build guide 기준으로 갱신한다.

release 산출물은 `Release Publish DMG` workflow를 공식 경로로 우선 사용한다. workflow 실행 전에는 로컬에서 release helper, YAML parse, version 문자열, release note template, universal 검증 helper를 확인한다. public release 실행 자체는 별도 승인 지점으로 두고, 실행 후 GitHub Release asset, checksum, Pages deployment, appcast, Sparkle EdDSA signature를 외부 공개 URL 기준으로 확인한다.

설치본 smoke는 작업지시자의 현재 `v0.1.0` 설치 상태를 검증 자산으로 사용한다. Sparkle 업데이트 감지 전에는 기존 설치본을 삭제하지 않는다. 업데이트 감지 확인 후에는 앱과 Finder extension 상태를 정리하고, `v0.1.1` public DMG를 새로 설치해 앱/Quick Look/Thumbnail smoke를 반복한다. 완전 삭제 절차가 사용자 환경에 영향을 줄 수 있으므로 삭제 범위는 단계 보고 시점에 다시 명시하고 승인받는다.

Homebrew는 이번 이슈에서 public DMG URL과 SHA256을 확정해 `Casks/alhangeul-macos.rb`와 #209 handoff에 필요한 정보를 기록하는 데 그친다. 실제 `postmelee/homebrew-tap` 공개 반영과 tap context 검증은 #209 범위로 넘긴다.

## 잠정 단계(3~6단계)

1. Release candidate 확정과 구현계획 작성
   - 선행 이슈와 release 기록, release workflow 기준, 현재 version/build 값을 확인한다.
   - `v0.1.1` release candidate commit 범위와 검증 gate를 확정한다.
   - 구현계획서(`task_m018_188_impl.md`)를 작성하고 승인받는다.
2. Version/release communication 정리
   - 앱/extension version과 build number를 `0.1.1` / `2` 이상으로 증가시킨다.
   - README, Pages 업데이트 문서, release 기록, release note template 입력값을 실제 후보 기준으로 보정한다.
   - release helper와 문서 검증을 실행한다.
3. Release candidate 로컬 검증
   - Rust lock, viewer asset, Debug/Release build, renderer smoke, universal slice, legal notice 포함 기준을 확인한다.
   - public release workflow 실행 전 필요한 tag, changelog, SHA/provenance 입력을 정리한다.
4. Public release 실행과 공개 URL 검증
   - 승인 후 `v0.1.1` tag와 `Release Publish DMG` workflow를 공식 release 기준으로 실행한다.
   - GitHub Release asset, `.sha256`, signed/notarized DMG, Pages deployment, appcast item, Sparkle EdDSA signature를 확인한다.
   - public DMG SHA256과 Cask handoff 정보를 기록한다.
5. 설치본 smoke와 최종 보고
   - 기존 `v0.1.0` 설치본에서 Sparkle 업데이트 확인이 `v0.1.1`을 감지하는지 확인한다.
   - 승인된 삭제 범위에 따라 기존 설치본을 완전히 제거한 뒤 `v0.1.1` public DMG를 새로 설치한다.
   - 앱 실행, 문서 열기, 창 확대/resize, Sparkle 수동 업데이트 확인, Quick Look/Thumbnail smoke를 반복하고 최종 보고서를 작성한다.

## 리스크

- GitHub Release 공개, Pages/appcast 갱신, Sparkle private key 사용, notarization은 외부 권한과 secret에 의존하므로 로컬에서 완결할 수 없다.
- `v0.1.0` 설치본을 먼저 삭제하면 Sparkle 업데이트 감지 검증 자산을 잃는다.
- 완전 삭제 과정은 사용자 Mac의 앱 설치, Quick Look/Thumbnail extension 등록 상태, Sparkle cache에 영향을 줄 수 있다.
- public release workflow가 실패하거나 appcast가 잘못 게시되면 이미 공개된 asset/Pages URL의 rollback 절차가 필요하다.
- `docs/appcast.xml`은 repository tracked snapshot과 Pages generated artifact의 관계를 혼동하기 쉽다. 실제 검증은 public Pages URL과 workflow artifact 기준으로 기록한다.
- Intel Mac 실기기 smoke는 접근 가능한 환경이 없으면 미실행 사유를 명시해야 한다.

5. 기존 설치본 완전 삭제 범위는 Sparkle 업데이트 감지 확인 후 단계 보고에서 다시 명시하고 승인받는 방식 승인
6. 승인 후 구현계획서(`mydocs/plans/task_m018_188_impl.md`) 작성
