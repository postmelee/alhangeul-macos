# Task #90 수행계획서 - 복학원서.hwp page bbox 밖 layout overflow 렌더 품질 조사

## 목적

`복학원서.hwp` 1페이지에서 일부 텍스트가 page bbox 오른쪽/아래 경계 밖에 배치되는 원인을 분리한다. Swift `CGTreeRenderer` 해석 문제인지, `rhwp` render tree/layout 산출 문제인지, upstream core layout 한계인지 판단하고 후속 조치 방향을 정리한다.

## 배경

Task #84와 Task #85에서 `복학원서.hwp`를 확인하는 동안 page 경계 밖 텍스트가 관찰됐다. 두 작업 모두 시작 전 baseline과 현재 산출 PNG가 byte 단위로 동일했고, `LAYOUT_OVERFLOW` 또는 `LAYOUT_OVERFLOW_DRAW` diagnostic이 함께 출력됐다. 따라서 최근 Viewer clipping 또는 Quick Look 전체 페이지 preview 구현이 만든 새 회귀라기보다, 기존 render tree/layout 산출 또는 native renderer 해석 문제일 가능성이 높다.

현재 HostApp, Quick Look, Thumbnail은 모두 `rhwp_render_page_tree`가 반환한 render tree를 Swift `RenderTree` 모델로 디코딩하고 `CGTreeRenderer`로 bitmap을 그리는 경로를 공유한다. 반면 `rhwp_render_page_svg`는 제품 기준 경로가 아니라 진단용 core SVG 비교 기준으로만 사용한다. 이번 작업은 두 산출물을 비교해 overflow가 어느 계층에서 생기는지 좁히는 조사 작업이다.

## 범위

포함:

- `samples/복학원서.hwp` 1페이지의 render tree JSON, core SVG, native PNG 산출물 재생성
- page size와 render tree root/page bbox, overflow node bbox, text run, parent 구조 분석
- `LAYOUT_OVERFLOW` 계열 diagnostic 발생 위치와 Swift native draw 결과의 관계 확인
- core SVG와 Swift native PNG의 overflow 재현 여부 비교
- Quick Look, Thumbnail, HostApp Viewer가 공유하는 render tree 기반 bitmap 경로 영향 범위 정리
- 앱 저장소에서 보정 가능한 문제인지, upstream `edwardkim/rhwp` 보고가 필요한 문제인지 결론 정리

제외:

- Quick Look 전체 페이지 preview 구조 변경
- HostApp Viewer page cache/LRU 정책 변경
- Thumbnail extension 표시 정책 변경
- 사용자용 PDF export 구현
- `edwardkim/rhwp` core 직접 수정
- release, signing, notarization, Homebrew Cask 관련 작업

## 설계 방향

기존 `render-debug-compare.sh`와 `render_core_native_compare_guide.md` 절차를 우선 사용한다. 제품 경로의 기준은 render tree 기반 native renderer로 유지하고, core SVG는 원인 분리용 비교 산출물로만 취급한다.

분석은 먼저 산출물을 재현 가능한 위치에 남기고, render tree JSON에서 page bbox 밖으로 나가는 노드를 구조적으로 찾는 방식으로 진행한다. Swift renderer가 render tree bbox를 그대로 그려 overflow가 보이는 경우와, render tree에는 clip 또는 container 정보가 있는데 Swift renderer가 이를 무시하는 경우를 구분한다.

앱 저장소에서 고칠 수 있는 문제로 판단되더라도 수행계획 승인 전에는 코드나 매뉴얼을 변경하지 않는다. 구현 단계에서 수정이 필요해지면 구현계획서에 별도 단계와 검증 샘플을 명시한다.

## 잠정 단계

1. Stage 1: 기준 산출물 재생성과 overflow 재현 확인
   - `복학원서.hwp` 1페이지 render tree JSON, core SVG, native PNG, summary를 생성한다.
   - 기존 #84/#85 관찰과 현재 `local/task90` 산출물이 같은 유형인지 확인한다.
2. Stage 2: overflow node와 page geometry 분석
   - page size, root bbox, text node bbox, parent table/paragraph 구조를 확인한다.
   - page bbox 밖으로 나가는 node와 diagnostic의 연결 관계를 정리한다.
3. Stage 3: Swift renderer와 core layout 책임 경계 판단
   - core SVG와 native PNG에서 overflow가 함께 나타나는지 비교한다.
   - render tree에 clip/container 정보가 있는데 Swift renderer가 누락하는지 확인한다.
   - 앱 저장소 수정 가능성 또는 upstream 보고 필요성을 결론낸다.
4. Stage 4: 검증과 후속 조치 자료 정리
   - 필요한 경우 최소 재현 명령, sample hash, 산출물 위치, upstream 보고 초안을 정리한다.
   - 최종 보고서에 Quick Look, Thumbnail, HostApp Viewer 영향 범위와 잔여 리스크를 기록한다.

## 리스크

- **core layout 한계 가능성 (중)**: overflow가 core SVG와 render tree에 모두 존재하면 앱 저장소에서 직접 고칠 수 있는 범위가 제한될 수 있다.
- **Swift clip/overflow 정책 오판 가능성 (중)**: page bounds clipping은 표시상 overflow를 숨길 수 있지만, layout 데이터 자체가 잘못된 문제를 해결하지는 않는다. 조사 결과에서 표시 보정과 layout 원인을 분리한다.
- **diagnostic 해석 불확실성 (중)**: `LAYOUT_OVERFLOW` 계열 메시지가 실제 사용자 표시 품질과 1:1로 대응하지 않을 수 있다. 산출물 비교와 node bbox 분석을 함께 사용한다.
- **pixel diff 환경 의존성 (저)**: `qlmanage` rasterize 또는 diff PNG 생성은 로컬 환경에서 실패할 수 있다. 필수 산출물은 render tree JSON, core SVG, native PNG, summary로 둔다.
