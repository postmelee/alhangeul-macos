# Task #84 수행계획서 - Viewer 다중 페이지 HWP 첫 페이지 초기 로드 누락 수정

## 목적

여러 페이지가 있는 `.hwp` 문서를 HostApp Viewer에서 열었을 때 첫 페이지가 초기 화면에서 로딩 상태로 남거나 보이지 않는 문제를 최소 변경으로 수정한다.

## 배경

분석 결과 HostApp Viewer는 ThumbnailExtension의 `HwpThumbnailRenderCache`나 `HwpPageImageRenderer.renderFirstPage`를 사용하지 않는다. Viewer 본문은 `DocumentViewerStore.pageTrees`에 페이지별 render tree를 보관하고, `DocumentPageContainer.onAppear`에서 `loadPage(_:)`를 호출한다.

현재 `DocumentPageContainer.onDisappear`는 페이지가 사라질 때 즉시 `store.unloadPage(page)`를 호출한다. 여러 페이지 문서에서는 SwiftUI `LazyVStack`/`ScrollView`의 초기 layout 및 virtualization 과정에서 첫 페이지가 조기에 disappear 처리될 수 있고, 이때 `pageTrees[0]`이 제거되면 첫 페이지가 `ProgressView` 상태로 되돌아갈 수 있다. 사용자가 아래 페이지로 이동했다가 돌아오면 `onAppear`가 다시 호출되어 로드되는 증상과도 일치한다.

## 범위

포함:

- HostApp Viewer의 페이지 tree unload 정책 최소 수정
- 첫 페이지 초기 표시 안정성 확인
- HostApp build와 render smoke 검증

제외:

- ThumbnailExtension cache 정책 변경
- Quick Look preview 경로 변경
- Rust core dependency 갱신
- 새 HWP/HWPX fixture 추가
- 대규모 page cache/LRU 설계
- renderer 시각 품질 보정

## 설계 방향

첫 페이지 누락을 직접 유발하는 `onDisappear` 즉시 unload 정책만 좁게 변경한다. 기본 방향은 `DocumentPageContainer.onDisappear`에서 `store.unloadPage(page)` 호출을 제거해, 한 번 로드된 render tree가 문서가 열려 있는 동안 유지되도록 하는 것이다.

문서를 새로 열 때는 기존처럼 `pageTrees.removeAll()`을 수행하므로 문서 간 캐시 오염은 발생하지 않는다. 페이지 tree가 큰 문서에서 메모리 정책이 필요해지면 이번 버그 수정과 분리해 지연 unload 또는 LRU/window eviction을 별도 타스크로 설계한다.

## 리스크

- 즉시 unload를 제거하면 문서가 열려 있는 동안 여러 페이지의 render tree가 유지되어 메모리 사용량이 늘 수 있다. 다만 이번 문제의 직접 원인이 aggressive eviction이므로, 첫 페이지 안정성 수정과 메모리 정책 설계는 분리한다.
- SwiftUI virtualization 동작은 OS 버전과 창 크기에 따라 달라질 수 있다. 자동 검증은 render pipeline의 정상 동작을 보강하고, 실제 UI 재현 검증은 가능한 환경에서 별도 확인한다.
