# Task #84 수행계획서 - Viewer 다중 페이지 HWP 첫 페이지 초기 로드 누락 수정

## 목적

여러 페이지가 있는 `.hwp` 문서를 HostApp Viewer에서 열었을 때 첫 페이지가 초기 화면에서 로딩 상태로 남거나 보이지 않는 문제를 최소 변경으로 수정한다.

## 배경

분석 결과 HostApp Viewer는 ThumbnailExtension의 `HwpThumbnailRenderCache`나 `HwpPageImageRenderer.renderFirstPage`를 사용하지 않는다. Viewer 본문은 `DocumentViewerStore.pageTrees`에 페이지별 render tree를 보관하고, `DocumentPageContainer.onAppear`에서 `loadPage(_:)`를 호출한다.

현재 `DocumentPageContainer.onDisappear`는 페이지가 사라질 때 즉시 `store.unloadPage(page)`를 호출한다. 여러 페이지 문서에서는 SwiftUI `LazyVStack`/`ScrollView`의 초기 layout 및 virtualization 과정에서 첫 페이지가 조기에 disappear 처리될 수 있고, 이때 `pageTrees[0]`이 제거되면 첫 페이지가 `ProgressView` 상태로 되돌아갈 수 있다. 사용자가 아래 페이지로 이동했다가 돌아오면 `onAppear`가 다시 호출되어 로드되는 증상과도 일치한다.

Stage 1에서 즉시 unload 제거 후 `/Users/melee/Documents/samples/table-vpos-01.hwp`를 실제 UI로 확인한 결과, 첫 번째와 두 번째 페이지가 초기에는 보이지 않다가 스크롤 후 늦게 보이는 현상이 남았다. 해당 파일의 page 1/2 render tree와 native PNG는 정상 생성됐다.

Stage 3에서 `DocumentPageNSView` redraw lifecycle을 보강했지만, 작업지시자가 첨부한 2026-04-29 07:11 테스트 영상에서 첫 페이지가 여전히 `ProgressView` 상태로 남는 것이 확인됐다. 따라서 남은 핵심 원인은 drawing invalidation이 아니라, 첫 화면에 필요한 page tree 준비를 SwiftUI `LazyVStack.onAppear`에만 맡긴 구조다. 문서를 연 직후 상단 페이지의 render tree는 store가 먼저 준비해야 한다.

## 설계 방향

첫 페이지 누락을 직접 유발하는 `onDisappear` 즉시 unload 정책을 좁게 변경한다. 기본 방향은 `DocumentPageContainer.onDisappear`에서 `store.unloadPage(page)` 호출을 제거해, 한 번 로드된 render tree가 문서가 열려 있는 동안 유지되도록 하는 것이다.

추가로 `DocumentPageNSView`는 고정 layer 설정을 초기화 경로에 두고, `configure` 시점에 drawing을 invalidation한다. 실제 해결 원인이 page tree 선로딩으로 확인된 뒤에는 window attach 이후 async redraw, frame size redraw, layer redraw policy 같은 추측성 lifecycle 보정은 제거한다.

`복학원서.hwp` 재검증에서 일부 render node가 page bbox 밖에 존재하고 Viewer에서 오른쪽/아래로 새는 회귀가 확인됐으므로, `draw(_:)`에서는 page view `bounds`를 명시적으로 clip한 뒤 렌더러를 호출한다.

최종 핵심 보정은 `DocumentViewerStore`가 문서 로드 완료 시점에 page 0과 page 1 render tree를 선로딩하는 것이다. 첫 화면 페이지 데이터 준비를 SwiftUI lazy lifecycle과 분리하고, `DocumentPageContainer.onAppear`는 이후 스크롤로 진입하는 페이지의 보조 로드 경로로 유지한다.

문서를 새로 열 때는 `pageTrees.removeAll()`로 render tree를 비우지만, SwiftUI/AppKit view 재사용 과정에서 `DocumentPageNSView` 내부 renderer가 유지될 수 있다. `CGTreeRenderer`의 `imageCache`는 HWP 문서 내부 `binDataId`를 key로 쓰므로 문서가 바뀌면 반드시 무효화한다. 페이지 tree가 큰 문서에서 메모리 정책이 필요해지면 이번 버그 수정과 분리해 지연 unload 또는 LRU/window eviction을 별도 타스크로 설계한다.

## 잠정 단계

1. Stage 1: Viewer page tree 즉시 unload 제거
   - `DocumentPageContainer.onDisappear`의 `store.unloadPage(page)` 호출을 제거한다.
   - Thumbnail/Quick Look 경로에는 손대지 않는다.
2. Stage 2: 빌드 및 렌더 smoke 검증
   - HostApp Debug build를 실행한다.
   - `validate-stage3-render.sh`와 대표 다중 페이지 `.hwp` render debug를 확인한다.
3. Stage 3: page view redraw lifecycle 보강
   - `DocumentPageNSView`의 frame size 변경, window attach, configure 시점 redraw 요청을 정리한다.
   - `/Users/melee/Documents/samples/table-vpos-01.hwp` 재현을 기준 검증 케이스에 포함한다.
4. Stage 4: 빌드 및 render debug 재검증
   - HostApp Debug build를 다시 실행한다.
   - 기존 대표 샘플과 `table-vpos-01.hwp` page 1/2 render debug를 확인한다.
   - 이후 실제 UI 영상에서 실패가 확인되어 자동 render debug만으로는 완료 판정하지 않는다.
5. Stage 5: 초기 페이지 render tree 선로딩 보정과 재검증
   - `DocumentViewerStore`가 문서 로드 직후 page 0/1 render tree를 준비한다.
   - `loadPage(_:)`에 page bounds guard를 추가한다.
   - HostApp build와 render debug를 다시 수행한다.
6. Stage 6: 추측성 redraw 보강 제거와 재검증
   - `DocumentPageNSView.viewDidMoveToWindow()`의 추가 redraw를 제거한다.
   - HostApp build와 `table-vpos-01.hwp` render debug를 다시 수행한다.
7. Stage 7: Viewer page bounds clipping 렌더 회귀 수정
   - `DocumentPageNSView`의 layer redraw policy, frame size redraw override를 제거한다.
   - `draw(_:)`에서 page bounds clip을 명시한다.
   - `복학원서.hwp`와 `table-vpos-01.hwp` 기준으로 재검증한다.
8. Stage 8: 문서 전환 시 renderer image cache 무효화
   - `CGTreeRenderer`가 렌더 대상 `RhwpDocument` identity 변경을 감지하면 `imageCache`를 비운다.
   - Viewer에서 문서를 연속으로 열 때 이전 문서 이미지가 새 문서 이미지 위치에 재사용되지 않도록 한다.
   - `복학원서.hwp`, `20250130-hongbo.hwp`, `aift.hwp` 기준 render smoke와 HostApp build를 수행한다.
9. Stage 9: 결과 보고와 잔여 리스크 정리
   - 단계 결과와 최종 보고서를 작성한다.
   - 오늘할일 상태를 갱신한다.

## 리스크

- 즉시 unload를 제거하면 문서가 열려 있는 동안 여러 페이지의 render tree가 유지되어 메모리 사용량이 늘 수 있다. 다만 이번 문제의 직접 원인이 aggressive eviction이므로, 첫 페이지 안정성 수정과 메모리 정책 설계는 분리한다.
- SwiftUI virtualization 동작은 OS 버전과 창 크기에 따라 달라질 수 있다. 자동 검증은 render pipeline의 정상 동작을 보강하고, 실제 UI 재현 검증은 가능한 환경에서 별도 확인한다.
