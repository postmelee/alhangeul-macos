# Task #148 수행계획서: v0.1 서명·공증 배포 수준 결정

## 목적

v0.1 공개 배포를 어떤 수준의 서명·공증·배포 채널로 진행할지 확정하고, 우선순위를 `Developer ID signed + notarized DMG`와 Homebrew Cask 배포로 맞춘다. 동시에 추후 App Store 배포를 별도 후속 경로로 분리하고, 배포 자동화와 배포 안내 문서 템플릿화를 현재 저장소 구조 안에서 어디까지 적용할 수 있는지 점검한다.

## 배경

Issue #148의 원래 범위는 unsigned, ad-hoc signed, Developer ID signed, notarized 배포 옵션 비교와 v0.1 최소 배포 수준 결정이다. 이번 작업지시에서는 Apple Developer 유료 계정이 있는 상태에서 우선 DMG와 brew 배포를 진행하고, 추후 App Store 배포 가능성까지 고려하도록 범위가 넓어졌다.

현재 저장소에는 공개 배포 기반 자산이 이미 일부 존재한다.

- `scripts/release.sh`: Developer ID 서명, app/DMG notarization, staple, Gatekeeper 검증, sha256 생성을 포함한 public DMG pipeline
- `scripts/package-release.sh`: 개발·검증용 Release zip package 생성
- `Casks/alhangeul-macos.rb`: Homebrew Cask 초안
- `mydocs/manual/release_distribution_guide.md`: Apple Developer Program 준비 상태와 release 절차

따라서 이번 작업은 새 배포 체계를 처음부터 만드는 작업이 아니라, 현재 스크립트·Cask·문서가 실제 v0.1 DMG/Homebrew 배포 판단과 맞는지 확인하고 부족한 자동화·안내 템플릿을 보강하는 작업이다.

## 범위

포함:

- v0.1 배포 수준을 `Developer ID signed + notarized DMG` 기준으로 둘 수 있는지 확인
- 현재 entitlements, sandbox, hardened runtime, extension bundle 구조가 서명·공증·Gatekeeper·Quick Look/Thumbnail 등록에 미치는 영향 정리
- DMG public release 절차, rehearsal 절차, 검증 명령, 실패 시 해석 기준 정리
- Homebrew Cask 배포 절차 점검: GitHub Release asset URL, sha256 고정, caveats, tap 운영 방식, 사용자 설치 안내
- 배포 자동화 가능 범위 점검: 버전 검증, DMG checksum 반영, Cask 갱신, release checklist, GitHub Release 게시 전 dry-run
- 배포 안내 문서 템플릿화 가능 범위 점검: release note, 설치 안내, Gatekeeper/확장 등록 안내, rollback/checksum 안내
- App Store 배포는 추후 경로로 분리하고 필요한 사전 조건과 차이를 정리

제외:

- 수행계획 승인 전 코드, 스크립트, 매뉴얼 본문 변경
- 승인 없는 실제 Git tag 생성, GitHub Release 게시, 원격 push, Homebrew tap PR 생성
- 승인 없는 실제 public notarization submission 실행
- App Store Connect 제출, App Store signing/export, `.p8`, app-specific password, `.p12` 같은 secret 생성·저장
- DMG artwork나 별도 auto-update 체계 도입

## 설계 방향

- v0.1의 1차 공개 배포 채널은 notarized DMG를 기준으로 정리한다.
- Homebrew Cask는 GitHub Release에 업로드된 notarized DMG와 고정 sha256을 참조하는 방식으로 둔다.
- rehearsal artifact와 public release artifact의 사용 경계를 문서와 스크립트 출력에서 계속 분리한다.
- secret은 저장소에 기록하지 않고, Team ID, signing identity 표시명, notarytool profile name처럼 비밀이 아닌 운영 식별자만 문서화한다.
- App Store 배포는 Developer ID DMG/Homebrew와 다른 lane으로 보고, sandbox, entitlement, distribution signing, App Store Connect metadata, review 대응을 후속 이슈 후보로 분리한다.
- 자동화는 우선 로컬 검증과 release 준비 자동화에 집중하고, credential이 필요한 게시 작업은 명시 승인 후 실행하는 구조로 둔다.

## 잠정 단계

### Stage 1: 현재 배포 자산과 credential 준비 상태 점검

- `scripts/release.sh`, `scripts/package-release.sh`, `Casks/alhangeul-macos.rb`, release guide, README의 현재 내용을 대조한다.
- Developer ID signing identity와 notarytool profile 확인 명령을 실행 가능한 범위에서 검증한다.
- v0.1 DMG/Homebrew 진행에 필요한 gap을 정리한다.

### Stage 2: v0.1 배포 수준과 사용자 안내 기준 확정

- unsigned/ad-hoc/Developer ID signed/notarized 옵션 비교를 현재 Apple Developer 계정 보유 상태 기준으로 갱신한다.
- v0.1 기본 배포 수준을 notarized DMG로 둘 때의 사용자 안내, Gatekeeper 안내, Quick Look/Thumbnail extension 등록 안내를 정리한다.
- App Store 배포는 후속 범위로 분리하고 차이점과 선행 조건만 남긴다.

### Stage 3: Homebrew Cask와 자동화 가능 범위 보강

- Cask의 URL, version, sha256, caveats, app stanza가 public DMG 기준과 맞는지 확인한다.
- Cask sha256 반영과 release 준비 검증을 자동화할 수 있는지 점검하고, 필요하면 최소 보조 스크립트 또는 체크리스트를 추가한다.
- GitHub Release 게시와 Homebrew tap 반영은 명시 승인 지점으로 분리한다.

### Stage 4: 배포 안내 문서 템플릿화

- release note, 설치 안내, checksum 검증, Gatekeeper/확장 등록 안내, rollback 안내 템플릿을 문서화한다.
- README는 공개 사용자용 최소 안내와 manual 링크 중심으로 유지하고, 운영 상세는 release guide에 둔다.

### Stage 5: 검증과 최종 정리

- shell script syntax, release help/fail-fast, Cask 문법, 문서 링크, grep 기반 stale wording 점검을 수행한다.
- 가능한 경우 rehearsal DMG 생성까지 검증하고, 실제 notarized public DMG 생성은 별도 승인 후 실행한다.
- 단계 보고서와 최종 결과보고서에 DMG/brew/App Store 후속 판단을 남긴다.

## 검증 계획

- `git diff --check`
- `bash -n scripts/release.sh scripts/package-release.sh`
- `./scripts/release.sh --help`
- credential 확인: `security find-identity -v -p codesigning`, `xcrun notarytool history --keychain-profile "alhangeul-notary"`
- Cask 문법 또는 정적 점검: 사용 가능한 Homebrew 환경에 맞춰 `brew style` 또는 `brew audit --cask --new Casks/alhangeul-macos.rb`
- 문서 정합성 점검: `rg`로 `sha256 :no_check`, `rehearsal`, `notarized`, `Homebrew Cask`, `App Store`, `Developer ID` 관련 문구 확인
- 필요 시 `./scripts/release.sh --skip-notarize 0.1.0`로 rehearsal DMG 생성 확인

실제 public notarization, GitHub Release 게시, Homebrew tap PR 생성은 별도 승인 후 수행한다.

## 리스크

- Issue #148의 원래 제외 범위에 Homebrew Cask와 릴리스 게시 작업이 들어 있었으므로, 이번 작업은 작업지시자의 최신 지시에 따라 범위를 확장한다.
- Homebrew 배포는 public GitHub Release asset과 고정 sha256이 필요하므로 실제 DMG 생성·업로드 전에는 최종 Cask 값을 확정할 수 없다.
- App Store 배포는 Developer ID DMG와 인증서, entitlement, sandbox, 심사 정책이 다르므로 v0.1 DMG/Homebrew 작업 안에서 실제 제출까지 묶으면 범위가 커진다.
- Notarization과 Homebrew audit은 로컬 credential, 네트워크, Homebrew 설치 상태에 따라 검증 가능 범위가 달라질 수 있다.
- secret 값이 문서나 shell history에 남지 않도록 명령 예시는 keychain profile과 환경변수 참조만 사용해야 한다.

## 승인 요청 사항

이 수행계획서 기준으로 Task #148의 구현계획서를 작성해도 되는지 승인 요청한다.

승인 시 다음 전제를 적용한다.

- v0.1 1차 공개 배포 기준은 `Developer ID signed + notarized DMG`로 검토한다.
- Homebrew Cask는 notarized DMG GitHub Release asset을 참조하는 방식으로 검토한다.
- App Store 배포는 이번 타스크에서 실제 제출하지 않고, 후속 준비 항목과 템플릿으로 분리한다.
- 실제 public notarization, GitHub Release 게시, Homebrew tap 반영은 구현 단계 중 별도 명시 승인 후에만 실행한다.
