# Task M013 #272 수행계획서

## 목적

GitHub Issue Forms 기반으로 사용자와 개발자가 선택할 수 있는 이슈 템플릿 필수 세트를 추가한다.

이 작업의 목적은 일반 사용자 제보, 문서 호환성 문제, Finder 통합 문제, 설치/업데이트 문제, 기능 제안, 개발자용 하이퍼-워터폴 타스크, 회귀 제보를 서로 다른 입력 흐름으로 분리해 triage 품질을 높이는 것이다.

## 배경

현재 저장소에는 `.github/pull_request_template.md`와 `.github/copilot-instructions.md`는 있지만 `.github/ISSUE_TEMPLATE/` 아래 이슈 템플릿은 없다.

알한글은 macOS 앱, Quick Look preview, Finder thumbnail, HWP/HWPX 문서 호환성, release/install 경로, 개발자 workflow가 함께 존재한다. 하나의 빈 이슈로 모든 제보를 받으면 macOS 버전, 앱 버전, 설치 경로, 문서 형식, 재현 절차, 첨부 가능 여부 같은 필수 triage 정보가 누락되기 쉽다.

이번 작업은 공개 사용자의 진입 장벽을 낮추기 위해 빈 이슈는 허용하되, 주요 제보 흐름에는 Issue Forms를 제공하는 방향으로 진행한다.

## 설계 방향

Issue Forms는 YAML 기반으로 작성한다. 파일명은 GitHub template chooser 정렬이 예측 가능하도록 두 자리 숫자 prefix를 사용한다.

템플릿은 사용자-facing 흐름과 개발자-facing 흐름을 분리한다.

- 사용자 앱 문제 제보: 앱이 열리지 않거나 일반 기능이 동작하지 않는 문제
- 문서 호환성 문제 제보: 특정 HWP/HWPX 파일의 열기, 렌더링, 저장, 내보내기 문제
- Quick Look/Thumbnail 문제 제보: Finder preview, thumbnail, extension 등록 문제
- 기능 제안: 사용자 가치, 사용 흐름, 대안 설명 중심
- 설치/업데이트/릴리즈 문제: DMG, Homebrew, Sparkle, Gatekeeper, notarization 체감 문제
- 개발자용 타스크: 하이퍼-워터폴 작업 등록 전 검토용 구조
- 회귀 제보: 이전 버전 또는 이전 커밋 대비 동작 저하

보안 취약점은 일반 issue template으로 받지 않는다는 안내만 둘 수 있다. 별도 `SECURITY.md` 신설은 이번 범위에서 제외한다.

## 잠정 단계

1. Stage 1: GitHub Issue Forms 문법과 기존 label 체계를 확인하고, 7개 필수 템플릿의 입력 항목/기본 라벨 설계를 확정한다.
2. Stage 2: `.github/ISSUE_TEMPLATE/`에 config와 7개 YAML Issue Form을 추가한다.
3. Stage 3: YAML 구조와 template chooser 정렬, label 참조, 필수 입력 항목을 검증하고 단계 보고서를 작성한다.
4. Stage 4: 최종 보고서와 오늘할일 상태를 정리하고 PR 게시 준비를 수행한다.

## 검증 계획

- `git diff --check`로 whitespace 오류를 확인한다.
- `ruby -e 'require "yaml"; ARGV.each { |f| YAML.load_file(f) }; puts "ok"' .github/ISSUE_TEMPLATE/*.yml`로 YAML parse를 확인한다.
- `rg -n "labels:|title:|name:|description:|blank_issues_enabled" .github/ISSUE_TEMPLATE`로 필수 top-level 항목과 blank issue 설정을 확인한다.
- 기존 label 목록과 템플릿의 `labels` 값이 맞는지 `gh api repos/postmelee/alhangeul-macos/labels` 조회 결과 기준으로 대조한다.
- GitHub UI 렌더링 자체는 로컬에서 완전히 검증할 수 없으므로, PR에서 template chooser 확인을 잔여 검증 항목으로 남긴다.

## 리스크

- Issue Forms YAML은 GitHub의 form schema 제약을 받으므로, 로컬 YAML parse가 통과해도 GitHub UI에서 schema 오류가 날 수 있다.
- 너무 많은 필드를 required로 두면 일반 사용자 제보가 줄어들 수 있다. 재현 절차, 환경, 문제 요약처럼 triage에 필요한 최소 항목만 필수로 둔다.
- 문서 호환성 제보에서 실제 HWP/HWPX 파일 첨부가 어려울 수 있다. 첨부 가능 여부와 개인정보 제거 여부를 분리해 묻는다.
- 개발자용 하이퍼-워터폴 타스크 템플릿이 실제 `task-register` 절차를 대체하는 것으로 오해될 수 있다. 이 템플릿은 제안/초안 입력용이며, 실제 작업 시작은 기존 승인 절차를 따른다는 문구가 필요하다.
- 빈 이슈를 허용하므로 일부 이슈는 구조화되지 않은 상태로 들어올 수 있다. 다만 공개 사용자 접근성을 우선해 이번 작업에서는 허용한다.
