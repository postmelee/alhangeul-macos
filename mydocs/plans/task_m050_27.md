# Issue #27 수행 계획서

## 목적

사용자에게 보이는 앱 이름을 `알한글` 기준으로 정리하고, Finder Quick Look/Thumbnail 검증 절차를 실제 운영에 맞게 정합화한다.

- 사용자 노출 이름은 `알한글`로 통일한다.
- 내부 식별자와 시스템 안정성에 필요한 ASCII 이름은 필요한 범위에서 유지한다.
- Finder 통합 검증은 정상 절차와 문제 복구 절차를 구분한다.
- 문서는 기존 내용을 필요한 부분만 수정한다.

## 배경

최근 점검에서 다음 불일치가 확인됐다.

1. `project.yml`의 실제 build 산출물은 `RhwpMac.app`인데, 일부 문서와 패키징 스크립트는 `알한글.app`을 기준으로 설명한다.
2. `Info.plist` 표시명은 한글이지만 extension 표시명 일부는 영어 표현이 섞여 있다.
3. Finder 통합 문서는 build 산출물을 바로 여는 오래된 절차와 `~/Applications/RhwpMac.app` 고정/`pluginkit -a` 재등록 같은 최근 우회 절차가 섞여 있다.
4. 최근 우회 절차는 시행착오로 얻은 복구 방법일 수 있으므로 표준 운영 절차로 고정하기 전에 성격을 나눠야 한다.

## 범위

### 포함

- 사용자에게 보이는 앱/extension 표시명 점검과 최소 수정
- 패키징 산출물 이름과 Homebrew Cask 앱 이름 정합화
- README와 관련 manual의 앱 실행/패키징/Finder 검증 설명 보정
- Finder 통합 검증 절차를 표준 절차와 troubleshooting 절차로 분리
- 아키텍처 문서의 앱 이름 경계 설명 보정

### 제외

- 릴리스/배포 실행
- 서명/공증 자동화 도입
- core submodule 변경
- Finder 자체 성능 문제 수정
- 과거 완료 보고서의 일괄 수정

## 설계 방향

### 1. 이름 정책

- 사용자 표시명, 배포 앱 번들명, Cask 설치 앱명은 `알한글`을 기준으로 한다.
- bundle identifier, executable name, Swift module name처럼 시스템 식별과 빌드 안정성에 영향을 주는 값은 변경 필요성이 확인된 경우에만 수정한다.
- extension 표시명도 사용자에게 노출될 수 있으므로 한글 기준으로 정리한다.

### 2. Finder 통합 운영 절차

- 표준 검증은 단일 설치본을 기준으로 수행한다.
- DerivedData 산출물을 직접 여는 절차는 단순 앱 실행 확인으로만 제한한다.
- `.app.disabled` 같은 조치는 중복 discovery가 실제로 확인된 경우의 troubleshooting 절차로 분리한다.
