# Task M013 #240 수행계획서

## 목적

Finder Quick Look/Thumbnail extension 검증 과정에서 Debug/테스트 빌드가 LaunchServices/PlugInKit에 중복 등록되는 문제를 반복하지 않도록, 기여자와 에이전트가 따라야 할 extension registration hygiene 규칙을 문서와 표준 검증 도구에 고정한다.

## 배경

#225 v0.1.2 배포 준비 중 Xcode가 `build.noindex/`와 DerivedData 아래의 `Alhangeul.app`을 LaunchServices에 등록하면서 System Settings 확장 목록과 Finder routing 후보가 중복되는 상황을 확인했다. Debug app은 HostApp GUI smoke에는 유효하지만 Finder Quick Look/Thumbnail 등록 검증의 진실 원천으로 쓰면 false positive와 환경 오염을 만들 수 있다.

이 문제는 개인 로컬 환경 정리만으로 끝낼 수 없다. 새 기여자나 에이전트가 같은 검증을 수행할 때 같은 실수를 반복하지 않도록, `CONTRIBUTING.md`, 빌드/실행 가이드, troubleshooting 문서, 필요 시 helper script에 명확한 규칙을 남겨야 한다.

## 범위

포함한다.

- `CONTRIBUTING.md`에 기여자용 Finder/Quick Look/Thumbnail 검증 주의사항 추가
- `mydocs/manual/build_run_guide.md`의 Finder 통합 확인 기준 보강
- `mydocs/troubleshootings/finder_integration_validation_pitfalls.md`의 중복 등록 진단/복구 절차 보강
- 개발 산출물 등록 잔존 여부를 확인하고 필요 시 파일 삭제 없이 registration만 정리하는 helper script 검토 및 추가
- PR 전 확인해야 할 registration hygiene checklist 문서화

제외한다.

- 제품 app/extension bundle identifier, UTI, signing policy 변경
- 전역 `lsregister -kill -r` 또는 `lsregister -delete`를 자동 실행하는 도구 추가
- 사용자 파일, 설치본, legacy app 파일을 자동 삭제하는 cleanup
- Finder/Quick Look renderer 동작 자체의 기능 수정

## 설계 방향

문서는 “짧은 강제 규칙”과 “상세 진단 절차”를 분리한다.

- `CONTRIBUTING.md`: 외부 기여자가 PR 전에 읽는 짧은 금지/허용 규칙과 checklist
- `build_run_guide.md`: Debug build, release package, smoke helper의 역할 차이와 표준 검증 명령
- `finder_integration_validation_pitfalls.md`: 중복 등록이 생겼을 때 확인할 명령, 판정 기준, cleanup-only 절차
- helper script: `pluginkit`, `lsregister`, `qlmanage -r cache`를 조합하되 파일 삭제와 전역 LaunchServices reset은 수행하지 않음

## 검증 계획

- 문서 변경은 `git diff --check`로 whitespace 오류를 확인한다.
- shell script를 추가하거나 수정하면 `bash -n`을 실행한다.
- helper script가 생기면 `--help` 또는 dry-run 성격의 기본 조회 모드를 실행해 destructive action 없이 동작하는지 확인한다.
- cleanup 옵션을 추가하는 경우 파일 삭제가 없고 `lsregister -u`, `pluginkit -r`, `qlmanage -r cache` 범위에 머무르는지 코드 리뷰로 확인한다.
- 최종적으로 `rg`로 문서에 Debug build와 Finder extension 검증 기준이 중복 없이 연결되어 있는지 확인한다.

## 리스크

- LaunchServices/PlugInKit 출력 형식은 macOS 버전에 따라 달라질 수 있다. helper는 실패를 과도하게 단정하지 않고 diagnostics를 남기는 방향으로 작성한다.
- cleanup 명령이 과하면 contributor의 정상 설치본 등록까지 흔들 수 있다. 기본 모드는 확인 전용으로 두고, cleanup은 명시 옵션으로만 수행한다.
- 문서가 여러 곳에 흩어지면 규칙이 불일치할 수 있다. 긴 설명은 troubleshooting 문서에 두고, 다른 문서는 링크와 핵심 규칙만 둔다.

## 승인 요청 사항

이 수행계획 기준으로 Stage 1을 진행해도 되는지 승인 요청한다. Stage 1에서는 문서와 smoke helper 현황을 조사하고 gap을 단계 보고서로 정리한다.
