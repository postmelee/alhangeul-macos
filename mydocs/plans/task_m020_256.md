# Task M020 #256 수행계획서

## 목적

`RhwpDocument`와 `HwpPageImageRenderer`에 Skia PNG backend를 선택적으로 호출할 수 있는 Swift 계층을 추가하고, 실패 시 기존 CoreGraphics render tree 경로로 fallback하는 Shared renderer contract를 만든다.

완료 후에는 Quick Look preview(#257)와 Finder thumbnail(#258)이 같은 Shared renderer contract로 `coreGraphicsOnly`와 `skiaOptIn`을 명시 선택할 수 있어야 한다. 제품 기본 renderer 전환은 이 이슈에서 결정하지 않는다.

## 배경

#255에서 RustBridge에 `native-skia` feature와 `rhwp_render_page_png` C ABI가 추가되었다. 이 ABI는 page index, scale, max-dimension option을 받고 PNG bytes를 반환하며, 실패 원인은 `RhwpRenderStatus`로 구분한다. 성공한 bytes는 `rhwp_free_bytes`로 해제해야 한다.

현재 Swift 경로는 `RhwpDocument.renderPageTree(at:)`가 render tree JSON을 받아 `HwpPageImageRenderer.renderPage`에서 `CGTreeRenderer`로 CoreGraphics bitmap을 만든다. Quick Look 단일 페이지 PNG, 다중 페이지 PDF, Thumbnail cache가 이 Shared renderer를 호출하므로 Skia 도입은 먼저 이 공통 계층에 opt-in backend와 진단 필드를 추가해야 한다.

## 범위

포함:

- `RhwpDocument`에 `rhwp_render_page_png` Swift wrapper 추가
- `RhwpRenderStatus`를 Swift fallback reason으로 매핑하는 구조 추가
- `HwpPageImageRenderer`에 backend 선택 API 추가
- 기본 호출은 기존 CoreGraphics 경로를 유지
- `skiaOptIn` 선택 시 Skia PNG bytes를 `CGImage`로 decode해 기존 `HwpRenderedPage` 계약으로 반환
- Skia 실패, 빈 PNG, PNG decode 실패, invalid page size에서 CoreGraphics fallback
- `backendUsed`, `fallbackReason`, `pngBytes`, `durationMs` 진단 필드를 후속 Quick Look/Thumbnail에서 재사용 가능한 형태로 노출
- embedded thumbnail policy와 50 MB 파일 크기 fallback 정책 보존
- 대표 샘플 1개 이상에서 Skia/CoreGraphics 산출물 차이 또는 byte/size 차이 기록

제외:

- Quick Look provider와 Thumbnail provider의 사용자-facing backend 정책 변경은 #257/#258에서 처리한다.
- Thumbnail cache key에 backend signature를 반영하는 작업은 #258에서 처리한다.
- 다중 페이지 PDF에서 Skia를 기본으로 사용하는 결정은 #257 또는 후속 gate에서 다룬다.
- `Skia first` 또는 `Skia default` 전환 판단은 #259에서 처리한다.
- `CGTreeRenderer` parity 개선, upstream `rhwp` 수정, HostApp viewer renderer 전환은 포함하지 않는다.

## 설계 방향

Swift bridge는 `Sources/RhwpCoreBridge`의 기존 FFI 수명 규칙을 따른다. `rhwp_render_page_png`가 반환한 Rust 소유 buffer는 즉시 Swift `Data`로 복사하고 `defer`에서 `rhwp_free_bytes`로 해제한다. 이 계층에는 AppKit/UIKit 의존성을 추가하지 않는다.

Shared renderer에는 backend 자체와 policy를 분리한다. 기본 policy는 기존 동작과 동일한 `coreGraphicsOnly`이며, `skiaOptIn`은 Skia를 먼저 시도하되 실패하면 CoreGraphics로 fallback한다. 후속 이슈가 명시적으로 opt-in을 전달하기 전까지 Quick Look/Thumbnail의 사용자-facing 결과는 바뀌지 않는다.

진단 정보는 최종 산출물을 기준으로 `backendUsed`를 기록하고, Skia를 시도했다가 CoreGraphics로 넘어간 경우 `fallbackReason`을 남긴다. `pngBytes`는 Skia PNG bytes를 받은 경우에만 byte count를 기록하고, 미사용 또는 실패 시 nil 정책을 우선한다. `durationMs`는 최소 전체 backend 소요 시간을 기록하고, 가능하면 Skia render와 PNG decode/CoreGraphics render 시간을 구분할 수 있는 구조로 둔다.

## 잠정 단계

1. Stage 1: Swift wrapper와 Shared renderer contract inventory
   - #255 ABI handoff와 현재 Swift FFI 패턴을 확인한다.
   - 기존 `HwpRenderedPage` 호출부와 Quick Look/Thumbnail 의존성을 정리한다.
   - 구현계획서(`task_m020_256_impl.md`)에 backend enum, policy, diagnostics, fallback taxonomy를 고정한다.
2. Stage 2: `RhwpDocument` Skia PNG wrapper 추가
   - `rhwp_render_page_png` 호출 wrapper와 Data copy/free 수명 관리를 구현한다.
   - `RhwpRenderStatus`와 Swift error/fallback reason 매핑을 추가한다.
   - wrapper 단위의 정상/실패 경로를 build와 smoke로 확인한다.
3. Stage 3: `HwpPageImageRenderer` backend abstraction과 CoreGraphics fallback 구현
   - `coreGraphicsOnly` 기본값을 유지하면서 `skiaOptIn` 경로를 추가한다.
   - Skia PNG decode를 `CGImageSource`/`CGImage`로 처리하고 실패 시 CoreGraphics fallback한다.
   - `HwpRenderedPage`에 후속 이슈가 읽을 수 있는 diagnostics를 연결한다.
4. Stage 4: 렌더 smoke와 산출물 비교
   - 대표 샘플에서 CoreGraphics와 Skia opt-in 결과를 생성한다.
   - backendUsed, fallbackReason, pngBytes, durationMs가 기대대로 기록되는지 확인한다.
   - 기존 Quick Look/Thumbnail 호출부가 기본 policy에서 회귀하지 않는지 빌드/검증한다.
5. Stage 5: 최종 보고서와 PR 준비
   - stage 산출물과 검증 결과, 잔여 위험을 최종 보고서에 정리한다.
   - 오늘할일 완료 처리와 PR 게시 준비를 수행한다.

## 리스크

- Skia PNG render 성공 후 Swift PNG decode 비용이 Quick Look/Thumbnail extension latency를 늘릴 수 있다.
- Skia와 CoreGraphics가 font fallback, antialiasing, image scaling에서 다른 결과를 만들 수 있다.
- `RhwpRenderStatus`를 Swift fallback reason으로 너무 거칠게 매핑하면 #257/#258 로그가 원인 추적에 부족할 수 있다.
- `HwpRenderedPage` contract를 넓히는 과정에서 기존 cache 또는 PDF 경로의 호출부가 불필요하게 바뀔 수 있다.
- `maximumPixelSize`를 Skia `max_dimension` 또는 `scale`로 변환하는 정책은 #258 요구와 연결되어 있어 초기 구현에서 과도하게 확정하면 후속 변경 비용이 생길 수 있다.
- memory/timeout fallback은 현재 별도 cancellable render 장치가 없으므로 이 이슈에서는 taxonomy와 진단 구조만 먼저 고정될 수 있다.
- 현재 별도 worktree(`/private/tmp/rhwp-mac-task256`)에서 작업을 시작했으므로 메인 worktree의 진행 중인 #274 변경과 오늘할일 파일 merge 충돌 가능성이 있다.

## 승인 요청 사항

1. #256을 `RhwpDocument` Swift wrapper와 `HwpPageImageRenderer` optional backend/fallback contract 작업으로 진행하는 범위 승인
2. 기본 제품 동작은 `coreGraphicsOnly`로 유지하고, Skia는 `skiaOptIn` 명시 선택으로만 시도하는 방향 승인
3. 다음 단계: 승인 후 구현계획서(`mydocs/plans/task_m020_256_impl.md`) 작성과 Stage 1 inventory 진행

승인 전에는 Swift source, Quick Look/Thumbnail provider 정책, build 설정 변경을 진행하지 않는다.
