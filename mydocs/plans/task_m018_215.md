# Task M018 #215 수행계획서

## 목적

Alhangeul macOS 앱 저장소의 주 저작권자 표기와 release legal notice 포함 기준을 현재 소유 구조에 맞게 정리한다.

완료 후에는 루트 `LICENSE`, README, third-party notice, 앱 bundle metadata, 외부 기여 문구가 서로 충돌하지 않아야 한다. 또한 public app bundle 안에서 저장소 license와 third-party notice를 확인할 수 있는 기준을 갖춰야 한다.

## 배경

루트 `LICENSE`는 앱 저장소 자체 license의 진실 원천인데, 기존 저작권자 표기가 upstream `rhwp` 저작권자인 Edward Kim 기준으로 되어 있었다. 작업지시자는 앱 저장소의 주 저작권자가 Taegyu Lee임을 확인했다.

기존 #147은 bundled `rhwp`, `rhwp-studio`, WOFF2 font의 third-party provenance를 정리했지만, Sparkle 의존성, 앱 bundle 내부 legal notice resource, `NSHumanReadableCopyright`, 외부 기여자의 저작권/라이선스 문구는 후속 보강으로 남아 있었다. #177에서 Sparkle이 추가되었으므로 third-party notice에도 반영해야 한다.

현재 대화 중 `LICENSE`의 저작권자 1줄은 이미 Taegyu Lee로 선반영되어 있다. 이 변경은 수행계획 승인 후 구현 단계 변경으로 검증하고 커밋한다.

## 범위(포함·제외)

포함:

- 루트 `LICENSE`의 앱 저장소 주 저작권자 표기 정정
- `THIRD_PARTY_LICENSES.md`에 앱 저장소와 upstream/third-party 저작권 경계 명시
- Sparkle 2.9.1 의존성과 MIT license 고지 추가
- app bundle 내부 legal notice resource 포함 기준 추가
- HostApp `Info.plist`의 `NSHumanReadableCopyright` 추가
- README License 섹션의 앱 license와 third-party notice 경계 보강
- `CONTRIBUTING.md`의 외부 기여 license/copyright 문구 보강
- 필요한 경우 `project.yml`의 HostApp resource 포함 설정 보강
- 수행계획서, 단계 보고서, 최종 보고서, 오늘할일 갱신

제외:

- 법률 자문 수준의 license 해석
- 새 third-party dependency 추가
- `edwardkim/rhwp` upstream source 또는 bundled `rhwp-studio` build output 내용 변경
- Sparkle 업데이트 기능 동작 변경
- public DMG 생성, signing, notarization, Homebrew 배포 실행
- Xcode project 직접 수정

## 설계 방향

루트 `LICENSE`는 Alhangeul macOS 저장소 자체의 license와 주 저작권자를 표시한다. `rhwp`와 `rhwp-studio`의 Edward Kim 저작권 표기는 third-party/upstream 고지로 유지하며, bundled viewer 내부의 `rhwp` about/copyright 문구는 변경하지 않는다.

사용자와 배포 검증자가 찾는 고지 흐름은 `README.md` -> `LICENSE` / `THIRD_PARTY_LICENSES.md` -> `FONTS.md`로 둔다. README에는 중복 표를 만들지 않고 경계와 진입점만 적는다.

app bundle에는 DMG root를 어지럽히지 않는 방식으로 legal notice를 포함한다. 후보 위치는 `Alhangeul.app/Contents/Resources/Legal/`이며, 저장소 license와 third-party notice, font notice를 텍스트 파일로 복사하는 방식을 우선 검토한다.

외부 기여 문구는 "기여물은 본 저장소의 MIT License로 제공된다"는 현재 정책을 유지하되, 별도 계약이 없는 한 기여자가 자신의 기여물 저작권을 보유한다는 점을 명확히 한다.

## 잠정 단계(3~6단계)

1. Stage 1: 현황 inventory와 legal notice 포함 경로 확정
   - `LICENSE`, README, `THIRD_PARTY_LICENSES.md`, `CONTRIBUTING.md`, `Info.plist`, `project.yml`의 현재 책임 경계를 확인한다.
   - app bundle legal notice resource 포함 방식을 결정한다.
2. Stage 2: 저장소 license와 문서 고지 보강
   - `LICENSE`, `THIRD_PARTY_LICENSES.md`, README, `CONTRIBUTING.md`를 정리한다.
   - Sparkle 고지와 앱/upstream 저작권 경계를 반영한다.
3. Stage 3: app bundle metadata와 legal notice resource 포함
   - `NSHumanReadableCopyright`와 bundle resource 포함 설정을 반영한다.
   - Debug/Release 산출물에서 legal notice 파일 포함 여부를 검증한다.
4. Stage 4: 통합 검증과 최종 정리
   - plist lint, XcodeGen parse/generate, resource 존재 확인, keyword scan, `git diff --check`를 수행한다.
   - 단계 보고서와 최종 보고서를 작성한다.

## 검증 계획

계획 단계:

```bash
git diff --check -- mydocs/orders/20260510.md mydocs/plans/task_m018_215.md
```

구현 단계 후보:

```bash
plutil -lint Sources/HostApp/Info.plist
xcodegen dump --type parsed-yaml
xcodegen generate
xcodebuild -project Alhangeul.xcodeproj \
  -scheme HostApp \
  -configuration Debug \
  -derivedDataPath build.noindex/DerivedData \
  CODE_SIGNING_ALLOWED=NO \
  build
test -f build.noindex/DerivedData/Build/Products/Debug/Alhangeul.app/Contents/Resources/Legal/LICENSE
test -f build.noindex/DerivedData/Build/Products/Debug/Alhangeul.app/Contents/Resources/Legal/THIRD_PARTY_LICENSES.md
rg -n "Taegyu Lee|Edward Kim|Sparkle|THIRD_PARTY_LICENSES|NSHumanReadableCopyright|Legal" \
  LICENSE README.md CONTRIBUTING.md THIRD_PARTY_LICENSES.md Sources/HostApp/Info.plist project.yml
git diff --check
```

필요 시 Release configuration build와 public DMG 내부 검증은 별도 승인 범위로 분리한다.

## 리스크

- app bundle에 legal notice 파일을 넣는 방식이 XcodeGen folder resource 규칙과 충돌할 수 있다.
- third-party notice와 README에 같은 정보를 중복하면 이후 core 또는 Sparkle version 변경 때 불일치가 생길 수 있다.
- 외부 기여 문구를 과하게 단정하면 실제 contributor license agreement가 있는 것처럼 오해될 수 있다.
- `rhwp-studio` 내부 upstream copyright 문구를 잘못 수정하면 upstream provenance를 훼손할 수 있다.
- 법률 자문 수준의 해석은 이번 범위에서 제외하므로, 배포 전 별도 법무 검토가 필요할 수 있다.
