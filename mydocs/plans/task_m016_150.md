# Task M016 #150 수행계획서

## 목적

WKWebView 기반 HostApp viewer가 bundled `rhwp-studio` asset, custom resource scheme, document scheme, WASM/JS runtime 중 어느 한 경로에서 실패해도 사용자가 빈 화면이나 무응답 상태를 보지 않도록 fallback과 진단 정보를 보강한다.

완료 후에는 정상 bundle에서는 HWP/HWPX 샘플이 기존처럼 열리고, asset 누락 또는 document scheme 실패를 인위적으로 만들었을 때 HostApp이 실패 원인과 재시도/다시 열기 흐름을 명확히 보여야 한다.

## 배경

현재 HostApp viewer는 `Sources/HostApp`이 소유한다. `DocumentViewerStore`가 문서 bytes와 revision을 `RhwpStudioDocumentPayload`로 보관하고, `RhwpStudioWebView`가 `WKWebView`를 구성한다. 정적 asset은 `alhangeul-studio://app` scheme의 `RhwpStudioResourceSchemeHandler`가 bundle의 `Resources/rhwp-studio`에서 제공하고, 문서 bytes는 `alhangeul-document://current?revision=...` scheme의 `RhwpStudioDocumentSchemeHandler`가 제공한다.

현재 `RhwpStudioResourceLocator.loadURL(for:)`는 로드 전에 `rhwp-studio/index.html` 존재까지만 확인한다. `scripts/verify-rhwp-studio-assets.sh`는 release/build-time 검증으로 JS, CSS, WASM, override CSS, manifest provenance를 확인하지만, 앱 런타임의 실패 메시지와 직접 연결되어 있지는 않다. 개별 asset 누락, MIME mismatch, `alhangeul-document` revision mismatch, JS/WASM 초기화 실패는 navigation error, timeout, injected bridge의 `"error"` 메시지, 또는 banner 수준 오류로 섞여 보일 수 있다.

#145 이후 release artifact/provenance 기준은 `rhwp`와 `rhwp-studio` `v0.7.10`으로 정리되었다. #150은 해당 bundle asset 기준을 제품 런타임 fallback과 smoke 절차에 연결하는 작업이다.

## 범위(포함·제외)

포함:

- HostApp runtime에서 `rhwp-studio` 필수 asset 존재를 사전 점검하는 경로 검토 및 보강
- `RhwpStudioResourceSchemeHandler`의 실패 메시지에 scheme, URL, 상대 asset path, 원인 구분을 남기는 진단 보강
- `RhwpStudioDocumentSchemeHandler`의 document fetch 실패를 asset failure와 구분해 사용자 메시지와 로그에 반영
- WKWebView navigation failure, load timeout, web content process termination, JS/WASM runtime error를 사용자 fallback 상태로 연결
- 재시도 또는 파일 다시 열기 같은 사용자가 즉시 취할 수 있는 recovery action 제공
- release bundle asset 검증 스크립트와 HostApp WKWebView smoke 절차의 연결 지점 정리
- 정상/asset 누락/document scheme 실패 시나리오 검증 결과를 단계 보고서에 기록

제외:

- 개발 서버 fallback을 제품 경로로 사용하는 구조
- `rhwp-studio` upstream 렌더 품질, WASM 내부 parser, editor UI 자체 수정
- Quick Look/Thumbnail native renderer 변경
- 손상·대용량 HWP/HWPX 파일 opening fallback 보강 (#149 범위)
- public DMG 생성, 서명, 공증, 배포 실행 (#166 범위)
- Xcode project 직접 수정 (`project.yml` 원본 규칙 유지)

## 설계 방향

실패 원인을 먼저 taxonomy로 나눈 뒤, 사용자에게는 간결한 fallback 상태를 보여주고 개발자 진단에는 충분한 URL/path/scheme 정보를 남긴다. 구분 후보는 다음과 같다.

| 구분 | 예시 | 사용자 메시지 방향 | 진단 정보 |
|------|------|------------------|-----------|
| resource preflight | `rhwp-studio` directory, `index.html`, JS/CSS/WASM 누락 | viewer asset이 설치본에 누락됨 | bundle path, missing file/count |
| resource scheme | `alhangeul-studio://app/assets/...` 응답 실패 | viewer asset을 읽지 못함 | scheme URL, relative path, underlying error |
| document scheme | current revision 불일치, payload 없음 | 문서 데이터를 viewer에 전달하지 못함 | document URL, requested/current revision |
| navigation/runtime | WebKit provisional navigation, timeout, process termination | 웹 viewer 로딩 실패 또는 중단 | loading URL, WebKit domain/code |
| JS/WASM | injected bridge `"error"`, unhandled rejection | viewer runtime 초기화 실패 | JS message, filename/line 가능 시 |

UI는 기존 banner만으로 끝내지 않고, 문서 화면을 대체할 수 있는 fallback 상태를 우선 검토한다. 사용자가 할 수 있는 조치는 "다시 시도", "다른 파일 열기", "Finder에서 원본 보기"처럼 HostApp이 이미 소유한 흐름을 재사용한다. 기존 overlay와 toolbar 동작을 크게 흔들지 않도록 `DocumentViewerStore`의 WebView loading/error 상태를 확장하는 방향을 우선한다.

빌드/배포 전 검증은 기존 `scripts/verify-rhwp-studio-assets.sh`를 계속 진실 원천으로 두되, 런타임 validator가 같은 기준 중 최소 필수 항목을 공유하거나 문서로 명확히 연결되게 한다. 제품 경로에서는 개발 서버나 네트워크 fallback을 사용하지 않는다.

## 잠정 단계(3~6단계)

1. Stage 1: 현행 WKWebView loading/failure 경로 inventory
   - `DocumentViewerStore`, `RhwpStudioWebView`, resource/document scheme handler, host bridge script, asset verify script를 대조한다.
   - 정상 로드와 실패가 현재 어떤 `errorMessage`/`webViewErrorMessage`/timeout으로 노출되는지 정리한다.
2. Stage 2: fallback 상태와 diagnostics 설계
   - resource/document/navigation/runtime failure taxonomy를 확정한다.
   - 사용자 fallback UI, recovery action, 로그/진단 문자열, smoke 시나리오를 구현 계획서에 구체화한다.
3. Stage 3: 런타임 asset/document failure 보강 구현
   - 필수 asset preflight 또는 runtime validator를 추가한다.
   - resource/document scheme error와 WebView navigation/runtime error를 구분해 store/UI로 전달한다.
   - 필요한 경우 JS error bridge를 보강한다.
4. Stage 4: negative smoke와 Release bundle 연결 검증
   - 정상 Debug/Release build에서 샘플 문서 열기 smoke를 수행한다.
   - 빌드 산출물 복사본에서 JS/CSS/WASM 또는 document scheme 실패를 인위적으로 만들어 fallback 문구와 recovery action을 확인한다.
   - `scripts/verify-rhwp-studio-assets.sh`와 release bundle 검증의 역할을 단계 보고서에 연결한다.
5. Stage 5: 최종 보고와 M16 release gate 연결
   - 남은 #149, #151, #146과 겹치지 않는 범위를 최종 보고서에 명시한다.
   - release note/known limitation에 넘겨야 할 fallback 문구 또는 smoke 결과를 정리한다.

## 리스크

- WKWebView 하위 resource load 실패는 항상 navigation failure로 올라오지 않을 수 있다. JS error/rejection bridge와 scheme handler diagnostics를 함께 봐야 한다.
- build 산출물에서 asset을 제거하는 negative smoke는 코드 서명 상태를 바꿀 수 있으므로, public signed/notarized artifact가 아니라 Debug 또는 복사본 기준으로만 수행해야 한다.
- runtime validator가 `scripts/verify-rhwp-studio-assets.sh`의 모든 release 검증을 중복하면 유지보수 비용이 커진다. 제품 런타임에는 사용자에게 필요한 최소 필수 항목만 넣고, provenance/hash 성격의 검증은 스크립트에 남기는 편이 낫다.
- fallback UI를 크게 바꾸면 #149 손상 파일 fallback이나 #146 한계 문서화와 메시지가 충돌할 수 있다. 이번 task는 asset/document delivery 실패에 집중한다.
- `rhwp-studio` upstream bundle 파일명이 hash 기반이라 asset 이름을 고정하면 다음 sync 때 깨질 수 있다. 파일명 고정보다 `index-*.js`, `index-*.css`, `rhwp_bg-*.wasm` 패턴 기준을 우선 검토한다.
