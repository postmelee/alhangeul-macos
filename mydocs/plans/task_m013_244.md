# Task M013 #244 수행계획서

## 목적

`devel-webview`에 누적된 실제 배포/제품 개발 라인을 외부 기여자가 자연스럽게 사용할 수 있는 `devel` 기준으로 승격하고, 기존 `devel`의 Swift native viewer/editor 장기 개발 라인은 별도 브랜치로 보존한다.

이 작업의 최종 목적은 일반 제품 변경 PR의 기본 base를 `devel`로 단순화하고, native renderer/editor 실험은 이름에서 성격이 드러나는 별도 라인으로 분리해 기여자와 에이전트가 PR 대상을 잘못 고르는 위험을 줄이는 것이다.

## 배경

현재 저장소의 실제 제품/배포 변경은 `devel-webview`에 누적되어 있다. 반면 `devel`은 #119/#123/#109 계열 Swift native renderer 작업을 포함한 장기 native 전환 라인으로 남아 있다. 이 구조는 내부적으로는 의도가 있었지만, 외부 기여자 관점에서는 `devel`이 일반 개발 기준처럼 보이므로 PR base 선택이 혼란스럽다.

사전 조사에서 다음을 확인했다.

- `origin/devel...origin/devel-webview`는 공통 조상 이후 `devel` 전용 33커밋, `devel-webview` 전용 432커밋으로 크게 분기되어 있다.
- `devel <- devel-webview` 직접 merge는 `project.yml`, `CGTreeRenderer.swift`, `FontFallback.swift`, `FontResourceRegistry.swift`, 생성 Xcode project, 일부 작업 문서에서 충돌이 발생한다.
- `origin/main <- origin/devel-webview` 가상 merge는 충돌 없이 자동 병합 가능했다.
- `origin/main`에는 v0.1.2 public release 이후 README/릴리스 문서 변경이 있고, `origin/devel-webview`에는 #230/#240 후속 작업이 있다. 새 제품 개발 기준은 두 라인의 관계를 함께 고려해야 한다.

## 범위

포함한다.

- 현재 `main`, `devel-webview`, `devel`의 commit topology와 충돌 범위 재확인
- 기존 `devel` native 라인 보존 브랜치 이름과 전환 방식 결정
- 새 제품 개발 기준 브랜치가 포함해야 할 commit 범위 결정

- 전환 작업자가 실행할 원격 branch migration runbook 작성 및 승인된 범위에서 실행

## 설계 방향

직접적인 `devel <- devel-webview` 병합은 피한다. 이미 분기된 native 라인을 보존해야 하고, 충돌도 renderer 공통 코드와 프로젝트 설정에 몰려 있어 “자동 병합 후 충돌 해결” 방식은 정책 전환 작업으로는 위험하다.

기본 방향은 다음과 같다.

- 기존 `devel`은 `native-viewer` 같은 명시적인 이름의 보존 브랜치로 남긴다.
- 제품 개발 기준은 `devel-webview`를 중심으로 하되, `main`의 public release 후속 문서/릴리스 변경까지 포함할지 Stage 1에서 확정한다.
- 외부 기여 안내에서는 일반 제품 변경 base를 `devel`로 안내하고, Swift native viewer/editor 변경만 native 보존 브랜치로 안내한다.
- `devel-webview`는 전환 직후 즉시 삭제하지 않고, legacy alias 또는 임시 호환 브랜치로 유지할지 별도 승인으로 결정한다.
- 원격 branch rename/update, branch protection/default PR base 변경은 Stage별 보고 후 작업지시자 명시 승인으로만 실행한다.

## 잠정 단계

1. Stage 1: 현재 branch topology, 원격 브랜치, workflow branch filter, 문서 참조 inventory를 정리하고 전환 선택지를 확정한다.
2. Stage 2: 제품 개발 기준 commit 범위와 native 보존 브랜치 이름을 확정하고, 원격 branch migration runbook을 작성한다.
3. Stage 3: README/CONTRIBUTING/AGENTS/manual/tech 문서와 PR template의 branch 안내를 새 정책에 맞게 갱신한다.
4. Stage 4: CI/release workflow branch filter와 automation 안내를 새 `devel` 기준으로 정렬하고 로컬 검증을 수행한다.
5. Stage 5: 작업지시자 승인 범위 안에서 원격 branch migration을 실행하거나, 실행 직전 runbook과 수동 설정 checklist를 최종 보고서에 고정한다.
6. Stage 6: 최종 검증, 최종 보고서, 오늘할일 상태 갱신, PR/후속 승인 절차를 정리한다.

## 검증 계획

- `git status --short --branch`로 메인 worktree의 `local/task243`와 분리 worktree의 `local/task244` 변경이 섞이지 않았는지 확인한다.
- `git rev-list --left-right --count`와 `git merge-tree --write-tree --name-only --messages`로 `main`, `devel-webview`, `devel` 관계와 충돌 파일을 기록한다.
- `rg -n "devel-webview|native viewer|native renderer|PR base|통합 브랜치|branch filter"`로 문서와 workflow 참조를 점검한다.
- workflow를 수정하면 YAML parse 또는 기존 repo 검증 방식으로 문법을 확인한다.
- shell helper를 수정하지 않는 한 제품 build는 필수 검증에서 제외하고, branch/workflow 문서 변경 검증에 집중한다.
- 원격 branch migration을 실행하는 경우 실행 전후 `git ls-remote --heads origin`과 `gh` 조회로 브랜치 ref가 의도대로 보존됐는지 확인한다.
- 모든 단계에서 `git diff --check`를 실행한다.

## 리스크

- `devel`을 비 fast-forward로 교체하거나 rename하면 기존 contributor clone과 열린 PR의 base가 흔들릴 수 있다. 원격 branch migration은 문서/공지와 보존 브랜치 생성을 먼저 끝낸 뒤 승인된 순서로 실행해야 한다.
- branch protection과 default branch 설정은 로컬 git 명령만으로 완전히 보정되지 않는다. GitHub repository setting 변경이 필요하면 별도 승인과 수동 확인 항목으로 남긴다.
- `main`과 `devel-webview`가 둘 다 release/product line 정보를 갖고 있어 새 `devel` 기준 commit 선택을 잘못하면 릴리스 문서가 후퇴할 수 있다.
- `devel-webview`를 너무 빨리 삭제하면 기존 문서 링크, 자동화 branch filter, 외부 PR base가 깨질 수 있다.
- native 라인 보존 이름이 모호하면 장기 Swift native viewer/editor 작업의 목적이 다시 흐려질 수 있다.

## 승인 요청 사항

이 수행계획 기준으로 Stage 1을 진행해도 되는지 승인 요청한다. Stage 1에서는 읽기 중심으로 branch topology, workflow filter, 문서 참조 inventory를 정리하고, 원격 branch 변경은 수행하지 않는다.
