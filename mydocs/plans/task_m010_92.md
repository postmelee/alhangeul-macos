# Task #92 수행계획서 - Viewer page tree cache LRU/window eviction 정책 도입

## 목적

HostApp Viewer에서 큰 다중 페이지 HWP/HWPX 문서를 열고 스크롤할 때 `DocumentViewerStore.pageTrees`가 무제한 증가하지 않도록 명시적인 page tree cache eviction 정책을 도입한다.

## 배경

Task #84에서 Viewer 첫 페이지 초기 로드 누락을 수정하면서 `DocumentPageContainer.onDisappear` 기반 즉시 unload를 제거하고, 문서 로드 직후 page 0/1 render tree를 선로딩하도록 보정했다.

이 변경은 초기 표시 correctness를 회복하는 데 필요했다. 다만 그 결과 한 번 로드된 page tree가 문서가 열려 있는 동안 계속 유지되어, 페이지 수가 많은 문서에서는 메모리 사용량이 증가할 수 있다.

`LazyVStack.onDisappear`를 eviction 신호로 직접 사용하는 방식은 현재 visible/current page가 초기 layout 또는 virtualization 과정에서 제거되어 `ProgressView`로 되돌아가는 회귀를 만들 수 있다. 따라서 eviction은 view lifecycle 이벤트에 즉시 결합하지 않고, store가 current page와 주변 page window를 기준으로 안전하게 판단해야 한다.

## 범위

포함:

- 현재 page와 주변 page window 유지 규칙 정의

- LRU 방식과 current-page distance/window 방식 비교

- 기존 `unloadPage(_:)` 재사용 또는 대체 API 검토
- 대표 다중 페이지 HWP 샘플 기반 render smoke와 실제 Viewer 수동 검증 기준 정리

## 설계 방향

첫 번째 원칙은 correctness 우선이다. 현재 화면에 표시 중인 page, `currentPage`, 초기 선로딩 대상 page 0/1은 eviction 대상에서 제외해야 한다. 사용자가 같은 위치로 돌아왔을 때 page tree가 과도하게 자주 재생성되어 flicker 또는 `ProgressView` 회귀가 생기지 않아야 한다.

현재 `currentPage`는 `DocumentPageContainer.onAppear`에서 갱신되므로, 엄밀한 의미의 viewport 중심 page라기보다 최근 appear된 page에 가깝다. 따라서 `currentPage`만을 eviction anchor로 쓰는 것은 위험하다. Stage 1에서 visible page 관측과 current page 상태를 분리해 평가하고, eviction 보호 집합은 다음 후보를 종합해 계산한다.

- 실제 화면에 나타난 것으로 관측된 visible page 집합
- `currentPage`와 그 주변 page window
- 방금 load/access된 page
- 문서 로드 직후 초기 안정성을 위한 page 0/1

정책 후보는 두 가지다.

- LRU: page load/access 시점을 기록하고 cache 상한을 넘으면 오래 접근하지 않은 page를 제거한다.
- 주변 page window: 현재 page를 기준으로 앞뒤 N페이지를 유지하고, window 밖 page를 제거한다.

Viewer의 스크롤/페이지 상태가 `currentPage` 중심으로 이미 관리되고 있으므로, 정책은 visible/current page 주변 window를 우선 보호하고, cache 상한을 넘었을 때 LRU 순서로 window 밖 page를 제거하는 혼합 정책으로 확정한다. 이 방식은 문서 viewer에서 흔한 주변 page 유지 모델에 가깝고, `LazyVStack` lifecycle에 cache 제거를 직접 묶지 않는다.

eviction은 page load/access 이후에만 수행한다. `setCurrentPage(_:)`, `onAppear`, `onDisappear`는 access 또는 visible page 관측 정보를 갱신할 수 있지만, `onDisappear`는 즉시 unload 트리거로 복귀시키지 않고 visible page 관측 정보 갱신에만 사용한다.

cache 상한은 우선 page count 기준 soft limit로 둔다. 기본 후보는 최대 12개 page tree, current/visible page 주변 반경 3페이지 보호다. protected page 수가 soft limit보다 많으면 correctness를 우선해 protected page를 제거하지 않고 일시적으로 limit 초과를 허용한다. page tree별 실제 메모리 산정은 현재 자동화되어 있지 않으므로 이번 단계에서 과도하게 복잡한 byte 기반 cache를 만들지 않는다. 필요하면 후속 profiling 작업으로 분리한다.

## 잠정 단계

1. Stage 1: 기존 Viewer page cache 흐름 분석과 정책 선택
   - `DocumentViewerStore`의 `pageTrees`, `currentPage`, `loadPage(_:)`, `unloadPage(_:)`, 초기 선로딩 흐름을 재확인한다.
   - `currentPage`가 실제 viewport 중심 page로 충분한지, visible page 관측 상태가 필요한지 판단한다.
   - protected page 집합과 cache 상한 기준을 명시한다.
   - visible/current 주변 window + LRU 제거 순서 혼합 정책을 선택한다.
   - eviction 대상에서 제외해야 하는 page 조건을 명시한다.
2. Stage 2: page tree cache eviction 구현
   - 선택한 정책에 필요한 상태와 helper를 `DocumentViewerStore`에 추가한다.
   - page load/access/current page 변경 흐름에서 안전하게 eviction을 실행한다.
   - `onDisappear` 즉시 unload 회귀를 만들지 않도록 view lifecycle 의존을 최소화한다.
3. Stage 3: build, render smoke, 대표 문서 재검증
   - HostApp Debug build를 수행한다.
   - `table-vpos-01.hwp`, `hwp-multi-001.hwp`, `20250130-hongbo.hwp`, `aift.hwp` 기준 render smoke를 수행한다.
   - 첫 페이지 초기 표시와 문서 전환 이미지 cache 회귀가 재발하지 않는지 확인한다.
4. Stage 4: 최종 보고와 후속 리스크 정리
   - 최종 보고서와 단계 보고서를 작성한다.
   - 오늘할일 상태를 갱신한다.
   - 큰 문서 실측이 부족하면 별도 performance profiling 후속 이슈로 분리한다.

## 리스크

- eviction 상한이 너무 작으면 사용자가 직전에 본 page로 돌아갈 때 render tree 재생성이 잦아져 체감 성능이 나빠질 수 있다.
- eviction 실행 시점이 SwiftUI layout lifecycle과 맞지 않으면 Task #84에서 고친 첫 페이지 초기 표시 문제가 재발할 수 있다.
- 자동 render smoke는 store cache eviction과 실제 스크롤 UI를 직접 검증하지 못하므로, 실제 Viewer 수동 검증이 필요하다.
- page tree 메모리 사용량 측정은 현재 자동화되어 있지 않다. 필요하면 별도 profiling 작업으로 분리한다.
