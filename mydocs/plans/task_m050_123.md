# Issue #123 수행 계획서

## 목적

Swift native renderer에서 `Body.clip_rect`와 `TableCell.clip` 적용 정책을 rhwp-studio의 Canvas renderer 동작에 가깝게 맞춘다. 본문 텍스트는 body 영역 안에 유지하되, 본문 좌우 경계를 벗어나도록 배치된 도형, 이미지, 표, 글상자 같은 control이 불필요하게 잘리지 않도록 보강한다.

## 배경

현재 HostApp, Quick Look, Thumbnail은 모두 `RhwpDocument`가 제공하는 render tree를 `RenderTree.swift`로 디코딩하고 `CGTreeRenderer`로 그리는 공통 native renderer 경로를 사용한다.

`CGTreeRenderer`의 현재 `Body` 분기는 `clip_rect`가 있으면 `ctx.clip(to:)`를 적용한 뒤 모든 children을 한 번에 렌더링한다. `TableCell` 분기도 `clip`이 true이면 셀 bbox로 단순 clipping한다. 이 방식은 본문 텍스트 clipping에는 안전하지만, rhwp-studio가 `WebCanvasRenderer.render_overflow_controls`에서 별도 pass로 다시 그리는 control overflow 정책과 다르다. 결과적으로 본문 좌우 여백 밖에 놓인 비-텍스트 객체가 Swift native 출력에서 잘릴 수 있다.

Issue #90에서는 `복학원서.hwp`의 하단 layout overflow가 core layout 산출 문제에 가깝다고 정리했다. 이번 이슈는 그 결론을 뒤집는 작업이 아니라, issue #123 본문에 명시된 body 좌우 control overflow와 table cell 우측 여유 폭 정책을 Swift renderer 해석 차원에서 보강하는 작업으로 범위를 한정한다.

## 범위

포함:

- rhwp-studio `WebCanvasRenderer.render_overflow_controls`, `ClipKind::Body`, `ClipKind::TableCell` 동작 확인
- Swift `RenderTree`/`CGTreeRenderer`의 body clip, table cell clip, control node 분류 방식 분석
- `Body.clip_rect` 적용 시 텍스트 계열과 비-텍스트 control 계열의 clipping/replay 정책 분리
- body 좌우 경계를 벗어나는 도형, 이미지, 표, 글상자 후보의 별도 렌더링 pass 구현
- `TableCell.clip`의 우측 여유 폭 처리 검토와 필요한 최소 보강
- 기존 텍스트 clipping, 다단, header/footer, footnote 영역 회귀 확인
- HostApp, Quick Look, Thumbnail이 공유하는 `HwpPageImageRenderer`/`CGTreeRenderer` 경로 검증

제외:

- 편집 모드 선택, resize overlay, hit testing 구현
- PageLayerTree 기본 경로 전환
- `edwardkim/rhwp` core layout engine 변경
- HWP/HWPX 저장 또는 편집 기능
- release, signing, notarization, Homebrew Cask 작업
- Xcode project 직접 수정

## 설계 방향

rhwp-studio의 Canvas renderer를 reference implementation으로 삼되, TypeScript/DOM/Canvas 구조를 Swift에 그대로 복제하지 않는다. Swift 경로에서는 `RenderNode` 트리와 CoreGraphics drawing의 책임 경계를 유지하며, body clip 안에서 일반 children을 그린 뒤 overflow control 후보만 필요한 조건에서 clip 바깥 pass로 다시 그리는 방향을 우선 검토한다.

분류 기준은 보수적으로 둔다. `TextRun`, `TextLine`, footnote marker 같은 텍스트 표시 계열은 기존 body clipping을 유지한다. rectangle, line, ellipse, path, image, table, group, form object, equation, 글상자에 해당할 수 있는 구조 노드는 rhwp-studio 기준과 render tree 필드를 대조해 overflow replay 대상 여부를 결정한다. 필드가 부족한 경우에는 bbox와 parent chain만으로 과도하게 추정하지 않고, 구현계획서에서 한계를 분리한다.

`TableCell.clip`은 셀 내부 텍스트 보호와 우측 여유 폭 허용 사이의 균형이 필요하다. 우측 여유 폭이 rhwp-studio와 같은 시각 결과를 만드는 데 필요한지 샘플로 확인하고, 필요한 경우 cell clip rect 계산 helper를 별도로 두어 기존 table border와 텍스트 clipping 영향을 좁힌다.

앱 저장소의 제품 기준 경로는 계속 `rhwp_render_page_tree` + Swift `RenderTree` + `CGTreeRenderer`로 둔다. core SVG 또는 rhwp-studio 출력은 비교 기준과 원인 분리 자료로 사용한다.

## 잠정 단계

1. Stage 1: rhwp-studio 기준 동작과 Swift 현행 경로 분석
   - `render_overflow_controls`, `ClipKind::Body`, `ClipKind::TableCell`의 조건을 정리한다.
   - Swift render tree에서 body, table cell, control 후보 node가 어떤 bbox와 parent 구조로 들어오는지 확인한다.
2. Stage 2: overflow control 분류와 renderer 구조 설계
   - 텍스트 계열과 비-텍스트 control 계열을 나누는 Swift 내부 helper를 설계한다.
   - body clip pass와 overflow replay pass의 순서, clip 상태, transform 적용 범위를 확정한다.
3. Stage 3: Body clip overflow control replay 구현
   - 기존 body clip 안 children 렌더링은 유지하고, body 좌우를 벗어나는 control 후보만 별도 pass로 다시 렌더링한다.
   - group/table/image/shape children 순회와 중복 렌더링 부작용을 확인한다.
4. Stage 4: TableCell clip 우측 여유 폭 보강
   - rhwp-studio 기준 샘플과 비교해 우측 여유 폭이 필요한 경우 cell clip rect 계산을 보정한다.
   - table border, 셀 내부 텍스트, vertical text 영향을 확인한다.
5. Stage 5: 공통 renderer 회귀 검증과 문서 정리
   - HostApp, Quick Look, Thumbnail 공통 renderer 경로에서 대표 샘플을 검증한다.
   - 단계 보고서와 최종 보고서에 비교 산출물, 잔여 한계, 후속 이슈 후보를 정리한다.

## 리스크

- rhwp-studio의 overflow replay 조건이 Swift render tree 필드만으로 완전히 복원되지 않을 수 있다.
- body clip 바깥 replay는 중복 drawing, z-order 변화, group/table 내부 children 재귀 처리 오류를 만들 수 있다.
- 텍스트 clipping과 control overflow를 잘못 분리하면 기존 다단, header/footer, footnote 표시가 회귀할 수 있다.
- `TableCell.clip` 우측 여유 폭 보강은 셀 내부 텍스트가 셀 경계 밖으로 새는 부작용을 만들 수 있다.
- 비교 기준 샘플이 부족하면 특정 HWP 문서에서만 필요한 control overflow 정책을 과도하게 일반화할 수 있다.
