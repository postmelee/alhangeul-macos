# Task #159 수행계획서

## 목적

GitHub Actions 기반 release 자동화 파이프라인을 구축해 M16 완료 직후 v0.1 첫 GitHub Release를 재현 가능하게 게시할 수 있는 운영 경로를 만든다. 동시에 upstream `edwardkim/rhwp` 최신 release 감지와 compatibility check를 release publish와 분리해, 잦은 core release 반영과 재배포를 안정적으로 반복할 수 있게 한다.

## 배경

M16은 release artifact, 서명·공증 수준, license/provenance, smoke gate, fallback, known limitations를 병렬로 정리 중이다. 이 작업은 M16의 문서와 기능 보강 범위를 건드리지 않고, 현재 저장소에 없는 `.github/workflows` release 자동화만 별도 경계로 추가한다.

현재 앱 저장소의 `rhwp-core.lock`과 `RustBridge/Cargo.toml`은 upstream `rhwp` `v0.7.9` 기준이다. 2026-05-06 확인 기준 upstream 최신 release는 `v0.7.10`이므로, 실제 public release publish 전에는 `v0.7.10` compatibility check와 core bump 여부를 명시적으로 판단해야 한다.

## 범위(포함·제외)

포함:

- release rehearsal DMG를 수동 생성하는 GitHub Actions workflow 추가
- protected/manual approval 후 signed/notarized public DMG를 생성하고 GitHub Release asset으로 업로드하는 workflow 추가
- upstream `rhwp` 최신 release를 감지하고 현재 lock과 비교하는 workflow 추가
- publish workflow가 floating latest가 아니라 `rhwp-core.lock`의 release tag와 resolved commit만 신뢰하도록 guard 추가
- `expected_rhwp_tag` 입력값과 lock 기준이 다르면 publish를 중단하는 기준 추가
- release note/provenance에 current core tag, resolved commit, DMG sha256을 포함하는 자동화 설계
- 필요한 경우 workflow에서 재사용할 CI helper script 추가

제외:

- 실제 public release 게시
- M16 이슈의 fallback, license, known limitations, smoke 문서 본문 변경
- `v0.7.10` core bump 자체 구현
- upstream `rhwp` 최신 release 자동 merge 또는 자동 배포
- Homebrew 공식 cask 등재 보장
- Apple credential 값 저장 또는 민감 정보 문서화

## 설계 방향

Release publish workflow는 현재 checkout의 lock 파일을 유일한 core provenance로 취급한다. 최신 upstream release가 존재하더라도 publish workflow가 임의로 dependency를 갱신하지 않고, 입력된 `expected_rhwp_tag`와 `rhwp-core.lock`의 `rhwp_release_tag`가 일치할 때만 진행한다.

Upstream check workflow는 release publish와 분리한다. 새 `rhwp` release가 발견되면 compatibility check 또는 경고 summary를 남기되, 앱 소스와 lock을 자동으로 수정하지 않는다. core 반영은 별도 하이퍼-워터폴 이슈와 `scripts/update-rhwp-core.sh --channel stable --tag <tag>` 경로로 처리한다.

M16 충돌 방지를 위해 이 작업의 기본 write scope는 `.github/workflows`와 CI helper에 둔다. `scripts/release.sh`, `Casks/alhangeul-macos.rb`, `README.md`, `mydocs/manual/release_distribution_guide.md`, `Info.plist`는 승인된 구현 단계에서 꼭 필요하다고 확인될 때만 최소 수정한다.

## 잠정 단계(3~6단계)

1. Stage 1: workflow 요구사항과 GitHub Actions secret/environment 입력값 설계
2. Stage 2: release rehearsal workflow 구현과 artifact upload 검증
3. Stage 3: publish workflow 구현, lock/expected tag guard, GitHub Release draft 생성 경로 검증
4. Stage 4: upstream `rhwp` release check workflow 구현과 `v0.7.10` 감지 summary 검증
5. Stage 5: 문법 검증, dry-run 가능한 job 검증, 최종 보고서 정리

## 검증 계획

- `git diff --check`
- workflow YAML 정적 검토
- `gh workflow list`는 workflow가 원격에 반영된 뒤 후속 확인
- `./scripts/release.sh --help`
- 가능하면 `./scripts/release.sh --skip-notarize 0.1.0`를 로컬 또는 Actions rehearsal 경로로 검증
- `./scripts/build-rust-macos.sh --verify-lock`
- `./scripts/update-rhwp-core.sh --check --channel stable --tag v0.7.10`는 publish 전 core bump 판단용 gate로 별도 기록

## 리스크

- GitHub-hosted macOS runner에 Developer ID certificate와 notarization credential을 안전하게 주입해야 하므로 protected environment와 secret 관리가 필요하다.
- public publish workflow를 너무 자동화하면 M16 완료 전 artifact가 게시될 수 있으므로 `workflow_dispatch`, environment approval, tag guard가 필요하다.
- `rhwp` 최신 release 감지와 publish를 결합하면 lock 재현성이 깨질 수 있다. 이 둘은 별도 workflow로 유지해야 한다.
- official Homebrew Cask 등재는 별도 리뷰와 acceptability 리스크가 있어 이번 자동화 완료만으로 보장할 수 없다.

## 승인 요청 사항

- 위 범위대로 `.github/workflows` 중심 구현을 시작해도 되는지 승인 요청한다.
- public release publish는 이 작업에서 실제 실행하지 않고, M16 완료 및 core 기준 확정 후 별도 명시 승인으로만 실행한다.
- `v0.7.10` 반영은 이 작업에서 직접 수행하지 않고, release 전 별도 core bump 판단 또는 후속 이슈로 분리한다.
