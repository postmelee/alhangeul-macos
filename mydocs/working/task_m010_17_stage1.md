# Issue #17 단계 1 완료 보고서

## 작업 내용

- `devel` 대상 PR 기반 워크플로우로 전환할 때 예상되는 단점을 다시 점검했다.
- 각 단점에 대해 무엇을 `AGENTS.md` 본문 규칙으로 넣고, 무엇을 운영 원칙 수준으로 남길지 구분했다.
- 이후 실제 문안 수정 시 유지해야 할 최소 운영 기준을 확정했다.

## 확정한 운영 기준

### 1. 브랜치 역할 단순화

- 로컬 작업 브랜치는 `local/task{N}`로 유지한다.
- 원격 게시 브랜치는 `publish/task{N}`로 고정한다.
- `publish/task{N}`는 PR merge 후 삭제한다.

이 기준은 `AGENTS.md`의 브랜치 표와 Git 워크플로우 본문에 직접 반영할 가치가 있다.

### 2. 소규모 작업의 속도 저하 완화

- 오탈자, 단순 문구 수정, 조사 메모처럼 예외 처리 대상은 경량 절차를 허용한다.
- 단, Issue와 최종 보고서는 유지해 추적성을 보존한다.
- 문서 전용 작업은 draft PR을 기본값으로 두는 방향이 적절하다.

이 기준은 `AGENTS.md`의 예외 처리 또는 운영 규칙 문장으로 짧게 반영할 수 있다.

### 3. 관리 포인트 증가 통제

- 절차 문구는 `Issue -> local/task -> publish/task -> devel PR -> merge`처럼 짧게 고정한다.
- 오늘 할 일, 계획서, 보고서, PR 본문이 같은 Issue 번호를 유지하도록 한다.

이 기준은 절차 요약과 브랜치 설명 문장에서 반영하는 것이 적절하다.

### 4. 1인 저장소 운영 부담 완화

- `devel` 대상 PR은 기본적으로 draft로 생성한다.
- PR 본문을 self-review 기록으로 사용해 검토 근거를 남긴다.
- ready 전환 여부는 최종 보고 후 판단하는 원칙이 적절하다.

이 기준은 운영상 유용하지만 세부도가 높으므로 `AGENTS.md`에는 핵심만 남기고 나머지는 설명 수준으로 유지하는 편이 낫다.

### 5. `devel` 히스토리 복잡화 통제

- merge 전략은 하나로 고정해야 한다.
- 기본 권장안은 merge commit 유지 또는 `--no-ff` 원칙이다.
- squash merge는 단계별 커밋 의미가 사라질 수 있으므로 기본값으로 두지 않는다.

이 기준은 `AGENTS.md` 메인테이너 워크플로우 예시에 녹여 반영하는 편이 자연스럽다.

## 결정 사항

- `AGENTS.md`에는 모든 장단점 설명을 장문으로 쓰지 않는다.
- 대신 아래 항목만 핵심 규칙으로 반영하는 방향을 유지한다.
  - `publish/task{N}` 원격 브랜치 규칙
  - `devel` 대상 PR 생성 흐름
  - draft PR 기본값
  - merge 전략 고정 원칙
- 장단점 자체의 해설은 계획서와 보고서에서 보존한다.
